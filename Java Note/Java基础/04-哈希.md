# 04-哈希

## 1. 哈希表是什么
- 哈希表是一种用空间换时间的数据结构。
- 它的核心目标是让查找、插入、删除尽量快。
- 本质上，哈希表会先把 key 通过哈希函数转换成一个整数，再映射到数组下标。
- 所以哈希表可以理解为：通过计算位置来直接定位数据，而不是挨个遍历查找。

## 2. 哈希表的基本结构
- 哈希表最经典的结构是：数组 + 冲突处理结构。
- 数组中的每个位置叫做一个桶（bucket）。
- 一个 key 先通过 `hashCode()` 得到哈希值，再根据规则映射到某个桶中。
- 当多个对象落到同一个桶时，就会发生哈希冲突。
- 常见处理方式是链地址法，也就是在桶后面挂链表。
- Java 8 之后，当冲突严重时，链表还可能转换成红黑树以提高效率。

## 3. 为什么哈希表查找快
- 普通查找通常要一个个比较，时间复杂度接近 O(n)。
- 哈希表会先计算 key 应该落到哪个桶里。
- 这样查找时不用扫描所有元素，而是只去某一个桶里找。
- 所以它快的原因是：先快速定位，再小范围比较。

## 4. 哈希冲突是什么
- 哈希冲突指不同的 key 经过哈希计算后，落到了同一个桶里。
- 这不是错误，而是哈希表设计中天然会发生的现象。
- 因为对象很多，但桶的数量有限，不同对象映射到同一个位置很正常。
- 所以哈希表一定需要冲突处理机制，比如链表或红黑树。

## 5. 哈希表插入元素的流程
1. 先调用 key 的 `hashCode()`，得到哈希值。
2. 根据哈希值计算数组下标，定位桶位置。
3. 如果该桶为空，直接插入。
4. 如果该桶不为空，说明发生冲突。
5. 再和桶中的已有元素比较，最终通过 `equals()` 判断是不是同一个逻辑对象。
6. 如果相等，就更新或去重；如果不相等，就继续挂到该桶中。

## 6. 哈希表查找元素的流程
1. 先调用 key 的 `hashCode()`。
2. 根据哈希值找到应该去的桶。
3. 如果桶为空，说明对象不存在。
4. 如果桶里有元素，就在桶内逐个比较。
5. 最终通过 `equals()` 判断目标对象是否真正匹配。

## 7. `hashCode()` 和 `equals()` 在哈希表中的分工
- `hashCode()` 的作用是快速定位桶的位置。
- `equals()` 的作用是精确判断两个对象逻辑上是否相等。
- 所以两者是配合关系，不是替代关系。
- 可以记成一句话：
  - `hashCode()` 负责“缩小范围”
  - `equals()` 负责“最终确认”

## 8. 为什么需要 `hashCode()`
- 如果没有 `hashCode()`，哈希集合查找时只能靠 `equals()` 一个个比。
- 这样效率就会很低，哈希表也就失去了“快查找”的意义。
- 有了 `hashCode()` 后，可以先把对象分桶，再在桶内少量比较。
- 所以 `hashCode()` 存在的核心目的，就是提高查找和去重效率。

## 9. `hashCode()` 和 `equals()` 的关系
- 如果两个对象通过 `equals()` 判断为 `true`，那么它们的 `hashCode()` 必须相同。
- 如果两个对象的 `hashCode()` 相同，它们的 `equals()` 不一定为 `true`，这叫哈希冲突。
- 所以：
  - `equals()` 相等 -> `hashCode()` 必须相等
  - `hashCode()` 相等 -> `equals()` 不一定相等
- 这也是为什么重写 `equals()` 时，通常也要同时重写 `hashCode()`。

## 10. 一句话总结
- 哈希表 = 数组 + 冲突处理。
- `hashCode()` 负责定位桶，`equals()` 负责桶内精确比较。
- 哈希表的核心思想是：先快速定位，再精确判断。
