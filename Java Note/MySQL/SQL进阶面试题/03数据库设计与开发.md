# 数据库设计与开发

## 1. 数据库基础（DB、DBMS、关系型数据库、完整性约束）

### 什么是数据库（DB）
数据库就是存储数据的仓库，按照一定的组织结构存放数据。

### 什么是 DBMS
数据库管理系统（Database Management System），就是管理数据库的软件，比如 MySQL、PostgreSQL、Oracle。

### 关系型数据库
数据以表格形式存储，表与表之间通过关系（外键）连接。

**特点：**
- 数据存储在二维表中
- 表之间通过主键和外键建立关系
- 支持 SQL 查询语言
- 支持事务（ACID）

### 完整性约束
保证数据的准确性和一致性。

**主要约束类型：**

1. **主键约束（PRIMARY KEY）**
   - 唯一标识每一行
   - 不能为 NULL
   - 一个表只能有一个主键

2. **外键约束（FOREIGN KEY）**
   - 建立表之间的关系
   - 引用另一个表的主键
   - 保证引用完整性

3. **唯一约束（UNIQUE）**
   - 列值不能重复
   - 可以为 NULL

4. **非空约束（NOT NULL）**
   - 列值不能为 NULL

5. **检查约束（CHECK）**
   - 限制列值的范围
   - 例如：年龄必须大于 0

6. **默认值约束（DEFAULT）**
   - 插入数据时未指定值，使用默认值

## 2. 数据库设计（数据库规范化、三范式、ER 图、数据类型）

### 数据库规范化
通过消除冗余和依赖，使数据库结构更合理。

### 三范式（1NF、2NF、3NF）

#### 第一范式（1NF）
每个字段都是不可分割的原子值。

```sql
-- ❌ 违反 1NF：联系方式包含多个值
CREATE TABLE employee_bad (
    emp_id INT,
    emp_name VARCHAR(50),
    contact VARCHAR(200)  -- "电话:123456,邮箱:abc@qq.com"
);

-- ✅ 符合 1NF：拆分成原子字段
CREATE TABLE employee_good (
    emp_id INT,
    emp_name VARCHAR(50),
    phone VARCHAR(20),
    email VARCHAR(100)
);
```

#### 第二范式（2NF）
在 1NF 基础上，非主键字段完全依赖于主键（消除部分依赖）。

```sql
-- ❌ 违反 2NF：订单明细表中商品名称只依赖商品ID，不依赖完整主键
CREATE TABLE order_item_bad (
    order_id INT,
    product_id INT,
    product_name VARCHAR(50),  -- 只依赖 product_id
    quantity INT,
    PRIMARY KEY (order_id, product_id)
);

-- ✅ 符合 2NF：拆分成两个表
CREATE TABLE order_item (
    order_id INT,
    product_id INT,
    quantity INT,
    PRIMARY KEY (order_id, product_id)
);

CREATE TABLE product (
    product_id INT PRIMARY KEY,
    product_name VARCHAR(50)
);
```

#### 第三范式（3NF）
在 2NF 基础上，非主键字段不依赖于其他非主键字段（消除传递依赖）。

```sql
-- ❌ 违反 3NF：部门名称依赖于部门ID，而部门ID又依赖于员工ID
CREATE TABLE employee_bad (
    emp_id INT PRIMARY KEY,
    emp_name VARCHAR(50),
    dept_id INT,
    dept_name VARCHAR(50)  -- 传递依赖
);

-- ✅ 符合 3NF：拆分成两个表
CREATE TABLE employee (
    emp_id INT PRIMARY KEY,
    emp_name VARCHAR(50),
    dept_id INT
);

CREATE TABLE department (
    dept_id INT PRIMARY KEY,
    dept_name VARCHAR(50)
);
```

### ER 图（实体关系图）
用于设计数据库结构，表示实体、属性和关系。

**三要素：**
- 实体（Entity）：矩形表示，如员工、部门
- 属性（Attribute）：椭圆表示，如姓名、工资
- 关系（Relationship）：菱形表示，如员工属于部门

**关系类型：**
- 一对一（1:1）：一个员工对应一个身份证
- 一对多（1:N）：一个部门有多个员工
- 多对多（M:N）：一个学生选多门课，一门课有多个学生

### 常用数据类型

#### 数值类型
```sql
INT           -- 整数，-2147483648 到 2147483647
BIGINT        -- 大整数
DECIMAL(10,2) -- 定点数，10位数字，2位小数（精确）
FLOAT         -- 浮点数（不精确，不推荐用于金额）
```

#### 字符串类型
```sql
CHAR(10)      -- 定长字符串，不足补空格
VARCHAR(50)   -- 变长字符串，最常用
TEXT          -- 长文本，最大 65535 字节
```

#### 日期时间类型
```sql
DATE          -- 日期，格式：YYYY-MM-DD
TIME          -- 时间，格式：HH:MM:SS
DATETIME      -- 日期时间，格式：YYYY-MM-DD HH:MM:SS
TIMESTAMP     -- 时间戳，自动更新
```

⚠️ **CHAR vs VARCHAR：**
- CHAR 定长，查询快但浪费空间
- VARCHAR 变长，节省空间但查询稍慢
- 固定长度的用 CHAR（如性别、手机号），变长的用 VARCHAR

## 3. DDL 语句（CREATE、ALTER、DROP、TRUNCATE）

### CREATE（创建）

```sql
-- 创建数据库
CREATE DATABASE company DEFAULT CHARACTER SET utf8mb4;

-- 创建表
CREATE TABLE employee (
    emp_id INT PRIMARY KEY AUTO_INCREMENT,
    emp_name VARCHAR(50) NOT NULL,
    sex CHAR(1) DEFAULT '男',
    dept_id INT,
    salary DECIMAL(10,2) CHECK (salary > 0),
    hire_date DATE,
    FOREIGN KEY (dept_id) REFERENCES department(dept_id)
);
```

### ALTER（修改）

```sql
-- 添加列
ALTER TABLE employee ADD COLUMN email VARCHAR(100);

-- 修改列类型
ALTER TABLE employee MODIFY COLUMN emp_name VARCHAR(100);

-- 修改列名
ALTER TABLE employee CHANGE COLUMN emp_name employee_name VARCHAR(50);

-- 删除列
ALTER TABLE employee DROP COLUMN email;

-- 添加主键
ALTER TABLE employee ADD PRIMARY KEY (emp_id);

-- 添加外键
ALTER TABLE employee ADD FOREIGN KEY (dept_id) REFERENCES department(dept_id);
```

### DROP（删除）

```sql
-- 删除表（结构和数据都删除）
DROP TABLE employee;

-- 删除数据库
DROP DATABASE company;

-- 删除索引
DROP INDEX idx_name ON employee;
```

### TRUNCATE（清空表）

```sql
-- 清空表数据，保留表结构
TRUNCATE TABLE employee;
```

⚠️ **DELETE vs TRUNCATE：**
- DELETE 可以带 WHERE 条件，可以回滚，慢
- TRUNCATE 清空整个表，不能回滚，快
- TRUNCATE 会重置自增主键，DELETE 不会

## 4. DML 语句（数据库事务、隔离级别、INSERT、UPDATE、DELETE、DROP）

### 数据库事务
一组 SQL 操作，要么全部成功，要么全部失败。

**ACID 特性：**
- **原子性（Atomicity）**：事务是不可分割的最小单位
- **一致性（Consistency）**：事务前后数据保持一致
- **隔离性（Isolation）**：多个事务互不干扰
- **持久性（Durability）**：事务提交后永久保存

```sql
-- 开启事务
START TRANSACTION;

-- 转账操作
UPDATE account SET balance = balance - 100 WHERE user_id = 1;
UPDATE account SET balance = balance + 100 WHERE user_id = 2;

-- 提交事务
COMMIT;

-- 或者回滚
ROLLBACK;
```

### 隔离级别
解决并发事务的问题。

**四种隔离级别（从低到高）：**

1. **READ UNCOMMITTED（读未提交）**
   - 可以读到其他事务未提交的数据
   - 问题：脏读

2. **READ COMMITTED（读已提交）**
   - 只能读到其他事务已提交的数据
   - 问题：不可重复读

3. **REPEATABLE READ（可重复读，MySQL 默认）**
   - 同一事务内多次读取结果一致
   - 问题：幻读

4. **SERIALIZABLE（串行化）**
   - 最高隔离级别，完全串行执行
   - 问题：性能最差

```sql
-- 查看当前隔离级别
SELECT @@transaction_isolation;

-- 设置隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

### INSERT（插入）

```sql
-- 插入单行
INSERT INTO employee (emp_name, sex, dept_id, salary, hire_date)
VALUES ('张三', '男', 1, 15000.00, '2024-01-15');

-- 插入多行
INSERT INTO employee (emp_name, sex, dept_id, salary) VALUES
('李四', '女', 1, 12000.00),
('王五', '男', 2, 13000.00);

-- 从其他表插入
INSERT INTO employee_backup SELECT * FROM employee WHERE dept_id = 1;
```

### UPDATE（更新）

```sql
-- 更新单行
UPDATE employee SET salary = 16000.00 WHERE emp_id = 1;

-- 更新多列
UPDATE employee SET salary = salary * 1.1, bonus = 2000 WHERE dept_id = 1;

-- 批量更新
UPDATE employee SET salary = salary * 1.05 WHERE hire_date < '2020-01-01';
```

⚠️ **注意：** 更新前先用 SELECT 验证 WHERE 条件，避免误更新。

### DELETE（删除）

```sql
-- 删除单行
DELETE FROM employee WHERE emp_id = 1;

-- 删除多行
DELETE FROM employee WHERE dept_id = 1;

-- 删除所有数据（不推荐，用 TRUNCATE）
DELETE FROM employee;
```

⚠️ **注意：** 删除前先用 SELECT 验证 WHERE 条件，避免误删除。

## 5. 索引优化（索引概念与类型、查看执行计划、优化查询性能、多表连接优化）

### 索引概念
索引就像书的目录，帮助快速定位数据。

**优点：**
- 加快查询速度

**缺点：**
- 占用存储空间
- 降低写入速度（INSERT、UPDATE、DELETE）

### 索引类型

1. **主键索引（PRIMARY KEY）**
   - 自动创建，唯一且非空

2. **唯一索引（UNIQUE）**
   - 列值唯一，可以为 NULL

3. **普通索引（INDEX）**
   - 最常用的索引

4. **全文索引（FULLTEXT）**
   - 用于全文搜索

5. **组合索引（多列索引）**
   - 多个列组合成一个索引

```sql
-- 创建普通索引
CREATE INDEX idx_name ON employee(emp_name);

-- 创建唯一索引
CREATE UNIQUE INDEX idx_email ON employee(email);

-- 创建组合索引
CREATE INDEX idx_dept_salary ON employee(dept_id, salary);

-- 删除索引
DROP INDEX idx_name ON employee;

-- 查看表的索引
SHOW INDEX FROM employee;
```

### 查看执行计划（EXPLAIN）

```sql
-- 查看查询的执行计划
EXPLAIN SELECT * FROM employee WHERE dept_id = 1;
```

**重要字段：**
- **type**：连接类型，从好到坏：system > const > eq_ref > ref > range > index > ALL
- **key**：实际使用的索引
- **rows**：扫描的行数，越少越好
- **Extra**：额外信息，如 Using index（覆盖索引）、Using filesort（文件排序）

### 优化查询性能

**1. 使用索引**
```sql
-- ✅ 使用索引
SELECT * FROM employee WHERE dept_id = 1;

-- ❌ 索引失效：对索引列使用函数
SELECT * FROM employee WHERE YEAR(hire_date) = 2020;

-- ✅ 改写查询
SELECT * FROM employee WHERE hire_date >= '2020-01-01' AND hire_date < '2021-01-01';
```

**2. 避免 SELECT ***
```sql
-- ❌ 查询所有列
SELECT * FROM employee WHERE dept_id = 1;

-- ✅ 只查询需要的列
SELECT emp_id, emp_name, salary FROM employee WHERE dept_id = 1;
```

**3. 使用 LIMIT**
```sql
-- 分页查询
SELECT * FROM employee ORDER BY emp_id LIMIT 10 OFFSET 20;
```

**4. 避免 OR，使用 IN**
```sql
-- ❌ 使用 OR
SELECT * FROM employee WHERE dept_id = 1 OR dept_id = 2 OR dept_id = 3;

-- ✅ 使用 IN
SELECT * FROM employee WHERE dept_id IN (1, 2, 3);
```

**5. 使用覆盖索引**
```sql
-- 创建组合索引
CREATE INDEX idx_dept_salary ON employee(dept_id, salary);

-- 查询只涉及索引列，不需要回表
SELECT dept_id, salary FROM employee WHERE dept_id = 1;
```

### 多表连接优化

**1. 小表驱动大表**
```sql
-- ✅ 小表在前
SELECT * FROM department d
JOIN employee e ON d.dept_id = e.dept_id;
```

**2. 使用索引连接**
```sql
-- 确保连接列有索引
CREATE INDEX idx_dept_id ON employee(dept_id);
```

**3. 避免笛卡尔积**
```sql
-- ❌ 没有连接条件
SELECT * FROM employee, department;

-- ✅ 使用 JOIN 和 ON
SELECT * FROM employee e JOIN department d ON e.dept_id = d.dept_id;
```

## 6. 视图（定义与作用、CREATE VIEW、可更新视图）

### 什么是视图
视图是虚拟表，基于 SQL 查询结果。不存储数据，只存储查询语句。

**作用：**
- 简化复杂查询
- 隐藏敏感数据
- 提供数据的不同视角

### CREATE VIEW

```sql
-- 创建视图：查询每个部门的员工数和平均工资
CREATE VIEW dept_summary AS
SELECT 
    d.dept_id,
    d.dept_name,
    COUNT(e.emp_id) AS emp_count,
    AVG(e.salary) AS avg_salary
FROM department d
LEFT JOIN employee e ON d.dept_id = e.dept_id
GROUP BY d.dept_id, d.dept_name;

-- 使用视图
SELECT * FROM dept_summary WHERE emp_count > 5;

-- 删除视图
DROP VIEW dept_summary;
```

### 可更新视图
如果视图满足以下条件，可以通过视图更新数据：
- 没有聚合函数（SUM、AVG、COUNT 等）
- 没有 DISTINCT
- 没有 GROUP BY
- 没有 UNION

```sql
-- 创建可更新视图
CREATE VIEW high_salary_emp AS
SELECT emp_id, emp_name, salary FROM employee WHERE salary > 15000;

-- 通过视图更新数据
UPDATE high_salary_emp SET salary = 16000 WHERE emp_id = 1;

-- 实际更新的是 employee 表
```

⚠️ **注意：** 大多数视图用于查询，不建议通过视图更新数据。

## 7. 存储过程（优点与使用场景、CREATE PROCEDURE、DROP PROCEDURE）

### 什么是存储过程
预编译的 SQL 语句集合，存储在数据库中，可以重复调用。

**优点：**
- 减少网络传输（一次调用执行多条 SQL）
- 提高性能（预编译）
- 代码复用
- 增强安全性（权限控制）

**缺点：**
- 难以调试
- 可移植性差（不同数据库语法不同）
- 业务逻辑分散（部分在代码，部分在数据库）

### CREATE PROCEDURE

```sql
-- 创建存储过程：给指定部门的员工涨工资
DELIMITER $$

CREATE PROCEDURE raise_salary(
    IN p_dept_id INT,
    IN p_percent DECIMAL(5,2)
)
BEGIN
    UPDATE employee 
    SET salary = salary * (1 + p_percent / 100)
    WHERE dept_id = p_dept_id;
    
    SELECT CONCAT('部门 ', p_dept_id, ' 涨薪 ', p_percent, '% 完成') AS result;
END$$

DELIMITER ;

-- 调用存储过程
CALL raise_salary(1, 10);  -- 给部门1涨薪10%
```

**带输出参数的存储过程：**
```sql
DELIMITER $$

CREATE PROCEDURE get_emp_count(
    IN p_dept_id INT,
    OUT p_count INT
)
BEGIN
    SELECT COUNT(*) INTO p_count
    FROM employee
    WHERE dept_id = p_dept_id;
END$$

DELIMITER ;

-- 调用
CALL get_emp_count(1, @count);
SELECT @count;  -- 输出结果
```

### DROP PROCEDURE

```sql
-- 删除存储过程
DROP PROCEDURE IF EXISTS raise_salary;

-- 查看存储过程
SHOW PROCEDURE STATUS WHERE db = 'sql_test';

-- 查看存储过程定义
SHOW CREATE PROCEDURE raise_salary;
```

## 8. 数据库触发器（概念与作用、CREATE TRIGGER、DROP TRIGGER）

### 什么是触发器
在特定事件（INSERT、UPDATE、DELETE）发生时自动执行的 SQL 代码。

**作用：**
- 自动记录日志
- 数据验证
- 级联更新
- 维护数据一致性

**缺点：**
- 难以调试
- 隐藏业务逻辑
- 可能影响性能

### CREATE TRIGGER

```sql
-- 创建日志表
CREATE TABLE employee_log (
    log_id INT PRIMARY KEY AUTO_INCREMENT,
    emp_id INT,
    action VARCHAR(10),
    old_salary DECIMAL(10,2),
    new_salary DECIMAL(10,2),
    change_time DATETIME
);

-- 创建触发器：员工工资变动时记录日志
DELIMITER $$

CREATE TRIGGER salary_change_log
AFTER UPDATE ON employee
FOR EACH ROW
BEGIN
    IF OLD.salary != NEW.salary THEN
        INSERT INTO employee_log (emp_id, action, old_salary, new_salary, change_time)
        VALUES (NEW.emp_id, 'UPDATE', OLD.salary, NEW.salary, NOW());
    END IF;
END$$

DELIMITER ;

-- 测试触发器
UPDATE employee SET salary = 20000 WHERE emp_id = 1;

-- 查看日志
SELECT * FROM employee_log;
```

**触发器类型：**
- **BEFORE INSERT/UPDATE/DELETE**：在操作前触发
- **AFTER INSERT/UPDATE/DELETE**：在操作后触发

**使用场景：**
```sql
-- 防止删除重要数据
DELIMITER $$

CREATE TRIGGER prevent_delete
BEFORE DELETE ON employee
FOR EACH ROW
BEGIN
    IF OLD.emp_id = 1 THEN
        SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = '不能删除CEO';
    END IF;
END$$

DELIMITER ;
```

### DROP TRIGGER

```sql
-- 删除触发器
DROP TRIGGER IF EXISTS salary_change_log;

-- 查看触发器
SHOW TRIGGERS;

-- 查看触发器定义
SHOW CREATE TRIGGER salary_change_log;
```

⚠️ **注意：** 触发器会增加数据库复杂度，能在应用层实现的逻辑尽量不用触发器。

## 面试回答模板

**Q: 什么是三范式？**

A: 
- 1NF：字段不可再分（原子性）
- 2NF：非主键字段完全依赖主键（消除部分依赖）
- 3NF：非主键字段不依赖其他非主键字段（消除传递依赖）
- 目的：减少数据冗余，提高数据一致性

**Q: 索引什么时候会失效？**

A:
- 对索引列使用函数或表达式
- 使用 LIKE 以 % 开头
- 使用 OR 连接条件（部分情况）
- 类型隐式转换
- 组合索引不遵循最左前缀原则

**Q: 事务的四大特性是什么？**

A: ACID
- 原子性：事务不可分割
- 一致性：事务前后数据一致
- 隔离性：多个事务互不干扰
- 持久性：提交后永久保存

**Q: 视图和表的区别？**

A:
- 表存储实际数据，视图不存储数据
- 视图是虚拟表，基于查询结果
- 视图用于简化查询、隐藏敏感数据
- 大多数视图只读，部分视图可更新

**Q: 什么时候使用存储过程？**

A:
- 复杂业务逻辑需要多次数据库交互
- 需要减少网络传输
- 需要在数据库层面做权限控制
- 但现代开发更倾向于在应用层实现业务逻辑，便于维护和测试
