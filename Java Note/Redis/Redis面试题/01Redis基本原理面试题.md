# Redis 基本原理面试题

---

## 1. Redis 有哪些常用数据类型？分别适用什么场景？

| 类型     | 底层编码                         | 典型使用场景                |
| ------ | ---------------------------- | --------------------- |
| String | int / embstr / raw (SDS)     | 缓存对象、计数器、分布式锁、Session |
| Hash   | ziplist / hashtable          | 存储对象字段（如用户信息）、购物车     |
| List   | ziplist / quicklist          | 消息队列、最新动态列表、任务队列      |
| Set    | intset / hashtable           | 标签系统、好友关系、去重、抽奖       |
| ZSet   | ziplist / skiplist+hashtable | 排行榜、带权重的消息队列、延迟队列     |

**补充：** Redis 5.0 引入 Stream 类型，专为消息队列设计，支持消费者组。

## 2. Redis 底层数据结构有哪些？

**SDS（Simple Dynamic String）动态字符串**
- 记录 `len`（已用长度）和 `alloc`（分配长度），O(1) 获取长度
- 二进制安全，可存储任意字节数据
- 扩容规则：小于 1MB 时翻倍，大于 1MB 时每次加 1MB

**IntSet 整数集合**
- 有序整数数组，支持 int16/int32/int64 编码
- 插入更大整数时自动升级编码，不可降级，倒序扩容。

**Dict 哈希表**
- 两个 hashtable（ht[0] 正常使用，ht[1] 用于 rehash）
- 扩容条件：`loadFactor >= 1`（无 BGSAVE/BGREWRITEAOF）或 `>= 5`（有后台任务）
- 缩容条件：`loadFactor < 0.1`
- **渐进式 rehash**：每次增删改查时迁移一个 bucket，避免阻塞

**ZipList 压缩列表**
- 连续内存，节省指针开销
- Entry 结构：`previous_entry_length` + `encoding` + `content`
- **连锁更新问题**：前一个 entry 长度变化可能导致后续所有 entry 的 `previous_entry_length` 字段级联扩展

**QuickList**
- ZipList 组成的双端链表
- 通过 `list-max-ziplist-size` 控制每个 ZipList 节点大小
- 通过 `list-compress-depth` 控制两端节点压缩深度

**SkipList 跳表**
- 多层有序链表，每层节点随机决定是否晋升
- 平均 O(log n) 查找，最坏 O(n)
- ZSet 同时使用跳表（范围查询）和哈希表（精确查询）

## 3. ZSet 为什么用跳表而不用红黑树？

**① 内存占用更灵活**

从内存占用上来比较，跳表比平衡树更灵活一些：平衡树每个节点包含 2 个指针（分别指向左右子树），而跳表每个节点包含的指针数目平均为 `1/(1-p)`，如果像 Redis 里的实现一样取 `p=1/4`，那么平均每个节点包含 1.33 个指针，比平衡树更有优势。

**② 范围查找更简单**

在做范围查找的时候，跳表比平衡树操作要简单：在平衡树上，找到指定范围的小值之后，还需要以中序遍历的顺序继续寻找其它不超过大值的节点。如果不不对平衡树进行一定的改造，这里的中序遍历并不容易实现。而在跳表上进行范围查找就非常简单，只需要在找到小值之后，对第 1 层链表进行若干步的遍历就可以实现。

**③ 算法实现更简单**

从算法实现难度上来比较，跳表比平衡树要简单得多。平衡树的插入和删除操作可能引发子树的调整，逻辑复杂，而跳表的插入和删除只需要修改相邻节点的指针，操作简单又快速。

## 4. Redis 的 String 类型底层是什么？为什么不用 C 字符串？

底层是 **SDS（Simple Dynamic String）**。C 字符串的问题：
- 获取长度需要 O(n) 遍历
- 不记录长度，不是二进制安全（遇到 `\0` 截断）
- 修改时频繁内存重分配

SDS 通过 `len` 字段解决以上所有问题，并通过预分配减少内存重分配次数。

---

## 5. Redis 是否线程安全？

Redis 的**命令执行**是单线程的，因此单个命令是原子的，不存在并发问题。但：
- 多个命令组合操作不是原子的（需要用 `MULTI/EXEC` 或 Lua 脚本）
- 客户端并发写入同一个 key 时，最终结果取决于命令到达顺序

## 6. 跳表的使用场景？

跳表在 Redis 中主要用于 **ZSet（有序集合）** 的底层实现，具体应用场景包括：

**① 排行榜系统**
- 存储用户分数，支持按分数范围查询（`ZRANGEBYSCORE`）
- 获取 Top N（`ZREVRANGE`）
- 跳表的有序性和范围查询能力完美适配排行榜需求

**② 延迟队列**
- 将任务的执行时间作为 score，任务作为 member
- `ZRANGEBYSCORE` 获取到期任务
- 跳表保证了范围查询的高效性

**③ 滑动窗口限流**
- 记录请求时间戳作为 score
- 通过 `ZRANGEBYSCORE` 统计时间窗口内的请求数
- 跳表的有序性支持高效的时间范围查询

**④ 社交关系**
- 存储好友列表，按亲密度排序
- 支持按亲密度范围查询
- 跳表的 O(log n) 查询效率保证性能

**⑤ 实时消息排行**
- 按点赞数、评论数排序
- 支持按分数范围筛选
- 跳表的有序性支持高效的排行查询

**核心优势：**
- 范围查询高效：O(log n) + O(m)，m 为结果数量
- 实现简单：比红黑树、AVL 树更容易实现和维护
- 内存友好：相比红黑树，指针开销更小

## 7. 讨论一下跳表在 Redis 中的使用规范

跳表在 Redis 中的使用遵循以下规范：

**① 数据结构组合**
- ZSet = 跳表 + 哈希表
- 跳表：支持范围查询（`ZRANGEBYSCORE`）
- 哈希表：支持 O(1) 精确查询（`ZSCORE`）
- 两者配合，兼顾范围查询和精确查询

**② 层级设计**
- 每个节点的层数随机决定，概率为 1/4
- 最大层数为 32（`ZSKIPLIST_MAXLEVEL`）
- 随机层级保证了查询效率的均匀分布

**③ 内存管理**
- 节点按 score 排序，相同 score 按 member 字典序
- 节点内存 = 指针数组 + member + score
- 大数据量时考虑压缩或分片

**④ 性能优化**
- 单个命令操作跳表是原子的（单线程）
- 范围查询尽量使用 `LIMIT` 限制结果数量
- 避免一次查询过大数据集（如 `ZRANGEBYSCORE` 不加 LIMIT）

**⑤ 使用限制**
- 跳表不支持范围删除的原子性
- 大 key 操作（如删除大 ZSet）会阻塞主线程
- 需要定期检查 ZSet 大小，避免内存膨胀

**⑥ 编码转换**
- 元素数量 < `zset-max-ziplist-entries`（默认 128）且 value 长度 < `zset-max-ziplist-value`（默认 64）时使用 ZipList
- 超过阈值自动转换为跳表
- 转换是单向的，不会回退

**⑦ 最佳实践**
- 热点数据尽量保持在内存中
- 使用 `ZUNIONSTORE` 等聚合操作时注意性能
- 大范围查询使用 `SCAN` 类命令（如 `ZSCAN`）迭代

## 8. 如何设计一个并发安全的跳表？

参考文章：[如何实现一个并发安全的跳表 - 小徐先生](https://zhuanlan.zhihu.com/p/622177029)

**场景：** 跳表是 Redis ZSet 的底层结构，单线程下没问题。但如果多个线程同时插入、删除、查找，可能把链表搞断、数据丢失。核心问题是多个线程同时修改跳表时怎么保证数据不乱。

**方案一：粗粒度锁**

整个跳表加一把大锁，所有操作串行执行。

> 想象一个单人洗手间，门口挂个锁——一个人进去，其他人全等着。

- 优点：实现简单，不容易出错
- 缺点：并发度为零，多线程退化成单线程
- 适用：读多写少，或对性能要求不高

**方案二：左边界节点锁模型（细粒度锁）**

> 来源：小徐先生的知乎文章

**核心思想：** 将锁的粒度从全局细化到节点维度，通过"左边界锁模型"实现并发操作。

**什么是左边界？**
跳表由多层单向链表组成，插入/删除节点本质上是调整"左边界"节点的 next 指针。

```
插入 key=8 的过程：

head → [0] → [2] → [4] → [7] → [10] → null
                        ↑
                      左边界（需要锁住这个节点）

步骤：
1. 构造新节点 {key: 8}
2. 令 {key: 8} 的 next 指向左边界 {key: 7} 的 next
3. 令 {key: 7} 的 next 指向 {key: 8}
```

**三类锁设计：**

| 锁类型 | 粒度 | 作用 |
|--------|------|------|
| deleteMutex | 全局读写锁 | delete 操作取写锁（全局互斥），get/put 取读锁（共享） |
| keyMutex | 每个 key 一把锁 | 同一个 key 的 put 操作串行化 |
| nodeMutex | 每个节点一把读写锁 | 作为左边界锁，get 加读锁，put 加写锁 |

**各操作流程：**

**① get 操作（读）**
- 取 deleteMutex 读锁（与 delete 互斥，与 put 共享）
- 逐层遍历，对左边界节点加读锁
- 找到目标返回，多个 get 可并发

**② put 操作（写）**
- 取 deleteMutex 读锁
- 取 keyMutex（同一个 key 的 put 互斥）
- 先 search 检查 key 是否存在（对左边界加读锁）
  - 存在：直接更新值
  - 不存在：随机新节点高度 → 对 head 加锁扩容（如需）→ 逐层对左边界加写锁 → 插入新节点

**③ delete 操作（删）**
- 取 deleteMutex 写锁（全局互斥）
- 无需对左边界加锁（因为 delete 需要绕过被删节点调整指针，会破坏左边界锁模型）
- 直接遍历调整指针实现删除

**优点：**
- 并发度高：不同区间的操作可以并行
- 读操作共享：get 之间不互斥
- 精确互斥：只有相同 key 的写操作才互斥

**缺点：**
- delete 操作退化为全局锁（左边界锁模型无法处理删除）
- 实现复杂，需要处理多种锁的组合

---

**方案三：节点级锁耦合（Hand-over-Hand Locking）**

每个节点一把锁，操作时只锁住"需要用到的节点"。从头节点开始，依次锁住当前节点和下一个节点，然后释放前一个节点，像接力一样往前走：

```
查找 key=50 的过程：

head → [10] → [30] → [50] → [70] → null

第一步：锁 head，锁 10，发现 50 > 10
第二步：释放 head，锁 30，发现 50 > 30
第三步：释放 10，锁 50，找到了！

同一时刻最多持有两把锁（当前节点和下一个节点）
```

插入时找到位置后锁住前后两个节点，插入新节点再释放；删除时锁住待删节点和前驱节点，修改指针跳过它。其他线程访问别的区间不受影响。

---

**方案四：读写锁优化**

跳表读操作远多于写操作，可以用读写锁：多个读线程同时进行，写线程独占。

> 类比图书馆的书——很多人可以同时翻阅，但修改内容时必须等所有人都放下。

可以给每层或每个节点加读写锁，粒度越细并发越高，但锁开销也越大。

---

**方案五：无锁跳表（Lock-Free）**

用 CAS（Compare-And-Swap）实现无锁并发，Java 里对应 `AtomicReference` + `compareAndSet`。插入时先创建新节点，用 CAS 尝试修改前驱节点的指针，失败则重试。

> 类比多人同时抢停车位——谁先把车停进去算谁的，其他人重新找。

- 优点：没有锁开销，高并发性能好
- 缺点：实现复杂，需处理 ABA 问题，CPU 可能空转

---

**面试简答版：**

并发安全的跳表有五种方案：①粗粒度锁，整表加锁，并发度低；②左边界节点锁模型，对操作目标的左边界节点加锁，不同区间可并行，但 delete 退化为全局锁；③节点级锁耦合，遍历时锁相邻两个节点接力前进，最多同时持有两把锁；④读写锁，读多写少场景下多个读可以并发；⑤无锁 CAS，性能最好但实现复杂。实际选择取决于场景——读多写少用读写锁，写多用节点锁耦合，极致性能考虑无锁。

## 9. Redis 为什么快？

Redis 快主要归功于四个设计：**内存操作 + 高效数据结构 + 单线程 + I/O 多路复用**。

**① 基于内存操作**

> 想象你去一家小餐馆，菜就在手边的冰柜里，不用跑去远处仓库拿——上菜自然快。

Redis 绝大部分操作在内存里完成，数据也存在内存中。内存读写速度是磁盘的几十万倍，少了磁盘 IO 这个最大瓶颈，自然就快。

**② 高效的数据结构**

> 厨师不是用普通锅铲，而是针对不同菜式用了特制厨具——切丝有切丝器，刨冰有刨冰机。

Redis 针对不同场景专门设计了底层结构：String 用 SDS、List 用 QuickList、Hash 用哈希表、ZSet 用跳表……每种结构都为特定操作优化过，读写效率远高于通用容器。

**③ 采用单线程**

> 一个厨师专做一道菜，不用在灶台之间来回切换，反而比多人手忙脚乱效率高。

单线程避免了多线程之间的锁竞争、上下文切换、CPU 缓存失效等问题。CPU 执行一条指令是纳秒级，网络 IO 是毫秒级——单线程完全够用，没必要引入多线程的复杂度。

**④ I/O 多路复用**

> 一个人同时盯多个灶台，哪个灶台有动静就去处理哪个，没动静就歇着。

虽然命令执行是单线程，但 Redis 用 epoll 实现了 I/O 多路复用——一个线程同时监听成千上万个客户端连接，哪个连接有请求就处理哪个，不浪费 CPU。

**⑤ 自建虚拟内存机制**

> 冰柜满了怎么办？把不常用的食材先挪到仓库去，把冰柜腾出来放常用的东西。

Redis 在内存不足时，会自己实现一套虚拟内存（VM）机制，把不常访问的冷数据从内存交换到磁盘，腾出宝贵的内存空间给热数据。这样冷热数据分离，避免了因为内存不足导致整体访问速度下降。

**提升容量的两种方式：**
- **分片**：将数据分散到多个 Redis Server 上（即集群方案）
- **虚拟内存**：把不常用的数据交换到磁盘上，热数据继续留在内存中

> 需要注意的是，Redis 并没有使用操作系统提供的 Swap 机制，而是自己实现了一套虚拟内存管理，效率更高。

**面试简答版：**

> Redis 快主要五个原因：一是基于内存操作，比磁盘快几个数量级；二是专门设计了高效的数据结构；三是单线程避免了锁竞争和上下文切换；四是 I/O 多路复用让单线程能处理大量并发连接；五是自建虚拟内存机制实现冷热数据分离，内存不足时依然能保持高性能。

---

## 10. Redis 为什么是单线程的？

Redis 的**核心命令处理**是单线程的，原因：
1. 纯内存操作，CPU 不是瓶颈，瓶颈在网络 IO
2. 单线程避免了锁竞争和上下文切换开销
3. 实现简单，不需要考虑并发安全

---

## 11. Redis 单线程为什么还这么快？

1. **纯内存操作**：读写速度极快
2. **IO 多路复用**：基于 epoll，单线程处理大量并发连接
3. **简单的数据结构**：专为高性能设计（SDS、跳表等）
4. **避免上下文切换**：单线程无线程切换开销
5. **命令处理简单**：大多数命令时间复杂度 O(1) 或 O(log n)

---

## 12. Redis 6.0 之前是单线程，6.0 之后是多线程吗？

参考文章：[追求性能极致：Redis6.0的多线程模型 - Brand](https://www.cnblogs.com/wzh2010/p/15886804.html)

**不完全准确。**

- **v4.0**：引入多线程处理**异步任务**（如 `UNLINK` 异步删除大 key、AOF 重写）
- **v6.0**：引入多线程处理**网络 IO**，但**命令执行仍是单线程**

---

### 6.0 之前的单线程模式

Redis 所谓的单线程并不是所有工作都只有一个线程执行，而是指**网络 IO 和键值对读写**由一个线程完成。

处理流程：获取 socket → 解析命令 → 执行命令 → 返回结果，全部由主线程串行处理。

其他功能（持久化、异步删除、集群数据同步等）由额外线程执行。

**为什么用单线程？**
- Redis 主要受限在内存和网络，CPU 几乎没有瓶颈
- 单线程避免了上下文切换、加解锁、死锁等性能损耗
- 通过 AE 事件模型 + IO 多路复用，单线程已足够高效

---

### 6.0 之后为什么引入多线程？

随着网络硬件性能提升，**网络 IO 成为瓶颈**：

```
单线程处理网络数据时，调用 epoll 的过程是阻塞的
并发量达到万级 QPS 时，网络 IO 读写速度跟不上
→ CPU 时间大部分浪费在网络 read/write 系统调用上
```

**引入多线程的目的：**
1. 充分利用多核 CPU 资源（单线程只能用一个核）
2. 分摊网络 IO 读写负荷，降低耗时

---

### Redis 6.0 多线程工作流程

```
1. 主线程建立连接，接收数据，将 socket 放入等待队列
2. 通过轮询将 socket 分配给 IO 线程
3. 主线程阻塞，等待 IO 线程完成 socket 读取和解析
4. IO 线程完成读取后，主线程执行 Redis 命令
5. 主线程阻塞，等待 IO 线程将结果写回 socket
6. 主线程清空队列，等待新请求
```

**本质：** 将网络 IO 读写独立出来，交给 IO 线程组并行处理，命令执行仍由主线程串行完成。

### 关键配置参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `io-threads` | 4 | IO 线程数（包含主线程） |
| `io-threads-do-reads` | `no` | 是否启用多线程读取（设为 `yes` 启用） |

```conf
# redis.conf 配置
io-threads-do-reads yes   # 开启多线程读
io-threads 5              # 线程数，8核机器建议设为 5~6
```

**线程数配置建议：**
- 4 核机器：2~3 个线程
- 8 核机器：5~6 个线程
- **线程数一定要小于机器核数**，超过 8 个意义不大

---

### 为什么命令执行仍是单线程？

- 纯内存操作，CPU 不是瓶颈
- 单线程避免锁竞争和上下文切换
- 保持原子性语义，不需要考虑并发安全

---

### 总结

Redis 的**命令执行始终是单线程**，多线程只用于：
- **v4.0+**：异步删除（UNLINK）、AOF 重写等后台任务
- **v6.0+**：网络 IO（读取请求、写回响应）

这样既提升了网络吞吐量，又保持了单线程的简单性和原子性。

## 13. Redis 使用的是哪种 IO 模型？

**IO 多路复用**，Linux 下使用 **epoll**。

五种 IO 模型对比：

| 模型 | 特点 |
|------|------|
| 阻塞 IO | 等待数据时线程阻塞 |
| 非阻塞 IO | 轮询，CPU 占用高 |
| IO 多路复用 | 一个线程监听多个 fd，有事件才处理 |
| 信号驱动 IO | 内核通知用户态，仍需同步读取 |
| 异步 IO | 全程异步，Linux 支持不完善 |

epoll 相比 select/poll 的优势：
- 无需每次传入全部 fd，使用事件驱动
- 支持边缘触发（ET）和水平触发（LT）
- 时间复杂度 O(1)，select/poll 是 O(n)

## 14. Redis 如何处理过期的 key？

过期删除有三种策略，Redis 实际选了其中两种组合使用：
![[Pasted image 20260522185733.png]]
**① 定时删除**

给每个过期 key 设置一个定时器，到期自动删除。听起来最完美——过期就删，内存释放最快。但问题在于：如果有大量 key 过期，就需要创建大量定时器，这会占用大量 CPU 时间，拖慢正常请求的处理。所以 Redis **没有采用**这种策略。

**② 惰性删除**

不主动删，等你来访问这个 key 的时候才检查它有没有过期。过期了就删掉返回空，没过期就正常返回。

> 比喻：像快递驿站不主动清理过期包裹，只有你来取件时才检查——你没来，它就一直放着。

- 优点：几乎不消耗额外 CPU，对服务器性能影响最小
- 缺点：如果一个 key 过期后一直没人访问，它就会一直赖在内存里，造成内存浪费

**③ 定期删除**

每隔一段时间，随机抽取一批 key 检查是否过期，过期的就删掉。

> 比喻：像保洁阿姨每隔一段时间来翻一翻储物柜，看到过期的东西就清理掉。

具体做法：每次从 expires 字典中随机抽取 20 个 key 检查，如果超过 25%（即 5 个以上）已经过期，就重复这个过程继续删。这样既限制了每次检查的耗时，又能逐步清理过期 key。

**Redis 为什么选"惰性 + 定期"？**

两种策略各管一个方面：
- 惰性删除：保证**CPU 性能**，不做多余检查
- 定期删除：保证**内存不会被过期 key 无限占用**，定期兜底清理

两者配合，在 CPU 开销和内存回收之间取得平衡。

## 15. Redis 过期删除策略和内存淘汰策略有什么区别？

这是一个高频面试题，很多人会混淆这两个概念。

| 对比项      | 过期删除策略            | 内存淘汰策略             |
| -------- | ----------------- | ------------------ |
| **触发条件** | key 被设置了过期时间（TTL） | 内存达到 maxmemory 上限  |
| **删除对象** | 已过期的 key          | 未过期的 key（可能）       |
| **目的**   | 回收过期 key 占用的内存    | 在内存满时腾出空间          |
| **策略**   | 惰性删除 + 周期删除       | 8 种淘汰算法（LRU、LFU 等） |

**举例说明：**

```
场景1：key 设置了 TTL，过期后被删除
→ 这是"过期删除策略"在工作

场景2：内存满了，但有些 key 还没过期
→ 这是"内存淘汰策略"在工作，会强制删除一些 key
```

**两者的关系：**
- 过期删除策略：主动清理已到期的 key
- 内存淘汰策略：被动兜底，内存不够时强制淘汰
- 两者配合使用，保证内存不会无限增长

---

## 16. Redis 内存满了怎么办？有哪些淘汰策略？

Redis 提供 **8 种**内存淘汰策略（通过 `maxmemory-policy` 配置）：

| 策略                | 说明                      |
| ----------------- | ----------------------- |
| `noeviction`      | 不淘汰，内存满时写操作报错（默认）       |
| `allkeys-lru`     | 从所有 key 中淘汰最近最少使用的      |
| `volatile-lru`    | 从设置了过期时间的 key 中淘汰 LRU   |
| `allkeys-random`  | 从所有 key 中随机淘汰           |
| `volatile-random` | 从设置了过期时间的 key 中随机淘汰     |
| `volatile-ttl`    | 淘汰剩余 TTL 最短的 key        |
| `allkeys-lfu`     | 从所有 key 中淘汰访问频率最低的（LFU） |
| `volatile-lfu`    | 从设置了过期时间的 key 中淘汰 LFU   |

**LRU vs LFU：**
- LRU（Least Recently Used）：淘汰最久未访问的，无法区分冷热数据中的偶发访问
- LFU（Least Frequently Used）：淘汰访问频率最低的，更能反映真实热度，Redis 使用近似 LFU

**推荐策略：** 缓存场景用 `allkeys-lru` 或 `allkeys-lfu`；有明确过期时间用 `volatile-lru`。

## 17. 如何设置 Redis 最大内存？生产环境一般设置多少？

```bash
# redis.conf
maxmemory 512mb

# 或运行时设置
CONFIG SET maxmemory 512mb
CONFIG SET maxmemory-policy allkeys-lru
```

**生产环境内存设置建议：**

| 场景 | 建议设置 | 原因 |
|------|----------|------|
| 纯缓存用途 | 物理内存的 **70%~80%** | 留空间给 fork 子进程（RDB/AOF 重写） |
| 混合用途（缓存+持久化） | 物理内存的 **50%~60%** | 避免 fork 时 Copy-On-Write 导致内存翻倍 |
| 有从节点的主节点 | 物理内存的 **50%** | 全量同步时 BGSAVE 需要额外内存 |

**注意事项：**
- 不要设置超过物理内存的 80%，否则可能触发 swap，性能急剧下降
- 使用 `INFO memory` 查看 `mem_fragmentation_ratio`（内存碎片率）
  - `> 1.5`：碎片率过高，考虑重启或执行 `MEMORY PURGE`
  - `< 1`：内存不足，Redis 使用了 swap，性能急剧下降

## 18. Redis 持久化有哪些方式？

![[Pasted image 20260522185146.png]]
**AOF（Append Only File）日志**

每执行一条写操作命令，就将该命令以追加的方式写入到 AOF 文件，然后在恢复时，以逐一执行命令的方式来进行数据恢复。用 AOF 日志的方式来恢复数据很慢，因为 Redis 执行命令由单线程负责的，AOF 日志恢复数据的方式是顺序执行日志里的每一条命令，如果 AOF 日志很大，这个过程就会很慢了。

三种刷盘策略：
- `always`：每条命令都刷盘，最安全，性能最差
- `everysec`：每秒刷盘，最多丢失 1 秒数据（推荐）
- `no`：由 OS 决定，性能最好，可能丢失较多数据

**RDB（Redis Database Backup）快照**

RDB 快照是记录某一个瞬间的内存数据，记录的是实际数据，而 AOF 文件记录的是命令操作的日志，而不是实际的数据。因此在 Redis 恢复数据时，RDB 恢复数据的效率会比 AOF 高些，因为直接将 RDB 文件读入内存就可以，不需要像 AOF 那样还需要额外执行操作命令的步骤才能恢复数据。

RDB 快照是全量快照，也就是说每次执行快照，都是把内存中的所有数据都记录到磁盘中。如果频率太频繁，可能会对 Redis 性能产生影响。如果频率太低，服务器故障时，丢失的数据会更多。通常可能设置至少 5 分钟才保存一次快照，这时如果 Redis 出现宕机等情况，意味着最多可能丢失 5 分钟数据。

触发方式：`SAVE`（同步，阻塞）、`BGSAVE`（异步，fork 子进程）、配置 `save` 规则自动触发

---

## 19. RDB 和 AOF 如何选择？

| 对比项 | RDB | AOF |
|--------|-----|-----|
| 数据安全性 | 低（可能丢失分钟级数据） | 高（最多丢失 1 秒） |
| 文件大小 | 小（二进制压缩） | 大（文本命令） |
| 恢复速度 | 快 | 慢 |
| 性能影响 | fork 时有短暂阻塞 | 持续写入，影响较小 |
| 适用场景 | 备份、容灾 | 数据安全要求高 |
![[Pasted image 20260522185459.png]]
**生产推荐：** 同时开启 RDB 和 AOF（Redis 4.0+ 支持混合持久化），AOF 优先用于恢复，RDB 用于备份。

## 20. Redis 混合持久化是什么？

Redis 4.0 引入，AOF-RDB 混用。

在 AOF 重写日志时，fork 出来的重写子进程会先将与主线程共享的内存数据以 RDB 方式写入到 AOF 文件，然后主线程处理的操作命令会被记录在重写缓冲区里，重写缓冲区里的增量命令会以 AOF 方式写入到 AOF 文件，写入完成后通知主进程将新的含有 RDB 格式和 AOF 格式的 AOF 文件替换旧的 AOF 文件。文件的前半部分是 RDB 格式的全量数据，后半部分是 AOF 格式的增量数据。

**好处：** 重启 Redis 加载数据的时候，由于前半部分是 RDB 内容，加载速度会很快。加载完 RDB 的内容后，才会加载后半部分的 AOF 内容，这里的内是 Redis 后台子进程重写 AOF 期间，主线程处理的操作命令，可以使得数据更少的丢失。

**缺点：** AOF 文件的可读性变差了。

开启方式：`aof-use-rdb-preamble yes`

## 21. 什么是 Redis 事务？原理是什么？

Redis 事务通过 `MULTI/EXEC` 实现命令的批量执行：

```
MULTI          # 开启事务
SET k1 v1      # 命令入队
SET k2 v2      # 命令入队
EXEC           # 执行所有入队命令
```

**事务原理：**

```
客户端发送 MULTI
    ↓
Redis 服务器记录事务状态，开启事务队列
    ↓
客户端发送命令（SET、GET、INCR 等）
    ↓
Redis 将命令放入事务队列（FIFO）
    ↓
客户端发送 EXEC
    ↓
Redis 顺序执行队列中的所有命令
    ↓
返回每条命令的执行结果
```

**核心机制：**
- **事务队列**：MULTI 后的命令不立即执行，而是放入队列
- **顺序执行**：EXEC 时按队列顺序逐条执行
- **原子性**：要么全部执行，要么全部不执行（但运行时错误不回滚）

**其他命令：**
- `DISCARD`：放弃事务，清空命令队列
- `WATCH key`：乐观锁，监视 key，若在 EXEC 前被修改则事务失败

---

## 22. Redis 事务的注意点有哪些？

**① 运行时错误不回滚**

```
MULTI
SET k1 v1
LPUSH k1 v2    # 类型错误！k1 是 String，不能 LPUSH
EXEC
```

- 语法错误：全部回滚（入队阶段就报错）
- 运行时错误：**只跳过出错命令，其他继续执行**
- 这和 MySQL 事务不同，MySQL 会回滚整个事务

**② 不支持回滚**

Redis 官方文档明确说明：不支持回滚，因为 Redis 认为失败通常是编程错误导致的。

**③ 事务中不能使用其他命令**

事务执行期间，其他客户端的命令会被阻塞，直到 EXEC 完成。

**④ WATCH 只检测 key 是否变化**

```
WATCH key
# 假设 key 被其他客户端修改
MULTI
SET key newvalue
EXEC  # 返回 nil，事务失败
```

- WATCH 只检查 key 的值是否变化，不检查是谁修改的
- EXEC 后 WATCH 自动失效

**⑤ 事务不是连续快照**

事务执行过程中，如果 Redis 崩溃，已执行的命令不会回滚。

---

## 23. 怎么理解 Redis 事务？

可以从三个角度理解：

**角度一：命令队列 + 顺序执行**

Redis 事务本质上是一个命令队列：
- MULTI：开始入队
- 命令：入队不执行
- EXEC：顺序执行所有命令

**角度二：弱事务 vs 强事务**

| 对比项 | Redis 事务       | MySQL 事务                  |
| --- | -------------- | ------------------------- |
| 原子性 | 部分满足（运行时错误不回滚） | 完全满足                      |
| 回滚  | 不支持            | 支持                        |
| 隔离性 | 满足（单线程）        | 可选（RU/RC/RR/Serializable） |
| 持久性 | 取决于持久化配置       | 满足（WAL）                   |

**角度三：替代方案更强大**

Redis 官方推荐用 **Lua 脚本** 替代事务：
- Lua 脚本整体原子执行
- 支持条件逻辑（if/else）
- 一次网络往返完成所有操作
- 不需要 MULTI/EXEC

**面试回答模板：**

> Redis 事务通过 MULTI/EXEC 实现命令的批量执行，原理是将命令放入队列，EXEC 时顺序执行。但它不支持回滚（运行时错误只跳过出错命令），是弱事务。生产中更推荐用 Lua 脚本实现原子操作，因为它支持条件逻辑且真正原子执行。

## 24. Redis 事务满足 ACID 吗？

| ACID 特性 | Redis 事务     | 说明                           |
| ------- | ------------ | ---------------------------- |
| 原子性     | **部分满足**     | 语法错误全部回滚；运行时错误只跳过出错命令，其他继续执行 |
| 一致性     | **部分满足**     | 运行时错误不回滚，可能导致数据不一致           |
| 隔离性     | **满足**       | 单线程执行，EXEC 期间不会被其他命令插入       |
| 持久性     | **取决于持久化配置** | 开启 AOF always 模式才能保证         |

**结论：** Redis 事务不完全满足 ACID，尤其是原子性（运行时错误不回滚）。

## 25. Redis 事务和 Lua 脚本有什么区别？

| 对比项 | MULTI/EXEC 事务 | Lua 脚本 |
|--------|----------------|---------|
| 原子性 | 运行时错误不回滚 | 脚本整体原子执行 |
| 条件逻辑 | 不支持 | 支持 if/else 等逻辑 |
| 网络往返 | 多次（入队 + EXEC） | 一次（发送脚本） |
| 复杂度 | 简单 | 灵活，可实现复杂逻辑 |

**推荐：** 需要原子性保证的复杂操作用 Lua 脚本（`EVAL`）。

## 26. Redis 执行 Lua 脚本的原理是什么？

**执行流程：**

```
客户端发送 EVAL 脚本
    ↓
Redis 服务器接收脚本
    ↓
脚本编译（首次执行时编译，后续直接执行编译后的字节码）
    ↓
在 Redis 服务器上执行 Lua 脚本
    ↓
脚本中可以调用 Redis 命令（通过 redis.call 或 redis.pcall）
    ↓
返回脚本执行结果
```

**关键特性：**

**① 原子执行**
- Lua 脚本执行期间，其他所有命令被阻塞
- 脚本中的所有 Redis 命令作为一个整体执行
- 不会被其他客户端的命令插入

**② 脚本缓存**
- 首次执行的脚本会被缓存（SHA1 校验和）
- 后续可以使用 `EVALSHA` 只发送 SHA1，减少网络传输
- 缓存不会自动过期，直到 Redis 重启

**③ 原子性保证**

```lua
-- 示例：原子性扣减库存
local stock = redis.call('GET', KEYS[1])
if stock and tonumber(stock) > 0 then
    redis.call('DECR', KEYS[1])
    return 1  -- 扣减成功
else
    return -1  -- 库存不足
end
```

**④ 脚本限制**
- 脚本执行时间不能超过 `lua-time-limit`（默认 5 秒）
- 超时会报错，但脚本可能已经部分执行
- 生产环境建议设置合理的超时时间

**⑤ 复制方式**
- 主从模式：脚本发送到从节点执行（`lua-replicate-commands yes`）
- 集群模式：脚本中所有 key 必须在同一个 slot

**与事务对比：**

| 对比项 | MULTI/EXEC | Lua 脚本 |
|--------|-----------|---------|
| 原子性 | 部分满足 | 完全满足 |
| 条件逻辑 | 不支持 | 支持 |
| 网络往返 | 多次 | 一次 |
| 执行效率 | 较高 | 较低（脚本解析） |

---

## 27. WATCH 命令的原理是什么？

`WATCH` 实现了**乐观锁**：
1. `WATCH key` 后，Redis 记录该 key 的版本
2. 执行 `MULTI` 开启事务，命令入队
3. `EXEC` 时检查被 WATCH 的 key 是否被修改
4. 若被修改，事务失败返回 `nil`；否则正常执行

适用于"读-改-写"的并发场景，比悲观锁性能更好。

## 28. Redis 单点吞吐量有多少？

**单点 Redis 吞吐量参考：**

| 操作类型 | QPS（每秒请求数） | 说明 |
|----------|------------------|------|
| 简单 GET/SET | **10 万~15 万** | 单个 key 读写 |
| 复杂命令（SORT、聚合） | **1 万~5 万** | 涉及多 key 或排序 |
| Pipeline 批量 | **50 万~100 万** | 多个命令批量发送 |
| Lua 脚本 | **5 万~10 万** | 复杂原子操作 |

**影响因素：**
- **网络延迟**：本机 < 1ms，同机房 < 1ms，跨机房 1~10ms
- **命令复杂度**：O(1) 的命令最快，O(N) 或 O(logN) 较慢
- **数据大小**：value 越大，网络传输越慢
- **客户端连接数**：过多连接会增加上下文切换开销

**优化建议：**
- 使用 Pipeline 减少网络往返
- 避免大 key（value 超过 10KB 考虑拆分）
- 合理设置连接池大小
- 使用连接复用

**基准测试：**
```bash
# 使用 redis-benchmark 测试
redis-benchmark -h 127.0.0.1 -p 6379 -c 50 -n 100000
# -c 50：50 个并发连接
# -n 100000：10 万次请求
```

## 29. 怎么测试 Redis 的连通性？

**方法一：使用 redis-cli ping**

## 30. 什么是缓存穿透？如何解决？

**问题：** 查询一个不存在的 key，每次都打到数据库。

**解决方案：**
- **缓存空值**：查询结果为空时也缓存，设置较短 TTL
- **布隆过滤器**：请求前先判断 key 是否存在，不存在直接拒绝

---

## 31. 什么是缓存雪崩？如何解决？

**问题：** 大量 key 同时过期，或 Redis 宕机，导致请求全部打到数据库。

**解决方案：**
- 设置随机 TTL，避免同时过期
- Redis 高可用（主从 + 哨兵 / 集群）
- 服务降级和限流

---

## 32. 什么是缓存击穿？如何解决？

**问题：** 热点 key 过期瞬间，大量并发请求同时打到数据库。

**解决方案：**
- **互斥锁**：只允许一个请求重建缓存，其他等待
- **逻辑过期**：不设置真实 TTL，在 value 中存过期时间，过期后异步重建

## 33. 为什么用 Redis 作为 MySQL 的缓存？

**核心原因：Redis 快，MySQL 慢**

| 对比项 | Redis | MySQL |
|--------|-------|-------|
| 存储介质 | 内存 | 磁盘 |
| 读写速度 | 微秒级（10万+ QPS） | 毫秒级（千级 QPS） |
| 数据模型 | Key-Value | 关系型表 |

**具体优势：**

**① 减少磁盘 I/O**
- MySQL 数据存在磁盘，每次查询都要访问磁盘
- Redis 数据存在内存，直接从内存读取，避免磁盘 I/O
- 内存读写速度是磁盘的几十万倍

**② 提高系统 QPS**
- MySQL 单机 QPS 通常在几千
- Redis 单机 QPS 可达 10 万+
- 使用 Redis + MySQL 组合，可以有效提高系统整体 QPS

**③ 减轻 MySQL 压力**
- 热点数据放 Redis，减少 MySQL 查询次数
- 避免大量请求直接打到 MySQL，防止数据库崩溃

**典型架构：**
```
客户端请求 → Redis（查缓存）
                ↓ 命中
              直接返回
                ↓ 未命中
              MySQL（查数据库）→ 写入 Redis → 返回
```

**面试回答模板：**

> MySQL 是数据库系统，数据存在磁盘，访问需要磁盘 I/O。Redis 数据存在内存，可以直接从内存获取，避免磁盘 I/O，提高操作速度。使用 Redis + MySQL 结合的方式，热点数据放 Redis，可以有效提高系统 QPS，减轻 MySQL 压力。

## 34. Redis 和 Memcached 的联系和区别？

### 共同点

- 都是内存数据库
- 性能都非常高
- 都有过期策略

### 区别

| 对比项 | Redis | Memcached |
|--------|-------|-----------|
| **数据结构** | 丰富（String、List、Hash、Set、ZSet、Stream） | 单一（仅 String） |
| **线程模型** | 单线程（v6.0 多线程IO） | 多线程 |
| **持久化** | 支持 RDB 和 AOF | 不支持 |
| **集群** | 支持主从复制、哨兵、Cluster | 仅单节点 |
| **内存淘汰** | 8种策略，可选 | 仅 LRU |
| **Value 大小** | 512MB | 1MB |
| **过期时间** | 无限制 | 最长 30 天 |
| **发布订阅** | 支持 | 不支持 |
| **Lua 脚本** | 支持 | 不支持 |

### 详细对比

**① 线程模型**

| | Redis | Memcached |
|--|-------|-----------|
| 模型 | 单线程处理命令 | 多线程处理请求 |
| 优点 | 无锁竞争，上下文切换少 | 可利用多核 CPU |
| 缺点 | 无法利用多核，复杂操作会阻塞 | 多线程切换和加锁有性能损耗 |

**② 数据结构**

```
Redis 支持：
├── String：缓存、计数器、分布式锁
├── List：消息队列
├── Hash：存储对象
├── Set：去重、标签
├── ZSet：排行榜
└── Stream：消息队列（v5.0+）

Memcached：
└── String：只能序列化存储，"整存整取"
```

**③ 淘汰策略**

- **Redis**：8种策略（noeviction、allkeys-lru、volatile-lru 等），可灵活选择
- **Memcached**：仅 LRU 淘汰，且刚写入的数据可能被优先淘汰（设计缺陷）

**④ 持久化**

- **Redis**：支持 RDB 快照和 AOF 日志，可恢复数据
- **Memcached**：不支持持久化，宕机数据全部丢失

**⑤ 集群**

- **Redis**：支持主从复制、哨兵、Cluster，高可用
- **Memcached**：仅单节点，无主从架构

### 选型建议

| 场景 | 推荐 |
|------|------|
| 简单缓存，纯 KV | Memcached（多线程，性能略高） |
| 需要复杂数据结构 | Redis |
| 需要持久化 | Redis |
| 需要高可用 | Redis |
| 需要发布订阅/Lua | Redis |

**面试回答模板：**

> Redis 和 Memcached 都是内存数据库，性能都很高。主要区别：Redis 数据结构更丰富，支持持久化和集群，是单线程；Memcached 只支持 String，不支持持久化，是多线程。生产环境推荐 Redis，功能更强大，生态更完善。

## 35. 如何理解 Redis 原子性操作原理？

Redis 命令的原子性可以从三个层面理解：

**① API 层面：单线程串行处理**

Redis 提供的所有 API 都是由单线程串行处理的。同一时刻只有一个命令在执行，不存在并发竞争，因此单个命令天然是原子的。

**② 网络模型：epoll 多路复用**

Redis 采用单线程 + epoll 的网络模型，用来处理多个 Socket 请求。虽然能同时监听大量连接，但命令执行仍然是串行的——epoll 只负责"接收请求"，不改变"串行执行"的本质。

**③ 请求处理：fork 子进程不影响主进程**

Redis 会 fork 子进程来执行 RDB 持久化、AOF 重写等耗时操作，这些操作在子进程中进行，不影响主进程的命令执行。主进程始终保证单线程串行，原子性不受影响。

**其他原子性保障：**

- **Lua 脚本**：脚本中的所有命令作为一个整体原子执行，期间不会被其他命令插入
- **MULTI/EXEC 事务**：命令入队后一次性执行，EXEC 期间不会穿插其他命令
- **Pipeline**：批量发送命令，但不是原子的——命令之间可能穿插其他客户端的命令

**面试回答模板：**

> Redis 的原子性基于单线程模型——所有 API 由单线程串行处理，同一时刻只有一个命令执行，天然保证原子性。网络层用 epoll 处理并发连接，但不影响命令的串行执行。RDB/AOF 等耗时操作由 fork 子进程完成，不阻塞主进程。此外，Lua 脚本和 MULTI/EXEC 事务也能实现多命令的原子操作。

## 36. AOF 的三种写回策略是什么？

Always、Everysec 和 No，这三种策略在可靠性上是从高到低，而在性能上是从低到高。

**Always**
每次写操作命令执行完后，同步将 AOF 日志数据写回硬盘。最安全，每条命令都不丢失，但性能最差。

**Everysec**
每次写操作命令执行完后，先将命令写入到 AOF 文件的内核缓冲区，然后每隔一秒将缓冲区内容写回到硬盘。兼顾安全和性能，最多丢失 1 秒数据，是推荐策略。

**No**
就是不控制写回硬盘的时机。每次写操作命令执行完后，先将命令写入到 AOF 文件的内核缓冲区，再由操作系统决定何时将缓冲区内容写回硬盘。性能最好，但丢失数据量不可控。

| 策略 | 写回时机 | 可靠性 | 性能 |
|------|----------|--------|------|
| Always | 每条命令同步写盘 | 最高 | 最差 |
| Everysec | 每秒写盘 | 较高（最多丢 1s） | 适中（推荐） |
| No | OS 自己决定 | 最低 | 最好 |

## 37. AOF 的重写机制是什么？

随着执行的命令越多，AOF 文件的体积自然也会越来越大，为了避免日志文件过大，Redis 提供了 AOF 重写机制。

**重写原理：**
- 它会直接扫描数据库中所有的键值对数据，然后为每一个值对生成一条写操作命令
- 接着将该命令写入到新的 AOF 文件
- 重写完成后，就替换现有的 AOF 日志

**重写具体过程：**

触发重写机制后，主进程会创建重写 AOF 的子进程，此时父子进程共享物理内存，重写子进程只会对这个内存进行只读。重写 AOF 子进程读取数据库里的所有数据，并逐一把内存数据的键值对转换成一条命令，再将命令记录到重写日志。

在发生写操作的时候，操作系统才会去复制物理内存，这样是为了防止 fork 创建子进程时，由于物理内存数据的复制时间过长而导致父进程长时间阻塞的问题。

**为什么用子进程而不是线程？**

- 子进程进行 AOF 重写期间，主进程可以继续处理命令请求，从而避免阻塞主进程
- 子进程带有主进程的数据副本，使用子进程而不是线程，因为如果是使用线程，多线程之间会共享内存，那么在修改共享内存数据的时候，需要通过加锁来保证数据的安全，而这样就会降低性能
- 创建子进程时，父子进程是共享内存数据的，不过这个共享的内存只能以只读的方式，而当父子进程任意一方修改了该共享内存，会发生写时复制，于是父子进程就有了独立的数据副本，不用加锁来保证数据安全

**触发方式：**
- `BGREWRITEAOF` 命令手动触发
- AOF 文件大小超过 `auto-aof-rewrite-percentage` 配置时自动触发

## 38. 为什么先执行 Redis 命令，再把数据写入 AOF 日志？

**好处：**

**① 保证正确写入**
如果当前的命令语法有问题，错误的命令记录到 AOF 日志里后可能还会进行语法检查。先执行 Redis 命令，再把数据写入 AOF 日志可以保证写入的都是正确可执行的命令。

**② 不阻塞当前写操作**
因为当写操作命令执行成功后才会将命令记录到 AOF 日志，避免写入阻塞。

**缺陷：**

**① 数据可能会丢失**
执行写操作命令和记录日志是两个过程，Redis 还没来得及将命令写入到硬盘时发生宕机，数据会有丢失的风险。

**② 阻塞其他操作**
不会阻塞当前命令的执行，但因为 AOF 日志也是在主线程中执行，所以当 Redis 把日志文件写入磁盘的时候，还是会阻塞后续的操作无法执行。

## 39. AOF 子进程的内存数据跟主进程的内存数据不一致怎么办？

Redis 设置了一个 **AOF 重写缓冲区**，这个缓冲区在创建 `bgrewriteaof` 子进程之后开始使用。

在重写 AOF 期间，当 Redis 执行完一个写命令之后，它会同时将这个写命令写入到 **AOF 缓冲区**和 **AOF 重写缓冲区**。当子进程完成 AOF 重写工作后，会向主进程发送一条信号。主进程收到该信号后，会调用一个信号处理函数，将 AOF 重写缓冲区中的所有内容追加到新的 AOF 的文件中，使得新旧两个 AOF 文件所保存的数据库状态一致；新的 AOF 的文件进行改名，覆盖现有的 AOF 文件。

## 40. RDB 在执行快照的时候，数据能修改吗？

可以。执行 `BGSAVE` 过程中，Redis 依然可以继续处理操作命令的，数据是能被修改的，采用的是**写时复制技术（Copy-On-Write，COW）**。

执行 `BGSAVE` 命令的时候，会通过 `fork()` 创建子进程，此时子进程和父进程是共享同一片内存数据的，因为创建子进程的时候，会复制父进程的页表，但是页表指向的物理内存还是一个，由于共享父进程的所有数据，可以直接读取主线程里的内存数据，并将数据写入到 RDB 文件。

此时如果主线程执行读操作，则主线程和 `BGSAVE` 子进程互相不影响。如果主线程要修改共享数据里的某一块数据，就会发生写时复制，数据的物理内存就会被复制一份，主线程在这个数据副本进行修改操作。与此同时，子进程可以继续把原来的数据写入到 RDB 文件。

## 41. Redis 持久化时对过期键会如何处理？

**RDB 持久化：**

分两个阶段处理：

- **生成阶段**（BGSAVE 写 RDB 文件时）：会对 key 做过期检查，已经过期的 key 不会被写入 RDB 文件，相当于生成的快照里天然就不包含过期 key
- **加载阶段**（Redis 启动加载 RDB 文件时）：要区分主从身份
  - **主服务器**：加载时会逐个检查 key 是否过期，过期的 key 不载入数据库
  - **从服务器**：不管过不过期，全部载入。因为从服务器的数据最终由主服务器控制，主从同步时会清空从库数据，所以过期 key 对从库没有实际影响

**AOF 持久化：**

也分两个阶段：

- **写入阶段**：如果某个过期 key 还没被惰性删除或定期删除清理掉，AOF 文件会保留这个 key。等到它被删除时（惰性删除或定期删除触发），Redis 会往 AOF 文件追加一条 `DEL` 命令来显式记录删除操作
- **重写阶段**：重写时会扫描所有 key，已过期的 key 不会被写入新的 AOF 文件，相当于重写本身就是一次"净化"

## 42. Redis 主从模式中，对过期键会如何处理？

从库不会主动进行过期扫描。即使从库中的 key 已经过期了，如果有客户端访问从库，依然可以拿到 key 对应的值（不会返回空）。

过期 key 的删除完全由主服务器控制：
1. 主库上的 key 到期时，会在 AOF 文件里记录一条 `DEL` 命令
2. 这条 `DEL` 命令会同步到所有从库
3. 从库通过执行这条 `DEL` 指令来删除过期的 key

> 简单说：从库自己不删过期 key，等主库告诉它"这个 key 删了"，它才跟着删。所以从库读到过期 key 的数据是"延迟过期"的。

## 43. Redis 是什么？简述它的优缺点？

Redis 本质上是一个基于 Key-Value 模型的内存数据库，整个数据库加载在内存中操作，定期通过异步方式将数据刷到硬盘上保存。因为是纯内存操作，性能非常出色，每秒可以处理超过 10 万次读写操作，是目前性能最快的 Key-Value 数据库之一。

**优点：**

| 优点 | 说明 |
|------|------|
| 读写性能极高 | 读速度约 11 万次/秒，写速度约 8 万次/秒 |
| 支持数据持久化 | 同时支持 AOF 和 RDB 两种持久化方式，宕机后可恢复数据 |
| 支持事务 | 单个命令天然原子，多个命令可通过 MULTI/EXEC 包装成事务 |
| 数据结构丰富 | 除了 String，还支持 Hash、List、Set、ZSet、Stream 等 |
| 支持主从复制 | 主机会自动将数据同步到从机，可进行读写分离 |
| 功能特性丰富 | 支持发布订阅、通知、key 过期等特性 |

**缺点：**

| 缺点 | 说明 |
|------|------|
| 容量受内存限制 | 不能用作海量数据的高性能读写，适合较小数据量的场景 |
| 数据一致性问题 | 主从异步复制，宕机前可能有数据未同步到从机，切换后数据不一致 |

## 44. 为什么要用 Redis 做缓存？

**从高并发角度：**

直接访问数据库能承受的请求数是有限的，而 Redis 的吞吐量是 MySQL 的几十倍甚至上百倍。把数据库中的热点数据转移到 Redis 缓存中，用户的大部分请求直接打到缓存，只有缓存没命中的才去查数据库，数据库的压力就小了很多。

**从高性能角度：**

用户第一次访问数据时从数据库加载（磁盘读取，较慢），然后把结果存到 Redis 中。下次再访问同样的数据时直接从 Redis 读取（内存操作，极快）。如果数据库中的数据更新了，同步更新缓存即可。

> 类比：你经常查一本字典，每次都去图书馆借太慢了。于是你把常用的几页复印一份放桌上，查的时候直接看桌上的复印版，不用跑去图书馆。只有桌上没有的内容才去图书馆查。

## 45. 为什么要用 Redis 而不用 Map/Guava 做缓存？

Map 和 Guava Cache 是本地缓存（存在 JVM 内存中），Redis 是分布式缓存（独立部署）。两者的核心区别：

| 对比项 | Redis | Map/Guava |
|--------|-------|-----------|
| 容量 | 可用几十 G 内存做缓存 | 受 JVM 内存限制，通常几个 G |
| 持久化 | 支持 RDB/AOF，重启不丢数据 | 内存对象，程序重启数据就没了 |
| 分布式 | 多个服务实例共享一份缓存，数据一致 | 每个实例各自存一份，缓存不一致 |
| 并发能力 | 专业的缓存服务，每秒百万级并发 | 只是一个普通对象，性能有限 |
| 过期机制 | 支持 key 过期、淘汰策略 | 本身无过期功能，需自己实现 |
| API 丰富度 | 丰富的数据结构和操作命令 | 相对简单 |
| 共享性 | 可单独部署，多个项目共享 | 只能在创建它的程序里使用 |
| 管理工具 | 有专门的管理工具查看缓存数据 | 无 |

**什么时候用 Map/Guava？**

单体应用、数据量小、对一致性要求不高、追求极低延迟的场景，本地缓存更合适（因为少了网络开销）。

**什么时候用 Redis？**

多实例部署、数据量大、需要分布式共享、需要持久化的场景，必须用 Redis。

**实际项目中的常见组合：** 本地缓存（Caffeine）+ Redis，形成多级缓存。热点数据放本地缓存（最快），其他数据放 Redis（共享+持久化），数据库兜底。

## 46. Redis 的常用场景有哪些？

| 场景 | 说明 | 用到的数据结构 |
|------|------|--------------|
| **缓存** | 将热点数据存在 Redis 中，减少数据库访问压力，提升响应速度 | String |
| **排行榜** | 电商月度销量榜、商品上新排行等，用 ZSet 的 score 天然排序 | ZSet |
| **计数器** | 商品浏览量、视频播放量等，每次 +1 操作用 INCR 原子递增 | String |
| **分布式会话** | 集群模式下，将 Session 存在 Redis 中，各服务实例共享登录状态 | String |
| **分布式锁** | 用 SETNX 实现互斥锁，解决库存扣减、秒杀等场景的并发问题 | String |
| **社交网络** | 点赞、关注/被关注、共同好友等，用 Set 的交集/并集运算 | Set |
| **最新列表** | 用 LPUSH 插入新内容 ID，LTRIM 限制列表数量，保持最新 N 条 | List |
| **消息系统** | 发布/订阅（Pub/Sub）实现简单的消息队列，用于业务解耦和流量削峰 | List/Pub/Sub |

> 注意：Redis 做消息队列只能用于简单场景，专业消息队列（Kafka、RocketMQ）在可靠性、吞吐量上更强。

## 47. Redis 的数据类型有哪些？

Redis 有五种常用数据类型和三种特殊数据类型。

**五种常用类型**（详见 [[01Redis基本原理面试题#1. Redis 有哪些常用数据类型？分别适用什么场景？|Q1]]）：String、Hash、Set、List、SortedSet。

**三种特殊类型：**

**① Bitmap（位图）**

Bitmap 本质是一个以位（bit）为单位的数组，每个位置只能存 0 或 1。通过 `SETBIT`、`GETBIT`、`BITCOUNT` 等命令操作。

> 类比：你在黑板上画了一排格子，每个格子只能打勾或不打勾。一个人签到了就打勾，没签到就不打勾。最后数一下勾的总数就是签到人数。

适用场景：
- **用户签到**：`SETBIT sign:202605:user1001 21 1`（5月21日签到）
- **在线状态**：`SETBIT online:20260522 1001 1`（用户1001在线）
- **商品有没有货**：`GETBIT stock:goods100 0`（0=缺货，1=有货）

优点是极省内存，1 亿个用户的状态只需要 12.5MB 内存。

**② HyperLogLog（基数统计）**

HyperLogLog 用于统计一个集合中**不重复元素的数量**（基数），无论元素有多少，每个 HyperLogLog 键只需要 12KB 内存，就可以计算接近 2^64 个不同元素的基数。

> 类比：你站在一个广场入口，想知道今天有多少个不同的人来过。你不需要记住每个人的名字，只需要拿一个特殊的计数器，人来一个就按一下，最后计数器显示的数字就是大概的人数（有误差，但在可接受范围内）。

适用场景：
- **网站 UV（独立访客）**：统计一天内有多少不同用户访问过网站
- **点赞去重**：统计一篇文章有多少不同用户点了赞

注意：HyperLogLog 的统计结果是有误差的，标准误判率是 0.81%。如果需要精确统计，应该用 Set。

```bash
PFADD page:uv user1 user2 user3   # 添加用户
PFCOUNT page:uv                    # 统计UV数（约数）
PFMERGE day1:uv day2:uv day3:uv   # 合并多天的UV
```

**③ Geospatial（地理位置）**

用于存储地理位置信息（经纬度），并支持距离计算、范围搜索等操作。底层是 Sorted Set，将经纬度转换为 score 存储。

> 类比：你在地图上标记了所有门店的位置，然后问"我附近 3 公里内有哪些门店？"或者"从我这里到最近的门店有多远？"——Geospatial 就是干这个的。

适用场景：
- **附近的人**：`GEOSEARCH` 查找附近 5 公里内的用户
- **打车距离计算**：计算乘客和司机之间的直线距离
- **门店定位**：查找最近的门店

```bash
GEOADD shops 116.40 39.90 "北京门店"   # 添加门店坐标
GEODIST shops "北京门店" "上海门店" km   # 计算距离（公里）
GEOSEARCH shops FROMLONLAT 116.40 39.90 BYRADIUS 5 km  # 搜索5km内的门店
```

## 48. Redis 持久化数据和缓存怎么做扩容？

Redis 的扩容方案取决于它的使用场景：**当缓存用** 和 **当持久化存储用**，策略完全不同。

**场景一：Redis 当缓存使用（允许数据丢失）**

使用**一致性哈希**实现动态扩容缩容。

一致性哈希将所有节点映射到一个哈希环上，key 通过哈希也落在环上，顺时针找到最近的节点。新增或移除节点时，只影响环上相邻的一小部分数据，不需要大规模数据迁移。

> 类比：一条环形跑道上站了几个人，每个人负责接住附近飞过来的球。新加一个人，只需要把旁边那个人负责的一部分球分给他，其他人的球不受影响。

**场景二：Redis 当持久化存储使用（不允许数据丢失）**

必须使用**固定的 keys-to-nodes 映射关系**，节点数量一旦确定就不能随意变化。因为持久化场景要求数据不丢失，随意增减节点会导致数据重新分布，风险很高。

如果确实需要动态变化节点数量（比如业务增长必须扩容），必须使用可以在运行时进行数据再平衡的系统。当前只有 **Redis Cluster** 能做到——通过 slot 在线迁移，将部分 slot 从旧节点迁移到新节点，过程中服务不中断。

**对比：**

| 场景 | 扩容方式 | 数据要求 |
|------|----------|----------|
| 缓存 | 一致性哈希，动态扩缩容 | 允许丢失，从数据库重新加载 |
| 持久化存储 | 固定节点数 + Redis Cluster slot 迁移 | 不允许丢失，需要数据再平衡 |

## 49. Redis 事务的概念？

Redis 事务不是传统意义上理解的事务。单个 Redis 命令的执行是原子的，但 Redis 没有在事务上增加任何维持原子性的机制，所以 **Redis 事务的执行并不是原子性的**。

事务可以理解为一个**打包的批量执行脚本**，但批量指令并非原子化操作——中间某条指令失败不会回滚前面已执行的指令，也不会造成后续指令不做。

> 相关详细内容见 [[01Redis基本原理面试题#21. 什么是 Redis 事务？原理是什么？|Q21]]、[[01Redis基本原理面试题#22. Redis 事务的注意点有哪些？|Q22]]、[[01Redis基本原理面试题#24. Redis 事务满足 ACID 吗？|Q24]]

**核心要点总结：**

1. 事务中如果某一条命令执行失败，之前的命令不会回滚，后续命令仍然继续执行
2. 事务中的所有命令都会序列化、按顺序执行，期间不会被其他客户端的命令打断
3. 网络断开的影响取决于断开的时机

**网络断开的两种场景：**

| 断开时机 | 结果 |
|----------|------|
| 事务开启前（MULTI 之前）网络断开 | 所有待执行的语句都不会被服务器执行 |
| 客户端执行 EXEC 命令之后网络断开 | 该事务中的所有命令都会被服务器执行 |

**事务的三个阶段：**

1. **MULTI 开启事务**：开启事务队列
2. **命令入队**：大量指令依次入队
3. **EXEC 执行事务**：截止此处一个事务已经结束

**其他事务相关命令：**

| 命令 | 作用 |
|------|------|
| `MULTI` | 开启事务，总是返回 OK，之后的命令入队不执行 |
| `EXEC` | 执行所有事务块内的命令，按先后顺序返回每条命令的结果；被打断时返回 nil |
| `DISCARD` | 清空命令队列，放弃执行事务，客户端退出事务状态 |
| `WATCH key` | 监视一个或多个 key，如果事务执行前 key 被修改，事务将被打断（乐观锁） |
| `UNWATCH` | 取消 watch 对所有 key 的监控 |

**AOF 模式下的崩溃恢复：**

使用 Append-Only 模式时，Redis 会将事务内的所有写操作一次性写入磁盘。但如果写入过程中系统崩溃（如电源故障），可能只有部分数据写入磁盘，另一部分丢失。Redis 重新启动时会执行一致性检测，发现问题后退出并给出错误提示，此时可以用 `redis-check-aof` 工具定位错误并回滚，修复后重新启动。

## 50. Redis 如何做内存优化？

**① 控制 key 的数量**

当使用 Redis 存储大量数据时，过多的 key 会消耗大量内存。Redis 本质是一个数据结构服务器，它提供了 hash、list、set、zset 等结构。使用 Redis 时不要陷入一个误区——大量使用 get/set 这样的 API，把 Redis 当成 Memcached 使用。对于存储相同的数据内容，利用 Redis 的数据结构降低外层键的数量，也可以节省大量内存。

> 类比：与其用 100 个 String 存 100 个用户信息，不如用 1 个 Hash 存，key 数量从 100 降到 1。

**② 缩减键值对象，降低 Redis 内存使用最直接的方式就是缩减 key 和 value 的长度**

- **key 长度**：在完整描述业务的情况下，键值越短越好
  - 好：`u:1001:n`（用户1001的名字）
  - 差：`user:1001:name`

- **value 长度**：值对象缩减比较复杂，常见需求是把业务对象序列化成二进制数组放入 Redis。首先应该在业务上精简业务对象，去掉不必要的属性避免存储无效数据。其次在序列化工具选择上，应该选择更高效的序列化工具来降低字节数组大小

**③ 编码优化**

Redis 对外提供了 string、list、hash、set、zset 等类型，但 Redis 内部针对不同类型存在编码的概念——所谓编码就是具体使用哪种底层数据结构来实现。编码不同将直接影响数据的内存占用和读写效率。

例如 Hash 类型：
- 元素少时用 ziplist（连续内存，紧凑）
- 元素多时用 hashtable（内存开销大但查询快）

可以通过配置阈值控制编码转换，让 Redis 在合适的时候自动选择最优编码。

## 面试回答模板

**Q: Redis 有哪些数据类型？**

A: Redis 有 5 种基本数据类型：String（缓存、计数器）、Hash（存储对象字段）、List（消息队列）、Set（去重、标签）、ZSet（排行榜）。底层编码各有不同，比如 ZSet 用跳表+哈希表实现。

**Q: Redis 为什么快？**

A: 四个原因：①基于内存操作，比磁盘快几个数量级；②专门设计了高效的数据结构（SDS、跳表等）；③单线程避免了锁竞争和上下文切换；④IO 多路复用让单线程能处理大量并发连接。

**Q: RDB 和 AOF 的区别？**

A: RDB 是定期快照，文件小恢复快，但可能丢数据；AOF 记录每条写命令，数据更完整但文件大。生产建议同时开启，Redis 4.0+ 支持混合持久化。

**Q: Redis 事务满足 ACID 吗？**

A: 不完全满足。隔离性满足（单线程），持久性取决于配置，但原子性只是部分满足——语法错误会回滚，运行时错误只跳过出错命令，其他继续执行。
