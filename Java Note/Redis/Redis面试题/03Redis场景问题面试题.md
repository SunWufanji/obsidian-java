# Redis 场景问题面试题

---

## 1. 缓存穿透、缓存击穿、缓存雪崩的区别？

| 问题 | 触发条件 | 影响范围 |
|------|----------|----------|
| 缓存穿透 | 查询不存在的 key，缓存和 DB 都没有 | 每次请求都打到 DB |
| 缓存击穿 | 热点 key 过期瞬间，大量并发请求涌入 | 单个 key 的并发压力打到 DB |
| 缓存雪崩 | 大量 key 同时过期，或 Redis 宕机 | 大规模请求打到 DB |

---

## 2. 缓存穿透如何解决？

**方案一：缓存空值**
- 查询结果为空时，也将 `null` 写入缓存，设置较短 TTL（如 5 分钟）
- 优点：简单；缺点：占用缓存空间，且恶意攻击时无效

**方案二：布隆过滤器（推荐）**
- 请求进入前先查布隆过滤器，判断 key 是否可能存在
- 不存在则直接返回，不查 DB
- 优点：内存占用极小；缺点：有误判率（false positive），不支持删除

```
请求 → 布隆过滤器 → 不存在 → 直接返回
                  → 可能存在 → 查缓存 → 命中 → 返回
                                      → 未命中 → 查 DB → 写缓存 → 返回
```

**方案三：接口层校验**
- 对非法参数（如负数 ID、超长字符串）直接拦截，不进入缓存和 DB

---

## 3. 缓存击穿如何解决？

**方案一：互斥锁（Mutex Lock）**
- 缓存失效时，只允许一个线程去查 DB 并重建缓存，其他线程等待或返回旧值
- 实现：用 `SETNX` 或 Redisson 分布式锁

```java
String value = redis.get(key);
if (value == null) {
    if (redis.setnx(lockKey, "1", 10)) {  // 获取锁
        try {
            value = db.query(key);
            redis.set(key, value, ttl);
        } finally {
            redis.del(lockKey);
        }
    } else {
        Thread.sleep(50);
        return get(key);  // 重试
    }
}
return value;
```

**方案二：逻辑过期（推荐热点 key）**
- 不设置真实 TTL，在 value 中存储过期时间字段
- 读取时判断是否逻辑过期，过期则异步重建缓存，当前请求返回旧值
- 优点：不阻塞请求；缺点：短暂返回旧数据（最终一致）

```java
// value 结构
{
  "data": {...},
  "expireTime": 1700000000
}
```

---

## 4. 缓存雪崩如何解决？

**针对大量 key 同时过期：**
- 设置随机 TTL：`TTL = 基础时间 + random(0, 300s)`，避免集中过期
- 热点数据永不过期，通过逻辑过期异步更新

**针对 Redis 宕机：**
- Redis 高可用：主从 + 哨兵，或 Redis Cluster
- 服务降级：Redis 不可用时，直接查 DB 或返回默认值
- 限流熔断：使用 Sentinel/Hystrix 限制打到 DB 的请求量

**通用方案：**
- 多级缓存：本地缓存（Caffeine）+ Redis，Redis 宕机时本地缓存兜底

---

## 5. 多级缓存如何保证数据一致性？

多级缓存架构：`本地缓存（JVM）→ Redis → DB`

**一致性问题：** 数据更新后，本地缓存和 Redis 可能仍有旧数据。

**解决方案：**

**方案一：短 TTL + 定期失效**
- 本地缓存设置较短 TTL（如 10s），接受短暂不一致
- 适合对一致性要求不高的场景

**方案二：消息通知（推荐）**
- 数据更新时，通过 MQ（如 Kafka/RocketMQ）广播失效消息
- 各服务节点收到消息后，主动删除本地缓存
- 适合集群部署场景

**方案三：Canal + MQ**
- Canal 监听 MySQL binlog，数据变更时发送 MQ 消息
- 消费者删除 Redis 缓存和本地缓存
- 与业务代码解耦，适合大规模系统

---

## 6. 如何保证 Redis 和 MySQL 的数据一致性？

核心问题：更新 DB 和更新/删除缓存这两步操作无法原子执行，存在不一致窗口。

**常见方案对比：**

| 方案 | 操作顺序 | 问题 |
|------|----------|------|
| 先更新缓存，再更新 DB | Cache → DB | DB 更新失败，缓存已是新值，不一致 |
| 先更新 DB，再更新缓存 | DB → Cache | 并发时可能覆盖更新，导致缓存是旧值 |
| 先删除缓存，再更新 DB | Del Cache → DB | 删除后、DB 更新前，其他线程读到旧 DB 值并写入缓存 |
| 先更新 DB，再删除缓存（推荐） | DB → Del Cache | 极小概率不一致（见下分析） |

---

## 7. 为什么推荐"先更新 DB，再删除缓存"？

**分析不一致场景：**
- 线程 A 更新 DB，线程 B 在 A 删除缓存前读到旧缓存值
- 这个窗口极短（A 更新 DB 到删除缓存之间），且 B 读到的是旧缓存，不是错误数据

**对比"先删缓存再更新 DB"的问题：**
1. 线程 A 删除缓存
2. 线程 B 读缓存未命中，查 DB 得到旧值，写入缓存
3. 线程 A 更新 DB
4. 结果：缓存是旧值，DB 是新值，不一致且持续到缓存过期

**延迟双删策略（增强版）：**
```
1. 删除缓存
2. 更新 DB
3. 延迟 500ms 再次删除缓存（消除步骤 2 期间写入的旧缓存）
```

---

## 8. 如何保证删除缓存操作的可靠性？

**问题：** 更新 DB 成功后，删除缓存失败（网络抖动、Redis 宕机），导致缓存长期是旧值。

**方案一：重试机制**

引入消息队列，删除缓存的操作由消费者来做。删除失败的话重新去消息队列拉取相应的操作，超过一定次数没有删除成功就向业务层报错。

> 类比：你发了一条消息对方没收到，自动重发几次，都失败了就告诉你"发送失败"。

**方案二：订阅 binlog（推荐）**

订阅 MySQL 的 binlog 日志，拿到具体要操作的数据，然后再执行缓存删除。

具体做法：可以让删除服务伪装成一个 MySQL 的从节点，向 MySQL 主节点发送 dump 请求。主节点收到请求后，就开始推送 binlog。删除服务解析 binlog 字节流之后，转换为便于读取的结构化数据，再进行删除。

> 类比：你不用自己去通知仓库出库（业务代码删缓存），而是监听仓库的出库记录（binlog），仓库一出货你就自动同步更新货架（删缓存）。

```
MySQL 更新 → binlog → Canal → MQ → 消费者删除 Redis 缓存
```

优势在于与业务代码完全解耦，即使业务代码漏删了缓存，Canal 也能兜底。

**方案三：设置合理 TTL**

即使删除失败，缓存过期后自然失效。作为兜底保障，不能作为主要方案。

## 9. 什么场景需要分布式锁？

单机锁（`synchronized`、`ReentrantLock`）只在单个 JVM 进程内有效，分布式场景下多个服务实例并发操作同一资源时需要分布式锁。

**典型场景：**
- 库存扣减：防止超卖
- 定时任务：防止多个节点重复执行
- 幂等控制：防止重复提交
- 分布式事务：协调多服务操作顺序

---

## 10. 分布式锁有哪些常见实现方案？

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| Redis SETNX | `SET key value NX EX ttl` | 性能高，实现简单 | 主从切换时可能丢锁 |
| Redisson | 基于 Redis，封装完善 | 支持可重入、看门狗续期 | 依赖 Redis |
| ZooKeeper | 临时顺序节点 + Watch | 强一致，不会丢锁 | 性能较低，运维复杂 |
| 数据库 | 唯一索引或 `SELECT FOR UPDATE` | 无额外依赖 | 性能差，不推荐 |

---

## 11. 用 Redis 实现分布式锁有哪些坑？

**坑一：锁没有过期时间**
- 持有锁的进程崩溃，锁永远不释放（死锁）
- 解决：`SET key value NX EX 30`，必须设置过期时间

**坑二：锁被别人释放**
- A 持有锁，锁过期后 B 获取锁，A 执行完后删除了 B 的锁
- 解决：value 存储唯一标识（如 UUID），释放时校验

```lua
-- 原子性校验 + 删除（Lua 脚本）
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

**坑三：锁过期但业务未完成**
- 业务执行时间超过锁的 TTL，锁自动释放，其他线程进入
- 解决：Redisson 看门狗机制，自动续期

**坑四：不可重入**
- 同一线程多次获取同一把锁时死锁
- 解决：Redisson 支持可重入锁（记录线程 ID 和重入次数）

## 12. Redisson 的看门狗机制是什么？

Redisson 获取锁时，若未指定 TTL（`leaseTime = -1`），会启动一个后台线程（看门狗）：
- 每隔 `lockWatchdogTimeout / 3`（默认 10s）检查锁是否仍被持有
- 若持有，则将锁的 TTL 重置为 `lockWatchdogTimeout`（默认 30s）
- 业务完成后，主动释放锁，看门狗停止

```java
RLock lock = redisson.getLock("myLock");
lock.lock();  // 不指定 TTL，看门狗自动续期
try {
    // 业务逻辑
} finally {
    lock.unlock();
}
```

**注意：** 若指定了 `leaseTime`，看门狗不会启动，锁到期自动释放。

## 13. RedLock 算法是什么？（红锁）

RedLock 是 Redis 官方提出的**多节点分布式锁**方案，解决单 Redis 节点主从切换时丢锁的问题。

**算法流程（N 个独立 Redis 节点，通常 N=5）：**
1. 记录当前时间 `t1`
2. 依次向 N 个节点请求加锁（相同 key、相同 value、相同 TTL）
3. 若在 `TTL` 时间内，超过 `N/2 + 1`（多数）节点加锁成功
4. 计算实际耗时 `t2 - t1`，若 `TTL - (t2-t1) > 0`，则加锁成功
5. 锁的有效时间 = `TTL - (t2-t1)`
6. 加锁失败时，向所有节点发送释放锁请求

**争议：** Martin Kleppmann 认为 RedLock 在时钟漂移场景下仍不安全，Antirez 对此有回应。实际生产中，大多数场景用单节点 Redisson 已足够。

## 14. Redis 分布式锁和 ZooKeeper 分布式锁如何选择？

| 对比项 | Redis | ZooKeeper |
|--------|-------|-----------|
| 性能 | 高（内存操作） | 较低（磁盘 + 网络） |
| 一致性 | AP（主从异步复制，可能丢锁） | CP（ZAB 协议，强一致） |
| 锁释放 | 依赖 TTL 或主动释放 | 客户端断开连接，临时节点自动删除 |
| 实现复杂度 | 简单 | 较复杂 |
| 适用场景 | 高并发、对一致性要求不极端 | 对一致性要求极高（如金融） |

**结论：** 大多数互联网场景选 Redis（Redisson），对强一致性有要求时选 ZooKeeper。

## 15. 什么是大 key？有什么危害？

**定义：**
- String 类型：value 超过 10KB（严格）或 1MB（宽松）
- 集合类型（Hash/List/Set/ZSet）：元素数量超过 5000 个，或总大小超过 10MB

**危害：**
- **内存不均衡**：集群中某个节点内存占用远高于其他节点
- **操作阻塞**：`DEL` 大 key 会阻塞 Redis 主线程（单线程）
- **网络带宽占用**：读取大 key 时传输数据量大
- **序列化/反序列化耗时**：客户端处理大 value 耗时长

---

## 16. 如何发现和处理大 key？

**发现大 key：**
```bash
# 扫描大 key（不阻塞，推荐）
redis-cli --bigkeys

# 或使用 SCAN + OBJECT ENCODING/STRLEN
redis-cli -h host -p port --bigkeys -i 0.1  # -i 控制扫描间隔，避免影响性能
```

**处理大 key：**

**方案一：拆分**
- Hash 大 key：按 field 哈希取模，拆分为多个小 Hash
  - `user:1:info` → `user:1:info:0`、`user:1:info:1`...
- List/Set 大 key：分页存储，`list:key:0`、`list:key:1`...

**方案二：压缩**
- 对 value 进行 gzip/snappy 压缩后存储，读取时解压

**方案三：异步删除**
- 删除大 key 时使用 `UNLINK`（异步删除，v4.0+），避免阻塞主线程
```bash
UNLINK bigkey  # 异步删除，不阻塞
DEL bigkey     # 同步删除，会阻塞
```

## 17. 什么是热 key？如何解决？

**定义：** 某个 key 的访问频率远高于其他 key，导致该 key 所在的 Redis 节点成为瓶颈。

**典型场景：** 热门商品详情、明星微博、秒杀商品库存

**危害：** 单节点 CPU/带宽打满，影响整个集群

**解决方案：**

**方案一：本地缓存（最有效）**
- 在应用层（JVM）缓存热 key，减少 Redis 请求
- 使用 Caffeine 等本地缓存框架，设置较短 TTL

**方案二：热 key 复制**
- 将热 key 复制到多个 Redis 节点，读取时随机选择节点
- `hotkey:1`、`hotkey:2`、`hotkey:3`，读时随机选

**方案三：京东 Hot Key 框架**
- 客户端上报 key 访问频率到 etcd
- 探测到热 key 后，自动推送到各应用节点的本地缓存
- 无需修改业务代码，自动化热 key 处理

**方案四：读写分离**
- 增加 Redis 从节点，热 key 读请求分散到多个从节点

## 18. 如何用 Redis 设计秒杀系统？

**秒杀核心挑战：**
- 瞬时高并发（QPS 可达百万级）
- 库存不能超卖
- 用户不能重复购买
- 系统不能崩溃

**整体架构：**
```
用户请求 → Nginx 限流 → 网关层 → 秒杀服务 → Redis → MQ → 订单服务 → DB
```

**Redis 在秒杀中的作用：**

**1. 库存预热**
- 秒杀开始前，将库存数量写入 Redis：`SET stock:goodsId 100`
- 所有库存操作在 Redis 中进行，避免直接打 DB

**2. 原子扣减库存（Lua 脚本）**
```lua
-- 原子性判断库存并扣减
local stock = tonumber(redis.call("get", KEYS[1]))
if stock <= 0 then
    return -1  -- 库存不足
end
redis.call("decr", KEYS[1])
return 1  -- 扣减成功
```

**3. 防止重复购买**
- 用 Set 记录已购买用户：`SADD bought:goodsId userId`
- 购买前先 `SISMEMBER bought:goodsId userId` 判断

**4. 异步下单**
- 扣减 Redis 库存成功后，将订单信息发送到 MQ
- 订单服务异步消费 MQ，写入 DB
- 避免秒杀流量直接打到 DB

---

## 19. 秒杀系统如何防止超卖？

**方案一：Redis 原子操作**
- 使用 `DECR` 命令原子扣减，判断返回值是否 >= 0
- 若 < 0，则 `INCR` 回滚并返回失败

**方案二：Lua 脚本（推荐）**
- 将"判断库存 + 扣减"封装为 Lua 脚本，保证原子性（见 Q18）

**方案三：分布式锁**
- 对每个商品加分布式锁，串行处理库存扣减
- 性能较低，适合库存量少、并发不极端的场景

**方案四：DB 乐观锁兜底**
- 即使 Redis 层有少量超卖，DB 层用版本号或 `WHERE stock > 0` 兜底
- 双重保障

---

## 20. 秒杀系统还需要注意哪些问题？

**限流：**
- 网关层：令牌桶/漏桶限流，限制每秒请求数
- 用户层：同一用户 1 秒内只允许 1 次请求（Redis 计数）
- 随机丢弃：库存售完后，直接在 Redis 层返回失败，不进入后续流程

**防刷：**
- 验证码：秒杀前输入验证码，打散请求时间
- 黑名单：对异常 IP/用户加入黑名单

**页面静态化：**
- 秒杀页面静态化，部署到 CDN，减少服务器压力

**库存预热时机：**
- 提前 5-10 分钟将库存写入 Redis，避免秒杀开始瞬间大量 DB 查询

## 21. Redis 使用有哪些最佳实践？

**key 命名规范：**
- 使用 `:` 分隔层级：`业务:模块:id`，如 `user:info:1001`
- 避免过长 key（影响内存和性能），建议不超过 44 字节（embstr 编码上限）
- 避免特殊字符，使用小写字母和数字

**value 设计：**
- 避免大 value，超过 10KB 考虑拆分或压缩
- 集合类型元素不超过 5000 个
- 优先使用 Hash 存储对象，而非 JSON 字符串（支持单字段更新）

**过期时间：**
- 所有缓存 key 必须设置 TTL，避免内存无限增长
- 避免大量 key 在同一时刻过期（加随机偏移）

**命令使用：**
- 禁止使用 `KEYS *`（全量扫描，阻塞主线程），改用 `SCAN`
- 禁止使用 `FLUSHALL`/`FLUSHDB`（清空数据库）
- 大集合删除用 `UNLINK` 替代 `DEL`
- 批量操作用 Pipeline 减少网络往返

**连接管理：**
- 使用连接池（JedisPool / Lettuce），避免频繁创建连接
- 设置合理的连接超时和读写超时

## 22. Redis 在生产环境中如何监控？

**关键监控指标：**

| 指标 | 命令 | 说明 |
|------|------|------|
| 内存使用 | `INFO memory` | `used_memory`、`mem_fragmentation_ratio` |
| 命中率 | `INFO stats` | `keyspace_hits / (hits + misses)` |
| 连接数 | `INFO clients` | `connected_clients` |
| 慢查询 | `SLOWLOG GET` | 默认记录超过 10ms 的命令 |
| 持久化状态 | `INFO persistence` | RDB/AOF 最后执行时间 |
| 主从延迟 | `INFO replication` | `master_repl_offset - slave_repl_offset` |

**慢查询配置：**
```bash
CONFIG SET slowlog-log-slower-than 10000  # 10ms
CONFIG SET slowlog-max-len 128
SLOWLOG GET 10  # 查看最近 10 条慢查询
```

**内存碎片率（mem_fragmentation_ratio）：**
- `> 1.5`：碎片率过高，考虑重启或执行 `MEMORY PURGE`（v4.0+）
- `< 1`：内存不足，Redis 使用了 swap，性能急剧下降

## 23. 缓存雪崩、击穿、穿透的详细解决方案？

**缓存雪崩的解决方案：**

针对大量 key 同时过期：
- **均匀设置过期时间**：给 TTL 加随机偏移量，避免同一时间集中失效
- **互斥锁**：缓存失效时加锁，保证同一时间只有一个请求去重建缓存，其他请求等待或返回空值
- **双 key 策略**：同时维护主 key（有过期时间）和备 key（永不过期），value 相同。主 key 失效时直接读备 key，更新时两个 key 同时更新
- **后台更新缓存**：缓存设为永不过期，由后台线程定时更新缓存数据，业务线程不负责更新

针对 Redis 故障宕机：
- **服务熔断/限流**：Redis 不可用时暂停对缓存服务的访问，直接返回错误或降级处理，保护数据库不被打垮。也可以启用限流，只放少量请求到数据库
- **构建高可靠集群**：通过主从 + 哨兵或 Redis Cluster 搭建集群，主节点故障时从节点自动切换，避免单点故障导致雪崩

**缓存击穿的解决方案：**
- **互斥锁**：同一时间只允许一个线程更新缓存，其他线程要么等待锁释放后重试读缓存，要么直接返回空值
- **不给热点数据设置过期时间**：由后台异步更新缓存，或者在热点数据即将过期前，提前通知后台线程更新缓存并重新设置过期时间

**缓存穿透的解决方案：**
- **非法请求拦截**：在 API 入口校验请求参数是否合理，是否含有非法值、请求字段是否存在，恶意请求直接返回错误，不进入缓存和数据库
- **缓存空值或默认值**：查询结果为空时，在缓存中设置一个空值或默认值，后续请求命中缓存后直接返回，不再查数据库
- **布隆过滤器**：写入数据库数据时，在布隆过滤器中做个标记。用户请求到来时，先通过布隆过滤器快速判断数据是否存在，不存在就不用查数据库

## 24. 布隆过滤器是怎么工作的？

布隆过滤器由一个初始值全为 0 的**位图数组**和 **N 个哈希函数**两部分组成。

**工作流程：**

写入数据时：
1. 用 N 个哈希函数分别对数据做哈希计算，得到 N 个哈希值
2. 将每个哈希值对位图数组长度取模，得到对应位置
3. 将这些位置的值设为 1

查询数据时：
- 用同样的 N 个哈希函数计算，检查对应位置是否都为 1
- 如果有任意一个位置为 0，说明数据**一定不存在**
- 如果全部为 1，说明数据**可能存在**（存在误判）

> 类比：想象一排格子（位图），你用几把不同的尺子（哈希函数）量一个东西，量完在对应的格子里画勾。下次查的时候再量一遍，如果所有格子都有勾，说明这个东西"可能存在"——因为别的东西也可能在这些格子里画了勾。

**缺陷：**

- **存在误判**：由于哈希冲突，不同的数据可能落在相同位置，导致查询说"存在"不一定真的存在。但查询说"不存在"就一定不存在
- **不支持删除**：删除一个 key 会影响其他 key（因为位可能被多个 key 共享）。改进方法是用 Counting Bloom Filter，用计数器数组代替位数组，支持删除
- **参数选择**：对于 n 个元素，数组大小 m 和哈希函数个数 k 的最优关系是 `k = (ln2) * (m/n)`，错误率最小时 `m >= n * lg(1/E)`

## 25. 如何保证数据库和缓存的一致性？

保证数据库和缓存的一致性有几种经典模式：

**① Cache Aside（旁路缓存，最常用）**

读写流程：
- **读**：先查缓存，命中直接返回；未命中查数据库，然后把数据写入缓存
- **写**：先更新数据库，再让缓存失效（删除缓存）

**存在的问题：** 假设有两个操作并发执行——一个读一个写。写操作先更新了数据库，但还没来得及删缓存，此时读操作可能读到旧缓存值。不过这个窗口极短（数据库读写比缓存慢得多），概率很低。还有一种更极端的情况：读操作缓存未命中去查数据库，此时写操作更新了数据库并删了缓存，但读操作还在用旧数据写回缓存，造成脏数据。概率同样很低，需要同时满足"读缓存未命中 + 并发写"的条件。

**② Read/Write Through（读写穿透）**

核心思想是把缓存作为数据库的代理，应用只跟缓存打交道：

- **Read Through**：缓存失效时，由缓存服务自己去数据库加载数据（对应用透明），而不是像 Cache Aside 那样由调用方负责加载
- **Write Through**：写数据时，先更新缓存，再由缓存同步更新数据库（同步操作）。如果缓存未命中，直接更新数据库

**③ Write Behind（异步写回）**

写数据时只更新缓存，不立即更新数据库。缓存会在后台异步、批量地把变更同步到数据库。

> 类比：你记笔记先写在便利贴上（缓存），等攒多了再统一抄到笔记本上（数据库）。

- 优点：写入速度极快，I/O 操作少
- 缺点：数据不是强一致的，缓存宕机可能丢失未同步的数据

**④ 2PC / Paxos**

用分布式事务协议保证一致性。2PC 太慢，Paxos 太复杂，实际项目中很少用。

**选型建议：**

| 模式 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|----------|
| Cache Aside | 最终一致 | 高 | 低 | 大多数互联网场景（推荐） |
| Read/Write Through | 强一致 | 中 | 中 | 缓存中间件支持时 |
| Write Behind | 弱一致 | 最高 | 高 | 写多读多、可容忍短暂不一致 |

---

## 26. 业务一致性要求高怎么办？

当业务对数据一致性要求很高时（如金融、支付场景），单纯靠"先更新 DB 再删缓存"可能不够，需要更严格的方案：

**① 延迟双删**

先删缓存，再更新 DB，延迟一段时间后再删一次缓存。第二次删除的目的是清理掉"更新 DB 期间"其他线程写入的脏缓存。

```
1. 删除缓存
2. 更新 DB
3. 延迟 500ms 再次删除缓存
```

**② Canal + MQ 兜底**

即使业务代码删缓存失败了，Canal 监听 binlog 也能自动触发删除。相当于给删缓存操作加了一道保险。

**③ 读请求强制走 DB**

对一致性要求极高的读操作，不走缓存，直接查数据库。牺牲性能换一致性。

**④ 使用 Read/Write Through 模式**

让缓存中间件负责同步更新数据库，应用只跟缓存交互。写操作是同步的，保证缓存和 DB 实时一致。但需要缓存中间件支持（如 Redis + MySQL 的同步插件）。

**⑤ 分布式事务（强一致）**

用 2PC、TCC 或 Saga 等分布式事务方案，保证更新 DB 和删除缓存是原子操作。性能代价大，适合核心交易链路。

**选型建议：**

| 方案 | 一致性 | 性能 | 适用场景 |
|------|--------|------|----------|
| 延迟双删 | 较高 | 高 | 大多数场景 |
| Canal + MQ | 较高 | 高 | 需要兜底保障 |
| 强制走 DB | 强 | 低 | 少量核心读 |
| Read/Write Through | 强 | 中 | 缓存中间件支持 |
| 分布式事务 | 最强 | 最低 | 金融支付等核心链路 |

**核心原则：** 一致性要求越高，性能代价越大。没有银弹，根据业务场景选择合适的方案。

## 27. 如何避免缓存失效？

缓存失效意味着大量请求打到数据库，需要从多个角度预防：

**① 合理设置过期时间**
- 避免大量 key 设置相同的 TTL，加随机偏移量打散过期时间
- 热点数据可以设置较长的 TTL，甚至永不过期（逻辑过期兜底）

**② 缓存预热**
- 系统启动或大促前，提前把热点数据加载到缓存
- 避免冷启动时大量请求直接打到数据库

**③ 多级缓存**
- 本地缓存（Caffeine）+ Redis + DB
- 即使 Redis 失效，本地缓存还能兜住一部分请求

**④ 熔断降级**
- 缓存层故障时，服务降级返回默认值或静态数据
- 用限流控制打到数据库的请求量，保护数据库不被打垮

**⑤ 缓存永不过期 + 异步更新**
- 缓存不设 TTL，由后台线程定时更新数据
- 永远有缓存可用，不存在"失效"的问题
- 代价是数据有延迟，适合对实时性要求不高的场景

具体实现有两种方式：
- **后台线程主动检测**：后台线程频繁检测缓存是否有效，检测到失效了马上从数据库读取数据并更新到缓存
- **业务线程通知更新**：业务线程发现缓存失效后，通过 MQ 发送消息通知后台线程更新缓存。后台线程收到消息后，在更新前判断缓存是否存在，存在就不执行更新；不存在就读取数据库数据并加载到缓存

**⑥ 缓存预热**
- 业务刚上线的时候，最好提前把数据缓存起来，而不是等待用户访问才来触发缓存构建
- 后台更新缓存的机制刚好也适合干这个事情，上线时主动触发一次全量缓存更新

**⑦ 使用互斥锁重建缓存**
- 缓存失效时只允许一个线程去查 DB 并重建缓存
- 其他线程等待或返回旧值，避免大量请求同时击穿到 DB

## 28. 如何实现延迟队列？

**什么是延迟队列？**

延迟队列是指把当前要做的事情，往后推迟一段时间再做：

- 淘宝、京东等购物平台下单，超过一定时间未付款，订单会自动取消
- 打车的时候，在规定时间内没有车主接单，平台会取消你的单
- 点外卖的时候，如果商家在 10 分钟还没接单，就会自动取消订单

**用 ZSet 实现延迟队列：**

ZSet 有一个 Score 属性可以用来存储延迟执行的时间。用 `ZADD score1 value1` 命令存入任务，再利用 `ZRANGEBYSCORE` 查询符合条件的所有待处理的任务，通过循环执行队列任务。

```
1. 生产者：ZADD delay_queue <执行时间戳> <任务内容>
2. 消费者：ZRANGEBYSCORE delay_queue 0 <当前时间戳>  （查到期任务）
3. 消费者：ZREM delay_queue <任务内容>  （处理完后删除）
```

> 类比：你在便利贴上写好任务和"几点钟做"，贴到墙上。每隔一会儿看一眼墙上的便利贴，把到期的撕下来执行。

**注意事项：**
- 消费者需要轮询 ZSet，检查是否有到期任务
- 可以用 `ZRANGEBYSCORE ... LIMIT 0 1` 每次只取一个，避免一次取太多
- 如果任务一直没被消费，需要有兜底机制（如告警或人工处理）

## 29. 如何设计一个缓存策略，动态缓存热点数据？

**核心思路：** 通过数据最新访问时间来做排名，过滤掉不常访问的数据，只留下经常访问的数据。用 ZSet 的排序能力来实现。

**以电商 Top 1000 商品为例：**

**① 维护排序队列**

在 Redis 中用 ZSet 维护一个"热门商品队列"，最多存 1000 个商品。每次有用户访问某个商品，就更新它在队列中的排名：

```
ZADD hot_goods <当前时间戳> <商品ID>
```

时间戳越大排名越靠前，越近访问的商品排越前面。

**② 定期淘汰冷门数据**

系统定期过滤掉队列中排名最后的 200 个商品（最久没被访问的），然后从数据库中随机读取 200 个新商品加入队列，保持队列始终有 1000 个商品。

**③ 请求到达时的处理流程**

用户请求到来时，先从队列中获取商品 ID：
- 命中队列 → 说明是热门商品 → 从另一个缓存结构中读取实际商品信息并返回
- 未命中队列 → 说明不是热门商品 → 走普通查询流程

> 类比：你开了一家超市，货架只有 1000 个位置。你每天统计哪些商品卖得最好，卖得差的下架，换新商品上去。顾客来了直接看货架（热门队列），有就拿，没有就去仓库找（查数据库）。

**为什么用 ZSet？**

ZSet 天然支持按 score 排序，每次访问更新时间戳就能自动调整排名，不用自己写排序逻辑。`ZADD` 更新 score 的时间复杂度是 O(log N)，性能很好。

## 30. Redis 如何实现分布式锁？

使用 SETNX 命令，只有当 key 不存在时才插入，如果 key 已存在则插入失败。key 插入成功代表加锁成功，否则加锁失败。

```
SET lock_key unique_value NX PX 10000
```

- `NX`：只有 key 不存在时才设置，保证互斥
- `PX 10000`：设置过期时间 10 秒，防止死锁
- `unique_value`：加锁客户端的唯一标识（如 UUID），解锁时用来校验身份

**解锁的过程：** 将 key 删除。但要保证执行解锁操作的客户端就是加锁的客户端，所以解锁时要先判断锁的 `unique_value` 是否为当前客户端，是才将 `lock_key` 删除。

**为什么要设置过期时间？** 以免客户端拿到锁后发生异常（如宕机），导致锁一直无法释放。

## 31. 如何保证加锁和解锁过程的原子性？

使用 **Lua 脚本**。因为 Redis 在执行 Lua 脚本时，可以以原子性的方式执行，保证了锁释放操作的原子性。

解锁的 Lua 脚本：

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

先校验 value 是否是自己加的锁，是才删除。两步操作在一个 Lua 脚本里原子执行，不会被其他线程插入。

## 32. 使用 Redis 实现分布式锁的优点和缺点？

**优点：**
- 性能高效（内存操作）
- 实现方便
- 避免单点故障（Redis 集群）

**缺点：**

**① 超时时间不好设置**
- 设置过长：锁被持有期间如果业务异常，其他线程要等很久
- 设置过短：业务还没执行完锁就过期了，其他线程进入导致并发问题

**② 主从切换时可能丢锁**
- Redis 主从模式下数据是异步复制的
- 如果在主节点获取到锁后，还没同步到其他节点时主节点宕机了
- 新的主节点依然可以获取锁，导致多个应用同时持有锁

## 33. 如何为分布式锁设置合理的超时时间？

**方案一：基于续约的方式**

先给锁设置一个超时时间，然后启动一个守护线程（看门狗），让守护线程在一段时间后重新设置这个锁的超时时间。

具体做法：写一个守护线程，定期检查锁的情况，当锁快失效的时候，再次进行续约加锁。当主线程执行完成后，销毁续约锁即可。

> 这就是 Redisson 看门狗机制的原理。不过这种方式实现起来相对复杂，建议直接用 Redisson。

**方案二：评估业务执行时间**

根据历史数据预估业务最大执行时间，锁的超时时间设为业务时间的 2-3 倍。比如业务通常 5 秒完成，锁可以设 15 秒。

**方案三：指定 leaseTime**

Redisson 支持指定 `leaseTime`，锁到期后自动释放，不会续期。适合业务执行时间可控的场景。

```java
RLock lock = redisson.getLock("myLock");
lock.lock(30, TimeUnit.SECONDS);  // 指定30秒后自动释放
```

## 34. Redis 如何解决集群环境下分布式锁的可靠性？

单节点 Redis 实现分布式锁有一个严重问题：主节点宕机后，如果数据还没同步到从节点，新主节点上锁就丢了，两个客户端可能同时持有同一把锁。

RedLock（红锁）就是为了解决这个问题而提出的方案。

**前提条件：** 准备 N 个独立的 Redis 实例（无主从关系），通常 N = 5。

**加锁流程（两步判断）：**

**第一步：** 记录当前时间 `t1`，然后依次向 N 个 Redis 实例发送加锁请求（相同的 key、相同的 value、相同的过期时间）。

**第二步：** 记录加锁完成的时间 `t2`，此时需要同时满足两个条件才算加锁成功：

| 条件 | 说明 |
|------|------|
| 多数节点加锁成功 | 成功数量 > N/2（如5个节点需要至少3个成功） |
| 锁的实际有效期 > 0 | `TTL - (t2 - t1) > 0`，即加锁耗时没有超过 TTL |

**锁的有效时间：** `TTL - (t2 - t1)`。比如 TTL 设了 10 秒，加锁过程花了 2 秒，那锁实际只能用 8 秒。

**为什么要同时满足两个条件？**
- 只有多数节点都加锁成功，才能保证即使一个节点宕机，锁的状态在其他节点上依然有效
- 如果加锁耗时超过了 TTL，锁可能在还没返回结果时就已经过期了，这样的锁没有意义

**加锁失败的处理：** 向所有 N 个节点发送释放锁的 Lua 脚本（原子操作），确保不会残留锁。

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
end
```

**为什么推荐用独立节点而不是集群？**
- Redis Cluster 的数据是按 slot 分片的，不同节点存不同数据
- 同一个 key 只会落在一个节点上，无法实现"多节点同时加锁"来提高可靠性
- 独立节点之间没有主从同步延迟，每个节点都是独立决策

## 35. Redis 管道（Pipeline）有什么用？

**什么是管道？**

正常情况下，客户端发一个 Redis 命令，要等服务器返回结果后才能发下一个。每次请求都要经历一次完整的网络往返（RTT），10 条命令就要跑 10 趟网络，效率很低。

管道技术就是把多条命令"打包"一次性发给服务器，服务器处理完后统一返回结果。这样 10 条命令只跑一趟网络，大大减少了网络开销。

> 类比：你去超市买东西，正常是买一件结一次账（普通模式）；管道就是把要买的东西全放进购物车，最后一起结账（管道模式）。省的是排队结账的次数，不是东西本身。

**使用场景：**
- 批量写入大量数据（如初始化缓存）
- 批量读取多个 key（如批量查用户信息）
- 减少高频操作的网络延迟

**注意事项：**
- 管道内命令不要太多，数据量过大可能导致网络阻塞
- 管道是**客户端功能**，不是 Redis 服务端的功能
- 管道内的命令没有原子性保证，如果中间有其他命令插入，可能影响结果

**Pipeline vs 批量命令：**
- `MSET`、`MGET` 等批量命令是 Redis 服务端提供的，一次执行多条操作
- Pipeline 是客户端把多条独立命令攒起来一起发，服务端逐条执行后统一返回
- Pipeline 更灵活，可以混合不同类型的命令

## 36. Redis 如何处理大 key？

**什么是大 key？**

- String 类型：value 大于 10KB
- 集合类型（Hash/List/Set/ZSet）：元素数量超过 5000 个

**大 key 会带来什么问题？**

| 问题 | 原因 |
|------|------|
| 客户端超时阻塞 | Redis 单线程处理，操作大 key 耗时长，客户端迟迟等不到响应 |
| 网络阻塞 | 读取大 key 时传输数据量大，占用大量网络带宽 |
| 阻塞工作线程 | 用 `DEL` 删除大 key 时会阻塞主线程，期间无法处理其他任何命令 |
| 内存分布不均 | 集群中某个节点存了大 key，内存占用远高于其他节点，导致负载不均衡 |

**如何处理？**

**情况一：String 类型的大 value**

先尝试压缩，用序列化+压缩算法（如 gzip、snappy）把 value 缩小到合理范围。序列化和反序列化会增加一些时间开销，但相比大 key 带来的阻塞要好得多。

如果压缩后仍然很大，就需要拆分——把一个大 key 拆成多个小 key，用 mget 等批量操作来读取。

**情况二：Hash 类型的大 key**

拆分为多个小 Hash，每个 field 代表一个具体的属性，用 `hget`/`hmget` 获取部分数据，`hset`/`hmset` 更新部分属性。

```
大key: user:1:info     → 拆分为:
       user:1:info:0    （基础信息）
       user:1:info:1    （订单信息）
       user:1:info:2    （行为数据）
```

**情况三：List/Set 等集合类型的大 key**

根据预估的数据规模进行分片，不同的元素按规则分配到不同的分片中。

```
大key: cart:user:1001   → 拆分为:
       cart:user:1001:0  （前1000个元素）
       cart:user:1001:1  （1001-2000个元素）
```

**删除大 key 的注意事项：**
- 千万不要用 `DEL`，会阻塞主线程
- 使用 `UNLINK`（v4.0+），异步删除，不阻塞

## 37. Redis 支持事务回滚吗？

**不支持。**

Redis 提供的 `DISCARD` 命令只能用来**主动放弃**事务执行——清空暂存的命令队列，让之前 `MULTI` 之后的命令全部不执行。但这不是回滚，而是"还没执行就取消了"。

**为什么 Redis 不支持回滚？**

官方的理由是：Redis 事务中出现的错误基本都是编程错误（比如对 String 类型执行 `HGETALL`），这类错误在开发阶段就能发现，很少会在生产环境出现。所以官方认为没有必要为 Redis 开发事务回滚功能。

**两种错误的区别：**

| 错误类型 | 是否支持回滚 | 举例 |
|----------|------------|------|
| 命令语法错误 | 整个事务取消（DISCARD） | 命令写错，如 `HGET` 写成 `HGETT` |
| 运行时类型错误 | 不回滚，继续执行后续命令 | 对 String 执行 `HGETALL`，该条报错但后续命令照常执行 |

> 类比：你写了一张购物清单（MULTI），去超市买了前两样东西，第三样发现货架空了。Redis 不会把前两样退回去，而是跳过第三样继续买第四样。MySQL 则会把三样全退回去（回滚）。

**实际生产中怎么办？**
- 用 **Lua 脚本**替代事务，Lua 脚本在 Redis 中是原子执行的，比事务更可靠
- 如果需要真正的事务语义，考虑用 MySQL 而不是 Redis

**缓存穿透：** 布隆过滤器 + 缓存空值

**缓存击穿：** 互斥锁 + 逻辑过期

**缓存雪崩：** 随机TTL + 高可用 + 限流

**数据一致性：** 先更新DB，再删除缓存（Canal兜底）

**分布式锁：** Redis SETNX + Lua脚本释放（Redisson 看门狗续期）

**秒杀：** Redis库存预热 + Lua原子扣减 + MQ异步下单

**大key：** UNLINK异步删除 + 拆分/压缩

**热key：** 本地缓存 + 复制 + Hot Key框架

## 38. 如果有个读超高并发的系统，用 Redis 来抗住大部分读请求，你会怎么设计？

**第一步：评估读并发的数量级**

Redis 单机的读 QPS 在万级（每秒几万没问题），先评估系统的读并发量是多少。

**方案一：一主多从 + 哨兵集群（读 QPS 10 万级）**

如果读并发在每秒 10 万级左右，使用一主多从 + 哨兵集群的缓存架构：
- 主库负责写
- 多个从库负责读，支持读写分离
- 哨兵集群保证高可用，解决单点故障问题

根据读请求的 QPS 来决定加多少个 Redis 从实例。如果读并发继续增加，只需要增加 Redis 从实例就行了。

**方案二：Redis Cluster（海量数据 + 高并发）**

如果需要缓存 1T+ 的数据，单机内存不够用，选择 Redis Cluster 模式：
- 每个主节点存一部分数据
- 假设一个 master 存 32G，那只需要 n * 32G >= 1T，n 个 master 节点就可以支持 1T+ 的海量数据

**Redis 单主的瓶颈：**

Redis 单主的瓶颈不在于读写的并发，而在于**内存容量**。即使是一主多从也不能解决这个问题，因为一主多从架构下，多个 slave 的数据和 master 的完全一样。假如 master 是 10G，那 slave 也只能存 10G 数据，所以数据量受单主的影响。

当需要缓存海量数据时，就必须得有多主了，并且多个主保存的数据还不能一样。Redis Cluster 模式完美地解决了这个问题。

**架构选型：**

```
读 QPS < 10万 → 一主多从 + 哨兵
数据量 > 单机内存 → Redis Cluster
读 QPS 极高 + 数据量大 → Redis Cluster + 本地缓存（多级缓存）
```
