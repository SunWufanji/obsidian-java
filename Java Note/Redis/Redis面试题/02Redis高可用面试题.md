# Redis 高可用面试题

---

## 1. Redis 主从复制的核心流程是什么？

主从复制分为**全量同步**和**增量同步**两个阶段：

**全量同步（首次连接）**
1. slave 向 master 发送 `PSYNC` 命令，携带 `replid` 和 `offset`
2. master 判断是否为第一次同步（replid 不匹配）
3. master 执行 `BGSAVE` 生成 RDB 快照，同时将期间的写命令记录到 `repl_backlog`
4. master 将 RDB 文件发送给 slave，slave 清空本地数据并加载 RDB
5. master 将 `repl_backlog` 中的增量命令发送给 slave

**增量同步（断线重连）**
1. slave 重连后发送 `PSYNC`，携带自己的 `replid` 和 `offset`
2. master 判断 replid 匹配，且 offset 在 `repl_backlog` 范围内
3. master 只发送 offset 之后的增量命令
4. 若 offset 已超出 `repl_backlog` 范围，则退化为全量同步

**关键概念：**
- `replid`：master 的复制 ID，slave 连接后会同步该 ID
- `repl_backlog`：环形缓冲区，默认 1MB，存储最近的写命令
- `offset`：复制偏移量，标记同步进度

---

## 2. 主从复制有什么作用？

- **读写分离**：master 负责写，slave 负责读，提升吞吐量
- **数据冗余**：slave 保存数据副本，防止单点故障数据丢失
- **故障恢复基础**：哨兵和集群模式都依赖主从复制

---

## 3. 主从复制的缺点是什么？

- master 宕机后需要手动切换，无法自动故障转移
- 复制延迟：slave 数据可能落后于 master（异步复制）
- `repl_backlog` 有限，断线时间过长会触发全量同步，开销大

---

## 4. 了解哨兵机制吗？

Redis Sentinel 是 Redis 官方提供的**高可用方案**，解决主从复制无法自动故障转移的问题。

**核心功能：**
- **监控**：持续检测 master 和 slave 是否正常运行
- **通知**：节点故障时通知管理员或其他应用
- **自动故障转移**：master 宕机时自动选举新 master，并通知 slave 切换
- **配置提供者**：客户端通过 Sentinel 获取当前 master 地址

**部署：** 通常部署奇数个 Sentinel（至少 3 个），避免脑裂，保证投票多数。

---

## 5. 如何判断主节点故障了？主观下线和客观下线的区别？

**主观下线（Subjective Down，SDOWN）**
- 单个 Sentinel 向 master 发送 `PING`，超过 `down-after-milliseconds`（默认 30s）未响应
- 该 Sentinel 单方面认为 master 下线，标记为主观下线
- 可能是网络抖动导致的误判

**客观下线（Objective Down，ODOWN）**
- 主观下线的 Sentinel 向其他 Sentinel 发起投票询问
- 当认为 master 下线的 Sentinel 数量达到 `quorum`（法定人数）时
- 才确认为客观下线，触发故障转移

**流程：**
```
单个Sentinel PING超时 → 主观下线 → 询问其他Sentinel → 
达到quorum → 客观下线 → 选举Leader Sentinel → 执行故障转移
```

---

## 6. 哨兵如何选举新的 master？

故障转移时，Leader Sentinel 从所有 slave 中选出新 master，选举规则（优先级从高到低）：

1. **slave-priority**：slave 优先级，值越小越优先（0 表示永不成为 master）
2. **复制偏移量 offset**：offset 越大说明数据越新，优先选
3. **Run ID**：字典序最小的 slave（兜底规则）

选出新 master 后：
1. 向新 master 发送 `SLAVEOF NO ONE`，使其成为独立 master
2. 向其他 slave 发送 `SLAVEOF <new_master>`，切换复制目标
3. 将旧 master 加入配置，等其恢复后作为新 master 的 slave

---

## 7. 哨兵模式有什么缺点？

- **写操作仍是单点**：只有一个 master 负责写，写性能受限
- **数据量受单机限制**：所有节点存储全量数据
- **故障转移期间短暂不可用**：选举和切换需要时间（通常几秒到几十秒）
- **脑裂风险**：网络分区时可能出现两个 master 同时接受写入

---

## 8. 如何搭建 Redis 主从集群？

Redis Cluster 是官方提供的**分布式方案**，支持数据分片和自动故障转移。

**核心特性：**
- 数据分片：使用 **16384 个哈希槽（slot）** 分布数据
- 每个 master 负责一部分 slot，slave 复制对应 master
- 无需代理，客户端直连，节点间通过 Gossip 协议通信

**最小集群配置：** 3 master + 3 slave（共 6 节点）

**搭建步骤（简述）：**
```bash
# 启动 6 个 Redis 实例（不同端口）
redis-server --port 7001 --cluster-enabled yes --cluster-config-file nodes-7001.conf

# 使用 redis-cli 创建集群
redis-cli --cluster create \
  127.0.0.1:7001 127.0.0.1:7002 127.0.0.1:7003 \
  127.0.0.1:7004 127.0.0.1:7005 127.0.0.1:7006 \
  --cluster-replicas 1
```

---

## 9. Redis Cluster 的数据分片原理是什么？

**哈希槽（Hash Slot）机制：**
1. Redis Cluster 共有 **16384 个 slot**（0–16383）
2. 每个 key 通过 `CRC16(key) % 16384` 计算所属 slot
3. 每个 master 负责一段连续的 slot 范围
4. 客户端请求时，若 key 不在当前节点，返回 `MOVED` 重定向

**为什么是 16384 个槽？**
- 16384 = 2^14，心跳包中用 bitmap 表示槽位，16384 bit = 2KB，足够小
- 集群节点数通常不超过 1000，16384 个槽分配足够均匀

---

## 10. Redis Cluster 如何处理节点故障？

1. 节点间通过 Gossip 协议互相发送 `PING/PONG` 检测存活
2. 某节点超时未响应，其他节点标记其为 **PFAIL**（疑似故障）
3. 超过半数 master 认为该节点故障，标记为 **FAIL**（确认故障）
4. 该节点的 slave 发起选举，获得多数 master 投票后晋升为新 master
5. 新 master 接管原来的 slot 范围，集群恢复正常

---

## 11. Redis Cluster 的缺点是什么？

- **不支持跨 slot 的多 key 操作**：`MGET`、`MSET`、事务等要求所有 key 在同一 slot
  - 解决：使用 `{}` 哈希标签强制同一 slot，如 `{user}:name`、`{user}:age`
- **客户端需要支持集群协议**：需要 cluster-aware 客户端（如 Jedis ClusterClient）
- **运维复杂度高**：扩缩容需要迁移 slot，操作复杂
- **最少需要 6 个节点**：成本较高

---

## 12. 你在使用 Redis 搭建主从集群时遇到过什么问题？

**常见问题及解决方案：**

**1. 复制不生效**
- 原因：slave 配置了 `bind 127.0.0.1`，master 无法连接
- 解决：修改 `bind` 为 `0.0.0.0` 或具体 IP，并配置 `protected-mode no`

**2. 全量同步频繁触发**
- 原因：`repl_backlog` 太小，slave 断线后 offset 超出范围
- 解决：增大 `repl-backlog-size`（如 `repl-backlog-size 10mb`）

**3. 主从数据不一致**
- 原因：异步复制存在延迟，读 slave 可能读到旧数据
- 解决：对一致性要求高的读操作走 master；或使用 `WAIT` 命令等待同步完成

**4. 脑裂问题**
- 原因：网络分区导致旧 master 和新 master 同时接受写入
- 解决：配置 `min-slaves-to-write 1` 和 `min-slaves-max-lag 10`，master 在无 slave 时拒绝写入

**5. 内存占用翻倍**
- 原因：`BGSAVE` 时 fork 子进程，Copy-On-Write 机制导致内存短暂翻倍
- 解决：预留足够内存，避免在内存紧张时触发全量同步

## 13. 切片集群以及主节点选举机制详细讲一下

**切片集群（Sharding Cluster）**

切片集群将数据分散到多个 master 节点，每个节点只存储部分数据，解决单机内存瓶颈。

Redis 官方实现是 **Redis Cluster**，使用哈希槽分片（见 Q9）。

**主节点选举机制（Raft-like）：**

当某个 master 被标记为 FAIL 后，其 slave 发起选举：

1. **发起选举**：slave 将自己的 `currentEpoch` 加 1，向所有 master 广播 `FAILOVER_AUTH_REQUEST`
2. **投票条件**：master 在同一 epoch 内只投一票，且只投给数据最新（offset 最大）的 slave
3. **获得多数票**：slave 获得超过半数 master 的投票后，晋升为新 master
4. **广播结果**：新 master 广播 `PONG` 通知集群，更新 slot 归属

**选举超时机制：**
- 每个 slave 的选举延迟时间不同：`delay = 500ms + random(0~500ms) + (master_repl_offset - slave_repl_offset) * 某系数`
- offset 越大的 slave 延迟越短，优先发起选举，保证数据最新的 slave 优先当选

**epoch（纪元）的作用：**
- 类似 Raft 的 term，防止旧选举结果干扰新选举
- 每次选举 epoch 递增，旧 epoch 的投票无效

## 14. Redis Cluster 扩容时如何迁移数据？

扩容时需要将部分 slot 从旧节点迁移到新节点：

```bash
# 使用 redis-cli 重新分配槽
redis-cli --cluster reshard <host>:<port>

# 或手动指定
redis-cli --cluster reshard 127.0.0.1:7001 \
  --cluster-from <source-node-id> \
  --cluster-to <target-node-id> \
  --cluster-slots 1000
```

**迁移过程（在线迁移，不影响服务）：**
1. 目标节点标记 slot 为 `IMPORTING` 状态
2. 源节点标记 slot 为 `MIGRATING` 状态
3. 逐个迁移 slot 中的 key（`MIGRATE` 命令）
4. 迁移完成后，通知集群更新 slot 归属
5. 客户端请求期间若 key 已迁移，返回 `ASK` 重定向（临时）或 `MOVED` 重定向（永久）

## 15. Redis 集群模式有哪些？

Redis 提供了三种主要的集群部署模式，分别解决不同的问题。

**主从模式（Master-Slave）：**

选择一台服务器作为主节点（Master），将数据复制到多台从节点（Slave），形成一主多从的架构。主节点负责读写操作，从节点一般只负责读取，实现读写分离。

当主节点有写操作时，会异步同步给从节点。由于是异步复制，主从之间无法保证强一致性——主节点写入后立刻宕机，从节点可能还没同步到最新数据。

**解决的问题：** 数据冗余、读写分离提高读性能

**哨兵模式（Sentinel）：**

主从模式有一个严重问题：主节点宕机后需要人工干预切换。哨兵模式在主从基础上增加了自动故障转移能力。

哨兵是一个独立运行的进程，负责监控主从节点的健康状态。当主节点宕机时，哨兵自动从从节点中选出一个作为新主节点，其他从节点重新指向新主节点，整个过程自动完成，无需人工介入。

**解决的问题：** 自动故障转移、高可用

**切片集群（Redis Cluster）：**

当数据量增长到单台服务器无法承载时，需要使用切片集群。它将数据分布在不同的服务器上，通过哈希槽（16384 个 slot）将 key 分配到不同节点，降低对单个主节点的依赖，提高整体读写性能。

每个节点负责一部分 slot，节点之间通过 Gossip 协议互相通信。客户端请求时，如果 key 不在当前节点，会收到重定向指引。

**解决的问题：** 数据量扩展、写性能瓶颈

**三种模式对比：**

| 模式 | 数据分布 | 故障转移 | 适用场景 |
|------|----------|----------|----------|
| 主从 | 全量复制 | 手动切换 | 数据量小、读多写少 |
| 哨兵 | 全量复制 | 自动切换 | 中小规模、高可用要求 |
| 切片集群 | 按 slot 分片 | 自动切换 | 大数据量、高并发写入 |

## 16. Redis 切片集群的工作原理？

切片集群使用**哈希槽（Hash Slot）**来决定数据分布在哪个节点上。

整个集群一共有 16384 个槽位，每个 key 存入 Redis 时，会通过以下公式计算它属于哪个槽：

```
槽位 = CRC16(key) % 16384
```

- `CRC16(key)`：对 key 做一次 CRC16 哈希算法，算出一个 16bit 的数字（范围 0~65535）
- `% 16384`：对 16384 取模，得到一个 0~16383 之间的数，就是这个 key 所属的槽位

然后每个主节点负责一段连续的槽位范围。比如 3 个主节点：
- Master1：负责 0 ~ 5460
- Master2：负责 5461 ~ 10922
- Master3：负责 10923 ~ 16383

客户端写入一个 key 时，先算出槽位，再把请求发给负责这个槽位的节点。如果发错了节点，会收到 `MOVED` 重定向，告诉客户端去正确的节点。

## 17. 哈希槽和 Redis 节点是如何对应的？

槽位分配到节点有两种方式：

**平均分配（自动）：**
集群创建时，Redis 自动将 16384 个槽位平均分配给所有主节点。比如 3 个节点就各分 5461/5461/5462 个槽。简单省事，适合大多数场景。

**手动分配：**
使用 `--cluster add-node` 或 `reshard` 命令指定每个节点负责哪些槽位。但要注意**必须把 16384 个槽位全部分完**，否则集群无法正常工作。

```bash
# 手动分配槽位
redis-cli --cluster reshard <host>:<port> \
  --cluster-from <source-id> \
  --cluster-to <target-id> \
  --cluster-slots 1000  # 分1000个槽给目标节点
```

手动分配适合需要精细控制数据分布的场景，比如想把热点数据集中到性能更好的机器上。

## 18. 主从模式的同步过程？

**第一次同步（全量复制）：**

从节点连上主节点后，整个同步过程分三步：

**第一步：连接协商**

从节点发送 `PSYNC` 命令给主节点，内容包含主节点的 `runID` 和从节点的复制进度 `offset`。主节点收到后返回自己的 `runID` 和 `offset`，从节点记录下这两个值，建立连接。

**第二步：主从数据同步**

主节点执行 `BGSAVE` 生成 RDB 文件，然后把 RDB 发送给从节点。从节点收到后先清空自己的旧数据，再载入 RDB 文件。

这个过程需要时间，为了保证数据一致，主节点在生成 RDB 期间收到的写命令会记录到 **replication buffer**（复制缓冲区）中。

**第三步：发送缓冲区数据**

RDB 发送完成后，主节点把 replication buffer 里的写命令发给从节点，从节点执行这些命令。至此第一次同步完成。

**命令传播：**

第一次同步完成后，双方保持一个 TCP 连接，后续主节点的每个写命令都通过这个连接实时发送给从节点，保证主从数据持续一致。

**压力分摊：**

如果主节点负载较高，生成和传输 RDB 的工作可以分摊给某个从节点来做，由这个从节点去生成 RDB 再同步给其他从节点，减轻主节点压力。

**增量复制：**

主从之间网络断开后恢复，不需要重新做全量同步，而是只同步断线期间的写命令（增量复制）。

---

## 19. 主服务器如何知道要将哪些增量数据发送给从服务器？

网络断开恢复后，从节点重新发送 `PSYNC`，带上自己的 offset。主节点根据 offset 差距判断该发什么：

- 从节点需要的数据**在 repl_backlog_buffer 范围内** → 只发增量数据（部分同步）
- 从节点需要的数据**不在 repl_backlog_buffer 范围内** → 退化为全量复制（重新生成 RDB）

**repl_backlog_buffer** 是一个环形缓冲区，主节点写入的每个写命令都会记录在这个缓冲区中。它有两个指针：
- **master_repl_offset**：主节点当前写到的位置
- **slave_repl_offset**：从节点上次同步到的位置

两个 offset 之间的差值就是从节点缺失的数据。只要这个差值没超出缓冲区大小，就能增量同步。

> 类比：你和朋友约好一起看同一本书。朋友掉线了一段时间，回来后问你"上次看到第几页了"。你翻到那一页，把后面的章节给他——这就是增量同步。但如果掉线太久，你已经翻到书的前面把前面的内容覆盖了（缓冲区写满了），就只能重新寄一本完整的书给他——这就是全量同步。

## 20. 如何避免主从数据的不一致？

由于主从复制是异步的，写入主节点后还没同步到从节点时去从节点读，可能读到旧数据。降低不一致影响的方法：

**方法一：部署在同一机房**
- 减少网络延迟，缩短主从同步的时间差

**方法二：外部程序监控复制进度差**
- 计算主从之间的 offset 差值
- 如果差值超过设定的阈值，让客户端不再从该从节点读取数据
- 等同步追上来后再恢复读取

**注意：** 阈值不能设太小，否则容易出现所有从节点都被禁止读取、客户端只能读主节点的情况。

## 21. 主从架构中过期 key 如何处理？

主从模式下，key 的过期处理有一个细节：**从节点不会主动删除过期 key，而是由主节点来决定删除。**

主节点处理过期 key 的流程：
1. 主节点发现某个 key 过期，执行删除操作
2. 主节点向从节点发送一条 `DEL` 命令
3. 从节点收到 `DEL` 后删除该 key

这样做的好处是保证主从行为一致——所有节点都按照主节点的决策来删除 key，避免从节点提前或延迟删除导致数据不一致。

## 22. 哨兵机制的工作原理？

哨兵机制的完整工作流程分为以下几个阶段：

**一、判断节点是否存活**

哨兵周期性地向所有主从节点发送 `PING` 命令。如果某个节点在 `down-after-milliseconds` 时间内没有响应，哨兵先将该节点标记为**主观下线**（单个哨兵的判断）。

然后该哨兵向其他哨兵发起投票，询问它们是否也认为该节点下线。当赞成票达到设定的阈值（quorum）后，该节点被标记为**客观下线**（多数哨兵的共识），此时才会触发故障转移。

**二、选举 Leader Sentinel**

哨兵集群中需要选出一个 leader 来负责执行主从切换。选举过程：

1. 判断主节点客观下线的哨兵成为候选者
2. 候选者向其他哨兵发送命令，请求投票成为 leader
3. 每个哨兵只有一票，可以投给自己或投给别人，但只有候选者才能把票投给自己
4. 候选者获得半数以上赞成票且票数大于 quorum，就成为 leader

**三、选出新主节点**

Leader Sentinel 从所有从节点中选出一个作为新主节点，需要先排除两类从节点：
- 已经下线的从节点
- 网络连接状态不好的从节点

然后对剩余从节点进行三轮考察，每轮胜出的从节点继续进入下一轮：

| 考察轮次 | 规则 | 说明 |
|----------|------|------|
| 第一轮 | slave-priority 优先级 | 值越小越优先 |
| 第二轮 | 复制偏移量 offset | offset 越大说明数据越新，越优先 |
| 第三轮 | Run ID | 字典序最小的获胜（兜底） |

**四、更换主节点**

Leader Sentinel 执行以下操作：
1. 向新主节点发送 `SLAVEOF NO ONE`，让它成为独立 master
2. 向其他从节点发送 `SLAVEOF <new_master_ip> <new_master_port>`，切换复制目标

**五、通知客户端**

客户端和哨兵建立连接后，会订阅哨兵提供的 `+switch-master` 频道。主从切换完成后，哨兵向该频道发布新主节点的 IP 和端口，客户端收到后自动切换到新主节点进行通信。

**六、旧主节点恢复**

当旧主节点重新上线时，哨兵会向它发送 `SLAVEOF` 命令，让它成为新主节点的从节点，重新加入集群。

## 23. 什么是集群的脑裂？

**脑裂（Split Brain）** 是指在主从架构中，由于网络问题导致集群分裂成两个"独立"的部分，各自认为自己是正常的，从而出现两个主节点同时接受写入的情况。

**脑裂发生的过程：**

1. 主节点的网络突然出现异常，和所有从节点都断开了连接
2. 但主节点和客户端之间的网络是正常的，客户端不知道内部出了问题，还在继续向这个失联的主节点写数据
3. 此时主节点会把新写入的数据缓存在本地缓冲区里
4. 同时，哨兵发现主节点失联了，就在从节点中选出一个新 leader 作为主节点
5. 此时集群中出现了两个主节点——旧主节点（失联但还在接受写入）和新主节点（哨兵选出的）

**网络恢复后的后果：**

哨兵之前已经选出新主节点了，所以会把旧主节点降级。旧主节点重新上线后，需要向新主节点请求全量同步。由于第一次同步是全量复制的方式，旧主节点会**清空自己本地的数据**，然后加载新主节点的 RDB。

问题就在这里：旧主节点在失联期间缓存的那些写入数据，在清空时全部丢失了。

```
时间线：
1. 旧master失联，客户端继续写入数据到旧master
2. 哨兵选出新master，旧master降级
3. 网络恢复，旧master向新master全量同步
4. 旧master清空本地数据（包括客户端写入的数据）→ 数据丢失
```

**如何防止脑裂？**

配置以下参数，让主节点在没有从节点时拒绝写入：

```bash
min-slaves-to-write 1      # 至少有1个从节点连接才允许写入
min-slaves-max-lag 10      # 从节点延迟不超过10秒才允许写入
```

这样当主节点和所有从节点失联时，它会自动停止接受写入，避免客户端在失联期间写入的数据在网络恢复后丢失。

## 24. 如何减少主从切换带来的数据丢失？

主从切换导致数据丢失主要有两个场景，针对每个场景有不同的应对方案。

**场景一：异步复制导致的同步丢失**

主节点写入数据后，在还没来得及同步给从节点之前就宕机了，这部分数据就会丢失。

**应对方案：**

配置一个阈值，一旦所有从节点的复制延迟都超过了这个阈值，主节点就拒绝接收任何写请求。这样可以保证在主节点宕机时，至少有一个从节点的数据是接近最新的。

当客户端发现主节点不可写时，可以采取降级措施：
- **方案一**：将数据暂时写入本地缓存和磁盘，等主节点恢复后重新写入
- **方案二**：将数据写入消息队列，等主节点恢复后从队列中消费并重新写入

**场景二：脑裂导致的数据丢失**

网络分区导致旧主节点和新主节点同时存在，旧主节点在失联期间缓存的写入数据，在网络恢复后会丢失。

**应对方案：**

配置主节点至少要连接 N 个从节点，并且从节点的 ACK 延迟不能超过 T 秒，否则主节点拒绝写入：

```bash
min-slaves-to-write 1      # 至少有1个从节点
min-slaves-max-lag 10      # ACK延迟不超过10秒
```

这样当从节点下线或通信超时的数量超过阈值时，主节点直接拒绝写入并返回错误给客户端。

**效果：**
- 新主节点上线后，只有新主节点能接收客户端请求，新写入的数据不会丢
- 旧主节点被降级为从节点，即使它之前缓存的数据被清空，也不会有新的数据丢失

**核心思路：** 与其等数据丢了再补，不如在写入时就保证"有足够的从节点跟上了"，没跟上就拒绝写入，把数据引导到安全的地方。

## 25. Redis 常见使用方式有哪些？

Redis 有五种常见的部署方式，分别适用于不同场景：

| 部署方式 | 特点 | 适用场景 |
|----------|------|----------|
| 单副本 | 单机部署，无备份 | 数据量小、开发测试 |
| 多副本（主从） | 读写分离，手动切换 | 数据量小、读多写少 |
| Sentinel（哨兵） | 自动故障转移 | 中小规模、高可用要求 |
| Redis Cluster | 数据分片 + 自动故障转移 | 大数据量、高并发 |
| Redis 自研 | 自定义架构 | 特殊需求、大规模定制 |

**各方式详细说明：**

**① Redis 单副本（单机）**

只部署一个 Redis 实例，所有数据存在一个节点上。

- 优点：部署简单，成本低
- 缺点：没有备份，宕机数据丢失
- 适用：数据量很少（几个 G 以内），主要承载高并发高性能的场景，比如简单缓存

**② Redis 多副本（主从模式）**

一主多从，主节点负责写，从节点负责读，实现读写分离。

- 优点：读性能提升，数据有备份
- 缺点：master 挂掉后需要手动指定新 master，可用性不高
- 适用：读多写少的场景，但对可用性要求不高时使用（生产中基本不用纯主从）

**③ Redis Sentinel（哨兵模式）**

在主从基础上加入哨兵进程，自动监控和故障转移。

- 优点：master 挂掉后哨兵自动选举新 master，可用性高
- 缺点：每个节点存储的数据是一样的（全量复制），浪费内存空间
- 适用：数据量不是很多，集群规模不大，需要自动容错容灾的场景

**④ Redis Cluster（切片集群）**

数据按 slot 分片存储在多个 master 上，每个 master 负责一部分数据。

- 优点：所有 master 的容量总和就是 Redis Cluster 可缓存的数据容量，支持海量数据
- 缺点：架构复杂，跨 slot 操作有限制
- 适用：海量数据 + 高并发 + 高可用的场景，数据量很大时推荐使用

**⑤ Redis 自研**

基于 Redis 源码进行二次开发，自定义架构满足特殊需求。

- 优点：完全可控，可以根据业务需求定制
- 缺点：开发和维护成本高
- 适用：大厂有特殊需求时使用

**选型建议：**

```
数据量小（< 几个G）→ 单副本
读多写少、可用性要求不高 → 主从模式
需要自动容错、数据量中等 → Sentinel
海量数据、高并发 → Redis Cluster
特殊定制需求 → 自研
```

## 26. 介绍下 Redis Cluster？

Redis 的哨兵模式虽然可以实现高可用和读写分离，但在这种模式下每台 Redis 服务器都存储相同的数据，很浪费内存。所以在 Redis 3.0 上加入了 Cluster 集群模式，实现了分布式存储，对数据进行分片，每台 Redis 节点上存储不同的内容。

Redis Cluster 是社区版推出的分布式集群解决方案，主要解决单机内存、并发和流量等瓶颈，起到负载均衡的目的。

**集群配置：**
- 最小配置 6 个节点（3 主 3 从）
- 主节点提供读写操作
- 从节点作为备用节点，不提供服务，只用于故障转移

**核心原理：** 采用虚拟槽分区，所有键通过哈希函数映射到 0 ~ 16383 个整数槽内，每个节点负责维护一部分槽以及槽所映射的键值数据。

**优点：**

| 优点 | 说明 |
|------|------|
| 无中心架构 | 去中心化的 P2P 架构，没有代理层 |
| 数据按 slot 分布 | 数据共享，可动态调整数据分布 |
| 可扩展性强 | 可线性扩展到 1000+ 个节点，支持动态添加/删除 |
| 高可用 | 部分节点不可用时集群仍可用，通过 Slave 实现自动 failover |
| 运维成本低 | 提高系统的扩展性和可用性 |

**缺点：**

| 缺点 | 说明 |
|------|------|
| Client 实现复杂 | 驱动要求实现 Smart Client，缓存 slots mapping 信息并及时更新，目前仅 JedisCluster 相对成熟 |
| 误判下线 | 节点因阻塞（阻塞时间大于 cluster-node-timeout）会被判断下线，导致不必要的 failover |
| 异步复制 | 数据通过异步复制，不保证强一致性 |
| 资源隔离差 | 多个业务使用同一套集群时，无法区分冷热数据，容易相互影响 |
| Slave 读能力弱 | Slave 充当"冷备"，不能缓解读压力，需通过 SDK 合理设计提高利用率 |
| 批量操作受限 | mset、mget 只支持相同 slot 值的 Key 批量操作，不支持跨 slot 查询 |
| 事务受限 | 只支持多 key 在同一节点上的事务操作，跨节点无法使用事务 |
| key 粒度限制 | 不能将一个很大的键值对象映射到不同节点 |
| 单数据库空间 | 集群模式下只能使用 db 0，不支持多数据库 |
| 复制结构单层 | 从节点只能复制主节点，不支持嵌套树状复制 |

**使用注意事项：**

- 避免产生 **hot-key**，导致某个主库节点成为系统瓶颈
- 避免产生 **big-key**，导致网卡撑爆、慢查询等问题
- 重试时间应该大于 `cluster-node-timeout` 时间
- 不建议使用 pipeline 和 multi-keys 操作，减少 max redirect 产生的场景

## 27. 介绍下 Redis 自研？

Redis 自研是指基于 Redis 源码进行二次开发，构建高可用解决方案。主要体现在配置中心、故障探测和 failover 的处理机制上，通常需要根据企业业务的实际线上环境来定制化。

**优点：**

| 优点 | 说明 |
|------|------|
| 高可靠性、高可用 | 根据业务需求定制故障转移策略 |
| 自主可控性高 | 不依赖第三方组件，完全掌控 |
| 贴切业务需求 | 可缩放性好，兼容性好 |

**缺点：**

| 缺点 | 说明 |
|------|------|
| 实现复杂 | 开发成本高，需要深入理解 Redis 内部机制 |
| 配套设施 | 需要建立配套的周边设施，如监控、域名服务、存储元数据信息的数据库等 |
| 维护成本高 | 长期维护需要持续投入人力 |

## 28. Redis 高可用方案具体怎么实施？

使用官方推荐的哨兵（Sentinel）机制就能实现高可用。当主节点出现故障时，由 Sentinel 自动完成故障发现和转移，并通知应用方，实现高可用性。

**Sentinel 的四个主要功能：**

| 功能 | 说明 |
|------|------|
| 集群监控 | 负责监控 Redis master 和 slave 进程是否正常工作 |
| 消息通知 | 如果某个 Redis 实例有故障，哨兵负责发送消息作为报警通知给管理员 |
| 故障转移 | 如果 master node 挂掉了，会自动转移到 slave node 上 |
| 配置中心 | 如果故障转移发生了，通知 client 客户端新的 master 地址 |

> 详细原理见 [[02Redis高可用面试题#22. 哨兵机制的工作原理？|Q22]]

**实施步骤：**

1. **部署哨兵集群**：至少部署 3 个 Sentinel 进程，避免单点故障
2. **配置监控目标**：在 sentinel.conf 中配置要监控的 master 地址和 quorum 数量
3. **配置故障转移**：设置 down-after-milliseconds（超时判定时间）和 failover-timeout（故障转移超时时间）
4. **客户端接入**：客户端通过 Sentinel 获取当前 master 地址，而非硬编码
5. **测试验证**：模拟 master 宕机，验证自动故障转移是否正常

## 29. 由于主从延迟导致读取到过期数据怎么处理？

主从异步复制存在延迟，从节点可能还没有同步到主节点的删除操作，导致从节点上读到已经过期的数据。

**方案一：SCAN 扫描淘汰**

通过 `SCAN` 命令扫库时，相当于访问了该 key，Redis 会对其做过期检测，充分发挥惰性删除的策略。这个方法能大大降低脏数据读取的概率，但缺点是会造成一定的数据库压力，影响线上业务效率。

**方案二：Redis 新特性（推荐）**

Redis 加入了新特性来解决主从不一致导致读取到过期数据的问题，增加了 key 是否过期以及对主从库的判断：
- 如果 key 已过期，当前访问的是 **master**，则返回 null
- 如果 key 已过期，当前访问的是 **slave**，且执行的是只读命令，也返回 null

这个特性从根源上避免了从节点返回过期数据的问题。

## 30. 主从复制的过程中如果因为网络原因停止复制了会怎么样？

如果出现网络故障断开连接了，主从之间会自动重连。从 Redis 2.8 开始，支持主从复制的**断点续传**，可以接着上次复制的地方继续复制，而不是从头开始复制。

**断点续传的原理：**

master 会在内存中创建一个 **backlog**（缓冲区），master 和 slave 都会保存一个 **replica offset** 和 **master id**，offset 就保存在 backlog 中。

```
master: backlog [....A..B..C..D..E....]  master_repl_offset: 100
slave:                                     slave_repl_offset: 80
                                                ↑
                                          还没同步到的位置
```

如果 master 和 slave 网络连接断掉了，slave 重连后会告诉 master 从上次的 replica offset 开始继续复制。master 在 backlog 中找到对应的 offset，只发送增量数据。

**如果找不到对应的 offset，就执行全量复制（resynchronization）。**

**多个 slave 同时重连：**

如果 master 发现有多个 slave node 都来重新连接，仅仅会启动一个 RDB save 操作，用一份数据服务所有 slave node，避免重复生成 RDB。

## 31. Redis Cluster 节点间通信是什么机制？

Redis Cluster 节点间采取 **Gossip 协议**进行通信，所有节点都持有一份元数据，不同的节点如果出现了元数据的变更，会不断地将元数据发送给其他节点，让其他节点进行数据变更。

**Gossip 的工作方式：**

节点互相之间不断通信，保持整个集群所有节点的数据是完整的。主要交换以下信息：
- 故障信息
- 节点的增加和移除
- hash slot 信息等

> 类比：公司里没有统一的公告板，每个人听到新消息后就口口相传，最终所有人都知道了一样的信息。

**优点：**

元数据的更新比较分散，不是集中在一个地方。更新请求会陆陆续续打到所有节点上去更新，有一定的延时，降低了压力。

**缺点：**

元数据更新有延时，可能导致集群的一些操作会有一些滞后。比如新加入一个节点后，其他节点需要一段时间才能感知到。

## 面试回答模板

**Q: Redis 主从复制的原理？**

A: 主从复制分全量和增量两种。首次连接时，master 执行 BGSAVE 生成 RDB 发给 slave，再把期间的写命令发过去。断线重连时，如果 offset 在 repl_backlog 范围内，只发增量命令，否则退化为全量同步。

**Q: 哨兵是怎么工作的？**

A: 哨兵负责监控、通知和自动故障转移。单个 sentinel 发现 master 主观下线后，询问其他 sentinel，达到 quorum 后确认客观下线，然后选举 leader sentinel 执行故障转移，从 slave 中选出新 master。

**Q: Redis Cluster 的数据分片原理？**

A: 使用 16384 个哈希槽，每个 key 通过 CRC16(key) % 16384 计算所属 slot，每个 master 负责一段 slot。客户端请求时如果 key 不在当前节点，返回 MOVED 重定向。

**Q: 主从集群脑裂怎么解决？**

A: 配置 min-slaves-to-write 1，master 在没有 slave 时拒绝写入，避免网络分区时旧 master 继续写入导致数据不一致。

## 面试简答版

**主从复制：**
全量同步（BGSAVE + RDB）+ 增量同步（repl_backlog + offset）

**哨兵：**
监控 + 自动故障转移，主观下线 → 客观下线 → 选举 leader → 故障转移

**Redis Cluster：**
16384 哈希槽分片，CRC16(key) % 16384，Gossip 协议通信，slave 选举晋升

**缺点：**
主从：手动切换、复制延迟
哨兵：写单点、数据量受限
集群：跨 slot 限制、最少 6 节点
