# 内存管理

## 1. 什么是分页？

### 先说场景

程序运行时需要用内存，但内存不是一整块永远空着等某个程序来用。

如果要求一个程序必须连续放在内存里，就会很麻烦：前面空一点、后面空一点，看起来总空闲不少，但就是凑不出一大块连续空间。

分页就是为了解决这个问题：**把程序和内存都切成一样大的小块，程序不用连续放，只要能通过一张表找到每一页实际放在哪就行。**

### 大白话解释

可以把分页想象成寄快递。

一个大包裹太占地方，仓库不一定能腾出连续的一大片位置，于是把它拆成几个大小差不多的小箱子，分别放到仓库不同格子里。

但是拆开之后，必须有一张清单记录：

```text
第 0 个小箱子 -> 仓库第 2 格
第 1 个小箱子 -> 仓库第 3 格
第 2 个小箱子 -> 仓库第 5 格
```

这张清单，在操作系统里就叫 **页表**。

### 对应到操作系统

| 概念 | 大白话理解 |
|------|------------|
| 页 Page | 程序被切出来的小块 |
| 页框 Frame | 物理内存被切出来的小块 |
| 页表 Page Table | 记录“第几页放在哪个页框”的映射表 |
| 页内偏移 | 进入某一页之后，具体找第几个字节 |

程序看到的是自己的逻辑地址，好像内存从 0 开始连续排着。

但真实内存里，它的第 0 页、第 1 页、第 2 页，可能分别放在物理内存的第 2 块、第 3 块、第 5 块。

### 地址是怎么找到的？

分页系统里，一个逻辑地址通常可以拆成两部分：

```text
逻辑地址 = 页号 + 页内偏移
```

访问内存时大概分两步：

```text
1. 根据页号查页表，找到对应的物理块号
2. 用 物理块号 + 页内偏移，拼出真正的物理地址
```

比如：

```text
页号 2 -> 页表里查到它在物理块 5
页内偏移是 100

最终访问的位置就是：物理块 5 里的第 100 个字节
```

### 为什么访问分页内存可能更慢？

因为它不是直接去内存里拿数据，而是要先查页表。

也就是说，最朴素的分页访问可能需要两次内存访问：

```text
第一次：访问页表，找到物理块号
第二次：访问真正的数据
```

这就像你去仓库拿东西，不能直接冲过去拿，得先看清单，再去对应格子找。

所以 CPU 里通常会加一个 **TLB**，也叫快表，可以理解成“页表的缓存”。

最近查过的页号和物理块号会放到 TLB 里，下次再访问同一页时，就不用每次都去内存里查页表了。

### 分页解决了什么问题？

分页最大的好处是：**程序不用连续存放在物理内存里。**

这样可以减少外部碎片，提高内存利用率。

| 问题 | 分页怎么解决 |
|------|--------------|
| 程序必须连续存放太麻烦 | 分页后每一页可以放在不同物理块里 |
| 外部碎片严重 | 页和页框大小固定，方便分配 |
| 地址转换复杂 | 通过页表完成逻辑地址到物理地址的映射 |

### ⚠️ 面试易错点

- **页是逻辑概念，页框是物理概念**：程序被切成“页”，内存被切成“页框”。
- **页和页框大小相同**：这样页才能刚好放进页框里。
- **分页解决的是外部碎片，不是内部碎片**：最后一页可能装不满，所以仍然可能有内部碎片。
- **页表一般放在内存中**：所以单纯分页访问可能要先查页表，再访问数据。
- **TLB 不是页表本身**：TLB 是页表的高速缓存，用来减少查页表的开销。

### 面试回答模板

分页是操作系统的一种内存管理方式。它把程序的逻辑地址空间划分成大小相等的页，也把物理内存划分成同样大小的页框，然后通过页表记录“页号到页框号”的映射关系。

这样程序就不需要连续存放在物理内存中，可以减少外部碎片，提高内存利用率。访问地址时，CPU 会根据逻辑地址中的页号查页表，找到物理块号，再结合页内偏移得到真正的物理地址。

不过分页也会带来查页表的开销，所以实际系统里通常会使用 TLB，也就是快表，来缓存最近使用过的页表项，加快地址转换。

### 面试简答版

分页就是把程序和物理内存都切成固定大小的小块，程序里的块叫页，内存里的块叫页框，再用页表记录每一页实际放在哪个页框里。

它的好处是程序不用连续存放，能减少外部碎片；缺点是访问内存时需要查页表，所以通常会用 TLB 来加速。

## 2. 什么是分段？

### 先说场景

分页主要是操作系统为了方便管理内存，把程序和内存都切成固定大小的小块。

但程序员写程序时，脑子里想的不是“第 0 页、第 1 页”，而是：

- 这块是代码
- 这块是全局变量
- 这块是堆
- 这块是栈
- 这块是共享库

这些部分的作用不一样，大小也不一样。分段就是为了更贴近程序本身的逻辑结构。

一句话说：**分段是把程序按逻辑功能分成多个段，每个段可以有自己的长度和权限。**

### 大白话解释

可以把一个程序想象成一本书。

分页像是把书硬生生按每 10 页切一刀，不管这一刀是不是刚好切在一章中间。

分段更像是按章节来分：

```text
第 0 段：正文
第 1 段：目录
第 2 段：附录
第 3 段：索引
```

每一章长度可以不同，而且每一章都有自己的起始位置。

操作系统要做的事情，就是用一张表记录：

```text
第 0 段 -> 从物理内存 40KB 开始，长度 30KB
第 1 段 -> 从物理内存 80KB 开始，长度 20KB
第 2 段 -> 从物理内存 120KB 开始，长度 15KB
```

这张表就叫 **段表**。

### 对应到操作系统

| 概念 | 大白话理解 |
|------|------------|
| 段 Segment | 程序里一个有独立意义的逻辑部分 |
| 段号 | 要访问第几个段 |
| 段内地址 | 进入这个段之后，再偏移多少 |
| 段表 | 记录每个段的起始地址和段长 |
| 基址 | 这个段在物理内存里的起点 |
| 段长 | 这个段最多能访问多大范围 |

分段系统里的逻辑地址通常是二维的：

```text
逻辑地址 = 段号 + 段内地址
```

比如：

```text
段号 = 1
段内地址 = 5KB
```

意思是：我要访问第 1 段里面，偏移 5KB 的位置。

### 地址是怎么转换的？

分段访问内存时，大概分三步：

```text
1. 根据段号查段表
2. 检查段内地址有没有超过段长
3. 用 段基址 + 段内地址 得到物理地址
```

比如段表里记录：

```text
第 1 段：基址 80KB，段长 20KB
```

现在访问：

```text
段号 1，段内地址 5KB
```

因为 5KB 没有超过段长 20KB，所以地址合法。

真正的物理地址就是：

```text
80KB + 5KB = 85KB
```

如果访问的是：

```text
段号 1，段内地址 25KB
```

那就越界了，因为第 1 段只有 20KB 长。

### 分段有什么好处？

分段的核心好处是：**符合程序的逻辑结构。**

代码段、数据段、栈段本来就是不同用途，分开管理会更自然。

| 好处 | 怎么理解 |
|------|----------|
| 方便共享 | 多个进程可以共享同一段代码 |
| 方便保护 | 代码段可以只读，数据段可以读写 |
| 符合程序结构 | 程序本来就有代码、数据、栈等逻辑部分 |
| 支持动态增长 | 栈段、堆段可以按需要变化 |

比如代码段通常不应该被修改，所以可以设置成只读。

这就像图书馆里的参考书，只能看，不能在上面乱写。

### 分段的问题

分段也有缺点：**容易产生外部碎片。**

因为每个段长度不一样，段和段之间又是离散分配的，用久了以后，内存里可能留下很多大小不一的小空洞。

这些小空洞单独看都有空间，但不一定能放下一个新的大段。

这点和分页不一样：分页因为块大小固定，所以更方便分配。

### ⚠️ 面试易错点

- **分页是固定大小，分段是长度可变**：页大小一样，段大小不一定一样。
- **分页偏操作系统管理，分段偏程序逻辑结构**：分页是为了提高内存利用率，分段是为了更符合代码、数据、栈这些逻辑需求。
- **分段会产生外部碎片**：因为每个段大小不固定，空闲空间可能被切得七零八落。
- **段表里记录的是基址和段长**：访问时要先判断段内地址是否越界，再加上基址得到物理地址。
- **分段地址是二维的**：段号 + 段内地址，不是单纯一个连续地址。

### 面试回答模板

分段是一种内存管理方式，它按照程序的逻辑结构把程序划分成多个段，比如代码段、数据段、栈段等。每个段的长度可以不同，并且每个段在内存中可以离散存放。

在分段系统中，逻辑地址由段号和段内地址组成。CPU 根据段号查段表，段表中保存了段的基址和段长。访问时先判断段内地址是否超过段长，如果没有越界，就用段基址加段内地址得到真正的物理地址。

分段的优点是符合程序的逻辑结构，方便共享和保护，比如代码段可以设置为只读，也可以被多个进程共享。缺点是段的长度不固定，容易产生外部碎片。

### 面试简答版

分段就是按照程序的逻辑功能把内存划分成多个段，比如代码段、数据段、栈段。逻辑地址由段号和段内地址组成，通过段表找到段的基址和段长，再计算出物理地址。

它的好处是符合程序结构，方便共享和保护；缺点是段大小不固定，容易产生外部碎片。

## 3. 分页和分段有什么区别？

### 先给结论

这道题最容易讲乱，因为分页和分段都在做“逻辑地址到物理地址的转换”。

但你记住一句话，基本就不会乱：

- **分页是按固定大小切，重点是方便内存管理**
- **分段是按逻辑模块切，重点是贴合程序结构**

也就是说，分页更像操作系统站在“怎么更好分配内存”的角度想问题；分段更像程序员站在“代码本来就分成不同功能块”的角度想问题。

### 用一个类比先记住

把一本书交给两个人处理：

第一个人按“每 10 页一刀”去切，这就是分页。

```text
第 1 份：1~10 页
第 2 份：11~20 页
第 3 份：21~30 页
```

它不关心你是不是刚好切在章节边界上，只在乎每份大小一样，方便管理。

第二个人按“目录章节”去切，这就是分段。

```text
第 1 段：前言
第 2 段：第一章
第 3 段：第二章
第 4 段：附录
```

每一段长短都可能不一样，但每段的意义非常明确。

### 区别 1：划分方式不同

分页是把程序和内存都切成**固定大小**的小块。

- 程序里的块叫页
- 内存里的块叫页框
- 页大小是固定的

分段是把程序按**逻辑功能**划分成不同的段。

- 比如代码段、数据段、栈段
- 每个段大小可以不一样
- 每个段都有自己的起始地址和长度

### 区别 2：地址结构不同

分页的逻辑地址通常是一维的，本质上是一个连续地址，只不过系统会把它拆成：

```text
页号 + 页内偏移
```

分段的逻辑地址更像二维地址：

```text
段号 + 段内偏移
```

因为你访问的不是“连续空间里的第几个字节”，而是“第几个逻辑段里的第几个字节”。

### 区别 3：目的不同

分页的重点是：

- 让内存分配更规整
- 减少外部碎片
- 提高内存利用率

分段的重点是：

- 让内存划分更符合程序逻辑
- 方便共享
- 方便保护

比如代码段可以设置为只读，数据段可以读写，这种保护在分段思路下就很自然。

### 区别 4：碎片问题不同

分页因为大小固定，**外部碎片少**，但最后一页可能装不满，所以会有**内部碎片**。

分段因为每段大小不同，**容易出现外部碎片**。

这个点特别常考，别记反了。

### 区别 5：对程序员是否“有感知”不同

分页通常对程序员是透明的。

你写代码时一般不会关心“我这个变量在第几页”，这些都是操作系统和硬件在背后处理。

分段更贴近程序本身的结构。

因为程序天然就有代码、数据、栈这些不同逻辑区域，所以分段更容易让人从程序结构的角度去理解。

### 对比总结

| 对比点 | 分页 | 分段 |
|--------|------|------|
| 划分依据 | 固定大小 | 逻辑模块 |
| 大小 | 页大小固定 | 段大小可变 |
| 地址形式 | 页号 + 页内偏移 | 段号 + 段内偏移 |
| 主要目的 | 提高内存利用率 | 满足程序逻辑和保护需求 |
| 碎片问题 | 可能有内部碎片 | 容易有外部碎片 |
| 共享和保护 | 相对没那么直观 | 更方便实现 |

### ⚠️ 面试易错点

- **不要只背“分页固定、分段可变”**：这只是表面区别，更核心的是“分页偏管理，分段偏逻辑”。
- **分页不是按功能分的**：它不关心这一块是代码还是数据，只关心大小统一。
- **分段不是为了提升内存利用率而设计的**：它更主要是为了让程序的不同逻辑部分独立管理。
- **分页主要解决外部碎片问题，分段反而容易产生外部碎片**。
- **分页和分段都要做地址映射**：只是一个查页表，一个查段表。

### 面试回答模板

分页和分段都是内存管理方式，但它们的出发点不一样。

分页是把程序和物理内存划分成固定大小的页和页框，通过页表完成映射，主要目的是提高内存利用率，减少外部碎片。它对程序员通常是透明的，不过可能会产生内部碎片。

分段是按照程序的逻辑结构来划分，比如代码段、数据段、栈段等，通过段表记录每个段的基址和段长。它的优点是更符合程序逻辑，方便共享和保护，但由于段大小不固定，容易产生外部碎片。

所以可以简单理解为：分页偏内存管理效率，分段偏程序逻辑组织。

### 面试简答版

分页是按固定大小切，分段是按逻辑模块切。分页主要为了提高内存利用率、减少外部碎片；分段主要为了满足程序逻辑、共享和保护需求。

分页大小固定，通常是一维地址；分段大小可变，通常是段号加段内偏移。分页容易有内部碎片，分段容易有外部碎片。

## 4. 什么是交换空间？

### 先说场景

内存是很宝贵的，但程序一多，物理内存总有不够用的时候。

这时候操作系统不会立刻让程序全崩掉，而是会想办法把**暂时不常用的数据先挪出去**，把内存位置腾出来给当前更需要运行的内容。

这个“临时挪出去”的地方，通常就是磁盘上的一块区域，也就是**交换空间**。

### 大白话解释

可以把内存想象成你的书桌，磁盘想象成旁边的大柜子。

书桌拿东西很快，但地方小；柜子地方大，但拿东西慢。

如果你桌上摆满了资料，你暂时不用的那几本书，就可以先放进柜子里，等需要时再拿回来。

```text
内存 = 书桌
交换空间 = 柜子里的临时存放区
交换 = 把东西从桌上搬到柜子里，或者再搬回来
```

所以交换空间不是让程序“消失”了，而是把一部分暂时不用的内容先放到更慢但更大的地方。

### 什么是交换空间，什么是交换？

这两个词很容易混。

- **交换空间（swap space）**：磁盘上的一块区域，专门用来临时存放从内存换出去的页
- **交换（swapping）**：把内存里的内容换到磁盘，或者再从磁盘换回内存的过程

你可以这样记：

```text
交换空间 = 地方
交换 = 动作
```

### 为什么需要交换空间？

最核心的原因就一个：**物理内存不够。**

当内存吃紧时，操作系统会把一些暂时不活跃的页换到交换空间里，这样就能释放出内存给当前活跃的程序使用。

常见场景有：

- 某些程序开着但长时间没操作
- 程序启动时加载了很多内容，后面其中一部分很久都用不到
- 系统同时运行的进程太多，RAM 不够

### 它和分页是什么关系？

交换空间通常是和分页机制一起工作的。

因为系统管理内存时，本来就常常是按“页”来管理，所以换出去、换回来的单位，通常也是**页**。

也就是说：

```text
页在内存里放不下
-> 某些不常用页被换到 swap 空间
-> 需要时再换回内存
```

所以交换空间本质上不是独立的一套新机制，而是虚拟内存管理里的一个重要配套手段。

### 它和虚拟内存是什么关系？

这个也是高频易混点。

你可以先这样理解：

- **物理内存**：真正的 RAM
- **交换空间**：磁盘上拿来临时顶内存压力的区域
- **虚拟内存**：操作系统给每个进程“看到的一套更大、更连续”的内存抽象

交换空间是虚拟内存能够工作的一个重要补充，但**交换空间不等于虚拟内存本身**。

虚拟内存是整体机制，交换空间只是其中一个帮手。

### 交换空间的优点和代价

它的优点很直接：

- 能缓解内存不足
- 让系统不至于因为 RAM 紧张立刻崩掉
- 能让更多进程同时存在

但代价也很明显：

- 磁盘速度远慢于内存
- 如果频繁换入换出，系统会明显变卡

这就像你一边干活一边不停从书桌跑去柜子里拿资料，肯定比直接在桌上翻慢得多。

如果系统大部分时间都在忙着换页，而不是忙着真正执行程序，这种现象就叫 **抖动（Thrashing）**。

### ⚠️ 面试易错点

- **交换空间在磁盘上，不在内存里**。
- **交换空间不等于虚拟内存**：它只是虚拟内存机制的一部分。
- **交换通常按页进行，不一定是整个进程一起搬**。
- **有 swap 不代表性能更好**：swap 的本质是“救场”，不是让磁盘变成和内存一样快。
- **频繁使用 swap 往往说明内存压力很大**，系统可能已经开始变慢。

### 面试回答模板

交换空间是磁盘上的一块区域，用来临时存放从内存中换出的页。当物理内存不足时，操作系统会把暂时不常用的页换到交换空间中，释放 RAM 给当前更活跃的程序使用，这个过程叫交换。

交换空间通常和分页机制一起工作，因为换入换出的基本单位往往是页。它可以缓解内存不足，让系统支持更多进程继续运行，但由于磁盘速度远慢于内存，如果频繁发生交换，系统性能会明显下降，严重时还会出现抖动。

所以可以把它理解成虚拟内存机制中的一个辅助区域，用来在内存紧张时临时“托底”。

### 面试简答版

交换空间就是磁盘上的一块区域，用来存放从内存换出去的页；交换就是把页在内存和磁盘之间来回搬的过程。

它的作用是缓解物理内存不足，但因为磁盘比内存慢很多，所以频繁使用交换空间会导致系统变卡。

## 5. 物理地址、逻辑地址、有效地址、线性地址、虚拟地址的区别？

### 先说场景

这道题很多人不是不会，而是**名词太多，脑子里没排队**。

明明都是“地址”，但它们其实是在说不同阶段的地址：

- 有的是程序里写出来的地址
- 有的是 CPU 处理中间阶段的地址
- 有的是内存里真正落点的地址

你只要先抓住一句话，就不容易乱：

**离程序最近的地址，不一定是真地址；离内存最近的地址，才是真地址。**

### 先给一个总图

你可以先粗暴地这样记：

```text
程序看到的地址
-> 经过分段 / 分页等转换
-> 最终变成物理地址
```

再对应到这些名词：

```text
逻辑地址 / 有效地址
    -> 线性地址
    -> 虚拟地址（开启分页时，线性地址也常被叫虚拟地址）
    -> 物理地址
```

这里面最关键的是：**物理地址一定是最后那个真正访问 RAM 的地址。**

### 1. 什么是物理地址？

物理地址就是内存条里真实存在的位置。

可以把它理解成你家的门牌号。快递最后必须送到这个门牌号，才能真正把东西送到你家。

不管程序里写了什么地址，中间怎么换算，**最后都得落到物理地址上**。

### 2. 什么是逻辑地址？

逻辑地址是程序“以为自己在访问”的地址。

它更像程序视角下的地址，不是内存条里最终的真实地址。

在分段模型里，逻辑地址通常可以理解成：

```text
段选择信息 + 段内偏移
```

也就是说，程序不是直接说“我要去物理内存 0x1234”，而是说“我要访问某个段里的某个位置”。

### 3. 什么是有效地址？

有效地址这个词，很多场景下你可以把它理解成：

**段内偏移地址**，也就是“在当前段里偏移多少”。

在 x86 语境里，它经常指指令计算出来、还没和段基址相加之前的那个偏移值。

所以它更像：

```text
我在这段里面，要找第几个字节
```

你可以把它看成逻辑地址里的“偏移那一半”。

### 4. 什么是线性地址？

线性地址是把**段基址 + 有效地址**算出来之后得到的地址。

也就是说，分段这一步处理完之后，就得到了线性地址。

```text
线性地址 = 段基址 + 有效地址
```

你可以把它理解成：

“先按段找到这一片区域，再在这一片区域里偏移过去，得到一个连续地址。”

如果系统**没有开启分页**，那这个线性地址就可以直接当成物理地址使用。

### 5. 什么是虚拟地址？

虚拟地址是程序使用的、并不直接对应真实 RAM 的地址。

它是操作系统和硬件一起给程序“包装”出来的一套地址视图。

在很多现代操作系统里，尤其是在开启分页之后，**线性地址通常就会继续作为虚拟地址，再经过页表映射到物理地址**。

所以很多资料里会看到：

```text
线性地址 ≈ 虚拟地址
```

这里不是说两个概念在所有语境下永远完全等价，而是说：

**在开启分页的常见讨论里，线性地址通常就是后面拿去做页表映射的那个虚拟地址。**

### 它们到底怎么串起来？

可以按这个流程记：

```text
程序给出逻辑地址
-> 取出其中的有效地址（偏移）
-> 和段基址组合，得到线性地址
-> 如果开启分页，线性地址再通过页表映射成物理地址
-> 如果没开分页，线性地址可直接作为物理地址
```

换成更口语的话：

```text
先决定“访问哪一段”
再决定“这段里的哪个位置”
然后看系统是否还要做分页转换
最后落到真正的物理内存
```

### 一句话区分

| 名词   | 一句话理解               |
| ---- | ------------------- |
| 物理地址 | 内存里真正的地址            |
| 逻辑地址 | 程序视角下使用的地址          |
| 有效地址 | 段内偏移地址              |
| 线性地址 | 段变换之后得到的连续地址        |
| 虚拟地址 | 程序可见、最终还要映射到物理地址的地址 |

### 为什么会有这么多名字？

因为 CPU 做地址转换不是一步到位，而是分阶段做的。

每做完一步，就会产生一个“中间版本的地址”，于是就有了不同的命名。

这就像寄快递：

- 你先写“公司三楼研发部小王”  
  这更像逻辑地址
- 快递站换成“某园区 A 栋 302”  
  这有点像线性地址
- 最后配送员按真实门牌号送到具体位置  
  这就是物理地址

名字多，不是因为它们神秘，而是因为地址在“翻译过程中”不断换名字。

### ⚠️ 面试易错点

- **物理地址一定是真实内存地址**，这是最终结果。
- **有效地址不是最终地址**，它更偏“段内偏移”。
- **线性地址是分段转换后的结果**，不是分页后的结果。
- **没开分页时，线性地址可以直接等于物理地址**。
- **开了分页后，线性地址通常又被当作虚拟地址，再去查页表映射到物理地址**。
- **逻辑地址、虚拟地址在很多资料里会混着说**，面试时最好主动说明你采用的是哪种区分口径。

### 面试回答模板

物理地址是内存中真实存在的地址，是 CPU 最终访问 RAM 时使用的地址。逻辑地址是程序视角下使用的地址，在分段机制里通常可以理解为某个段加上段内偏移。

其中，有效地址一般指段内偏移；段基址和有效地址相加之后得到线性地址。如果系统没有开启分页，那么线性地址可以直接作为物理地址使用；如果开启了分页，线性地址会继续作为虚拟地址，通过页表映射后，最终得到物理地址。

所以可以简单理解为：逻辑地址和有效地址更偏程序视角，线性地址是分段转换后的中间结果，虚拟地址是分页机制里程序看到的地址，而物理地址才是真正落到内存上的地址。

### 面试简答版

物理地址是真实内存地址；逻辑地址是程序使用的地址；有效地址通常指段内偏移；线性地址是段基址加偏移后的结果；虚拟地址是开启分页后还要继续映射到物理内存的地址。

没开分页时，线性地址可以直接等于物理地址；开了分页后，线性地址通常再经过页表映射，最终得到物理地址。

## 6. 页面替换算法有哪些？

### 先说场景

程序运行时，如果访问的页不在内存里，就会发生缺页。

这时候系统得把新页调进来。但如果物理内存已经满了，就必须先赶走一个旧页，腾出位置。

问题来了：

**到底该赶走谁？**

这就是页面替换算法要解决的问题。

### 大白话解释

可以把内存想成书桌，页想成你手边摊开的资料。

桌子放不下更多资料时，你得先拿走一份旧资料，才能把新资料摆上来。

如果你总是扔错，比如把下一秒就要用的资料扔掉，那你马上又得重新拿回来，效率会很差。

页面替换算法，就是在决定“该把哪份旧资料先拿走”。

### 常见页面替换算法

你截图里列得比较全，这里我帮你按“面试常考”和“补充了解”分成两层记。

第一层，高频一定要会：

- 最优算法 `OPT`
- `FIFO`
- 第二次机会算法
- `Clock`
- `LRU`

第二层，知道思路会很加分：

- `NRU`
- `NFU`
- 老化算法
- 工作集算法
- `WSClock`

你可以先看这个总表：

| 算法 | 核心思路 | 一句话评价 |
|------|----------|------------|
| 最优算法 OPT | 淘汰未来最久不会再用的页 | 理论最好，但实现不了 |
| NRU | 按最近是否访问、是否修改分优先级 | 简单，但比较粗糙 |
| FIFO | 最早进来的页先淘汰 | 最简单，但容易误杀 |
| 第二次机会 | FIFO 基础上给“最近访问过”的页一次机会 | 比 FIFO 更合理 |
| Clock | 第二次机会的高效实现 | 实际里很常见 |
| LRU | 淘汰最近最久没用的页 | 效果好，但精确实现贵 |
| NFU | 按访问次数累计判断冷热 | 比较粗，带历史包袱 |
| 老化算法 | 用衰减方式近似 LRU | 比 NFU 更聪明 |
| 工作集 | 保留一段时间内真正活跃的页 | 思路好，但实现重 |
| WSClock | 工作集 + Clock 的折中版 | 实用性很强 |

### 1）最优算法 OPT

OPT 的想法非常理想化：

**把“未来最久都不会再访问”的那个页换出去。**

这当然是最优的，因为它最不容易导致马上又缺页。

但问题也很明显：

你得提前知道“未来”。

现实系统做不到，所以它一般拿来当理论上的对照标准。

### 2）FIFO

FIFO 很好理解：

谁先进内存，谁先被换出去。

这就像排队出门，最早进来的先离开。

它的优点是实现非常简单，但问题是：

**早进来的页不一定就是不重要的页。**

有些页虽然进来得早，但可能一直在被频繁使用，这时 FIFO 就会把它误杀掉。

### 3）NRU

`NRU` 全称可以理解成“最近未使用”。

它通常会参考两个标记位：

- `R` 位：最近有没有被访问过
- `M` 位：有没有被修改过

系统会把页大致分成几类，优先从“最近没访问过、而且最好也没修改过”的那类里挑页淘汰。

为什么会考虑 `M` 位？

因为被修改过的页如果要换出去，通常还得先回写磁盘，成本更高。

所以 `NRU` 的思路其实是：

**优先淘汰最近不活跃、而且处理成本还低的页。**

它的优点是简单，但它没有 LRU 那么细，只能做一个比较粗的冷热判断。

### 4）第二次机会算法

第二次机会算法可以看作是 `FIFO` 的改良版。

它的想法很朴素：

> 本来 FIFO 是“轮到你了你就走”，  
> 但如果发现这页最近刚被访问过，那就先别急着赶走，再给它一次机会。

也就是说，它在淘汰页之前会先看访问位：

- 如果最近没访问过，换掉
- 如果最近访问过，就把访问位清掉，先放过它，继续看下一个

所以它比纯 `FIFO` 更合理，因为它不再完全只看“谁来得早”。

### 5）Clock

Clock 算法可以理解成 LRU 的“便宜实用版”。

它通常会配合访问位一起工作：

- 最近访问过的页先给一次机会
- 没访问过的页更容易被换出去

所以它不像真正的 LRU 那么贵，但效果又比纯 FIFO 更靠谱。

这也是为什么实际系统里常常会采用它或者它的变种。

你也可以把它理解成：

**第二次机会算法的循环指针版 / 高效实现版。**

### 6）LRU

`LRU` 的想法最符合人的直觉：

**最近最久没有被使用的页，未来短时间内再次使用的概率也更低。**

它的效果通常很好，所以一直很经典。

这个思路很符合局部性原理，所以效果通常比较好。

但问题还是那个问题：

想精确知道“谁是最近最久没用的”，系统得维护很重的访问信息，真正严格实现成本高。

所以很多实际算法，都是在“模仿 LRU”，而不是完美实现它。

### 7）NFU

`NFU` 可以理解成“最不经常使用”。

它会给每个页记一个计数器，每次统计周期里，如果页被访问过，就给它加分。

最后谁分最低，谁更可能被淘汰。

它的问题是：

**太看重历史累计。**

也就是说，一个页只要很久以前火过，分数就可能一直高，但它现在也许早就不活跃了。

所以 `NFU` 虽然看起来像 LRU，但没有“最近”这个味道。

### 8）老化算法

老化算法可以理解成对 `NFU` 的升级。

它不是只会傻傻累计总次数，而是让旧的访问记录慢慢“贬值”。

这就像你判断最近谁最常用，不是看“历史总访问量”，而是看“最近这段时间谁更活跃”。

所以老化算法比 `NFU` 更接近 `LRU`，也更实用。

你可以把它记成：

**老化 = 带时间衰减的 NFU = 更像 LRU 的近似算法**

### 9）工作集算法

工作集算法的核心思路是：

**一个进程在某一段时间里真正频繁使用的那些页，应该尽量留在内存里。**

这组“最近一段时间真正活跃的页”，就叫工作集。

它的思想其实很强，因为它直接围绕“当前活跃页集合”来做决策。

但问题是实现起来比较重：

- 要记录时间窗口
- 要判断哪些页属于当前工作集
- 管理成本比较高

所以它理论上很好，但直接完整实现并不轻松。

### 10）WSClock

`WSClock` 可以看作：

**工作集算法 + Clock 算法的折中版。**

它一边保留了工作集“关注当前活跃页”的思路，一边又借了 `Clock` 的高效扫描方式。

所以它通常被认为是一个：

- 效果不错
- 实现成本又比较能接受

的方案。

如果面试官追问“实际里比较像样的折中方案有哪些”，提 `Clock / 老化 / WSClock` 都很加分。

### 怎么记最核心结论？

你可以先这样记：

```text
OPT：理论最好，但做不到
FIFO：最简单，但不聪明
第二次机会：FIFO 的改良版
Clock：第二次机会的高效实现，实际里常见
LRU：很经典，效果好，但实现贵
NRU / NFU：思路不难，但都比较粗
老化：比 NFU 更像 LRU
工作集：理论漂亮，但实现重
WSClock：工作集和 Clock 的折中版
```

### ⚠️ 面试易错点

- **OPT 只是理论最优，不是实际系统真的能直接用。**
- **FIFO 简单不代表好**，它可能把常用页换掉。
- **LRU 不是“最近最少使用次数”**，而是“最近最久没有使用”。
- **第二次机会和 Clock 是一脉相承的**，Clock 通常可以看作第二次机会的高效实现。
- **NFU 不是 LRU**，它更偏“累计访问次数”，缺少“最近性”。
- **老化算法比 NFU 更接近 LRU**，这个区别最好别混。
- **Clock 常被看作 LRU 的近似实现**，这是实际里很重要的点。
- **工作集和 WSClock 更偏进阶补充**，会说出思想就已经不错了。

### 面试回答模板

页面替换算法是在发生缺页、并且内存没有空闲页框时，用来决定淘汰哪一个旧页的策略。常见算法包括 OPT、NRU、FIFO、第二次机会、Clock、LRU、NFU、老化算法、工作集算法和 WSClock。

其中，OPT 理论最优，但无法预测未来访问，所以只能作为基准；FIFO 最简单，但可能误淘汰仍然常用的页；第二次机会和 Clock 是对 FIFO 的改进；LRU 效果通常较好，但精确实现成本高；NFU 和老化算法是对访问历史的近似统计，其中老化算法更接近 LRU；工作集和 WSClock 更关注当前真正活跃的页，其中 WSClock 是兼顾效果和实现成本的折中方案。

### 面试简答版

常见页面替换算法有 OPT、NRU、FIFO、第二次机会、Clock、LRU、NFU、老化、工作集和 WSClock。

面试重点先抓住这几个：OPT 理论最好但做不到；FIFO 简单但不聪明；第二次机会和 Clock 是 FIFO 的改进；LRU 效果好但实现贵；老化和 WSClock 是比较实用的折中方案。

## 7. 什么是虚拟内存？

### 先说场景

如果程序必须一次性把自己所有内容都装进内存，很多大程序根本跑不起来。

而且有些数据程序虽然“属于自己”，但当前时刻根本用不到，没必要一直占着宝贵的 RAM。

虚拟内存就是为了解决这个问题。

### 大白话解释

可以把虚拟内存理解成一种“看起来比实际更大的内存”。

对程序来说，它会觉得自己拿到了一块很大、很连续的内存空间。

但实际上：

- 真正常用的部分放在物理内存里
- 暂时不用的部分可以放在磁盘上

也就是说，程序看到的是一个“理想化视图”，系统背后再想办法把它映射到真实资源上。

### 它到底是什么？

虚拟内存本质上是一种内存管理机制。

它让进程看到的是自己的虚拟地址空间，而不是直接操作真实物理内存。

当程序访问地址时，系统会通过地址映射，把虚拟地址转换成物理地址。

如果访问的那一页当前不在内存里，还会触发缺页，由系统把它调进来。

### 它解决了什么问题？

虚拟内存主要解决这几个问题：

| 问题       | 虚拟内存怎么解决        |
| -------- | --------------- |
| 物理内存不够大  | 只把当前需要的部分放进 RAM |
| 程序要求连续空间 | 给程序提供连续的虚拟地址空间  |
| 多进程内存隔离  | 每个进程有自己的地址空间    |
| 内存利用率低   | 按需装入，不用的页不常驻内存  |

### 为什么它看起来像“内存变大了”？

因为程序不再受“必须一次性全装进 RAM”这个限制。

它可以只先装入当前用得到的部分，剩下的以后再说。

所以从程序视角看，好像自己拥有了比物理内存更大的可用空间。

### 它和交换空间是什么关系？

交换空间是虚拟内存机制里的一个辅助区域。

你可以把 [[03内存管理#4. 什么是交换空间？|交换空间]] 理解成“内存不够时暂时托底的地方”，而虚拟内存是更大的整体机制。

所以：

- 虚拟内存是整套思想
- 交换空间是其中一个配套手段

### ⚠️ 面试易错点

- **虚拟内存不是“真的创造出了新内存”**，它只是做了地址抽象和按需调度。
- **虚拟内存不等于交换空间**，交换空间只是它的一部分。
- **虚拟内存的核心不只是扩容**，还有地址隔离、连续视图、按需装入这些价值。

### 面试回答模板

虚拟内存是一种内存管理机制，它为每个进程提供独立、连续的虚拟地址空间，而不是让进程直接操作物理内存。程序访问地址时，系统会把虚拟地址映射到物理地址；如果目标页当前不在内存中，还会通过缺页中断按需调入。

虚拟内存的作用不仅是让程序看起来能使用比物理内存更大的空间，还能提高内存利用率，并实现多进程之间的地址隔离。

### 面试简答版

虚拟内存就是给进程提供一套独立、连续的虚拟地址空间，背后再映射到真实物理内存。

它的作用是按需装入、提高内存利用率、实现内存隔离，并让程序看起来能用到比 RAM 更大的空间。

## 8. 虚拟内存的实现方式有哪些？

### 先说场景

既然虚拟内存的核心是：

- 给程序一套虚拟地址空间
- 实际访问时再映射到物理内存

那接下来的问题就是：

**这个映射到底怎么做？**

### 三种常见实现方式

虚拟内存常见的实现方式主要有三种：

| 方式 | 核心思路 |
|------|----------|
| 请求分页 | 按页进行，页面用到时再装入 |
| 请求分段 | 按段进行，段用到时再装入 |
| 请求段页式 | 先分段，再在段内分页，并按需装入 |

### 1）请求分页

这是最常见、也是最容易考的方式。

程序按页划分，只有某一页真正被访问时，才把它调入内存。

它的优点是：

- 粒度小
- 内存利用率高
- 适合现代系统

### 2）请求分段

请求分段是按逻辑段来装入。

比如代码段、数据段、栈段，哪个段需要了，再把哪个段装入。

它更符合程序逻辑结构，但段大小不固定，管理起来没有分页那么规整。

### 3）请求段页式

它相当于把分段和分页结合起来。

先按逻辑模块分段，再在每个段里面继续分页。

这样既保留了“逻辑结构清晰”的特点，又能利用分页带来的灵活分配能力。

### 现代系统里更常见哪种？

实际里最常见的是基于分页的方式，也就是请求分页，或者以分页为核心的段页式管理。

因为分页更适合现代硬件和操作系统实现。

### ⚠️ 面试易错点

- **不是只有请求分页一种**，经典理论上还有请求分段、请求段页式。
- **“请求”两个字的重点是按需装入**，不是一开始全装进去。
- **现代系统里分页方案更常见**，这一点最好顺手带出来。

### 面试回答模板

虚拟内存的实现方式主要有请求分页、请求分段和请求段页式三种。它们的共同点都是按需装入，也就是程序运行时只把当前真正需要的部分调入内存，而不是一开始全部装入。

其中，请求分页最常见，它按页为单位进行管理，粒度小、实现灵活；请求分段更符合程序逻辑结构；请求段页式则结合了分段和分页两者的特点。现代操作系统中更常见的是基于分页的实现方式。

### 面试简答版

虚拟内存常见实现方式有三种：请求分页、请求分段、请求段页式。

核心思想都是按需装入，其中现代系统最常见的是请求分页。
