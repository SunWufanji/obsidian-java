# 进程管理

## 1. 进程和线程的区别？

### 调度角度

把进程想象成**一家公司**，线程就是公司里的**员工**。

- 进程：资源分配的基本单位（公司有办公室、电脑、打印机）
- 线程：CPU 调度的基本单位（员工干活，真正执行任务的是员工）

一家公司可以有多个员工，但资源是分配给公司整体的，不是直接分给个人。

### 切换速度

**员工换班** vs **公司搬迁**

- 线程切换：只换个人干活，桌上的文件不用动 → **快**
- 进程切换：要搬走所有资料、电脑、重新布置办公室 → **慢**

所以线程切换开销远小于进程切换。

### 拥有资源

- **进程**：独立拥有资源（内存空间、文件描述符、I/O 设备等）
- **线程**：不拥有系统资源，但可以访问进程的资源

类比：员工不拥有公司的电脑，但可以使用公司的电脑干活。

这里可以再顺手记两个高频区别：

- **独立性**：进程彼此隔离，一个进程崩了不一定带崩别的进程；线程共享同一进程资源，一个线程乱写内存，可能把整个进程一起带崩
- **通信方式**：进程间通信通常要靠管道、消息队列、共享内存、Socket 等机制；线程间通信更直接，因为天然共享地址空间，但也更容易出现竞争问题

### 系统开销

| 操作 | 进程 | 线程 |
|------|------|------|
| 创建/销毁 | 要分配/回收资源，开销大 | 几乎不分配资源，开销小 |
| 上下文切换 | 要切换内存映射、文件等 | 只切换寄存器和栈，开销小 |

更细一点说，进程更重，主要重在这些地方：

- 要准备独立地址空间
- 要建立和维护进程控制块 `PCB`
- 切换时要动页表、地址映射、内存管理状态

线程更轻，通常只需要维护自己的执行上下文和线程控制信息。

### ⚠️ 面试易错点

- 进程间通信（IPC）需要专门机制（管道、共享内存等），因为地址空间独立
- 同一进程内的线程共享地址空间，可以直接读写变量
- Java 中 `new Thread()` 创建的是线程，不是进程

### 面试回答模板

> 进程和线程主要有四点区别：
> 1. **调度**：进程是资源分配的基本单位，线程是 CPU 调度的基本单位
> 2. **切换**：线程切换比进程切换快得多，因为不需要切换地址空间
> 3. **资源**：进程拥有独立的资源（内存、文件等），线程共享进程的资源
> 4. **开销**：创建、销毁、切换进程的开销都比线程大得多

## 2. 协程与线程的区别？

### 谁来调度

**本质区别**：谁决定什么时候切换执行

- **线程**：操作系统内核调度（**抢占式**）→ OS 说了算，随时可能被切换
- **协程**：用户程序自己调度（**非抢占式**）→ 你说了算，什么时候切换由你决定

类比：
- 线程 = 公共厨房做饭，厨师长随时可能让你换灶台（OS 调度）
- 协程 = 自己家厨房，你做完这道菜再做下一道（用户调度）

### 执行方式

**协程同一时间只有一个在运行**，相当于单线程能力。

```
线程：A和B同时跑（宏观并行）
协程：A暂停 → B跑 → B暂停 → A继续（交替执行）
```

所以协程不会产生线程竞争问题，不需要加锁！

### 数量关系

```
进程 → 可以有多个线程 → 每个线程可以有多个协程
```

协程是比线程更轻量的存在，数量可以很多。

### 状态保留

**协程能记住自己执行到哪了**（挂起时保存状态，恢复时接着执行）

```kotlin
// Kotlin协程示例
suspend fun fetchData() {
    println("开始")
    delay(1000)  // 挂起，不阻塞线程
    println("1秒后继续")  // 恢复时从这里接着
}
```

线程做不到这一点——线程一挂起，栈就没了。

### 核心区别总结

| 特性 | 线程 | 协程 |
|------|------|------|
| 调度者 | OS 内核 | 用户程序 |
| 切换方式 | 抢占式 | 非抢占式（主动让出） |
| 状态保存 | 由 OS 管理 | 自己保存，可恢复 |
| 并发模型 | 真并行 | 交替执行（单线程内） |
| 资源开销 | 大（栈空间 MB 级） | 小（栈空间 KB 级） |
| 是否需要锁 | 需要 | 不需要 |

### ⚠️ 面试易错点

- **协程 ≠ 轻量线程**：虽然都叫并发，但调度机制完全不同
- **协程不是 OS 概念**：是程序自己实现的，OS 根本不知道协程存在
- **协程直接利用线程/线程池执行**，不是替代线程

### 补充：Java 对协程的支持

Java 21 正式引入了**虚拟线程（Virtual Thread）**，本质上就是协程：

```java
// Java 21 虚拟线程
Thread.startVirtualThread(() -> {
    System.out.println("我在虚拟线程里");
});
```

虚拟线程由 JVM 调度，不占 OS 线程资源，百万级并发不是梦。

### 面试回答模板

> 协程和线程有三个核心区别：
> 1. **调度方式**：线程是抢占式调度，由 OS 内核控制；协程是非抢占式，由用户程序自己控制
> 2. **执行模型**：线程可以真正并行；协程在同一时刻只有一个在执行，通过挂起/恢复交替进行
> 3. **资源开销**：协程不占 OS 线程资源，创建/切换开销小，可以轻松创建百万级协程

## 3. 并发和并行有什么区别？

### 一句话区分

- **并发**：**宏观上同时**，微观上轮流（一个人同时做多件事）
- **并行**：**真正的同时**，多个事情一起发生（多个人同时做事）

### 生活类比

**排队买奶茶**：

| 场景 | 对应 |
|------|------|
| 1 个收银员，快速在两个顾客间切换 | **并发** |
| 2 个收银员，同时服务两个顾客 | **并行** |

收银员 = CPU 核心

### 图示理解

```
并发（单核）：   A → B → A → B → A → B （快速切换）

并行（多核）：   A → A → A → A → A → A
                B → B → B → B → B → B （真正同时）
```

### 关键区别

| 维度 | 并发 | 并行 |
|------|------|------|
| 同一时刻 | 只有一个任务在执行 | 多个任务同时执行 |
| 硬件要求 | 单核 CPU 即可 | 需要多核 CPU |
| 实现方式 | 时间片轮转、上下文切换 | 多核同时运行 |
| 目标 | 提高资源利用率 | 提高处理速度 |

### ⚠️ 面试易错点

- **并发不等于并行**：并发是"看起来同时"，并行是"真的同时"
- **多线程程序在单核上是并发，多核上才是并行**
- **并发是手段，并行是能力**：并发解决的是"如何高效利用资源"，并行解决的是"如何更快处理"

### 补充：并发的常见实现

| 机制 | 说明 |
|------|------|
| 多进程 | OS 分配资源，进程间切换 |
| 多线程 | 同一进程内，共享地址空间 |
| 协程 | 用户态，更轻量 |
| 异步 IO | 非阻塞，事件驱动 |

### 面试回答模板

> 并发和并行的区别在于：
> 1. **并发**：宏观上多个任务同时进行，但微观上是交替执行的，单核 CPU 就能实现
> 2. **并行**：同一时刻有多个任务真正在同时执行，需要多核 CPU 支持
> 3. 简单说：并发是"轮流做"，并行是"一起做"

## 4. 进程与线程的切换流程？

### 进程切换：两步走

```
进程切换
  ├── 第1步：切换页表（地址空间）
  │     └── 一旦切换，之前缓存的内存地址全作废
  └── 第2步：切换内核栈和硬件上下文
        └── 保存/恢复寄存器、程序计数器等
```

**类比**：
- 第 1 步 = 搬家（换地址空间）
- 第 2 步 = 把个人物品打包带走（保存现场）

### 线程切换：只做一半

因为**线程共享进程的地址空间**，所以：

| 步骤 | 进程切换 | 线程切换 |
|------|---------|---------|
| 第 1 步：切换页表 | ✅ 需要 | ❌ 不需要 |
| 第 2 步：切换上下文 | ✅ 需要 | ✅ 需要 |

**线程切换 = 只打包物品，不搬家** → 快得多！

### 为什么第 1 步开销大？

切换页表会导致**TLB（快表）失效**：

```
切换前：TLB缓存着进程A的地址映射 → 访问内存很快
切换后：TLB全部作废 → 要重新从内存查页表 → 访问变慢
```

TLB 是 CPU 访问内存的"缓存"，一失效就像导航软件突然清空了所有收藏地址。

### 线程共享什么，不共享什么

| 共享 | 不共享 |
|------|--------|
| 地址空间（代码段、数据段） | 栈（每个线程独立） |
| 全局变量 | 寄存器值 |
| 文件描述符 | 程序计数器（PC） |
| 进程 ID | 线程 ID |

如果你想再展开一点，可以这样记：

- **线程共享的**：虚拟地址空间、堆、全局变量、文件描述符、信号处理方式、环境变量
- **线程独有的**：线程栈、寄存器上下文、程序计数器、线程局部存储

所以线程之间通信会更方便，但同步问题也更明显，因为大家摸的是同一份资源。

### ⚠️ 面试易错点

- **切换的"上下文"是什么？** → 寄存器、程序计数器、栈指针等 CPU 状态
- **切换不等于创建**：切换是已有线程/进程间的跳转，创建是分配资源
- **用户态线程切换不需要陷入内核**：协程切换开销更小

### 面试回答模板

> 进程切换分两步：
> 1. **切换页表**：使用新的地址空间，会导致缓存的内存地址作废
> 2. **切换内核栈和硬件上下文**：保存/恢复寄存器等 CPU 状态
>
> 线程切换时，因为共享进程的地址空间，**第 1 步不需要做**，只需做第 2 步。所以线程切换比进程切换快得多。

如果继续细分，线程上下文切换主要是在做这些事：

- 保存当前线程的寄存器状态
- 保存程序计数器和栈指针
- 保存线程状态和调度相关信息
- 从下一个线程的控制块里恢复这些状态
- 把 CPU 执行权切给新线程

它比进程切换轻，核心就在于**不用换地址空间，也不用切页表**。

如果你看更底层一点，它本质上就是在线程控制块里来回保存和恢复这些信息：

- `PC`：下一条要执行的指令位置
- `SP`：当前线程栈顶位置
- 通用寄存器和标志寄存器
- 线程状态、优先级等调度信息

所以线程上下文切换的重点，是“换执行现场”，不是“换整套内存世界”。

## 5. 为什么虚拟地址空间切换会比较耗时？

### 核心原因：TLB 失效

```
进程运行时：
  虚拟地址 → 查TLB（很快！命中） → 物理地址

进程切换后：
  虚拟地址 → 查TLB（全空了！） → 查页表（很慢） → 物理地址
```

**类比**：
- TLB = 手机导航的"收藏夹"
- 进程切换 = 换个城市，收藏夹里的地址全没用了
- 要重新搜索地址 = 重新查页表

### 为什么慢？

| 步骤 | 耗时 |
|------|------|
| TLB 命中 | 1 个时钟周期 |
| TLB 未命中，查页表 | 几十到几百个时钟周期 |

差了**几十倍到上百倍**！

### 面试回答模板

> 虚拟地址空间切换耗时的原因是**TLB 失效**：
> 1. 进程有自己的页表，TLB 缓存了常用的地址映射
> 2. 进程切换后，TLB 内容全部作废
> 3. 后续每次访问内存都要重新查页表，速度变慢
> 4. 线程切换不会导致 TLB 失效，因为线程共享地址空间

## 6. 进程间通信方式有哪些？

### 管道（Pipe）

**生活类比**：水管，单向流动

```
进程A → [写入] 管道 → [读出] 进程B
```

| 类型 | 特点 |
|------|------|
| 匿名管道 | 只能在父子进程间用 |
| 命名管道 | 可以在任意进程间用，像文件一样 |

**限制**：半双工（不能同时收发）、容量有限

### 信号（Signal）

**类比**：手机推送通知，随时可能来

```bash
kill -9 进程ID  # 发送SIGKILL信号
```

常用信号：

| 信号 | 作用 |
|------|------|
| SIGKILL | 强制终止 |
| SIGTERM | 请求终止（可捕获） |
| SIGINT | Ctrl+C 中断 |
| SIGSEGV | 段错误 |

### 消息队列（Message Queue）

**类比**：快递柜，消息排队等你取

- 消息有类型，可以按类型读取
- 比管道更灵活，支持格式化数据
- 缺点：容量有限，需要内核维护

### 共享内存（Shared Memory）

**类比**：两个人共用一张桌子，直接看对方的文件

```
进程A ←──→ [共享内存区域] ←──→ 进程B
```

**最快的 IPC 方式**！但需要自己处理同步问题（加锁）。

### 信号量（Semaphore）

**类比**：红绿灯，控制谁能通行

- 本质是个计数器
- 用于**同步**，不是传递数据
- P 操作（等待/减 1），V 操作（通知/加 1）

### Socket

**类比**：打电话，可以跨网络

- 不仅能本机进程通信，还能跨机器通信
- Web 服务、微服务通信的基础

### 总结对比

| 方式 | 速度 | 复杂度 | 适用场景 |
|------|------|--------|---------|
| 管道 | 慢 | 低 | 简单父子进程 |
| 信号 | 快 | 中 | 通知/控制 |
| 消息队列 | 中 | 中 | 异步通信 |
| 共享内存 | 最快 | 高 | 大量数据传输 |
| 信号量 | - | 高 | 同步控制 |
| Socket | 慢 | 中 | 跨网络通信 |

### ⚠️ 面试易错点

- **管道和消息队列的区别**：管道是字节流，消息队列有消息边界
- **共享内存需要同步**：多进程同时写会有竞争问题
- **Socket 不只用于网络**：本地 Unix Socket 也可以用于本机进程通信

### 面试回答模板

> 进程间通信主要有 6 种方式：
> 1. **管道**：单向流动，简单但受限
> 2. **信号**：异步通知，常用于进程控制
> 3. **消息队列**：内核维护的消息链表，灵活
> 4. **共享内存**：最快，但需要自己加锁
> 5. **信号量**：用于同步，控制并发访问
> 6. **Socket**：支持跨网络，最通用

## 7. 进程间同步的方式有哪些？

### 信号量（Semaphore）

**类比**：停车场的剩余车位显示屏

```
信号量 = 3（还有3个车位）
车开走 → +1（V操作）
有车进来 → -1（P操作）
```

- **P 操作**：申请资源，资源不足则阻塞
- **V 操作**：释放资源，唤醒等待的进程

### 管程（Monitor）

**类比**：银行叫号系统，一次只叫一个号

- 一个时刻只有一个进程能在管程内执行
- 其他进程排队等待
- Java 的 `synchronized` 本质就是管程

### 消息传递

**类比**：微信聊天，发消息同步信息

```
进程A → send(消息) → 消息队列 → receive(消息) → 进程B
```

分布式系统常用，不同机器上的进程也能通信。

### 屏障（Barrier）

**类比**：等所有人到齐再出发

```
线程1 → [屏障] → 等待...
线程2 → [屏障] → 等待...
线程3 → [屏障] → 全部到齐！一起继续
```

并行计算中常用，保证阶段同步。

### 条件变量

**类比**：快递到了会通知你取

- 等待条件满足时阻塞
- 条件满足时被唤醒
- 通常配合互斥锁使用

## 8. 线程同步的方式有哪些？

### 互斥锁（Mutex）

**类比**：厕所钥匙，一次只能一个人用

```java
synchronized(lock) {
    // 临界区，同一时间只有一个线程能执行
}
```

### 信号量（Semaphore）

**类比**：图书馆有 3 本相同的书，可以同时 3 人借阅

```java
Semaphore sem = new Semaphore(3);  // 允许3个并发
sem.acquire();  // 借书
// 使用资源
sem.release();  // 还书
```

### 条件变量

**类比**：排队取号，叫到你号了再办

```java
lock.lock();
while (!条件满足) {
    condition.await();  // 排队等待
}
// 条件满足，继续执行
lock.unlock();
```

### 读写锁

**类比**：图书馆借书规则

| 场景 | 规则 |
|------|------|
| 读 | 多人同时借同一本书 ✅ |
| 写 | 一个人在修改，其他人不能动 ❌ |

适合**读多写少**的场景。

### 原子操作

**类比**：ATM 取款，扣款和出钞是一体的，不会中间被打断

```java
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();  // 原子自增，线程安全
```

### 对比总结

| 方式 | 粒度 | 性能 | 适用场景 |
|------|------|------|---------|
| 互斥锁 | 粗 | 一般 | 临界区保护 |
| 信号量 | 可调 | 好 | 控制并发数量 |
| 读写锁 | 细 | 好 | 读多写少 |
| 原子操作 | 最细 | 最好 | 简单变量操作 |
| 条件变量 | - | 好 | 复杂条件等待 |

### ⚠️ 面试易错点

- **互斥锁 vs 信号量**：互斥锁是二值信号量（只能 0/1），信号量可以是任意值
- **死锁的四个条件**：互斥、持有等待、不可抢占、循环等待
- **Java 中**：`synchronized`是互斥锁，`ReentrantLock`功能更强，`CountDownLatch`是屏障

### 面试回答模板

> 进程同步主要有 5 种方式：
> 1. **信号量**：计数器控制资源访问，P/V 操作
> 2. **管程**：同一时刻只有一个进程执行
> 3. **消息传递**：通过消息队列通信
> 4. **屏障**：等待所有进程到齐再继续
> 5. **条件变量**：等待条件满足再执行
>
> 线程同步更常用：互斥锁、信号量、条件变量、读写锁、原子操作。

## 9. 线程的分类？

### 两种线程

| 类型    | 用户级线程（ULT） | 内核级线程（KLT） |
| ----- | ---------- | ---------- |
| 管理者   | 用户程序（线程库）  | OS 内核       |
| 创建/销毁 | 用户态完成      | 需要系统调用     |
| 调度    | 程序自己调度     | OS 调度       |
| 感知者   | OS 不知道它的存在  | OS 完全感知     |

### 生活类比

**ULT = 你自己记的待办清单**

- 你自己决定先做哪件事
- 老板（OS）不知道你在干嘛
- 你挂了（线程库出问题），OS 还以为你在工作

**KLT = 公司正式安排的工作任务**

- 老板（OS）给你排任务
- 你挂了，老板立刻知道

### 关键区别

```
ULT：用户程序 → 线程库 → 一个OS线程内跑多个ULT
KLT：用户程序 → 每个线程映射到一个OS线程
```

| 特性 | ULT | KLT |
|------|-----|-----|
| OS 是否感知 | ❌ 不知道 | ✅ 知道 |
| 能否利用多核 | ❌ 不能（一个 OS 线程） | ✅ 能 |
| 创建开销 | 小 | 大 |
| 切换开销 | 小 | 大 |

### 混合模型

现代语言通常用**混合模型**：ULT + KLT

```
Java线程 → ULT → 映射到 → KLT（OS线程）
         （JVM管理）      （OS调度）
```

- Java 8 及之前：1 个 Java 线程 = 1 个 OS 线程
- Java 21 虚拟线程：多个虚拟线程共享少量 OS 线程

### ⚠️ 面试易错点

- **ULT 无法利用多核**：因为 OS 只看到一个线程，无法分配到多个 CPU 核心
- **Go 语言的 goroutine 是 ULT**：由 Go 运行时调度，不是 OS 调度
- **Java 线程是 KLT**：`new Thread()` 创建的是 OS 级线程（Java 21 之前）

### 面试回答模板

> 线程分为两种：
> 1. **用户级线程（ULT）**：由用户程序管理，OS 不知道，切换开销小，但无法利用多核
> 2. **内核级线程（KLT）**：由 OS 管理，切换开销大，但能真正并行
>
> 现代语言通常采用混合模型，如 Java 的线程是 KLT，Go 的 goroutine 是 ULT。

## 10. 什么是临界区，如何解决冲突？

### 核心概念

| 概念 | 含义 | 类比 |
|------|------|------|
| **临界资源** | 一次只能一个人用的资源 | 厕所、打印机、共享变量 |
| **临界区** | 访问临界资源的代码段 | "上厕所"这段代码 |

### 冲突场景

```
进程A：我要用打印机！
进程B：我也要！
→ 同时用 → 打印混乱
```

### 解决冲突的三个原则

| 原则 | 含义 | 类比 |
|------|------|------|
| **互斥进入** | 一次只允许一个进程在临界区 | 厕所一次进一个人 |
| **有限等待** | 进入后必须在有限时间内退出 | 不能霸占厕所不走 |
| **让出 CPU** | 进不去时不能空转等待 | 等厕所时别堵门口干站着 |

### 实现方式

```java
// 方式1：互斥锁
synchronized(lock) {
    // 临界区代码
}

// 方式2：信号量
semaphore.wait();   // 进入
// 临界区代码
semaphore.signal(); // 退出

// 方式3：自旋锁（忙等）
while (!tryLock()) {
    // 自旋等待（违反原则3，不推荐）
}
```

### ⚠️ 面试易错点

- **临界区是代码段，不是资源本身**：资源是打印机，临界区是"使用打印机的那段代码"
- **忙等（自旋）vs 阻塞**：忙等浪费 CPU，阻塞让出 CPU 更高效
- **Java 的 synchronized**：底层用 Monitor 实现，本质是管程

### 面试回答模板

> 临界区是访问临界资源的代码段，临界资源是一次只能被一个进程使用的资源。
>
> 解决冲突的三个原则：
> 1. **互斥进入**：一次只允许一个进程在临界区
> 2. **有限等待**：进入后必须在有限时间内退出
> 3. **让出 CPU**：进不去时阻塞等待，不能忙等浪费资源

## 11. 什么是死锁？死锁产生的条件？

### 死锁是什么

**类比**：两个人在窄巷子里相遇，都不肯让路

```
A：你先让我过！
B：你先让我过！
A：你不让我我就不走！
B：你不让我我也不走！
→ 永远卡住
```

**定义**：两个或多个进程互相等待对方持有的资源，谁也无法继续执行。

### 四个必要条件

**缺一不可**，破坏任何一个条件就能防止死锁：

| 条件 | 含义 | 类比 |
|------|------|------|
| **互斥** | 资源一次只能一个人用 | 厕所一次只能进一个人 |
| **请求与保持** | 拿着一个资源，还要去拿别的 | 拿着饭碗去抢菜 |
| **不可剥夺** | 别人不能抢你手里的资源 | 你不能抢别人正在用的碗 |
| **循环等待** | A 等 B，B 等 A | 互相盯着对方的饭 |

### 代码示例

```java
// 死锁示例
Object lockA = new Object();
Object lockB = new Object();

// 线程1：先拿A，再拿B
new Thread(() -> {
    synchronized(lockA) {
        Thread.sleep(100);
        synchronized(lockB) { /* ... */ }
    }
}).start();

// 线程2：先拿B，再拿A（顺序相反）
new Thread(() -> {
    synchronized(lockB) {
        Thread.sleep(100);
        synchronized(lockA) { /* ... */ }
    }
}).start();
// 两个线程可能互相等待对方的锁 → 死锁
```

### 如何防止死锁

**破坏四个条件之一**：

| 条件 | 破坏方法 |
|------|---------|
| 互斥 | 资源设为共享（有些资源做不到） |
| 请求与保持 | 一次性申请所有资源 |
| 不剥夺 | 允许 OS 强制回收资源 |
| **循环等待** | **按固定顺序申请资源**（最常用） |

### ⚠️ 面试易错点

- **死锁 vs 活锁**：活锁是都在动但没进展（让路让到同一边），死锁是完全不动
- **死锁 vs 饥饿**：饥饿是某个进程永远抢不到资源，不是互相等待
- **银行家算法**：用于**避免**死锁，不是检测死锁

### 面试回答模板

> 死锁是指两个或多个进程互相等待对方持有的资源，导致都无法继续执行。
>
> 死锁产生的四个必要条件：
> 1. **互斥**：资源一次只能被一个进程使用
> 2. **请求与保持**：持有资源的同时请求新资源
> 3. **不可剥夺**：不能强制回收已分配的资源
> 4. **循环等待**：进程间形成环形等待链
>
> 防止死锁的关键是破坏其中一个条件，最常用的是按固定顺序申请资源。

## 12. 如何处理死锁问题？

### 五种处理策略

| 策略     | 核心思想      | 类比              |
| ------ | --------- | --------------- |
| **预防** | 提前破坏条件    | 门口检查，不让冲突的人同时进去 |
| **避免** | 运行时判断是否安全 | 进门前问"你会打架吗"     |
| **检测** | 允许发生，定期检查 | 有摄像头，出事再查       |
| **解除** | 发生了就解决    | 打架了就拉架          |
| **鸵鸟** | 装作没看见     | 假装问题不存在         |

### 死锁预防（静态）

**破坏四个必要条件之一**：

| 条件       | 破坏方法        | 缺点           |
| -------- | ----------- | ------------ |
| 互斥       | 共享资源        | 不现实（打印机不能共享） |
| 请求与保持    | 一次性申请所有资源   | 浪费，降低并发      |
| 不剥夺      | 允许强制回收      | 实现复杂         |
| **循环等待** | **按顺序申请资源** | **最常用**      |

### 死锁避免（动态）

**银行家算法**：分配资源前检查"给了会不会不安全"

```
系统状态安全 = 存在一个安全序列，能让所有进程都完成
不安全 ≠ 必死锁，但有风险
```

### 死锁检测

**资源分配图检测环路**：

```
有环 → 可能死锁
无环 → 一定没死锁
```

检测到死锁后，可以终止进程或抢占资源。

### 死锁解除

| 方法 | 说明 |
|------|------|
| 终止进程 | 杀掉一个或多个死锁进程 |
| 资源抢占 | 强制回收资源再分配 |

### 鸵鸟策略

**假装没看见**（Unix、Linux、Windows 都是这么干的）

原因：解决死锁的代价 > 死锁发生的影响

### 面试回答模板

> 处理死锁有五种策略：
> 1. **预防**：破坏四个必要条件之一，最常用按顺序申请资源
> 2. **避免**：银行家算法，分配前检查安全性
> 3. **检测**：允许死锁发生，定期检测资源分配图的环路
> 4. **解除**：终止进程或资源抢占
> 5. **鸵鸟策略**：忽略问题，实际操作系统常用

## 13. 进程调度策略有哪几种？

### 先来先服务（FCFS）

**类比**：排队买奶茶，先到先得

```
A(10ms) → B(2ms) → C(1ms)
B和C要等A做完，平均等待时间很长
```

✅ 简单公平
❌ 短作业等太久（**护航效应**）

### 短作业优先（SJF）

**类比**：快餐店，做起来快的先做

```
B(2ms) → C(1ms) → A(10ms)
平均等待时间最短
```

✅ 平均等待时间最短
❌ 长作业可能**饿死**

### 最短剩余时间优先

**SJF 的抢占版**：新来的任务如果比当前任务剩余时间短，直接插队

### 时间片轮转

**类比**：每人限时发言，时间到就换人

```
A(时间片) → B(时间片) → C(时间片) → A(时间片)...
```

| 时间片大小 | 效果 |
|-----------|------|
| 太小 | 切换太频繁，开销大 |
| 太大 | 退化成 FCFS，响应慢 |

### 优先级调度

**类比**：VIP 通道

⚠️ 低优先级可能饿死 → **解决**：随等待时间增加优先级

### 对比总结

| 算法 | 公平性 | 响应时间 | 吞吐量 | 复杂度 |
|------|--------|---------|--------|--------|
| FCFS | ✅ 公平 | 慢 | 一般 | 低 |
| SJF | ❌ 不公平 | 快 | 高 | 中 |
| 时间片轮转 | ✅ 公平 | 快 | 一般 | 低 |
| 优先级调度 | 看实现 | 看实现 | 看实现 | 中 |

### 面试回答模板

> 进程调度策略主要有 5 种：
> 1. **先来先服务**：按到达顺序，简单但短作业等太久
> 2. **短作业优先**：执行时间短的先做，平均等待最短但长作业可能饿死
> 3. **最短剩余时间**：SJF 的抢占版
> 4. **时间片轮转**：每人一段 CPU 时间，最常用
> 5. **优先级调度**：按优先级分配，需要防止低优先级饿死

## 14. 进程有哪些状态？

### 五种状态

```
创建 → 就绪 ⇄ 运行 → 终止
            ↗     ↘
          阻塞
```

| 状态 | 含义 | 类比 |
|------|------|------|
| **创建** | 进程刚被创建 | 员工刚入职 |
| **就绪** | 准备好了，等 CPU | 在办公室等着被安排活 |
| **运行** | 正在 CPU 上执行 | 正在干活 |
| **阻塞** | 等 I/O 或其他事件 | 等快递，没法干活 |
| **终止** | 执行完毕或被杀 | 离职 |

### 状态转换

| 转换 | 触发条件 |
|------|---------|
| 就绪→运行 | 被调度选中 |
| 运行→就绪 | 时间片用完 |
| 运行→阻塞 | 请求 I/O 等待 |
| 阻塞→就绪 | I/O 完成 |

### ⚠️ 面试易错点

- **阻塞→运行不可能直接转换**：必须先到就绪状态
- **就绪→阻塞也不可能**：就绪状态的进程没有在执行，不会发起 I/O 请求

### 面试回答模板

> 进程有 5 种状态：
> 1. **创建**：进程刚被创建
> 2. **就绪**：已准备好，等待 CPU 调度
> 3. **运行**：正在 CPU 上执行
> 4. **阻塞**：等待 I/O 等事件，无法执行
> 5. **终止**：执行完毕或被终止

## 15. 进程和程序的根本区别？

### 核心区别

| 维度 | 程序 | 进程 |
|------|------|------|
| 本质 | 静态的代码文件 | 动态的执行过程 |
| 存储位置 | 磁盘 | 内存 |
| 状态 | 不变的 | 一直在变（创建→运行→终止） |
| 生命周期 | 永久存在 | 用完就销毁 |
| 数量关系 | 一个程序可以对应多个进程 | 每个进程都对应一个程序 |

### 生活类比

| 程序 | 进程 |
|------|------|
| 菜谱 | 做菜的过程 |
| 建筑图纸 | 盖房子的过程 |
| 乐谱 | 演奏音乐会 |
| 剧本 | 拍电影 |

**同一本菜谱可以同时做 10 份** → 同一程序可以创建多个进程

### 本质差异

```
程序 = 程序员写的代码（.exe、.class文件）

进程 = 程序被加载到内存后：
        分配了资源（内存、文件描述符）
        有了PCB（进程控制块）
        正在被CPU执行或等待执行
```

### ⚠️ 面试易错点

- **程序不等于进程**：程序是死的代码，进程是活的执行
- **进程有 PCB，程序没有**：PCB 是进程的"身份证"，包含 PID、状态、资源等
- **程序可永久保存，进程用完就没了**

### 面试回答模板

> 进程和程序的根本区别在于：
> 1. **程序是静态的**：存储在磁盘上的代码文件
> 2. **进程是动态的**：程序被加载到内存后的一次执行实例
>
> 简单说：程序是菜谱，进程是做菜的过程。一个程序可以对应多个进程。

## 16. 什么操作会导致创建新进程？

### 常见触发场景

| 操作 | 说明 | 例子 |
|------|------|------|
| **运行程序** | 双击 exe、命令行启动 | 打开 Chrome |
| **fork()** | Linux 系统调用，复制当前进程 | `fork()` |
| **exec()** | 加载新程序替换当前进程 | `exec("ls")` |
| **Shell 命令** | 命令行执行命令 | `ls -la` |
| **Java ProcessBuilder** | Java 启动外部程序 | `Runtime.exec()` |

### Linux 下创建进程

```bash
# 方式1：fork（复制当前进程）
pid = fork();  // 返回两次：父进程返回子进程ID，子进程返回0

# 方式2：exec（加载新程序）
exec("/bin/ls");  // 当前进程被替换为ls程序

# 方式3：Shell自动创建
ls -la  # Shell会fork一个子进程来执行ls
```

### Windows 下创建进程

```c
// CreateProcess是Windows创建进程的API
CreateProcess(
    "notepad.exe",  // 要运行的程序
    // ...
);
```

### Java 中创建进程

```java
// 方式1：Runtime.exec()
Process p = Runtime.getRuntime().exec("notepad.exe");

// 方式2：ProcessBuilder（推荐）
ProcessBuilder pb = new ProcessBuilder("notepad.exe");
Process p = pb.start();
```

⚠️ **注意**：Java 的`new Thread()`创建的是**线程**，不是进程！

### 总结

| 层级 | 创建方式 |
|------|---------|
| **OS 层** | fork、exec、CreateProcess |
| **Shell 层** | 执行命令时自动创建 |
| **应用层** | Java Runtime.exec()、Python subprocess |

### ⚠️ 面试易错点

- **fork 创建的是副本**：子进程复制父进程的资源
- **exec 会替换进程**：不是创建新进程，而是让当前进程运行新程序
- **Java 线程≠进程**：`new Thread()`是创建线程，不是进程

### 面试回答模板

> 创建新进程的常见操作：
> 1. **运行程序**：双击 exe、命令行启动
> 2. **fork()**：Linux 系统调用，复制当前进程
> 3. **exec()**：加载新程序替换当前进程
> 4. **Shell 命令**：执行命令时自动创建子进程
> 5. **Java ProcessBuilder**：启动外部程序
>
> 注意：Java 的`new Thread()`创建的是线程，不是进程。

## 17. 为什么需要线程？

### 先说场景

如果一个程序里只有一条执行线，那它做事就只能一件一件排队来。

这在简单场景下没问题，但一旦程序既要处理用户输入，又要做计算，还要等网络或磁盘，单线程就很容易把自己卡住。

所以线程存在的核心意义就是：

**让同一个进程里可以同时推进多件事。**

### 大白话解释

可以把进程想成一家餐馆，线程就是店里的员工。

如果餐馆里只有一个员工，那他既要点单、又要做菜、又要收银，任何一步卡住，整家店都慢下来。

如果有多个员工，就可以分工：

- 一个人收银
- 一个人做菜
- 一个人打包

程序里用线程，思路也类似。

### 为什么需要线程？

线程最主要的价值，通常有这几个：

1. **提高并发能力**

同一个进程里可以同时推进多个任务。

2. **提高响应性**

比如 GUI 程序里，一个线程处理界面，另一个线程做后台任务，这样后台再慢，界面也不至于完全卡死。

3. **更好利用多核 CPU**

多个线程可以被调度到不同核心上执行，提高吞吐量。

4. **等待 IO 时别把整个程序拖住**

一个线程在等网络、磁盘、数据库时，别的线程还能继续做别的事。

5. **同进程内共享资源方便**

线程共享地址空间，线程间交换数据通常比进程间通信更轻量。

6. **编程模型更灵活**

很多场景下，把任务拆成多个线程，比硬塞进一个大循环里更自然，比如前台响应、后台处理、异步等待可以分开推进。

### ⚠️ 面试易错点

- **线程不是为了“看起来高级”而存在**，核心是并发、响应性和资源利用率。
- **多线程不等于一定更快**，它也会带来同步、调度和切换开销。
- **线程共享资源是优势，也是风险来源**。

### 面试回答模板

之所以需要线程，是因为很多程序在运行时需要同时推进多个任务，比如处理用户请求、执行后台计算、等待网络或磁盘 IO。线程允许同一个进程内并发执行多条执行流，从而提高程序的响应性、资源利用率和整体吞吐量。

另外，线程共享进程资源，所以同进程内的数据共享和协作通常比多进程更轻量，这也是线程被广泛使用的重要原因。

### 面试简答版

需要线程，是因为它能让同一个进程同时处理多件事，提高响应性、并发能力和多核利用率。

## 18. 多线程是不是越多越好，太多会有什么问题？

### 先给结论

不是。

线程不是越多越好，线程数一旦过多，收益会下降，甚至反过来拖慢程序。

### 为什么太多线程会出问题？

1. **上下文切换开销变大**

线程多了，CPU 要更频繁地在不同线程之间切来切去。

切换本身就要保存和恢复上下文，这些都不是免费操作。

2. **内存开销会增加**

每个线程都要有自己的栈和控制信息，线程太多会吃掉不少内存。

3. **资源竞争更激烈**

线程共享同一进程里的资源，线程一多，锁竞争、数据竞争、死锁风险都会变大。

4. **调试和维护复杂度上升**

线程越多，执行顺序越不确定，问题越难复现，排查成本也会上升。

5. **缓存命中率可能变差**

线程太多时，CPU Cache 里的热点数据可能被更频繁地挤掉，反而拖慢访问速度。

### 那合理线程数怎么想？

通常要看任务类型：

- **CPU 密集型**：线程数通常接近 CPU 核数就够了
- **IO 密集型**：线程数可以适当多一些，因为很多线程会在等待 IO

所以没有一个“永远正确”的固定数字，关键要看业务场景。

### 怎么优化？

最常见的办法是：

- 用线程池管理线程
- 不要无脑为每个任务都单独开线程
- 根据 CPU / IO 特性调线程数量

线程池特别重要，因为它能避免线程的频繁创建和销毁，把这部分额外成本降下来。

### ⚠️ 面试易错点

- **线程多不等于吞吐量一定高**。
- **线程问题不只在切换开销，还在锁竞争、内存占用、调试复杂度。**
- **合理线程数和任务类型强相关**。

### 面试回答模板

多线程不是越多越好。线程太多会带来上下文切换开销、内存占用增加、锁竞争加剧、调试复杂度提升等问题，严重时反而会降低程序性能。

一般来说，CPU 密集型任务的线程数更适合接近 CPU 核数，而 IO 密集型任务可以适当增加线程数。实际开发中通常会结合线程池和任务类型来控制线程规模。

### 面试简答版

线程不是越多越好。太多线程会增加切换开销、内存消耗和资源竞争，反而拖慢程序。

## 19. 什么时候用单线程，什么时候用多线程？

### 先说结论

这个问题本质上不是“哪个更高级”，而是：

**你的任务到底更适合简单执行，还是更适合并发推进。**

### 更适合单线程的场景

单线程通常更适合：

- 逻辑简单、任务量不大
- 不需要明显并发
- 状态很多、并发反而容易出错
- 对时序要求很强，希望流程简单可控

比如：

- 简单脚本
- 小工具程序
- 顺序数据处理

### 更适合多线程的场景

多线程通常更适合：

- 需要同时处理多个任务
- 需要提高响应性
- 存在大量 IO 等待
- 希望更好利用多核 CPU
- 希望把前台交互和后台任务拆开

比如：

- Web 服务
- 图形界面程序
- 后台任务系统
- 网络爬虫、文件下载器

### 怎么抓核心判断标准？

你可以主要看这几个维度：

| 维度 | 更偏单线程 | 更偏多线程 |
|------|-----------|-----------|
| 任务关系 | 强顺序 | 可并发 |
| 响应要求 | 一般 | 高 |
| IO 等待 | 少 | 多 |
| CPU 利用 | 单核也够 | 希望更好利用多核 |
| 复杂度容忍度 | 低 | 可以接受更复杂设计 |

### ⚠️ 面试易错点

- **多线程不是默认更先进**，很多时候单线程反而更稳定、更好维护。
- **是否使用多线程，要看任务特征，不是看流行程度。**
- **多线程收益和复杂度是一起到来的**。

### 面试回答模板

选择单线程还是多线程，主要取决于任务特性和性能目标。单线程更适合逻辑简单、任务顺序性强、并发需求不高的场景，优点是实现和调试都更简单。多线程更适合需要同时处理多个任务、存在大量 IO 等待、需要提高响应性或利用多核 CPU 的场景。

因此，这个问题的关键不是“线程越多越先进”，而是根据任务是否适合并发来选择合适模型。

### 面试简答版

顺序强、逻辑简单、并发需求低时更适合单线程；需要并发、响应快、IO 多或想利用多核时更适合多线程。
