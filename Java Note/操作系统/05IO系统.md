# IO系统

## 1. 讲一讲 IO 多路复用？

### 先说场景

服务器同时面对很多连接时，最大的问题不是“单个连接读得够不够快”，而是：

**我怎么用尽量少的线程，盯住很多连接，谁有数据我就先处理谁。**

这就是 IO 多路复用想解决的问题。

### 大白话解释

可以把它想成一个服务员盯很多桌客人。

如果一桌配一个服务员，成本很高。

更聪明的办法是：一个服务员同时看很多桌，谁举手了就先去服务谁。

IO 多路复用就是这个思路：

**用一个线程 / 进程，同时监听多个文件描述符，哪个就绪了就处理哪个。**

### 它到底是什么？

IO 多路复用本质上是一种“事件通知机制”。

程序把多个文件描述符交给内核，内核帮它盯着：

- 哪个可读
- 哪个可写
- 哪个出异常

一旦有事件发生，再通知程序来处理。

常见实现就是：

- `select`
- `poll`
- `epoll`

### 为什么它叫“多路复用”？

因为它把“等待多个 IO”的工作，复用到一个或少量线程上完成了。

你不用为每个连接都开一个线程去傻等。

### 它适合什么场景？

典型场景就是高并发网络服务器，比如：

- 聊天服务
- 网关
- Redis / Nginx 这类高并发场景

它的优势不是让单个连接更快，而是**让系统用更少资源处理更多连接**。

### ⚠️ 面试易错点

- **IO 多路复用不是“同时真正并行处理多个 IO”**，而是高效地等待和分发多个 IO 事件。
- **它的优势主要是连接数多时系统开销更小**，不是单个请求延迟一定更低。
- **`select`、`poll`、`epoll` 都属于 IO 多路复用机制**。

### 面试回答模板

IO 多路复用是一种 IO 事件监听机制，它允许一个线程或进程同时监听多个文件描述符，把“等待多个 IO 是否就绪”的工作交给内核完成。当某个文件描述符可读或可写时，内核再通知应用程序处理。

它的核心价值在于用更少的线程处理更多连接，减少线程切换和资源开销，典型实现包括 `select`、`poll` 和 `epoll`。

### 面试简答版

IO 多路复用就是一个线程同时监听多个 IO，哪个就绪了就处理哪个，常见实现是 `select`、`poll`、`epoll`。

## 2. Unix 常见的 IO 模型有哪些？

### 先说场景

一次 IO 读数据，通常会经历两件事：

1. 等数据准备好
2. 把数据从内核拷贝到用户空间

不同 IO 模型，本质上就是在解决：

**这两个阶段，进程到底要不要等、怎么等、由谁来通知。**

### 常见五种模型
  
[2021-Java后端工程师必会知识点-(操作系统)面试指南系列，很多情况下不会去深挖细节，是小六六以被面试者的角色去回 - 掘金](https://juejin.cn/post/6942686874301857800#heading-13)
Unix 常见 IO 模型一般有五种：

| 模型 | 核心特点 |
|------|----------|
| 阻塞 IO | 调用后一直等，直到数据准备好并拷贝完成 |
| 非阻塞 IO | 调用后如果没准备好，立刻返回，再反复询问 |
| IO 多路复用 | 先统一等多个描述符就绪，再去读写 |
| 信号驱动 IO | 数据就绪后，内核用信号通知进程 |
| 异步 IO | 连“拷贝到用户空间”都由内核完成，完成后再通知 |

### 怎么理解它们的区别？

最简单的抓法就是看：

- 进程是不是一直自己等
- 内核有没有主动通知
- 数据拷贝这一步是谁负责

### 重点怎么记？

- **阻塞 IO**：最省脑子，但会卡住线程
- **非阻塞 IO**：线程不傻等，但会频繁轮询
- **IO 多路复用**：适合很多连接一起等
- **信号驱动 IO**：靠信号通知
- **异步 IO**：最彻底，内核把准备和拷贝都做完再通知

### 它和 IO 多路复用是什么关系？

IO 多路复用是五种 IO 模型里的其中一种。

所以你可以把上一题理解成这题的一个子集展开。

### ⚠️ 面试易错点

- **`select` / `poll` / `epoll` 不是五种模型里的三种新模型**，它们都属于 IO 多路复用这一类。
- **同步 / 异步 和 阻塞 / 非阻塞 不是完全同一维度**。
- **异步 IO 的关键不是“快”，而是数据准备和拷贝都由内核完成后再通知应用。**

### 面试回答模板

Unix 常见 IO 模型主要有五种：阻塞 IO、非阻塞 IO、IO 多路复用、信号驱动 IO 和异步 IO。它们的主要区别在于：等待数据的方式不同、内核通知应用的方式不同，以及数据从内核拷贝到用户空间这一步到底由谁来完成。

其中，阻塞 IO 最直接，非阻塞 IO 会不断轮询，IO 多路复用适合同时监听多个连接，信号驱动 IO 通过信号通知，而异步 IO 则由内核把整个 IO 操作做完后再通知应用。

### 面试简答版

Unix 常见 IO 模型有五种：阻塞 IO、非阻塞 IO、IO 多路复用、信号驱动 IO、异步 IO。

它们区别主要在于等待方式和通知方式不同。

## 3. select、poll 和 epoll 之间的区别？

### 先说场景

这三个东西本质上都在做一件事：

**帮你同时监听多个文件描述符，看谁已经就绪。**

所以它们都是 [[05IO系统#1. 讲一讲 IO 多路复用？|IO 多路复用]] 的实现方式。

### 先抓总主线

你不用一上来背很多细节，先记住这三句话：

- `select`：老办法，能用，但限制多
- `poll`：本质和 `select` 很像，只是去掉了一些限制
- `epoll`：更适合高并发场景，Linux 下最常见

### select 是什么特点？

`select` 会把你关心的 fd 集合交给内核，然后内核检查一遍，返回后你自己还得再扫一遍，找出到底哪个 fd 就绪了。

它的问题主要有两个：

- 有最大连接数限制
- 每次都要把 fd 集合从用户态拷到内核态，再整体扫描

连接一多，成本就上来了。

### poll 比 select 强在哪？

`poll` 和 `select` 的核心思路其实差不多，还是：

- 把 fd 列表交给内核
- 内核遍历检查
- 返回后应用再处理

它改进的地方主要是：

- 没有 `select` 那种固定最大 fd 数量限制

但它本质上还是要遍历全部 fd，所以连接很多时，性能问题依然在。

### epoll 为什么更适合高并发？

`epoll` 的核心优势是：

**它不是每次都把所有 fd 重新全量扫描一遍，而是更偏事件驱动。**

你先把关心的 fd 注册给内核，后面哪个 fd 真正就绪了，内核就把它放进就绪队列，你来取就行。

所以在大量连接但只有少量活跃连接的场景里，`epoll` 会非常有优势。

### 对比总结

| 对比点 | select | poll | epoll |
|--------|--------|------|-------|
| fd 数量限制 | 有 | 没有固定上限 | 没有固定上限 |
| 每次调用是否要全量遍历 | 要 | 要 | 不需要像前两者那样全量扫描 |
| 适合场景 | 连接数少 | 比 select 稍灵活 | 高并发、大量连接 |
| 常见平台 | 通用 | 通用 | Linux |

### 为什么大家常说 epoll 是 O(1)？

面试里经常会听到这个说法，但别死背成绝对真理。

更稳妥的说法是：

**在高并发场景下，epoll 不需要像 select / poll 那样每次都无差别扫描全部 fd，所以整体效率通常更高。**

这样回答更稳，也更不容易被追问卡住。

### ⚠️ 面试易错点

- **`poll` 不是本质上完全不同的新机制**，它和 `select` 思路非常接近。
- **`epoll` 的优势主要在高并发、大量连接、少量活跃连接的场景。**
- **不要把 “epoll 一定 O(1)” 背成绝对口号**，面试时说它减少了无差别遍历，更稳。
- **这三者都属于同步 IO**，只是等待事件的机制不同。

### 面试回答模板

`select`、`poll` 和 `epoll` 都是 IO 多路复用的实现方式。`select` 和 `poll` 的核心思路比较接近，都是每次调用时把待监听的 fd 集合交给内核，内核检查后返回结果，因此当连接数很多时，遍历成本较高。

`poll` 相比 `select` 去掉了固定最大 fd 数量的限制，但性能模型本质没有变化。`epoll` 则更适合高并发场景，它采用事件驱动方式，先把 fd 注册到内核中，后续只返回真正就绪的 fd，因此在大量连接场景下效率通常更高。

### 面试简答版

`select`、`poll`、`epoll` 都是 IO 多路复用机制。

`select` 限制多，`poll` 比它灵活一些，但两者都要遍历全部 fd；`epoll` 更适合高并发场景，因为它更偏事件驱动，只关心真正就绪的 fd。
