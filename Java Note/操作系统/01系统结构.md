# 系统结构

## 1. 什么是缓冲区溢出？有什么危害？

### 先说场景

程序给一块内存留了固定大小的空间，本来只够放 10 个字符，结果外面传进来 20 个字符，还硬往里塞。

这时候，后面的数据就可能把原来合法的数据给顶掉，这就是缓冲区溢出。

### 大白话解释

可以把缓冲区想象成一个杯子。

杯子只能装 300ml 水，你却往里倒 500ml，超出来的水一定会洒到别处。

缓冲区溢出也是一样：本来该放在这一小块内存里的数据太多了，多出来的部分会覆盖后面的内存内容。

### 它到底是什么？

缓冲区溢出说白了就是：

**写入的数据量，超过了缓冲区本身的容量。**

常见原因有：

- 没有检查用户输入长度
- 用了不安全的字符串处理函数
- 拷贝数据时没有做边界判断

### 有什么危害？

最常见的危害有两类：

| 危害 | 大白话理解 |
|------|------------|
| 程序崩溃 | 把不该改的内存改坏了 |
| 被利用执行恶意代码 | 攻击者故意覆盖关键地址或控制数据 |

比如函数返回地址被覆盖，就可能导致程序跳到攻击者指定的位置去执行。

### 为什么它很危险？

因为它不是“报个错就结束”那么简单。

有些情况下只是程序异常退出，但严重时它会变成安全漏洞，让攻击者拿到程序控制权。

所以缓冲区溢出既是稳定性问题，也是安全问题。

### ⚠️ 面试易错点

- **缓冲区溢出不只是程序崩溃**，还可能演变成代码执行漏洞。
- **根因通常是边界检查不足**，不是“内存突然不够了”。
- **它和内存泄漏不是一回事**：溢出是写过界，泄漏是申请了内存没释放。

### 面试回答模板

缓冲区溢出是指程序向缓冲区写入数据时，写入的数据超过了缓冲区本身的容量，导致多余的数据覆盖了后面的合法内存内容。

它的危害一方面是可能导致程序崩溃、数据损坏，另一方面在严重情况下还可能被攻击者利用，覆盖返回地址或关键控制信息，从而执行恶意代码。

它产生的根本原因通常是程序没有做好输入长度检查或边界保护。

### 面试简答版

缓冲区溢出就是写进缓冲区的数据太多，超过了它本身容量，结果把后面的内存覆盖掉了。

危害是程序可能崩溃，严重时还可能被利用执行恶意代码。

## 2. 中断的处理过程？

### 先说场景

CPU 正在老老实实执行当前程序，这时候外设来了一个“紧急通知”，比如键盘有输入、网卡收到数据、定时器时间到了。

CPU 不可能完全不理它，所以会先暂停手头工作，先去处理这个更紧急的事，这就是中断处理的核心思路。

### 大白话解释

你可以把中断想成你正在写题，突然手机响了。

你会先把笔放下，记住自己写到哪了，接电话，电话打完再回来继续写。

中断处理也是这个流程：

```text
先保存现场
-> 去处理中断
-> 处理完恢复现场
-> 继续执行原来的程序
```

### 具体流程

1. **保存现场**

CPU 会先把当前程序的重要执行信息保存起来，比如程序计数器、寄存器等。

2. **根据中断号找到中断处理程序**

系统会根据中断类型，跳转到对应的中断服务程序。

3. **执行中断处理程序**

比如读取设备状态、搬运数据、更新时间片等。

4. **恢复现场**

把刚才保存的寄存器、程序计数器等恢复回来。

5. **继续执行原程序**

程序会从被打断的位置继续往下跑。

### 为什么要先保存现场？

因为 CPU 处理中断时也要用寄存器、程序计数器这些东西。

如果不先把原来的内容存起来，中断处理一跑，原程序的状态就丢了，回来时就不知道从哪接着执行。

### ⚠️ 面试易错点

- **中断不是程序“自己主动调用”**，很多时候是硬件或异常触发的。
- **核心流程一定有保存现场和恢复现场**，这是中断能返回原程序的关键。
- **中断处理程序不是随便跳过去执行一段代码**，它有固定入口和处理机制。

### 面试回答模板

中断处理的核心流程是：当中断发生时，CPU 先暂停当前程序，保存现场，然后根据中断号找到对应的中断服务程序并执行。执行完成后，再恢复现场，最后回到原来被打断的位置继续执行。

这样做的目的，是让 CPU 既能及时响应外部事件，又不会破坏原来程序的执行状态。

### 面试简答版

中断处理过程就是：保存现场、处理中断、恢复现场、继续执行原程序。

## 3. 中断和轮询有什么区别？

### 先说场景

CPU 想知道设备有没有准备好，可以有两种办法：

- 自己隔一会儿就去问一次
- 让设备准备好时主动来通知它

前者就是轮询，后者就是中断。

### 大白话解释

轮询像你不停问同学：

```text
“好了吗？”
“好了吗？”
“好了吗？”
```

中断像同学准备好了以后主动拍你一下：

```text
“好了，你现在可以处理了。”
```

### 两者区别

| 对比点 | 轮询 | 中断 |
|--------|------|------|
| 谁更主动 | CPU 主动去问 | 设备主动通知 CPU |
| CPU 开销 | 更高 | 相对更省 |
| 响应方式 | 周期性检查 | 事件触发 |
| 适用场景 | 简单场景、实现容易 | 更常见、更高效 |

### 为什么轮询效率低？

因为设备可能根本没准备好，但 CPU 还是要反复去查。

这会浪费很多时间在“白问”上。

### 中断就一定更好吗？

大多数情况下，中断更高效，但也不是完全没成本。

因为每次中断都要保存现场、切换到中断处理程序、再恢复现场，如果中断特别频繁，也会带来系统开销。

### ⚠️ 面试易错点

- **轮询不是错，只是效率通常没中断高**。
- **中断不是零成本**，频繁中断也会有上下文切换开销。
- **核心区别不是“一个快一个慢”**，而是“一个主动问，一个被动通知”。 

### 面试回答模板

轮询和中断的核心区别在于设备状态的获知方式不同。轮询是 CPU 主动不断检查设备是否就绪，而中断是设备在准备好后主动通知 CPU。

轮询实现简单，但 CPU 会浪费时间反复检查；中断更高效，能够减少无效等待，不过也会带来中断处理和现场切换的开销。

### 面试简答版

轮询是 CPU 主动不断去问设备好没好；中断是设备好了之后主动通知 CPU。

轮询简单但浪费 CPU，中断更高效。

## 4. 什么是用户态和内核态？

### 先说场景

不是所有代码都应该有同样大的权限。

如果一个普通应用随便就能直接改内存、操作硬盘、控制网卡，那系统很容易被玩坏。

所以操作系统把运行状态分成了两种：

- 权限低一点的用户态
- 权限高一点的内核态

### 大白话解释

可以把它理解成公司里的两种权限：

- 普通员工账号
- 管理员账号

普通员工能做日常工作，但不能随便改公司底层配置。

管理员账号权限大，能改系统级内容，但也更危险。

### 用户态是什么？

用户态就是普通应用程序平时运行的状态。

这个状态下，程序权限受限：

- 不能直接访问硬件
- 不能随便访问任意内存
- 不能执行某些特权指令

这样做是为了保护系统安全和稳定。

### 内核态是什么？

内核态就是操作系统内核运行时的状态。

这个状态下权限很高，可以：

- 访问硬件
- 管理内存
- 调度进程
- 执行特权指令

### 为什么要分成两种状态？

核心目的就是：

**把高风险操作关进“高权限区域”，防止普通程序乱来。**

这样即使某个应用出问题，也不至于直接把整个系统一起带崩。

### ⚠️ 面试易错点

- **用户态不是“给用户用的模式”**，而是普通程序运行的受限状态。
- **内核态不是一直在运行**，只有执行系统内核相关操作时才进入。
- **权限差异是两者最核心的区别**，不是名字本身。

### 面试回答模板

用户态和内核态是操作系统的两种运行状态。用户态权限较低，普通应用程序一般运行在用户态，不能直接访问硬件或执行特权指令。内核态权限较高，操作系统内核运行在内核态，可以访问所有内存和硬件资源。

之所以要区分这两种状态，是为了做权限隔离，保证系统的安全性和稳定性，防止普通程序直接进行高风险操作。

### 面试简答版

用户态是普通程序运行的低权限状态，内核态是操作系统运行的高权限状态。

区分它们是为了做权限隔离，保护系统安全。

## 5. 用户态和内核态是如何切换的？

### 先说场景

应用程序自己不能直接操作硬盘、网卡、文件系统，但它又确实需要这些能力，比如：

- 读文件
- 发网络请求
- 申请内存

这时候它就得“请内核帮忙”，于是就发生了用户态到内核态的切换。

### 大白话解释

你可以把它理解成前台员工没权限开机房门，但他可以提交工单给管理员。

管理员接单后进去处理，处理完再把结果返回。

这就很像：

```text
用户态程序发起请求
-> 进入内核态
-> 内核完成操作
-> 再返回用户态
```

### 最常见的切换方式

最常见的是 **系统调用**。

比如程序调用 `read`、`write`、`open` 这些接口时，本质上就是在请求内核帮它完成操作。

执行系统调用时，CPU 会通过陷入指令（trap）从用户态切到内核态，进入内核代码执行。

执行完之后，再返回用户态。

### 除了系统调用，还有哪些情况？

除了系统调用，以下情况也可能触发切换：

- **中断**：比如键盘输入、网卡收包
- **异常**：比如缺页异常、除零异常

也就是说，切换不一定总是程序主动要求的，也可能是系统被动响应事件。

### 切换时发生了什么？

切换过程中，一般会做这些事：

1. 保存当前用户态现场
2. 切换 CPU 到内核态
3. 跳到对应的内核处理入口
4. 内核处理完后恢复现场
5. 返回用户态继续执行

这里顺手注意一个很容易混的点：

**用户态 / 内核态切换，不等于进程上下文切换。**

- 用户态 / 内核态切换：重点是**权限级别变了**
- 进程上下文切换：重点是**CPU 换了一个进程来运行**

两者有时候会一起发生，但不是同一个概念。

### 为什么切换有开销？

因为这不是普通函数调用。

它涉及：

- 权限级别变化
- 现场保存与恢复
- 可能的缓存和流水线影响

所以系统调用太频繁时，性能会受影响。

### ⚠️ 面试易错点

- **用户态切到内核态最常见的方式是系统调用**。
- **中断和异常也会触发切换**，不是只有系统调用。
- **切换是有成本的**，不是简单跳一下函数。
- **从内核态返回用户态时要恢复现场**，不然程序没法接着正常跑。

### 面试回答模板

用户态和内核态的切换，最常见是通过系统调用实现的。应用程序在用户态执行时，如果需要访问文件、网络、设备等受保护资源，就会通过系统调用触发 trap，CPU 从用户态切换到内核态，进入内核代码执行。

此外，中断和异常也可能触发这种切换。切换过程中需要保存现场、切换权限级别、执行内核逻辑，最后再恢复现场并返回用户态，所以用户态和内核态切换是有一定开销的。

### 面试简答版

用户态和内核态最常见通过系统调用切换。程序发起系统调用后，CPU 进入内核态执行内核代码，执行完再返回用户态。

中断和异常也会触发这种切换。

## 6. 为什么计算机要给存储结构分级？

### 先说场景

我们当然希望存储器同时满足这几个条件：

- 速度特别快
- 容量特别大
- 价格特别便宜

但现实里基本做不到“三个愿望一次满足”。

越快的存储，通常越贵、容量越小；越便宜、容量越大的存储，通常又越慢。

所以计算机只能折中，把存储做成一层一层的结构。

### 大白话解释

你可以把存储分级想成家里放东西：

- 手里拿着的东西，拿得最快，但数量很少
- 桌上的东西，稍微慢一点，但能放多一些
- 柜子里的东西，容量更大，但拿起来更慢
- 地下室里的东西，最多，但最慢

计算机也一样：

- 寄存器最快，但特别小
- Cache 很快，但容量也不大
- 内存更大，但比 Cache 慢
- SSD / 硬盘容量很大，但速度又慢很多

### 为什么要这样设计？

核心原因就是：

**用少量的高成本高速存储，配合大量的低成本大容量存储，在性能和成本之间做平衡。**

这样常用数据尽量放在快的地方，不常用数据放在慢但大的地方。

### 它带来的好处

| 好处 | 大白话理解 |
|------|------------|
| 提高速度 | 常用数据离 CPU 更近 |
| 控制成本 | 不需要把所有存储都做成最贵最快的 |
| 扩大容量 | 慢存储负责兜住海量数据 |
| 提高能效 | 不同层级做不同事情，更划算 |

### 常见层级

常见存储层级一般可以理解成：

```text
寄存器
-> L1 / L2 / L3 Cache
-> 主存 RAM
-> SSD / HDD
-> 更外层存储
```

越往上越快、越贵、越小；越往下越慢、越便宜、越大。

### ⚠️ 面试易错点

- **分级不是为了复杂而复杂**，而是因为速度、容量、成本无法同时拉满。
- **越快不代表越好用**，如果全都用最快存储，成本会高得离谱。
- **分级思想的核心是“让常用数据待在更快层”**。

### 面试回答模板

计算机之所以要对存储结构分级，是因为不同存储介质在速度、容量和成本上存在明显矛盾。越靠近 CPU 的存储通常速度越快，但容量更小、成本更高；越远离 CPU 的存储容量更大、成本更低，但速度更慢。

因此，计算机采用分层存储结构，把高频访问的数据尽量放到更快的层级，比如寄存器、Cache 和内存中，而把低频数据放到容量更大但更慢的 SSD 或硬盘中，从而在性能、容量和成本之间取得平衡。

### 面试简答版

存储分级的原因是速度、容量、成本三者很难同时最优。

所以计算机会把常用数据放在快但贵的小存储里，把不常用数据放在慢但便宜的大存储里。

## 7. CPU cache L1、L2、L3 读取数据的时间量级差距有多大？

### 先说结论

这道题不用死记某个精确数字，因为不同 CPU 差别会很大。

面试里更重要的是记住：

**L1 最快，L2 次之，L3 更慢，而主存比它们都慢一大截。**

### 大致量级怎么记？

你可以先记一个“够用版”范围：

| 层级 | 大致延迟量级 |
|------|--------------|
| L1 | 几个时钟周期 |
| L2 | 十个时钟周期左右 |
| L3 | 几十个时钟周期 |
| 主存 RAM | 上百个时钟周期，甚至更多 |

如果你想带一点数字感，可以这么说：

- `L1`：通常大概 `1~4` 个周期
- `L2`：通常大概 `4~12` 个周期
- `L3`：通常大概 `20~50` 个周期
- `RAM`：通常大概 `100+` 个周期

这已经足够应付大多数面试场景了。

### 为什么差这么多？

因为这些存储离 CPU 的远近、设计目标、容量大小都不一样。

- `L1` 离核心最近，容量最小，专门追求极致低延迟
- `L2` 比 L1 大一些，速度稍慢
- `L3` 更大，常常还是多核共享，所以又慢一些
- 主存虽然容量大得多，但离 CPU 更远，访问链路也更长

### 怎么用一句话说清楚？

你可以直接说：

**L1 到 L3 是逐级变慢的，但 RAM 相比 Cache 会再慢一个明显量级。**

### ⚠️ 面试易错点

- **不要把数字背得太死**，不同架构和不同代 CPU 差异很大。
- **重点不是某个精确值，而是量级差**。
- **真正最关键的是：访问主存比访问 Cache 慢很多很多。**

### 面试回答模板

L1、L2、L3 Cache 的访问延迟通常是逐级增加的。一般来说，L1 只需要几个时钟周期，L2 大约十个周期左右，L3 往往是几十个周期，而主存访问通常是上百个时钟周期。

所以真正重要的结论是：Cache 层级之间已经有明显差距，而主存相对 Cache 又会慢一个更大的量级，这也是为什么 CPU Cache 会对性能影响很大。

### 面试简答版

L1 最快，通常几个周期；L2 十个周期左右；L3 几十个周期；主存通常上百个周期。

重点不是具体数，而是 RAM 比 Cache 慢很多。

## 8. CPU 缓存对程序性能的影响都体现在哪些方面？

### 先说场景

CPU 算得很快，但如果每次都得苦等内存把数据送过来，CPU 再强也会被“喂不饱”。

所以 Cache 的作用，本质上就是：

**尽量别让 CPU 老去等慢吞吞的主存。**

### 先给结论

CPU 缓存对程序性能的影响，主要体现在这几个方面：

- 数据访问延迟
- 局部性利用效果
- 吞吐量
- 指令执行流畅度
- 多核性能表现

### 1）数据访问速度更快

这是最直接的一点。

如果数据命中 L1 / L2 / L3，CPU 取数据的速度会比直接去主存快很多。

所以命中率高，程序通常就更快。

### 2）局部性利用得好不好

Cache 能发挥作用，靠的是局部性原理：

- **时间局部性**：刚访问过的数据，接下来可能还会访问
- **空间局部性**：访问了某个位置，附近的位置也很可能很快会被访问

所以像循环、数组顺序访问、热点代码，通常就更吃 Cache 红利。

### 3）吞吐量会提升

当 CPU 等内存的时间减少后，它单位时间内就能干更多事。

所以 Cache 不只是“单次访问更快”，它还会直接影响整个程序的吞吐量。

### 4）指令执行会更顺

现代 CPU 有流水线、乱序执行、预取这些机制。

如果数据和指令能稳定地从 Cache 里拿到，CPU 的执行就更流畅，不容易频繁停下来等内存。

### 5）多核场景也会受影响

多核 CPU 往往会涉及共享缓存，特别是 `L3`。

这时候 Cache 不只是影响“单核快不快”，还会影响：

- 核之间共享数据的成本
- 缓存一致性带来的开销
- 多线程程序的扩展性

### 为什么有些程序特别吃 Cache？

因为它们的数据访问模式很集中、很有规律。

比如：

- 大量循环
- 顺序扫描数组
- 热点函数反复调用

而如果程序总是乱跳地址、数据访问非常分散，Cache 效果就会差很多。

### ⚠️ 面试易错点

- **Cache 不只是影响“速度”，也影响吞吐量和多核表现。**
- **决定 Cache 效果的关键，不只是 Cache 大小，还有访问模式。**
- **命中率高不代表所有程序都一样收益大**，和局部性强弱关系很大。

### 面试回答模板

CPU Cache 对程序性能的影响主要体现在降低数据访问延迟、提升吞吐量、提高指令执行流畅度以及改善多核程序的整体表现。因为 Cache 比主存快很多，所以如果程序访问的数据和指令能够更多地命中 Cache，CPU 等待内存的时间就会显著减少。

此外，Cache 的效果和局部性原理密切相关。具有较强时间局部性和空间局部性的程序，更容易从 Cache 中获益；而访问模式离散、随机性强的程序，Cache 效果通常会差一些。

### 面试简答版

CPU Cache 会影响数据访问延迟、吞吐量、指令执行流畅度和多核性能。

本质上就是：命中 Cache 越多，CPU 等主存越少，程序通常就越快。

## 9. 什么是硬中断和软中断？

### 先说场景

CPU 被打断，不一定都是同一种“打断”。

有些是外部硬件突然来敲门，比如键盘、网卡、定时器；有些则是程序或系统内部主动触发，要求 CPU 转去处理某件事。

所以中断还要再分成硬中断和软中断。

### 大白话解释

可以这样理解：

- **硬中断**：外面有人突然敲门
- **软中断**：你自己在系统里发起一个“请处理这个事”的请求

一个更偏外部事件触发，一个更偏软件内部触发。

### 什么是硬中断？

硬中断通常由外部硬件设备触发，比如：

- 键盘输入
- 鼠标动作
- 网卡收到数据
- 定时器到时

它的特点是：

- 来自硬件
- 往往比较突然
- 更强调及时响应

### 什么是软中断？

软中断通常由程序或操作系统内部主动触发。

它常见于：

- 系统调用
- 某些异常处理
- 软件内部调度某项后续工作

你可以把它理解成：

**不是外面设备来打断你，而是软件自己“制造了一个需要内核处理的事件”。**

### 两者区别

| 对比点 | 硬中断 | 软中断 |
|--------|--------|--------|
| 触发来源 | 外部硬件 | 软件 / 系统内部 |
| 触发方式 | 异步到来 | 通常是程序逻辑触发 |
| 典型场景 | 键盘、网卡、时钟 | 系统调用、软件内部处理 |
| 核心特点 | 实时性更强 | 可控性更强 |

### 为什么要区分？

因为不同来源的事件，处理方式和优先级往往不同。

硬中断通常更强调快速响应，避免硬件事件丢失；软中断则更像把某些后续处理工作交给系统继续完成。

### 为什么还要有软中断？

很多同学会问：

既然已经有硬中断了，为什么还要再搞一个软中断？

核心原因是：

**不是所有需要内核处理的事情，都适合直接靠外部硬件中断来完成。**

软中断存在的价值，主要体现在这几个方面：

1. **承接系统调用和内核服务请求**

用户程序在用户态下权限不够，很多操作必须交给内核处理，比如文件操作、内存管理、网络通信。

这时候就需要一种“由软件主动发起、请求内核接手”的机制，软中断正好承担这个角色。

2. **处理异常和错误**

比如除零、非法访问、缺页异常，本质上都不是“外部硬件通知”，而是程序执行过程中内部触发的问题。

系统需要靠软中断 / 异常机制及时接管处理，避免错误继续扩散。

3. **把一些后续工作交给内核分阶段完成**

有些事情不适合全压在硬中断那一刻做完，否则会拖太久。

这时候常见思路是：

- 硬中断先快速响应
- 更重一点的后续处理，交给软中断或内核后续机制

这样可以兼顾实时性和系统整体效率。

4. **更灵活、更可控**

硬中断很多是外部突然打进来的，你没法决定它什么时候来。

软中断则更像系统自己安排的处理机制，灵活性更高，也更适合做系统内部协作。

5. **做权限控制和资源访问控制**

用户程序不能直接碰很多底层资源，所以系统需要通过软中断这类机制，把请求先收进内核，再由内核判断：

- 这件事该不该做
- 当前进程有没有权限做
- 应该按什么规则访问资源

这也是为什么系统调用、异常处理这些事情，通常都离不开它。

6. **支持调试、跟踪和日志等软件控制流程**

有些软件级事件不是硬件突然打进来的，而是系统或程序自己故意触发，用来做调试、跟踪、异常定位、日志记录等工作。

这种场景更适合通过软中断 / trap / 软件异常机制来完成。

### ⚠️ 面试易错点

- **硬中断重点是“硬件触发”**。
- **软中断重点是“软件主动触发”**。
- **软中断不等于普通函数调用**，它依然属于一种进入系统处理流程的机制。
- **有些资料会把系统调用、异常、trap 一起讲**，面试时只要把“外部硬件触发”和“软件内部触发”区分清楚就够了。

### 面试回答模板

硬中断和软中断的核心区别在于触发来源不同。硬中断通常由外部硬件设备触发，比如键盘输入、网卡收包、时钟中断等，特点是异步到来、对实时响应要求较高。软中断则通常由程序或操作系统内部主动触发，常见于系统调用、异常处理或系统内部任务调度。

因此，可以简单理解为：硬中断偏外部事件通知，软中断偏软件内部请求或处理机制。

### 面试简答版

硬中断由硬件触发，比如键盘、网卡、时钟；软中断由软件或系统内部触发，比如系统调用或内部处理请求。

核心区别就是：一个来自外部硬件，一个来自软件内部。

## 10. 说一下 Linux 启动的流程

### 先说场景

按下电源键之后，Linux 不是一下子就“直接能用”了。

它中间要经历一串接力：

```text
硬件上电
-> 固件自检
-> 引导加载程序
-> 内核加载
-> 内核初始化
-> 启动 init / systemd
-> 启动服务
-> 进入登录或桌面
```

所以 Linux 启动流程，本质上就是“控制权一步一步往后交接”的过程。

### 大白话解释

你可以把它想成开一家店：

- 先检查电、水、门锁是否正常
- 再打开总开关
- 再把店里的基础设备准备好
- 再叫员工上班
- 最后才开始正式营业

Linux 启动也是这个思路。

### 1）POST / BIOS / UEFI

开机后，最先工作的不是 Linux 内核，而是主板固件，也就是 `BIOS` 或 `UEFI`。

它会先做基本检查和初始化，比如：

- 检查硬件是否正常
- 初始化内存
- 确定从哪个设备启动

这一步可以理解成“开机自检 + 找启动入口”。

### 2）启动引导程序 Bootloader

固件找到可启动设备后，会把控制权交给引导加载程序，比如：

- `GRUB`
- `systemd-boot`

引导加载程序的主要作用是：

- 找到可启动的系统
- 让用户选择启动项
- 把 Linux 内核装进内存

它相当于“正式把操作系统拉起来的中间人”。

### 3）加载 Linux 内核

Bootloader 会把内核镜像加载到内存中，然后把控制权交给内核。

这时 Linux 内核才算真正开始接管机器。

### 4）内核初始化

内核接手后，会做很多底层初始化工作，比如：

- 初始化 CPU、内存管理
- 初始化中断和调度机制
- 加载必要驱动
- 挂载根文件系统

这一步很关键，因为它决定系统能不能真正把硬件和软件世界连起来。

### 5）启动第一个用户态进程

当内核把自己的事情准备得差不多后，会启动第一个用户态进程。

传统上是 `init`，现代 Linux 更常见的是 `systemd`。

这个进程非常关键，因为后续大部分用户空间服务，都是它继续拉起来的。

### 6）启动系统服务和守护进程

`init` 或 `systemd` 会继续启动各种后台服务，比如：

- 日志服务
- 网络服务
- 定时任务
- 显示管理器

这些服务起来之后，系统才逐步进入“可用状态”。

### 7）进入登录界面或 Shell

最后，系统会进入：

- 图形登录界面
- 或命令行登录界面

用户登录之后，Shell、桌面环境、应用程序才会继续启动。

如果是图形界面系统，通常还会继续拉起显示管理器和桌面环境；如果是服务器环境，更多会直接进入终端登录或目标服务运行状态。

### 怎么记最核心链路？

你可以先记这一条：

```text
BIOS/UEFI
-> Bootloader
-> Kernel
-> init/systemd
-> 服务启动
-> 用户登录
```

面试里能把这条顺下来，基本就够用了。

### ⚠️ 面试易错点

- **Linux 不是一开机就先跑内核**，最开始是 BIOS / UEFI。
- **Bootloader 不是内核本身**，它的任务是“把内核拉起来”。
- **内核初始化完成后，才会启动第一个用户态进程**。
- **`systemd` / `init` 属于用户空间，不是内核本身。**

### 面试回答模板

Linux 启动流程大致分为几个阶段：首先 BIOS 或 UEFI 上电自检并确定启动设备；然后启动引导程序，比如 GRUB，由它把 Linux 内核加载到内存中；接着内核开始初始化 CPU、内存、驱动和根文件系统；内核初始化完成后，会启动第一个用户态进程，传统上是 init，现代系统中通常是 systemd；最后由它继续拉起各类系统服务和守护进程，并进入登录界面或 Shell。

所以从整体上看，Linux 启动就是控制权从固件到引导程序，再到内核，最后到用户空间服务的逐步交接过程。

### 面试简答版

Linux 启动流程可以概括为：

`BIOS/UEFI -> Bootloader -> 内核加载与初始化 -> init/systemd -> 启动服务 -> 用户登录`
