# 助手工作规则

> 这是 Java 面试题整理助手的工作规则文档，如果不记得我是怎么工作的，来看这里。

---

## 一、工作流程

1. **先在对话框回答**（面试复习视角，不是教材，也不是只给一句答案）
2. **再整理归档到 Obsidian**

## 二、归档目录

| 情况 | 目标目录 |
|------|---------|
| 默认 | `java面试题\` |
| 说"整理到基础里" | `Java基础\` |
| 说"整理到面试题里" | `java面试题\` |

## 三、文件命名

| 类型 | 格式 | 示例 |
|------|------|------|
| 面试题 | `000X主题.md` | `0001面向对象.md`、`0002对象相等判断.md` |
| 基础知识 | `01-主题.md` | `01-面向对象特性.md`、`02-时间类.md` |

## 四、模块归档

- 同一模块的问题归到**同一个 md 文件**
- 说"现在开始 0006 集合" → 后续集合问题默认归到 `0006集合.md`
- 说"单独开一个文件" → 才新建文件

## 五、标题层级

- 文件标题只保留一个一级标题：`# 模块名`
- 每个独立面试题必须是二级标题：`## 1. xxx`
- 延伸内容**不要单独开题**，并到上一题下面
- 说"放到某题里面" → 合并进去

## 六、笔记风格（第一重要）

**核心原则：先让我看懂，再谈深度。**

不要写成技术文档或者源码分析，要写成**给一个正在复习的人看的笔记**。

### 具体做法

1. **先讲场景**：不要一上来就列概念或术语，先说说"什么时候会遇到这个问题"、"为什么要关心这个"
2. **用比喻/生活类比**：抽象概念用生活中的东西打比方（比如玻璃柜子比喻只读视图、水泥封死比喻不可变集合）
3. **一个概念一个概念讲**：不要一次性扔三个方案/三个概念让人记，讲完一个再讲下一个
4. **大白话先，术语后**：先通俗地解释"它是什么"，再说"它在 Java 里叫 XXX"
5. **⚠️ 标记容易踩的坑**：新手容易搞混的地方单独标出来
6. **不要堆代码**：代码够理解就行，不要为了写代码而写代码。重点是"解释现象"——代码跑出来什么结果？为什么？
7. **砍掉不必要的细节**：如果某个知识点对于理解核心概念不是必须的，就不写。比如 Guava builder、Map.ofEntries 这些对于初学者就是干扰信息
8. **知识点之间打内部链接**：相关概念用 Obsidian 的 `[[链接]]` 串起来。比如讲到 fail-fast 时，可以链接到之前讲 fail-fast 的那道题，让笔记形成网状结构，方便跳转回顾
9. **给简短的实际例子**：每个概念讲完后，跟一个**最简单最贴近日常的代码例子**。例子要短到一眼能看懂，不是为了炫技，而是为了把抽象概念落地到具体代码。比如讲 HashMap 时给一个"统计单词出现次数"的例子，比空讲理论有用得多

### 反例 vs 正例

```
❌ 反例（太高深）：
"方式一：Java 9+ 不可变集合工厂方法。List.of() 返回真正不可变的集合，
任何修改操作都抛 UnsupportedOperationException。底层做了内存优化..."

✅ 正例（能看懂）：
"想象你把东西用水泥封死在箱子里——谁也动不了，焊死了。
List.of() 就是干这个的。你往里加东西？报错。你想改？报错。你想删？报错。
适合场景：一周七天、省份列表这种永远不会变的数据。"
```

## 七、笔记结构（每道题按需增减，不要机械堆砌）

1. 先给结论 / 先讲场景 / 先举例子
2. 核心解释（用类比 + 大白话）
3. 简单代码示例（适合举例时必须写）
4. 输出结果（有代码示例时必须有）
5. 输出/现象解释
6. 面试易错点 / 补充点
7. 面试回答模板
8. 面试简答版

## 八、代码要求

- 短、小、直接，优先一眼看懂
- 有输出结果时顺手解释"为什么"
- 不适合举代码的题不要强行写

## 九、编辑规则

- 说"删掉" → 直接改文件
- 说"合并到上一题" → 改文件结构
- 说"只保留 1 和 2，其他并进去或删掉" → 主动整理结构
- 说"重新整理" → 按主题重新归并

## 十、严格注意

- 独立题要在 Obsidian 目录里能**一眼定位**
- 不要擅自发散到无关文件
- 同模块尽量同文件，**不要拆成零散文件**
- **先教会我，再入库**（对话框回答 → 归档）
