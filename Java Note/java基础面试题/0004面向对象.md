# 面向对象

## 1. 面向对象和面向过程的区别

### 核心区别
- 面向过程：关注“怎么做”，把问题拆成一个个步骤和函数。
- 面向对象：关注“谁来做”，先抽象出对象，再给对象分配职责。
- 面向过程以流程为中心；面向对象以类和对象为中心。

### 优缺点
- 面向过程优点：简单直接，适合小型、线性的程序。
- 面向过程缺点：代码容易堆在一起，复用性和扩展性较差。
- 面向对象优点：更贴近现实世界，便于复用、维护和扩展。
- 面向对象缺点：设计成本更高，简单场景下可能显得偏重。

### 适用场景
- 面向过程：脚本、小工具、一次性处理逻辑。
- 面向对象：中大型项目、业务系统、需要长期维护的程序。

### 面试简答版
- 面向过程关注的是解决问题的步骤，强调“怎么做”。
- 面向对象关注的是参与问题的对象，强调“谁来做”。
- 相比面向过程，面向对象更适合复杂系统开发，因为它具有更好的封装性、复用性和扩展性。

### 为什么面向对象更适合大型项目
- 因为大型项目业务复杂、模块多、变化频繁。
- 面向对象可以通过封装控制复杂度，通过继承和组合复用代码，通过多态提升扩展能力。

## 2. 面向对象三大特性

### 2.1 封装

#### 定义
- 封装就是把属性和方法放到类中，对外隐藏内部实现细节，只暴露必要的访问接口。

#### 常见实现方式
- 使用 `private` 修饰成员变量。
- 通过 `getter/setter` 提供受控访问。
- 对外只暴露业务需要的方法，不暴露内部实现。

#### 好处
- 保护数据安全，避免外部随意修改。
- 降低耦合，提高类的独立性。
- 使用者不需要关心内部实现，只关心如何调用。

#### 面试关键词
- 隐藏细节
- 保护数据
- 降低耦合

### 2.2 继承

#### 定义
- 继承是子类获取父类属性和方法的机制，同时子类还可以扩展自己的特有功能。

#### 作用
- 提高代码复用性。
- 建立类与类之间的层次关系。
- 便于功能扩展和统一管理。

#### Java 中的特点
- 类和类之间使用 `extends` 继承。
- Java 类只支持单继承，不支持多继承。
- 一个类可以实现多个接口，从而间接达到类似“多继承”的效果。

#### 面试关键词
- 代码复用
- `extends`
- 单继承

### 2.3 多态

#### 定义
- 多态指同一个父类引用指向不同子类对象时，调用同名方法会表现出不同的行为。

#### 前提条件
- 有继承或实现关系。
- 子类对父类方法进行了重写。
- 父类引用指向子类对象。

#### 作用
- 提高程序的灵活性和扩展性。
- 便于面向接口编程。
- 符合开闭原则，新增子类时通常不用改原有调用逻辑。

#### 本质
- 编译时看引用类型。
- 运行时看实际对象类型。
- 本质上是运行时动态绑定。

#### 面试关键词
- 父类引用指向子类对象
- 方法重写
- 动态绑定

## 3. Java 语言是如何实现多态的

### 3.1 两种多态

#### 编译时多态
- 编译时多态也叫静态多态，典型体现是方法重载（`overload`）。
- 编译器会根据方法名和参数列表，在编译阶段就确定调用哪个方法。
- 它本质上不是运行时“动态选择实现”，而是提前决定调用版本。

#### 运行时多态
- 运行时多态也叫动态多态，典型体现是方法重写（`override`）。
- 编译时只知道引用类型，运行时才根据对象的实际类型决定调用哪个方法。
- 面试里通常说的“多态”主要指运行时多态。

### 3.2 Java 实现运行时多态的三个条件
- 有继承或实现关系。
- 子类重写父类方法。
- 父类引用指向子类对象，即向上转型。

### 3.3 实现原理
- 编译阶段：先检查父类引用能否调用这个方法，语法上是否合法。
- 运行阶段：JVM 根据引用实际指向的对象类型，找到真正要执行的方法实现。
- 因此同样是父类引用调用方法，不同子类对象会表现出不同结果。

### 3.4 本质理解
- 编译时看左边，运行时看右边。
- 左边指引用类型，决定“能不能调”；右边指实际对象，决定“调谁”。
- 这背后依赖的是运行时动态绑定机制。

### 3.5 面试补充
- 重载是编译时多态，重写是运行时多态。
- `static`、`private`、`final` 方法不属于典型的运行时多态范围，因为它们不能被正常重写或走动态绑定。
- 多态的价值在于统一接口、屏蔽差异、增强扩展性。

### 3.6 面试简答版
- Java 中多态分为编译时多态和运行时多态。
- 编译时多态主要通过方法重载实现，在编译阶段就确定调用哪个方法。
- 运行时多态主要通过继承、方法重写和向上转型实现。
- JVM 在运行时根据对象的实际类型进行动态绑定，从而决定最终执行哪个重写后的方法。

## 4. 实现多态的方法举例

#### 方式一：方法重载实现编译时多态
- 同一个类中，方法名相同，参数列表不同，编译器会在编译阶段决定调用哪个方法。

```java
class PrintDemo {
    public void show(int a) {
        System.out.println("int: " + a);
    }

    public void show(String a) {
        System.out.println("String: " + a);
    }
}
```

- 调用 `show(10)` 时会匹配 `show(int)`。
- 调用 `show("hello")` 时会匹配 `show(String)`。
- 这属于编译时多态。

#### 方式二：方法重写 + 向上转型实现运行时多态
- 父类引用指向子类对象，调用被重写的方法时，运行时根据实际对象类型决定执行哪个实现。

```java
class Animal {
    public void speak() {
        System.out.println("动物发声");
    }
}

class Dog extends Animal {
    @Override
    public void speak() {
        System.out.println("汪汪");
    }
}

class Cat extends Animal {
    @Override
    public void speak() {
        System.out.println("喵喵");
    }
}

public class Test {
    public static void main(String[] args) {
        Animal a1 = new Dog();
        Animal a2 = new Cat();

        a1.speak();
        a2.speak();
    }
}
```

- `a1` 和 `a2` 的引用类型都是 `Animal`，但实际对象不同。
- 运行结果分别是 `汪汪` 和 `喵喵`。
- 这属于运行时多态。

#### 方式三：接口实现多态
- 接口引用指向实现类对象，也是一种非常常见的多态形式。

```java
interface Usb {
    void work();
}

class Mouse implements Usb {
    public void work() {
        System.out.println("鼠标工作");
    }
}

class Keyboard implements Usb {
    public void work() {
        System.out.println("键盘工作");
    }
}
```

```java
public class Test {
    public static void main(String[] args) {
        Usb u1 = new Mouse();
        Usb u2 = new Keyboard();

        u1.work();
        u2.work();
    }
}
```

- 这里统一用 `Usb` 接口引用调用 `work()`，但不同实现类会表现出不同结果。
- 实际开发中，接口多态比继承多态更常见。

#### 面试简答版
- Java 实现多态主要有两种方式：重载和重写。
- 重载是同一个类中方法名相同、参数不同，属于编译时多态。
- 重写是父类引用指向子类对象，调用重写方法时根据实际对象类型执行，属于运行时多态。
- 除了继承多态，接口引用指向实现类对象也是常见的多态写法。

## 5. 重载（Overload）和重写（Override）的区别是什么？

#### 重载（Overload）
- 发生在同一个类中，方法名相同，参数列表不同。
- 参数列表不同可以体现在参数个数、类型或顺序不同。
- 返回值类型不能单独作为区分重载的依据。
- 重载属于编译时多态。

#### 重写（Override）
- 发生在父子类之间，子类重新实现父类已有方法。
- 方法名、参数列表必须相同。
- 返回值类型一般相同，或者是其子类型。
- 访问权限不能比父类更严格。
- 重写属于运行时多态。

#### 构造器是否可以被重写
- 不能。
- 构造器不属于普通成员方法，不能被子类继承，因此也就不能被重写。
- 构造器名必须和类名一致，而父类和子类类名不同，从语法上也不可能形成“同名同参数”的重写关系。
- 子类只能通过 `super(...)` 调用父类构造器，完成父类部分的初始化。
- 构造器虽然不能重写，但可以重载。

#### 面试简答版
- 构造器不能被重写，因为构造器不会被继承，它只负责本类对象的初始化。
- 子类可以调用父类构造器，但这不叫重写，通常通过 `super(...)` 完成。
- 构造器只能重载，不能重写。

#### 核心区别
- 范围不同：重载在同类中，重写在父子类之间。
- 参数要求不同：重载必须参数不同，重写必须参数相同。
- 多态阶段不同：重载是编译时多态，重写是运行时多态。
- 调用依据不同：重载看参数列表，重写看实际对象类型。

#### 面试易错点
- 只有返回值不同，不能构成重载。
- `static` 方法谈不上标准意义上的重写，更准确说是隐藏。
- `private` 方法不能被重写，因为子类无法直接访问。
- `final` 方法不能被重写。

#### 面试简答版
- 重载发生在同一个类中，方法名相同但参数列表不同，属于编译时多态。
- 重写发生在父子类之间，方法名和参数列表相同，子类重写父类实现，属于运行时多态。
- 简单记忆：重载看参数，重写看继承。

## 6. 重载的方法能否仅根据返回值类型区分？

#### 结论
- 不能。
- Java 判断方法是否构成重载，看的主要是方法名和参数列表，不看返回值类型。

#### 原因
- 如果两个方法方法名相同、参数列表也相同，只是返回值不同，编译器在调用时无法仅凭调用语句判断该选哪个方法。
- 例如 `test()` 同时存在 `int test()` 和 `double test()`，当你直接写 `test();` 时，调用目标会产生歧义。

#### 正确认知
- 重载成立的条件是参数列表不同，包括参数个数、类型或顺序不同。
- 返回值类型可以不同，但前提是参数列表本身已经不同；返回值不是区分重载的唯一依据。

#### 面试补充
- Java 方法签名在面试语境下通常关注“方法名 + 参数列表”。
- 因此“只改返回值，不改参数”不算重载，会直接编译报错。

#### 面试简答版
- 不能仅根据返回值类型区分重载方法。
- 因为 Java 的重载主要看方法名和参数列表，返回值类型不是判断重载的依据。
- 只有返回值不同、参数完全相同的方法不能同时存在。

## 7. 抽象类和接口的区别是什么？

#### 相同点
- 都不能直接实例化。
- 都可以包含抽象方法，用来约束子类或实现类必须提供具体实现。
- 都体现了 Java 的抽象能力，是实现多态的重要手段。

#### 语法层面的区别
- 抽象类使用 `abstract class` 定义；接口使用 `interface` 定义。
- 抽象类可以有普通成员变量、构造器、普通方法和抽象方法。
- 接口不能有构造器，也不能保存实例状态。
- 接口中的成员变量默认是 `public static final` 常量。
- 接口中的方法早期通常是 `public abstract`；从 Java 8 开始还可以有 `default`、`static` 方法，从 Java 9 开始还可以有 `private` 方法。

#### 继承和实现上的区别
- 一个类只能继承一个抽象类，因为 Java 类是单继承。
- 一个类可以实现多个接口，因此接口更适合做能力扩展和规范约束。

#### 设计层面的区别
- 抽象类更强调“是什么”的抽象，适合提取一类对象的共同属性和共同行为。
- 接口更强调“能做什么”的抽象，适合定义行为规范和能力标准。
- 抽象类适合做模板复用；接口适合做解耦和扩展。

#### 如何选择
- 多个子类之间有公共代码、公共状态时，优先考虑抽象类。
- 只想定义规范，不关心具体实现，或者希望支持多实现时，优先考虑接口。
- 实际开发中常见做法是：接口定义能力，抽象类提供部分通用实现。

#### 面试易错点
- “接口里只能有抽象方法”是旧说法，现代 Java 中接口还可以有 `default`、`static`、`private` 方法。
- 接口中的变量本质上都是常量，不能定义普通实例变量。
- 抽象类不一定非要有抽象方法，但有抽象方法的类必须是抽象类。

#### 抽象类能使用 `final` 修饰吗
- 不能。
- `abstract` 表示这个类不能直接实例化，通常需要被继承后由子类补全实现。
- `final` 表示这个类不能被继承。
- 两者语义完全冲突：一个要求“可以被继承”，一个要求“不能被继承”。
- 因此 `abstract final class` 在 Java 中是非法的，会直接编译报错。

#### 面试补充
- 不仅抽象类不能用 `final` 修饰，抽象方法也不能用 `final` 修饰。
- 因为抽象方法本来就是给子类重写的，而 `final` 方法禁止重写，这两者同样矛盾。

#### 面试简答版
- 抽象类和接口都用于抽象和实现多态，但侧重点不同。
- 抽象类更适合抽取一类事物的共性，可以有成员变量、构造器和普通方法。
- 接口更适合定义行为规范，一个类可以实现多个接口，便于解耦和扩展。
- 简单记忆：抽象类表示“是什么”，接口表示“能做什么”。

## 8. 一句话区分三大特性
- 封装：隐藏细节，对外提供接口。
- 继承：复用父类，扩展子类能力。
- 多态：统一调用，不同实现。

## 9. Java 创建对象有哪几种方式？

#### 面试常答的 4 种方式
- `new` 关键字创建对象。
- 通过反射创建对象。
- 通过 `clone()` 克隆对象。
- 通过反序列化创建对象。

#### 方式一：`new` 创建对象
- 这是最常见、最直接的创建方式。
- 例如：`User user = new User();`

#### 运行方式
- JVM 先检查类是否已经加载，没有的话先进行类加载。
- 然后在堆内存中为对象分配空间，并先给成员变量赋默认值。
- 接着执行构造器，对对象进行显式初始化。
- 最后返回对象引用。

#### 特点
- 一定会调用构造器。
- 适合绝大多数日常开发场景。

#### 方式二：反射创建对象
- 反射允许在运行时动态获取类信息并创建对象。
- 常见写法是通过 `Constructor.newInstance()`。

```java
Class<?> clazz = Class.forName("com.demo.User");
Object obj = clazz.getDeclaredConstructor().newInstance();
```

#### 运行方式
- 先拿到类的 `Class` 对象。
- 再定位构造器。
- 最后通过反射调用构造器完成对象创建。
- 本质上仍然是在堆中创建对象，只是“调用构造器”的方式变成了反射。

#### 特点
- 比 `new` 更灵活，适合框架、配置驱动、IOC 容器等场景。
- 一般性能略低于直接 `new`，而且代码可读性差一些。

#### 方式三：`clone()` 克隆对象
- 克隆是基于已有对象复制出一个新对象。
- 一般需要实现 `Cloneable` 接口并重写 `clone()` 方法。

```java
class User implements Cloneable {
    @Override
    protected Object clone() throws CloneNotSupportedException {
        return super.clone();
    }
}
```

#### 运行方式
- JVM 以当前对象为模板，复制其字段值，生成一个新对象。
- `super.clone()` 默认是浅拷贝，即基本类型直接复制，引用类型只复制地址。
- 克隆通常不会走普通的构造器初始化流程。

#### 特点
- 适合已有对象状态已经准备好，只想快速复制一个相似对象的场景。
- 需要特别注意浅拷贝和深拷贝问题。

#### 方式四：反序列化创建对象
- 把字节流恢复为对象，也是一种创建对象的方式。

```java
ObjectInputStream ois = new ObjectInputStream(new FileInputStream("user.obj"));
User user = (User) ois.readObject();
```

#### 运行方式
- JVM 从字节流中读取对象数据。
- 然后根据序列化信息在内存中恢复对象状态。
- 对于实现 `Serializable` 的类，反序列化时通常不会调用该类的普通构造器。
- 这是它和 `new`、反射最容易混淆但也最值得记住的区别。

#### 面试补充
- 面试里通常先答这 4 种就够了。
- 如果追问更底层，还可以提到 `Unsafe.allocateInstance()`，但这不是常规业务开发创建对象的方式。

#### 面试简答版
- Java 常见的创建对象方式有 4 种：`new`、反射、克隆、反序列化。
- `new` 和反射本质上都会创建新对象，通常会涉及构造器调用。
- `clone()` 是复制已有对象，默认是浅拷贝，通常不走构造器。
- 反序列化是从字节流恢复对象，对于 `Serializable` 类通常不会调用普通构造器。

## 10. 什么是不可变对象？好处是什么？

#### 定义
- 不可变对象指对象一旦创建完成，其状态就不能再被修改。
- 所谓“状态”，通常就是对象内部的成员变量值。
- 如果修改看起来发生了，本质上往往是创建了一个新对象，而不是改原对象。

#### 常见例子
- `String`
- `Integer`、`Long` 等包装类
- `LocalDate`、`LocalDateTime`
- `BigInteger`

#### 不可变对象通常怎么设计
- 类一般用 `final` 修饰，避免被继承破坏不可变性。
- 成员变量使用 `private final` 修饰。
- 不提供 `setter` 方法。
- 如果成员里包含可变对象，要做防御性拷贝。

#### 好处一：线程安全
- 因为状态不能变，所以多个线程共享同一个对象时，不会出现“改来改去”的并发问题。
- 这也是 `String` 天然线程安全的重要原因之一。

#### 好处二：更适合做缓存和共享
- 不可变对象不会被中途修改，所以可以放心复用。
- 例如字符串常量池、本地缓存、配置对象，都很适合使用不可变对象。

#### 好处三：适合做哈希键
- 像 `HashMap` 的键，如果状态变化会影响 `hashCode()` 和 `equals()`，容易出问题。
- 不可变对象状态稳定，更适合作为键。

#### 好处四：更容易理解和维护
- 可变对象常常会出现“谁把它改了”的问题。
- 不可变对象一旦创建完成，状态固定，程序行为更容易推理和排查。

#### 面试易错点
- `final` 引用不等于对象不可变。
- `final List<String> list` 只是说明 `list` 这个引用不能再指向别的对象，不代表列表内容不能改。

#### 面试简答版
- 不可变对象是指对象创建后状态不能再改变的对象。
- 它的好处主要有：线程安全、适合共享和缓存、适合作为哈希键、代码更容易维护。
- 常见的不可变对象有 `String`、包装类和 Java 8 的时间类。

## 11. 能否创建一个包含可变对象的不可变对象？

#### 结论
- 可以，但要非常小心。
- 关键不是“成员是不是可变对象”，而是“这个可变对象能不能被外部继续修改到”。

#### 为什么很多人会写错
- 比如下面这种写法并不是真正的不可变：

```java
final Person[] persons = new Person[10];
```

- `final` 只能保证 `persons` 这个引用不再指向别的数组。
- 但数组里的元素仍然可以改，数组本身的内容也可以改。
- 所以这只是“引用不可变”，不是“对象状态不可变”。

#### 真正要做到不可变，需要满足什么
- 构造时不要直接保存外部传入的可变对象引用，要做防御性拷贝。
- `getter` 不要直接返回内部可变对象本身。
- 如果内部元素本身也是可变对象，必要时还要做深拷贝。

#### 典型例子
```java
final class Team {
    private final List<String> members;

    public Team(List<String> members) {
        this.members = new ArrayList<>(members);
    }

    public List<String> getMembers() {
        return new ArrayList<>(members);
    }
}
```

#### 为什么这个例子更接近不可变
- 构造器里复制了一份外部传入的列表，避免外部继续拿原列表修改内部状态。
- `getMembers()` 返回的是新列表，而不是内部那份列表本身。
- 这样外部拿到返回值后怎么改，都不会影响对象内部状态。

#### 还要注意的点
- 如果列表里放的不是 `String` 这种不可变对象，而是 `Person` 这种可变对象，仅复制列表还不够。
- 因为虽然列表壳子变了，但列表里的元素对象可能还是同一批。
- 这时如果要严格不可变，就需要对元素也做深拷贝，或者要求元素本身也是不可变对象。

#### 面试简答版
- 可以创建包含可变对象的不可变对象，但必须通过防御性拷贝来隔离内部状态。
- 也就是说，构造时拷贝一份，返回时再拷贝一份，不能直接暴露内部可变对象引用。
- 如果内部元素本身也是可变的，还要进一步考虑深拷贝，否则它就不是真正严格意义上的不可变对象。

## 12. 值传递和引用传递的区别是什么？为什么说 Java 中只有值传递？

#### 先给结论
- Java 中只有值传递，没有 C++ 那种真正意义上的引用传递。
- 基本类型传递的是“值的副本”。
- 对象传递的不是对象本身，而是“引用这个对象的地址值的副本”。
- 所以很多人觉得 Java 传对象像“引用传递”，其实本质仍然是值传递。

#### 什么是值传递
- 值传递指：方法调用时，把实参的值复制一份传给形参。
- 方法内部操作的是这份副本，不是外面的原变量本身。
- 因此如果在方法里给形参重新赋值，不会影响方法外的原变量。

#### 什么是引用传递
- 引用传递指：把变量本身传进去，方法参数相当于原变量的别名。
- 如果在方法内部修改这个“别名”的指向，外部变量也会跟着变。
- Java 并不支持这种意义上的参数传递方式。

#### 基本类型为什么显然是值传递
```java
public class Test {
    public static void change(int x) {
        x = 100;
    }

    public static void main(String[] args) {
        int a = 10;
        change(a);
        System.out.println(a); // 10
    }
}
```

- 调用 `change(a)` 时，传进去的是 `a` 的值 `10` 的一份副本。
- 方法里的 `x` 改成 `100`，只是改了副本。
- 外面的 `a` 还是原来的 `10`。

#### 为什么对象也还是值传递
```java
class Person {
    String name;
}

public class Test {
    public static void change(Person p) {
        p.name = "Tom";
    }

    public static void main(String[] args) {
        Person person = new Person();
        person.name = "Jack";

        change(person);
        System.out.println(person.name); // Tom
    }
}
```

- 这里传进去的不是 `person` 对象本身，而是 `person` 这个引用变量里保存的地址值副本。
- 外面的 `person` 和方法里的 `p`，虽然是两个不同的引用变量，但它们此时保存的是同一个对象地址。
- 所以通过 `p.name = "Tom"` 修改对象内容，外面的 `person` 看到的对象也被改了。
- 这就是为什么很多人误以为“对象是引用传递”。

#### 真正能证明 Java 不是引用传递的例子
```java
class Person {
    String name;
}

public class Test {
    public static void change(Person p) {
        p = new Person();
        p.name = "Tom";
    }

    public static void main(String[] args) {
        Person person = new Person();
        person.name = "Jack";

        change(person);
        System.out.println(person.name); // Jack
    }
}
```

- 如果 Java 是引用传递，那么方法里执行 `p = new Person();` 后，外面的 `person` 也应该指向新对象。
- 但实际并没有，外面的 `person` 还是原来那个对象，输出仍然是 `Jack`。
- 这说明方法中的 `p` 只是拿到了地址值的一份副本，重新给 `p` 赋值，只会改掉这份副本的指向，不会影响外面的 `person`。

#### 如何从内存角度理解
- 基本类型变量直接保存具体值，比如 `10`、`3.14`、`true`。
- 引用类型变量保存的是对象地址值，可以理解成“对象的位置”。
- Java 调用方法时，不管是基本类型还是引用类型，传的都是变量中保存内容的一份拷贝。
- 对于基本类型，拷贝的是数值本身。
- 对于引用类型，拷贝的是地址值本身。

#### 一句话解释对象参数为什么“能改内容但不能改指向”
- 因为传进去的是“地址值的副本”。
- 两个引用变量虽然不是同一个变量，但它们一开始指向同一个对象，所以能通过它修改对象内容。
- 但是一旦在方法里让参数引用重新指向新对象，只是改了副本，不会影响外部引用。

#### 面试最容易混淆的点
- `person.name = "Tom"` 改的是对象内容，不是把外部引用变量 `person` 本身改掉。
- `p = new Person()` 改的是方法参数 `p` 的指向，不是外部变量 `person` 的指向。
- 所以“能通过参数改对象内容”不等于“Java 是引用传递”。

#### 面试回答模板
- 值传递是把实参的值复制一份传给形参，方法内部修改的是副本；引用传递则是把变量本身传进去，形参相当于原变量别名。
- Java 中基本类型传递的是值副本；引用类型传递的是对象地址值的副本。
- 因此外部对象内容可以通过这个副本引用被修改，但如果在方法里让参数重新指向新对象，外部引用不会受影响。
- 所以 Java 中本质上只有值传递，没有真正意义上的引用传递。

#### 面试简答版
- Java 只有值传递。
- 基本类型传的是值的副本；引用类型传的是地址值的副本。
- 因此方法里可以通过这个副本修改对象内容，但不能改变外部引用本身的指向。
