# IO

## 1. Java 的 IO 流分为几种？

#### 先给结论
- 从数据单位看，Java IO 流分为两大类：
  - 字节流
  - 字符流
- 从流向看，又分为：
  - 输入流
  - 输出流
- 所以面试里最常见的答法是 4 个核心抽象类：
  - `InputStream`
  - `OutputStream`
  - `Reader`
  - `Writer`

#### 核心解释
##### 1）按处理单位划分
- 字节流：以字节为单位处理数据，适合图片、音频、视频、文件等一切二进制数据。
- 字符流：以字符为单位处理数据，适合文本数据，比如 `.txt`、`.java`、`.json`。

##### 2）按流向划分
- 输入流：把数据“读进来”。
- 输出流：把数据“写出去”。

##### 3）按功能再细分
- 节点流：直接连接数据源，比如 `FileInputStream`、`FileReader`
- 处理流 / 包装流：对节点流做增强，比如 `BufferedInputStream`、`BufferedReader`

#### 一个面试里常用的整体图
- 字节输入流：`InputStream`
- 字节输出流：`OutputStream`
- 字符输入流：`Reader`
- 字符输出流：`Writer`

#### 面试易错点 / 补充点
- 不要只答“分两种：输入流和输出流”，这太浅了。
- 更稳的是先按“字节流 / 字符流”答，再补“每类又分输入和输出”。
- 如果面试官继续追问，还可以补“节点流 / 处理流”。

#### 面试回答模板
- Java IO 流最核心可以从两个维度来分。按处理单位分为字节流和字符流；按流向分为输入流和输出流。因此最基础的四个抽象类分别是 `InputStream`、`OutputStream`、`Reader` 和 `Writer`。如果再细分，还可以分成节点流和处理流。

#### 面试简答版
- 按单位：字节流、字符流。
- 按方向：输入流、输出流。
- 四个核心抽象类：`InputStream`、`OutputStream`、`Reader`、`Writer`。

## 2. 字节流如何转为字符流？

#### 先给结论
- 字节流转字符流，核心就是加一个“解码桥梁”：
  - `InputStreamReader`
- 它可以把字节流按指定字符集解码成字符流。

#### 核心解释
- 字节流读到的是原始字节。
- 但字符流处理的是字符。
- 所以中间必须知道“这些字节该按什么编码规则解释”，比如 UTF-8、GBK。
- `InputStreamReader` 的作用就是：
  - 接收一个 `InputStream`
  - 按指定编码把字节解码成字符
  - 对外提供 `Reader` 能力

#### 简单代码示例
```java
import java.io.ByteArrayInputStream;
import java.io.InputStream;
import java.io.InputStreamReader;
import java.io.Reader;
import java.nio.charset.StandardCharsets;

public class Demo {
    public static void main(String[] args) throws Exception {
        byte[] data = "你好".getBytes(StandardCharsets.UTF_8);

        InputStream in = new ByteArrayInputStream(data);
        Reader reader = new InputStreamReader(in, StandardCharsets.UTF_8);

        char[] buf = new char[10];
        int len = reader.read(buf);
        System.out.println(new String(buf, 0, len));
    }
}
```

#### 输出结果
```text
你好
```

#### 输出 / 现象解释
- 原始数据本来是 UTF-8 编码后的字节数组。
- `InputStreamReader` 按 UTF-8 把这些字节重新解码成字符，所以最终能正确输出“你好”。

#### 补充
- 反过来，字符流转字节流一般用：
  - `OutputStreamWriter`

#### 面试易错点 / 补充点
- 不要漏掉“编码”这个关键字。
- 字节流转字符流不是简单包一层就完了，核心是“按什么字符集解码”。
- 如果编码不一致，就会出现乱码。

#### 面试回答模板
- Java 中字节流转字符流通常使用 `InputStreamReader`。它本质上是一个桥接类，可以把 `InputStream` 读取到的字节按照指定字符集解码成字符流，所以实际使用时最好显式指定编码，比如 UTF-8，避免乱码问题。

#### 面试简答版
- 用 `InputStreamReader`。
- 本质：字节 -> 按编码解码 -> 字符。

## 3. 字符流与字节流的区别？

#### 先给结论
- 字节流按字节处理数据。
- 字符流按字符处理数据。
- 文本优先考虑字符流，二进制数据优先考虑字节流。

#### 核心解释
##### 1）处理单位不同
- 字节流一次处理 1 个字节或字节数组。
- 字符流一次处理 1 个字符或字符数组。

##### 2）适用场景不同
- 字节流适合所有类型数据，尤其是图片、音频、视频、压缩包等二进制文件。
- 字符流更适合文本，比如源码、配置文件、日志文件。

##### 3）是否涉及编码不同
- 字节流不直接处理字符编码，它看到的就是原始字节。
- 字符流内部通常会涉及编码和解码。
- 所以字符流本质上很多时候还是建立在字节流基础上的。

#### 简单代码示例
```java
public class Demo {
    public static void main(String[] args) {
        System.out.println("A".getBytes().length);
        System.out.println("你".getBytes(java.nio.charset.StandardCharsets.UTF_8).length);
    }
}
```

#### 输出结果
```text
1
3
```

#### 输出 / 现象解释
- 字符“`A`”在 UTF-8 下通常占 1 个字节。
- 字符“`你`”在 UTF-8 下通常占 3 个字节。
- 这也说明“字符”和“字节”不是一回事，所以 Java 才把字节流和字符流分开设计。

#### 面试易错点 / 补充点
- 不要说“字符流效率一定比字节流高”，这不是绝对结论。
- 字符流更大的意义是“处理文本方便”，不是绝对性能优势。
- 图片、视频这类内容不能用字符流乱读，否则数据会被错误解释。

#### 面试回答模板
- 字节流和字符流的核心区别在于处理单位不同：字节流按字节处理，字符流按字符处理。字节流适合处理所有二进制数据，字符流更适合处理文本。字符流通常会涉及编码解码，因此很多时候本质上也是对字节流的一层封装。

#### 面试简答版
- 字节流：按字节处理，适合二进制。
- 字符流：按字符处理，适合文本。
- 字符流通常涉及编码解码。

## 4. BIO、NIO、AIO 的区别？

#### 先给结论
- `BIO`：同步阻塞 IO
- `NIO`：同步非阻塞 IO
- `AIO`：异步非阻塞 IO

#### 核心解释
##### 1）BIO
- 一个连接、一个线程的思路比较常见。
- 读写时线程会阻塞住，直到数据准备好。
- 编程模型简单，但并发量高时线程开销大。

##### 2）NIO
- 引入了 `Channel`、`Buffer`、`Selector`
- 线程不一定一直阻塞等待某一个连接。
- 一个线程可以轮询多个通道，适合高并发场景。
- 但编程模型比 BIO 复杂。

##### 3）AIO
- 异步非阻塞。
- 发起 IO 操作后，不需要自己一直等结果。
- 等数据准备好后，系统会回调通知你，或者以异步结果形式返回。
- 理论上并发能力更强，但实际项目里使用频率通常没有 NIO 那么高。

#### 一个直观类比
- BIO：你去窗口排队，没办完之前不能走。
- NIO：你不断去看多个窗口哪个空了，再去处理。
- AIO：你办完申请就先走，等好了别人主动通知你。

#### 面试易错点 / 补充点
- NIO 是“同步非阻塞”，不是“异步非阻塞”。
- 很多人容易把 NIO 和 AIO 混掉，这是高频易错点。
- NIO 的核心关键词通常要会说：`Channel`、`Buffer`、`Selector`

#### 面试回答模板
- BIO、NIO、AIO 的核心区别在于阻塞方式和是否异步。BIO 是同步阻塞 IO，线程会一直等到读写完成；NIO 是同步非阻塞 IO，引入 `Channel`、`Buffer` 和 `Selector`，一个线程可以管理多个连接；AIO 是异步非阻塞 IO，发起操作后线程不必等待，等 IO 完成后通过回调或异步结果通知处理。

#### 面试简答版
- BIO：同步阻塞
- NIO：同步非阻塞
- AIO：异步非阻塞

## 5. Java IO 都有哪些设计模式？

#### 先给结论
- Java IO 里最经典、最常考的设计模式主要有：
  - 装饰器模式
  - 适配器模式
  - 工厂模式
  - 模板方法模式

#### 核心解释
##### 1）装饰器模式
- 典型类：`BufferedInputStream`、`BufferedReader`
- 作用：在不改变原始流接口的前提下，给流增强能力，比如加缓冲、提高读取效率。
- 比如你可以在 `FileInputStream` 外面再包一层 `BufferedInputStream`。

##### 2）适配器模式
- 典型类：`InputStreamReader`、`OutputStreamWriter`
- 作用：把字节流适配成字符流，或者把字符流适配成字节流。
- 这就是前面“字节流转字符流”的本质。

##### 3）工厂模式
- 典型场景：`Files.newInputStream()`、`Files.newBufferedReader()`
- 作用：由工厂方法帮你创建合适的流对象，调用者不需要关心太多底层细节。

##### 4）模板方法模式
- 比如 `InputStream`、`Reader` 这类抽象父类定义了统一的读写规范。
- 子类只需要实现具体底层行为。
- 也就是“顶层约定流程，底层子类补充实现”。

#### 简单代码示例
```java
import java.io.BufferedReader;
import java.io.ByteArrayInputStream;
import java.io.InputStreamReader;
import java.nio.charset.StandardCharsets;

public class Demo {
    public static void main(String[] args) throws Exception {
        ByteArrayInputStream in =
                new ByteArrayInputStream("hello".getBytes(StandardCharsets.UTF_8));

        BufferedReader reader =
                new BufferedReader(new InputStreamReader(in, StandardCharsets.UTF_8));

        System.out.println(reader.readLine());
    }
}
```

#### 输出结果
```text
hello
```

#### 输出 / 现象解释
- `InputStreamReader` 把字节流适配成字符流，体现的是适配器模式。
- `BufferedReader` 又在外面做了缓冲增强，体现的是装饰器模式。
- 这段代码是 Java IO 设计思想非常典型的组合写法。

#### 面试易错点 / 补充点
- 面试最稳的是先答装饰器模式和适配器模式，这两个几乎是必答点。
- 工厂模式和模板方法模式属于加分项。
- 不要把“继承体系很多”直接等同于“全是模板方法模式”，要说清具体体现。

#### 面试回答模板
- Java IO 中最典型的设计模式是装饰器模式和适配器模式。比如 `BufferedInputStream`、`BufferedReader` 属于装饰器模式，用来增强原始流能力；`InputStreamReader`、`OutputStreamWriter` 属于适配器模式，用来完成字节流和字符流之间的转换。另外，`Files` 提供的一些创建方法体现了工厂模式，而 `InputStream`、`Reader` 等抽象基类也体现了模板方法思想。

#### 面试简答版
- 最常考：装饰器、适配器。
- 可补充：工厂模式、模板方法模式。
