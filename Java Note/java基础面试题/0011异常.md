# 异常

## 1. Error 和 Exception 区别是什么？

#### 先给结论
- `Throwable` 是 Java 异常体系的根类，下面主要分成两大类：`Error` 和 `Exception`。
- `Error` 一般表示系统级、JVM 级严重问题，程序通常很难自己处理。
- `Exception` 一般表示程序运行过程中出现的问题，通常是程序员应该关注和处理的。

#### 核心解释
- `Error`
  - 更偏“系统出问题了”。
  - 比如内存溢出、栈溢出、类定义加载失败等。
  - 这种问题往往不是你写个 `try-catch` 就能真正解决的。
- `Exception`
  - 更偏“程序执行时出现异常情况”。
  - 比如空指针、数组越界、文件不存在、类型转换失败。
  - 这类问题通常可以通过代码修复、规避或者捕获处理。

#### 一个直观理解
- `Error`：更像“环境/系统层面崩了”。
- `Exception`：更像“业务代码或运行逻辑出问题了”。

#### 面试易错点 / 补充点
- 不要说 “Error 不能捕获”，语法上是可以捕获的，因为它也继承自 `Throwable`。
- 更准确的说法是：`Error` 通常不建议作为常规业务处理对象，因为多数情况下程序已经处于不可靠状态。

#### 面试回答模板
- Java 中 `Throwable` 主要分为 `Error` 和 `Exception`。`Error` 表示 JVM 或系统级的严重问题，比如内存溢出、栈溢出，这类问题通常不适合作为常规业务异常处理；`Exception` 表示程序运行过程中出现的问题，比如空指针、IO 异常等，通常需要程序员关注和处理。

#### 面试简答版
- `Error`：系统级严重问题，通常不作为常规处理对象。
- `Exception`：程序运行异常，通常需要处理。

## 2. 非受检查异常（运行时异常）和受检查异常（一般异常）区别是什么？

#### 先给结论
- 受检查异常：编译器会强制你处理。
- 非受检查异常：编译器不强制处理。
- 对应关系通常是：
  - 受检查异常：`Exception` 下面除 `RuntimeException` 及其子类之外的大多数异常
  - 非受检查异常：`RuntimeException` 及其子类

#### 核心解释
##### 1）受检查异常（Checked Exception）
- 编译阶段就会检查。
- 如果方法可能抛出这类异常，你要么 `try-catch`，要么 `throws`。
- 常见例子：`IOException`、`FileNotFoundException`、`SQLException`

##### 2）非受检查异常（Unchecked Exception）
- 编译器不强制你显式处理。
- 往往表示程序逻辑错误、参数错误、调用方式错误。
- 常见例子：`NullPointerException`、`ArrayIndexOutOfBoundsException`、`ClassCastException`、`ArithmeticException`

#### 简单代码示例
```java
import java.io.FileInputStream;

public class Demo {
    public static void main(String[] args) {
        int a = 10 / 0; // 运行时异常，编译不报必须处理
        System.out.println(a);
    }
}
```

#### 输出结果
```text
Exception in thread "main" java.lang.ArithmeticException: / by zero
```

#### 输出 / 现象解释
- `ArithmeticException` 属于运行时异常。
- 编译器不会强制你写 `try-catch`，但程序运行到这里会直接抛异常。

#### 受检查异常的对比例子
```java
import java.io.FileInputStream;

public class Demo {
    public static void main(String[] args) throws Exception {
        FileInputStream in = new FileInputStream("a.txt");
    }
}
```

#### 输出 / 现象解释
- 如果你不写 `throws Exception` 或 `try-catch`，这段代码在编译阶段就过不去。
- 这就是“受检查异常”的核心特点：编译器强制你处理。

#### 面试易错点 / 补充点
- “运行时异常”不等于“不用管”，只是编译器不强制你处理。
- 受检查异常也不是“更严重”，它只是“编译器要求显式处理”。

#### 面试回答模板
- 受检查异常和非受检查异常的核心区别在于编译器是否强制处理。受检查异常在编译阶段必须通过 `try-catch` 或 `throws` 处理，常见如 `IOException`；非受检查异常主要是 `RuntimeException` 及其子类，编译器不强制处理，常见如空指针、数组越界、算术异常等。

#### 面试简答版
- Checked：编译器强制处理。
- Unchecked：编译器不强制处理。
- `RuntimeException` 及其子类通常属于非受检查异常。

## 3. throw 和 throws 的区别是什么？

#### 先给结论
- `throw` 是“真正抛出一个异常对象”。
- `throws` 是“在方法声明上说明这个方法可能抛出哪些异常”。
- 一个写在方法体里，一个写在方法签名上。

#### 核心解释
##### 1）`throw`
- 用于手动抛异常。
- 后面跟的是一个异常对象。
- 常见写法：`throw new IllegalArgumentException("参数非法");`

##### 2）`throws`
- 用于方法声明。
- 表示“我这个方法可能抛出这些异常，调用者自己决定怎么处理”。

#### 简单代码示例
```java
public class Demo {
    public static void checkAge(int age) {
        if (age < 0) {
            throw new IllegalArgumentException("age < 0");
        }
        System.out.println("ok");
    }

    public static void main(String[] args) {
        checkAge(-1);
    }
}
```

#### 输出结果
```text
Exception in thread "main" java.lang.IllegalArgumentException: age < 0
```

#### 输出 / 现象解释
- 这里是 `throw`，因为代码里真的 new 了一个异常对象并抛出去了。

#### `throws` 示例
```java
import java.io.IOException;

public class Demo {
    public static void read() throws IOException {
        throw new IOException("io error");
    }
}
```

#### 输出 / 现象解释
- `throws IOException` 是告诉调用者：这个方法可能抛出 `IOException`。
- 真正把异常对象抛出去的动作，还是 `throw new IOException(...)`。

#### 面试易错点 / 补充点
- `throw` 后面跟一个异常对象。
- `throws` 后面跟一个或多个异常类型。
- 不要把它们记成“一个抛异常，一个声明异常”这么表面，最好再补一句“一个在方法体里，一个在方法签名上”。

#### 面试回答模板
- `throw` 和 `throws` 的区别在于：`throw` 用于在方法体内手动抛出一个异常对象；`throws` 用于写在方法声明后，表示该方法可能抛出哪些异常类型，由调用者决定捕获还是继续向上抛。

#### 面试简答版
- `throw`：抛异常对象。
- `throws`：声明可能抛出的异常类型。

## 4. NoClassDefFoundError 和 ClassNotFoundException 区别？

#### 先给结论
- `NoClassDefFoundError` 属于 `Error`。
- `ClassNotFoundException` 属于 `Exception`，而且是受检查异常。
- 前者更偏“类在运行时没法正常找到/使用了”，后者更偏“主动加载类时没找到这个类”。

#### 核心解释
##### 1）`ClassNotFoundException`
- 常见于反射、类加载器动态加载类。
- 比如 `Class.forName("com.demo.User")` 时找不到类。
- 这是一个受检查异常，编译器要求你处理。

##### 2）`NoClassDefFoundError`
- 编译时可能是有这个类的，所以代码能编过。
- 但运行时类不见了、依赖缺失了，或者类初始化失败了，结果 JVM 真正使用它时出问题。
- 这更像“运行环境和编译环境不一致”导致的问题。

#### 简单代码示例
```java
public class Demo {
    public static void main(String[] args) throws Exception {
        Class.forName("abc.NotExist");
    }
}
```

#### 输出结果
```text
Exception in thread "main" java.lang.ClassNotFoundException: abc.NotExist
```

#### 输出 / 现象解释
- 这里是程序主动去加载一个类，所以没找到时抛的是 `ClassNotFoundException`。

#### 面试易错点 / 补充点
- 不要只背“一个编译期，一个运行期”，这个说法太粗。
- 更稳的说法是：
  - `ClassNotFoundException`：主动加载类时找不到
  - `NoClassDefFoundError`：类在运行时无法正常被 JVM 使用
- `NoClassDefFoundError` 不一定只是 class 文件丢了，也可能是类初始化阶段失败导致后续无法定义。

#### 面试回答模板
- `ClassNotFoundException` 一般发生在程序主动加载类时，比如 `Class.forName()`，属于受检查异常；`NoClassDefFoundError` 属于 `Error`，通常表示某个类在编译时存在，但运行时 JVM 真正使用它时无法找到定义或无法正常完成加载、链接、初始化，常见于依赖缺失或环境不一致。

#### 面试简答版
- `ClassNotFoundException`：主动加载类没找到，属于异常。
- `NoClassDefFoundError`：运行时类定义不可用，属于错误。

## 5. Java 常见异常有哪些？

#### 先给结论
- 面试里这题一般不是让你背全家桶，而是看你能不能分门别类地说。
- 最稳的答法：按“运行时异常”和“受检查异常”分开说。

#### 常见运行时异常
- `NullPointerException`：空指针
- `ArithmeticException`：算术异常，比如除 0
- `ArrayIndexOutOfBoundsException`：数组下标越界
- `StringIndexOutOfBoundsException`：字符串下标越界
- `ClassCastException`：类型转换异常
- `NumberFormatException`：数字格式异常
- `IllegalArgumentException`：参数非法

#### 常见受检查异常
- `IOException`：IO 异常
- `FileNotFoundException`：文件找不到
- `SQLException`：数据库操作异常
- `ClassNotFoundException`：类找不到
- `ParseException`：解析异常

#### 一个面试视角的回答思路
- 如果是程序员代码逻辑问题，常见是运行时异常。
- 如果是外部资源、文件、数据库、网络这类不确定性问题，常见是受检查异常。

#### 面试易错点 / 补充点
- 别只说异常名，不解释场景，容易显得死记硬背。
- 最好能挑 2~3 个高频异常顺手解释一下触发场景。

#### 面试回答模板
- Java 常见异常我一般分成两类来答。运行时异常包括空指针、数组越界、类型转换异常、数字格式异常、算术异常等，通常和代码逻辑有关；受检查异常包括 `IOException`、`SQLException`、`ClassNotFoundException` 等，通常和文件、数据库、类加载、外部资源访问有关。

#### 面试简答版
- 运行时异常：空指针、越界、类型转换、数字格式、算术异常。
- 受检查异常：IO、SQL、类加载相关异常。

## 6. try-catch-finally 中哪个部分可以省略？

#### 先给结论
- `catch` 可以省略。
- `finally` 也可以省略。
- 但 `catch` 和 `finally` 不能同时都省略。
- 也就是说，`try` 后面至少要跟一个 `catch` 或 `finally`。

#### 核心解释
- 合法写法有：
  - `try-catch`
  - `try-finally`
  - `try-catch-finally`
- 不合法写法是只有一个裸 `try`。

#### 简单代码示例
```java
public class Demo {
    public static void main(String[] args) {
        try {
            System.out.println("try");
        } finally {
            System.out.println("finally");
        }
    }
}
```

#### 输出结果
```text
try
finally
```

#### 输出 / 现象解释
- 这说明 `catch` 是可以省略的。
- `finally` 常用于资源释放或收尾逻辑。

#### 面试易错点 / 补充点
- 很多人会脱口而出“finally 可以省略”，但忘了 `catch` 也能省略。
- 更完整的答法是：`catch` 和 `finally` 都可以单独省略，但不能同时省略。

#### 面试回答模板
- 在 `try-catch-finally` 结构中，`catch` 和 `finally` 都可以省略，但不能同时省略。也就是说，`try` 后面至少要有一个 `catch` 或一个 `finally`，所以合法形式包括 `try-catch`、`try-finally` 和 `try-catch-finally`。

#### 面试简答版
- `catch` 可省略。
- `finally` 也可省略。
- 但不能都省略。

## 7. try-catch-finally 中，如果 catch 中 return 了，finally 还会执行吗？

#### 先给结论
- 会执行。
- 即使 `catch` 里有 `return`，`finally` 通常也还是会执行。
- 但如果 `finally` 里自己也有 `return`，会把前面的返回结果覆盖掉。

#### 核心解释
- `return` 不是说一写出来，方法立刻什么都不干直接结束。
- 更准确地说是：
  - 先确定返回值
  - 然后执行 `finally`
  - 最后再真正返回
- 所以 `finally` 在很多情况下都能执行到。

#### 简单代码示例
```java
public class Demo {
    public static int test() {
        try {
            int a = 1 / 0;
            return 1;
        } catch (Exception e) {
            System.out.println("catch");
            return 2;
        } finally {
            System.out.println("finally");
        }
    }

    public static void main(String[] args) {
        System.out.println(test());
    }
}
```

#### 输出结果
```text
catch
finally
2
```

#### 输出 / 现象解释
- `catch` 里虽然执行了 `return 2`，但 `finally` 仍然执行了。
- 最后方法返回值还是 2。

#### 补充：finally 里也 return 会怎样
```java
public class Demo {
    public static int test() {
        try {
            return 1;
        } finally {
            return 3;
        }
    }
}
```

#### 输出 / 现象解释
- 最终返回的是 3，不是 1。
- 因为 `finally` 里的 `return` 把前面的返回结果覆盖掉了。

#### 面试易错点 / 补充点
- “`finally` 一定执行”这个说法也别说得太绝对。
- 像 `System.exit(0)`、进程被强杀、机器断电这类极端情况，`finally` 也可能执行不到。
- 日常面试里一般默认讨论正常 JVM 执行流程。

#### 面试回答模板
- 在 `try-catch-finally` 中，如果 `catch` 里执行了 `return`，`finally` 通常还是会执行。因为 JVM 会先保存返回结果，再执行 `finally`，最后真正返回。但如果 `finally` 里自己又写了 `return`，会覆盖前面 `try` 或 `catch` 的返回值，所以实际开发中不建议在 `finally` 中写 `return`。

#### 面试简答版
- 会执行。
- 但 `finally` 里如果再 `return`，会覆盖前面的返回值。

## 8. JVM 是如何处理异常的？

#### 先给结论
- JVM 处理异常的核心流程可以概括成 4 步：
  - 发生异常，创建异常对象
  - 按当前方法的异常表查找匹配的处理器
  - 如果当前方法处理不了，就沿调用栈向上抛
  - 一直没人处理，最终交给线程默认异常处理机制，程序结束或线程结束

#### 核心解释
##### 1）异常发生
- 当代码执行出错，比如除 0、空指针、数组越界，JVM 会创建对应异常对象并准备抛出。
- 如果是你手动 `throw`，本质也是把一个异常对象抛出。

##### 2）当前方法查找能不能处理
- 编译后的字节码里有异常表，可以理解成“这段代码出问题时，跳到哪个 `catch` 处理”。
- JVM 会先在当前方法里找有没有匹配的 `catch`。

##### 3）当前方法处理不了，就栈展开
- 如果当前方法没有匹配的处理器，就退出当前方法，把异常继续抛给调用者。
- 这个一层层往上找处理器的过程，就叫栈展开。

##### 4）最终没人处理
- 如果一路抛到最顶层还没人处理，线程会终止。
- 我们平时看到控制台打印异常栈信息，本质上就是默认异常处理机制在工作。

#### 简单代码示例
```java
public class Demo {
    public static void a() {
        b();
    }

    public static void b() {
        int x = 1 / 0;
    }

    public static void main(String[] args) {
        try {
            a();
        } catch (Exception e) {
            System.out.println("main catch");
        }
    }
}
```

#### 输出结果
```text
main catch
```

#### 输出 / 现象解释
- 异常是在 `b()` 里产生的。
- `b()` 没处理，就往上抛给 `a()`。
- `a()` 也没处理，再抛给 `main()`。
- `main()` 里的 `catch` 接住了，所以程序没有直接崩掉。

#### 面试易错点 / 补充点
- JVM 不是“看到异常就立刻退出程序”，而是会先找有没有匹配的异常处理器。
- 异常处理和正常流程是两套执行路径，异常路径会触发栈展开。
- `finally` 之所以很多时候能执行，就是因为编译器/JVM 会在异常路径和正常路径里都安排收尾逻辑。

#### 面试回答模板
- JVM 处理异常时，首先会生成对应的异常对象，然后根据当前方法字节码中的异常表查找是否有匹配的 `catch` 处理器；如果当前方法处理不了，就会退出当前栈帧并把异常继续抛给上层调用者，这个过程叫栈展开。如果一路向上都没有匹配的处理器，最终就由线程默认异常处理机制输出异常栈并结束线程或程序。

#### 面试简答版
- 发生异常 -> 查当前方法异常表 -> 当前方法处理不了就栈展开 -> 顶层还没人处理就终止并打印异常栈。
