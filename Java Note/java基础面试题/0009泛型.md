# 泛型

## 1. Java 中的泛型是什么？

#### 先给结论
- 泛型本质上就是“把类型也参数化”，让代码在编译期就约束你只能操作某种类型的数据。
- 它的核心价值不是语法好看，而是：类型安全、减少强转、提升代码复用性。
- 泛型可以用在类、接口、方法上，所以常见有泛型类、泛型接口、泛型方法。

#### 核心解释
- 你可以把泛型理解成“给类型留一个占位符”，真正使用时再把具体类型填进去。
- 比如 `List<E>` 里的 `E`，本质上就是一个类型变量，等你写成 `List<String>` 时，才明确这个集合里放的是 `String`。
- 没有泛型时，很多容器只能按 `Object` 来处理，放什么都能放，但取出来时要自己强转，既麻烦又不安全。
- 有了泛型后，编译器会帮你在编译阶段检查类型，很多问题不用等到运行时才暴露。

#### 简单代码示例
### 1）不使用泛型
```java
import java.util.ArrayList;
import java.util.List;

public class Demo {
    public static void main(String[] args) {
        List list = new ArrayList();
        list.add("hello");
        list.add(123);

        String s = (String) list.get(0);
        System.out.println(s);
    }
}
```

输出结果：
```java
hello
```

输出 / 现象解释：
- 这段代码能运行，是因为第 0 个元素刚好是 `"hello"`。
- 但这个 `list` 里其实也混进了 `123`，说明它没有类型约束。
- 如果你去拿第 1 个元素再强转成 `String`，运行时就会报 `ClassCastException`。

### 2）使用泛型
```java
import java.util.ArrayList;
import java.util.List;

public class Demo {
    public static void main(String[] args) {
        List<String> list = new ArrayList<>();
        list.add("hello");
        // list.add(123);  编译直接报错

        String s = list.get(0);
        System.out.println(s);
    }
}
```

输出结果：
```java
hello
```

输出 / 现象解释：
- 这里 `List<String>` 已经明确规定只能放 `String`。
- 所以 `list.add(123)` 在编译阶段就过不去。
- 同时取出元素时也不需要再手动强转。

### 3）泛型方法
```java
public class Demo {
    public static <T> void print(T value) {
        System.out.println(value);
    }

    public static void main(String[] args) {
        print("abc");
        print(123);
    }
}
```

输出结果：
```java
abc
123
```

输出 / 现象解释：
- 这里的 `<T>` 不是类上的泛型，而是方法自己的泛型参数。
- 说明泛型不只可以写在类上，也可以写在方法上。

#### 泛型一般有哪些形式
### 1）泛型类
```java
class Box<T> {
    private T value;

    public void set(T value) {
        this.value = value;
    }

    public T get() {
        return value;
    }
}
```

### 2）泛型接口
```java
interface Generator<T> {
    T next();
}
```

### 3）泛型方法
```java
public static <T> T getFirst(T a, T b) {
    return a;
}
```

#### 面试易错点 / 补充点
- 泛型的重点不是“能存不同类型”，而是“在使用前先约束类型”。
- 泛型的类型检查主要发生在编译期，不是运行期。
- Java 泛型本质上是伪泛型，底层依赖类型擦除，这个通常是下一问高频追问点。
- `List<String>` 和 `List<Integer>` 在运行时的类信息其实是一样的，这和类型擦除有关。

#### 面试回答模板
- 泛型是 Java 1.5 引入的特性，本质上就是把类型参数化，也就是把原来写死的具体类型变成一个类型变量。
- 它的主要作用是提升类型安全、减少强制类型转换，并提高代码复用性。
- 泛型可以用在类、接口和方法上，所以常见有泛型类、泛型接口和泛型方法。
- 比如 `List<String>` 就表示这个集合里只能放 `String`，这样编译器会在编译期帮我们做类型检查。

#### 面试简答版
- 泛型就是把类型参数化，让类型在使用时再确定。
- 它的核心作用是：类型安全、减少强转、提高复用性。
- 泛型可以用于类、接口、方法，比如 `List<String>`、`Class<T>`、`<T> void test(T t)`。

## 2. 使用泛型的好处是什么？

#### 先给结论
- 泛型最核心的好处有 3 个：类型安全、减少强制类型转换、提升代码复用性和可读性。
- 面试里最稳的回答顺序是：先说编译期类型检查，再说减少强转，最后说更通用的抽象能力。
- 不要把“性能更高”当成泛型的核心好处，Java 泛型主要是编译期能力，底层有类型擦除。

#### 核心解释
- 没有泛型时，很多通用容器和通用方法只能用 `Object` 接数据。
- 这样做的问题是：放进去时没有约束，取出来时要自己强转，错误往往拖到运行时才暴露。
- 有了泛型后，编译器可以提前知道“这里应该是什么类型”，很多类型错误在编译阶段就被拦下来了。
- 所以泛型的价值，本质上是把很多原本运行时的类型问题，前移到编译期解决。

#### 简单代码示例
### 1）提高类型安全
```java
import java.util.ArrayList;
import java.util.List;

public class Demo {
    public static void main(String[] args) {
        List<String> list = new ArrayList<>();
        list.add("hello");
        // list.add(123);  编译报错

        System.out.println(list.get(0));
    }
}
```

输出结果：
```java
hello
```

输出 / 现象解释：
- `List<String>` 明确规定集合里只能放 `String`。
- 如果你想塞一个 `Integer`，编译器会直接报错。
- 这就是泛型最大的价值：把类型错误尽量提前到编译期发现。

### 2）减少强制类型转换
```java
import java.util.ArrayList;
import java.util.List;

public class Demo {
    public static void main(String[] args) {
        List<String> list = new ArrayList<>();
        list.add("java");

        String s = list.get(0);
        System.out.println(s);
    }
}
```

输出结果：
```java
java
```

输出 / 现象解释：
- 如果没有泛型，通常要写成 `(String) list.get(0)`。
- 有了泛型后，编译器已经知道返回值就是 `String`，所以不需要你手动强转。
- 这会让代码更干净，也降低了写错强转类型的概率。

### 3）提高代码复用性
```java
public class Demo {
    public static <T> T getFirst(T a, T b) {
        return a;
    }

    public static void main(String[] args) {
        System.out.println(getFirst("A", "B"));
        System.out.println(getFirst(10, 20));
    }
}
```

输出结果：
```java
A
10
```

输出 / 现象解释：
- 同一个方法不需要分别写 `String` 版、`Integer` 版。
- 用一个泛型方法就能处理多种类型。
- 这就是泛型在“通用代码设计”上的价值。

#### 泛型带来的几个实际收益
### 1）编译期类型检查，更安全
- 泛型的第一价值就是类型安全。
- 它能把很多本该运行时出现的 `ClassCastException`，提前到编译期暴露。
- 这符合“越早发现问题越好”的工程原则。

### 2）减少显式强转，代码更清晰
- 不用到处写 `(String)`、`(User)` 这种类型转换。
- 代码可读性更好，也更不容易写错。

### 3）增强通用性和复用性
- 泛型让类、接口、方法都可以写得更抽象。
- 比如集合框架、比较器、工具类，大量依赖泛型提升复用性。

### 4）API 语义更明确
- `List<String>` 一眼就知道是“字符串列表”。
- `Map<String, Integer>` 一眼就知道 key 和 value 的类型。
- 这其实也是泛型的重要价值：增强接口表达能力。

#### 面试易错点 / 补充点
- 泛型的核心收益是编译期类型安全，不是运行时性能优化。
- Java 泛型有类型擦除，所以很多泛型信息在运行时并不会完整保留。
- “减少强转”不是独立于“类型安全”的另一套原理，它其实是编译期类型约束带来的直接结果。
- 面试时如果说“泛型提高性能”，要非常谨慎，最多说它减少了一些手写强转和错误概率，不要说成主要优势。

#### 面试回答模板
- 使用泛型的主要好处有三个：第一是提高类型安全，编译器可以在编译阶段检查类型是否正确；第二是减少强制类型转换，让代码更简洁、更不容易出错；第三是提升代码复用性，可以把类、接口、方法设计得更通用。
- 比如 `List<String>` 表示集合中只能放 `String`，这样既避免了运行时的错误类型混入，也不需要在取值时再手动强转。
- 所以泛型本质上是把很多类型问题从运行时提前到编译期解决。

#### 面试简答版
- 泛型的好处主要是：类型安全、减少强转、提高复用性。
- 它能让很多类型错误在编译期就暴露，而不是拖到运行时报 `ClassCastException`。
- 同时像 `List<String>` 这种写法，语义也更清晰。

## 3. Java 泛型的原理是什么？什么是类型擦除？

#### 先给结论
- Java 泛型的底层原理就是类型擦除。
- 所谓类型擦除，就是：泛型信息主要在编译期生效，编译后很多具体类型参数会被擦掉，字节码里不会按 `List<String>`、`List<Integer>` 这种形式保留两套不同类型。
- 所以 Java 泛型本质上是“编译期泛型”，不是为每种具体类型都重新生成一套独立运行时代码。

#### 核心解释
- 你写代码时可以写 `List<String>`、`List<Integer>`，编译器会用这些信息帮你做类型检查。
- 但编译完成后，JVM 看到的很多地方其实是“普通类 + 编译器补上的类型转换”。
- 简单说就是：编译器先利用泛型保证你写代码时更安全，然后在生成字节码时把大部分泛型类型信息抹平。
- 这就是为什么我们常说：泛型主要存在于编译阶段。

#### 类型擦除到底擦掉了什么
### 1）擦掉类型参数
- 比如 `T`、`E`、`K`、`V` 这种类型变量，在编译后不会原样保留成运行时真实类型。
- 如果没有上界，通常会擦除成 `Object`。
- 如果有上界，比如 `T extends Number`，通常会擦除成它的上界 `Number`。

### 2）编译器会在需要的地方补强转
- 泛型擦除后，编译器为了保证类型语义一致，会在取值等位置自动帮你插入必要的强制类型转换。
- 所以你平时写 `String s = list.get(0);` 看起来没强转，但编译器底层可能帮你补了。

### 3）运行时很多泛型信息不再可区分
- 比如 `List<String>` 和 `List<Integer>`，运行时通常是同一个 `Class`。
- 这也是很多泛型限制的根源，比如不能直接 `new T()`，不能 `T.class`，不能判断 `obj instanceof List<String>`。

#### 简单代码示例
### 1）无边界泛型擦除成 `Object`
```java
class Box<T> {
    private T value;

    public T get() {
        return value;
    }
}
```

你可以把它粗略理解成编译后更接近：

```java
class Box {
    private Object value;

    public Object get() {
        return value;
    }
}
```

输出结果：
- 这不是运行示例，而是“编译后大致会变成什么样”的理解示意。

输出 / 现象解释：
- 这里 `T` 没有上界，所以擦除后通常按 `Object` 处理。
- 这就是很多文章里说“泛型最终会变成 `Object`”的来源。

### 2）有边界泛型擦除成上界类型
```java
class Box<T extends Number> {
    private T value;

    public T get() {
        return value;
    }
}
```

粗略理解成编译后更接近：

```java
class Box {
    private Number value;

    public Number get() {
        return value;
    }
}
```

输出结果：
- 同样这是结构示意，不是直接运行输出。

输出 / 现象解释：
- 这里 `T` 有上界 `Number`，所以擦除后不是 `Object`，而是 `Number`。
- 这点很重要，面试里经常会追问“是不是所有泛型都擦成 `Object`”，答案是不是。

### 3）运行时类信息相同
```java
import java.util.ArrayList;
import java.util.List;

public class Demo {
    public static void main(String[] args) {
        List<String> list1 = new ArrayList<>();
        List<Integer> list2 = new ArrayList<>();

        System.out.println(list1.getClass() == list2.getClass());
    }
}
```

输出结果：
```java
true
```

输出 / 现象解释：
- `List<String>` 和 `List<Integer>` 编译期不同。
- 但运行时拿到的类对象是同一个。
- 这正是类型擦除最经典的现象。

### 4）编译器自动补强转
```java
import java.util.ArrayList;
import java.util.List;

public class Demo {
    public static void main(String[] args) {
        List<String> list = new ArrayList<>();
        list.add("java");

        String s = list.get(0);
        System.out.println(s);
    }
}
```

输出结果：
```java
java
```

输出 / 现象解释：
- 源码里你没写显式强转。
- 但编译器在底层可能已经替你补上了从 `Object` 到 `String` 的转换逻辑。
- 所以泛型并不是“运行时天然知道类型”，而是“编译器提前帮你把类型规则处理好了”。

#### 为什么 Java 要这样设计
- 因为 Java 当年引入泛型时，需要尽量兼容老版本字节码和已有类库。
- 如果为每种类型都重新生成一套完全不同的运行时代码，兼容成本会更高。
- 所以 Java 选择了“编译期检查 + 运行期擦除”的折中方案。

#### 类型擦除带来的典型限制
### 1）不能直接 `new T()`
```java
class Demo<T> {
    public T create() {
        // return new T();  编译报错
        return null;
    }
}
```

原因：
- 因为运行时并不知道 `T` 到底是什么具体类型。

### 2）不能写 `T.class`
```java
class Demo<T> {
    public void test() {
        // System.out.println(T.class);  编译报错
    }
}
```

原因：
- 擦除后运行时拿不到这个具体类型字面量。

### 3）不能判断 `instanceof List<String>`
```java
Object obj = new ArrayList<String>();
// if (obj instanceof List<String>) { }  编译报错
```

原因：
- 运行时无法区分 `List<String>` 和 `List<Integer>`。

#### 面试易错点 / 补充点
- “泛型只存在于编译期”这句话要稍微严谨一点：更准确地说，是大部分泛型类型参数在编译后会被擦除，运行时主要依赖擦除后的类型。
- 不是所有泛型都擦成 `Object`；有上界时，会擦成上界类型。
- 类型擦除不等于“泛型完全没用”，泛型的价值主要体现在编译期类型检查和更清晰的 API 设计。
- 很多文章把“反编译后长得像 `Object`”直接等价成全部原理，这不完整；你还要补一句“编译器会自动插入必要的类型转换逻辑”。

#### 面试回答模板
- Java 泛型的底层原理是类型擦除，也就是泛型主要在编译期生效，编译器利用泛型做类型检查，编译完成后再把很多具体类型参数擦除掉。
- 如果类型变量没有上界，通常会擦除成 `Object`；如果有上界，比如 `T extends Number`，通常会擦除成它的上界类型。
- 同时，编译器会在需要的地方自动补充强制类型转换，以保证源码层面的泛型语义成立。
- 所以 Java 泛型本质上是编译期泛型，这也是为什么 `List<String>` 和 `List<Integer>` 在运行时是同一个 `Class`。

#### 面试简答版
- Java 泛型的原理就是类型擦除。
- 编译器先用泛型做类型检查，编译后再把很多类型参数擦掉，无边界通常擦成 `Object`，有上界就擦成上界类型。
- 所以泛型主要是编译期能力，运行时很多具体泛型类型并不能直接拿到。

## 4. 什么是泛型中的限定通配符和非限定通配符？

#### 先给结论
- 通配符就是 `?`，表示“某种未知类型”。
- 非限定通配符是 `<?>`，意思是“类型未知，但不关心具体是什么”。
- 限定通配符分两种：`<? extends T>` 和 `<? super T>`。
- `extends` 适合读数据，`super` 适合写数据，这个规律面试里直接记 `PECS`：Producer Extends，Consumer Super。

#### 核心解释
- 我们先区分“类型参数”和“通配符”：
  - `List<T>` 里的 `T` 是类型参数，一般出现在类、接口、方法定义时。
  - `List<?>`、`List<? extends Number>`、`List<? super Integer>` 里的 `?` 是通配符，一般出现在“使用别人定义好的泛型类型”时。
- 通配符的核心作用是：当我们不知道具体类型，或者只关心类型范围时，用它来描述“可以接受哪些泛型对象”。
- 所以这题的重点不是背定义，而是搞清楚三种写法分别能“读什么、写什么、接收什么”。

#### 1）非限定通配符 `<?>` 是什么
- `<?>` 表示“可以是任意类型”，但具体类型未知。
- 它常用于：我只想把这个泛型对象当成一个“泛型容器”来处理，但不关心里面到底是 `String`、`Integer` 还是别的。

```java
import java.util.Arrays;
import java.util.List;

public class Demo {
    public static void printList(List<?> list) {
        for (Object obj : list) {
            System.out.println(obj);
        }
    }

    public static void main(String[] args) {
        printList(Arrays.asList("A", "B"));
        printList(Arrays.asList(1, 2));
    }
}
```

输出结果：
```java
A
B
1
2
```

输出 / 现象解释：
- `printList(List<?>)` 可以接收 `List<String>`，也可以接收 `List<Integer>`。
- 因为这里我们不关心具体元素类型，只是把它们当 `Object` 读出来。
- 但你不能往 `List<?>` 里随便加元素，因为你不知道它真实的元素类型到底是什么。

#### 2）上界通配符 `<? extends T>` 是什么
- `<? extends T>` 表示“某个未知类型，但它一定是 `T` 或 `T` 的子类”。
- 它适合从集合里读数据，因为读出来至少可以当成 `T` 用。
- 但不适合往里写数据，因为你不知道它到底是 `T`、`T` 的哪个子类。

```java
import java.util.Arrays;
import java.util.List;

public class Demo {
    public static double sum(List<? extends Number> list) {
        double result = 0;
        for (Number n : list) {
            result += n.doubleValue();
        }
        return result;
    }

    public static void main(String[] args) {
        System.out.println(sum(Arrays.asList(1, 2, 3)));
        System.out.println(sum(Arrays.asList(1.5, 2.5)));
    }
}
```

输出结果：
```java
6.0
4.0
```

输出 / 现象解释：
- `List<? extends Number>` 可以接收 `List<Integer>`、`List<Double>`。
- 因为它们的元素都至少能当成 `Number` 来读。
- 但你不能写 `list.add(1)`，因为如果真实类型是 `List<Double>`，那加 `Integer` 就不安全了。

#### 3）下界通配符 `<? super T>` 是什么
- `<? super T>` 表示“某个未知类型，但它一定是 `T` 或 `T` 的父类”。
- 它适合写数据，因为你往里面放 `T` 一定安全。
- 但读数据时就不够精确，只能按 `Object` 来接。

```java
import java.util.ArrayList;
import java.util.List;

public class Demo {
    public static void addNumbers(List<? super Integer> list) {
        list.add(1);
        list.add(2);
    }

    public static void main(String[] args) {
        List<Number> list = new ArrayList<>();
        addNumbers(list);
        System.out.println(list);
    }
}
```

输出结果：
```java
[1, 2]
```

输出 / 现象解释：
- `List<? super Integer>` 可以接收 `List<Integer>`、`List<Number>`、`List<Object>`。
- 因为这些类型都能安全地装下 `Integer`。
- 但如果你从里面读元素，编译器只能保证那是 `Object`，不能直接当成 `Integer`。

#### 三种通配符的核心区别
### 1）`<?>`
- 表示任意未知类型。
- 适合“只看不改”或者“我不关心具体类型”的场景。

### 2）`<? extends T>`
- 表示上界，接收 `T` 及其子类。
- 适合读，不适合写。

### 3）`<? super T>`
- 表示下界，接收 `T` 及其父类。
- 适合写，读出来通常只能当 `Object`。

#### 一句话记忆
- `<?>`：不知道是什么类型
- `<? extends T>`：最多把它当 `T` 来读
- `<? super T>`：可以往里放 `T`
- 口诀：`PECS` -> Producer Extends，Consumer Super

#### 一个最容易记住的对比例子
```java
import java.util.ArrayList;
import java.util.List;

public class Demo {
    public static void main(String[] args) {
        List<? extends Number> list1 = new ArrayList<Integer>();
        List<? super Integer> list2 = new ArrayList<Number>();

        // list1.add(1);   编译报错
        list2.add(1);

        Number n = list1.isEmpty() ? 0 : list1.get(0);
        Object o = list2.get(0);

        System.out.println(n);
        System.out.println(o);
    }
}
```

输出结果：
```java
0
1
```

输出 / 现象解释：
- `list1` 是 `extends`，所以适合读成 `Number`，但不适合写。
- `list2` 是 `super`，所以适合写入 `Integer`，但读出来只能按 `Object` 接。
- 这就是上下界通配符最本质的区别。

#### 面试易错点 / 补充点
- `List<Object>` 不是 `List<String>` 的父类型，泛型本身默认不具备协变关系。
- `<?>` 不是“什么都能放进去”，恰恰相反，它基本不能安全地放任何非 `null` 元素。
- `extends` 不是“继承专用语法”，在通配符这里表示上界限制。
- `super` 不是“只能接父类本身”，而是可以接 `T` 以及 `T` 的父类。
- 真正常见的工程经验就是一句话：需要读，用 `extends`；需要写，用 `super`。

#### 面试回答模板
- 通配符就是 `?`，表示未知类型。非限定通配符 `<?>` 表示任意未知类型，一般用于我们不关心具体泛型参数的场景。
- 限定通配符分为两种：`<? extends T>` 表示上界，接收 `T` 及其子类，适合读；`<? super T>` 表示下界，接收 `T` 及其父类，适合写。
- 这类问题面试里通常可以用 `PECS` 来记忆，也就是 Producer Extends，Consumer Super。

#### 面试简答版
- `<?>`：任意未知类型。
- `<? extends T>`：上界，适合读。
- `<? super T>`：下界，适合写。
- 记忆口诀：`PECS`。

## 5. `List<? extends T>` 和 `List<? super T>` 之间有什么区别？

#### 先给结论
- `List<? extends T>` 表示元素类型是 `T` 或 `T` 的某个子类，适合读，不适合写。
- `List<? super T>` 表示元素类型是 `T` 或 `T` 的某个父类，适合写，不适合精确读。
- 它们最大的区别，本质上就是：一个强调“往上兼容读取”，一个强调“往下兼容写入”。

#### 核心解释
- `extends` 表示上界：你只知道这个集合里的东西，至少可以当成 `T` 来看。
- 所以从里面取数据时是安全的，因为无论真实类型是 `T` 还是 `T` 的子类，读出来至少都是 `T`。
- 但你不能随便往里放 `T`，因为真实类型可能是某个更具体的子类。
- `super` 表示下界：你只知道这个集合一定能接收 `T`。
- 所以往里面放 `T` 是安全的，但取出来时只能保守地当成 `Object`。

#### 1）`List<? extends T>` 的特点
- 能接收 `List<T>` 和 `List<T 的子类>`。
- 适合读取。
- 不适合写入除 `null` 之外的元素。

```java
import java.util.Arrays;
import java.util.List;

public class Demo {
    public static double sum(List<? extends Number> list) {
        double result = 0;
        for (Number n : list) {
            result += n.doubleValue();
        }
        return result;
    }

    public static void main(String[] args) {
        System.out.println(sum(Arrays.asList(1, 2, 3)));
        System.out.println(sum(Arrays.asList(1.5, 2.5)));
    }
}
```

输出结果：
```java
6.0
4.0
```

输出 / 现象解释：
- `List<? extends Number>` 可以接收 `List<Integer>`、`List<Double>`。
- 因为这些元素读出来都至少能当成 `Number`。
- 但不能写 `list.add(1)`，因为如果真实类型是 `List<Double>`，往里放 `Integer` 就不安全。

#### 2）`List<? super T>` 的特点
- 能接收 `List<T>` 和 `List<T 的父类>`。
- 适合写入。
- 读取时通常只能拿到 `Object`。

```java
import java.util.ArrayList;
import java.util.List;

public class Demo {
    public static void addIntegers(List<? super Integer> list) {
        list.add(1);
        list.add(2);
    }

    public static void main(String[] args) {
        List<Number> list = new ArrayList<>();
        addIntegers(list);
        System.out.println(list);
    }
}
```

输出结果：
```java
[1, 2]
```

输出 / 现象解释：
- `List<? super Integer>` 可以接收 `List<Integer>`、`List<Number>`、`List<Object>`。
- 因为这些类型都能安全地容纳 `Integer`。
- 但如果从里面 `get()`，编译器只能保证拿到的是 `Object`，不能直接当成 `Integer`。

#### 3）最直观的对比
```java
import java.util.ArrayList;
import java.util.List;

public class Demo {
    public static void main(String[] args) {
        List<? extends Number> list1 = new ArrayList<Integer>();
        List<? super Integer> list2 = new ArrayList<Number>();

        // list1.add(1);  编译报错
        list2.add(1);

        // Integer a = list2.get(0);  编译报错
        Object b = list2.get(0);

        System.out.println(b);
    }
}
```

输出结果：
```java
1
```

输出 / 现象解释：
- `list1` 是 `extends`，所以可以安全地按 `Number` 读，但不能安全地写。
- `list2` 是 `super`，所以可以安全地写入 `Integer`，但读出来只能按 `Object` 接。

#### 一张表记住
| 写法 | 含义 | 适合读/写 | 读出来是什么 | 能不能写入 `T` |
| --- | --- | --- | --- | --- |
| `List<? extends T>` | `T` 或子类 | 适合读 | `T` | 不能 |
| `List<? super T>` | `T` 或父类 | 适合写 | `Object` | 能 |

#### 面试易错点 / 补充点
- `extends` 不是“只能接子类，不接自己”，它也能接 `T` 本身。
- `super` 也不是“只能接父类，不接自己”，它同样能接 `T` 本身。
- `extends` 不能写，不是因为语法小气，而是因为编译器不知道真实子类型到底是谁。
- `super` 读不出 `T`，不是因为里面没有 `T`，而是因为编译器只能保守推断到 `Object`。
- 面试里如果只记一句，就记 `PECS`：Producer Extends，Consumer Super。

#### 面试回答模板
- `List<? extends T>` 和 `List<? super T>` 的核心区别在于读写能力不同。
- 前者表示元素类型是 `T` 或 `T` 的子类，适合读取，因为取出来至少可以当成 `T`；但不适合写入，因为真实类型可能是某个更具体的子类。
- 后者表示元素类型是 `T` 或 `T` 的父类，适合写入，因为往里面放 `T` 一定安全；但读取时只能保守地当成 `Object`。
- 所以这两个写法通常用 `PECS` 来记忆：Producer Extends，Consumer Super。

#### 面试简答版
- `? extends T`：上界，适合读，不适合写。
- `? super T`：下界，适合写，读出来通常是 `Object`。
- 口诀：`PECS`。

## 6. 可以把 `List<String>` 传递给一个接受 `List<Object>` 参数的方法吗？

#### 先给结论
- 不可以。
- `List<String>` 不是 `List<Object>` 的子类型。
- 虽然 `String` 是 `Object` 的子类，但泛型容器本身在 Java 里默认不是协变的。

#### 核心解释
- 很多人看到 `String extends Object`，就会自然以为 `List<String>` 也应该能当成 `List<Object>` 用。
- 但 Java 泛型不是这么设计的。
- 如果允许这样传，类型安全会被破坏。
- 所以 Java 明确规定：`List<String>` 和 `List<Object>` 是两个没有继承关系的不同泛型类型。

#### 为什么不能这样传
- 假设真的允许把 `List<String>` 当成 `List<Object>` 传进去。
- 那么方法内部就可以往这个 `List<Object>` 里加入任意 `Object`，比如 `Integer`。
- 这样外面原本的 `List<String>` 就被塞进了一个 `Integer`，类型约束就彻底失效了。
- 所以编译器会直接禁止这种赋值和传参。

#### 简单代码示例
### 1）直接传会编译报错
```java
import java.util.ArrayList;
import java.util.List;

public class Demo {
    public static void print(List<Object> list) {
        System.out.println(list);
    }

    public static void main(String[] args) {
        List<String> list = new ArrayList<>();
        list.add("A");

        // print(list);  编译报错
    }
}
```

输出结果：
- 这段代码运行不了，因为在编译阶段就报错了。

输出 / 现象解释：
- 报错的根本原因不是 `String` 不能转成 `Object`。
- 而是 `List<String>` 不能转成 `List<Object>`。
- 这里要区分“元素的继承关系”和“泛型类型本身的继承关系”，这两个不是一回事。

### 2）如果真允许，会发生什么问题
```java
import java.util.ArrayList;
import java.util.List;

public class Demo {
    public static void addObject(List<Object> list) {
        list.add(123);
    }

    public static void main(String[] args) {
        List<String> strings = new ArrayList<>();
        strings.add("hello");

        // 如果这一句被允许：
        // addObject(strings);
    }
}
```

输出结果：
- 这段代码同样不能按注释那样真正运行，因为编译器会阻止。

输出 / 现象解释：
- 如果 `addObject(strings)` 被允许，那方法内部就能往本来只该存 `String` 的列表里塞入 `123`。
- 这会直接破坏 `List<String>` 的类型安全。
- 这就是为什么 Java 不允许 `List<String>` 传给 `List<Object>` 参数。

#### 那什么写法可以接收 `List<String>`？
### 1）如果只是读取，用 `List<?>`
```java
import java.util.Arrays;
import java.util.List;

public class Demo {
    public static void print(List<?> list) {
        for (Object obj : list) {
            System.out.println(obj);
        }
    }

    public static void main(String[] args) {
        print(Arrays.asList("A", "B"));
    }
}
```

输出结果：
```java
A
B
```

输出 / 现象解释：
- `List<?>` 可以接收 `List<String>`、`List<Integer>` 等各种具体泛型列表。
- 因为这里我们只读，不尝试往里乱写。

### 2）如果需要某个范围，用通配符
- 想读 `Number` 体系的数据，用 `List<? extends Number>`。
- 想安全写入 `Integer`，用 `List<? super Integer>`。
- 真正的解决方案通常不是写 `List<Object>`，而是根据需求选择合适的通配符。

#### 一个最容易记住的对比
```java
List<Object> list1 = new ArrayList<Object>();
List<String> list2 = new ArrayList<String>();

// list1 = list2;  编译报错
```

输出结果：
- 编译报错，不能赋值。

输出 / 现象解释：
- 这就说明：泛型在 Java 里默认是不协变的。
- 子类元素类型，不会自动推出“子类泛型容器也是父类泛型容器”。

#### 面试易错点 / 补充点
- `String` 是 `Object` 的子类，不等于 `List<String>` 是 `List<Object>` 的子类。
- `List<Object>` 并不是“万能参数”，很多时候真正该写的是 `List<?>`。
- 这类问题本质上考的是“泛型不变性”，也就是 Java 泛型默认不支持这种父子替代关系。
- 数组和泛型这里要分开：数组在 Java 里是协变的，但泛型不是。面试官有时会顺手追问这个区别。

#### 面试回答模板
- 不可以把 `List<String>` 传给一个接受 `List<Object>` 参数的方法，因为 `List<String>` 不是 `List<Object>` 的子类型。
- 虽然 `String` 继承自 `Object`，但 Java 泛型默认是不变的，不会因为元素类型有继承关系，就让泛型类型也自动产生继承关系。
- 这样设计是为了保证类型安全，否则方法内部就可能往原本只允许存 `String` 的列表里加入其他 `Object`，比如 `Integer`。
- 如果只是想兼容不同类型的列表，通常应该使用 `List<?>` 或者带上下界的通配符。

#### 面试简答版
- 不可以。
- 因为 `List<String>` 不是 `List<Object>` 的子类，Java 泛型默认不协变。
- 如果允许这么传，会破坏类型安全。

## 7. Array 中可以用泛型吗？

#### 先给结论
- 面试里这题最稳的回答是：Java 里不能直接创建泛型数组。
- 例如 `new T[10]`、`new List<String>[10]` 都会编译报错。
- 更准确一点说：可以声明“元素类型带泛型信息”的数组引用，但不能直接安全地创建这种泛型数组。

#### 先补一个背景：Array 和 List 是什么
### 1）Array 是什么
- Array 就是数组，是 Java 语言内置的数据结构。
- 特点是：长度固定、按下标访问快、可以存基本类型，也可以存引用类型。

```java
public class Demo {
    public static void main(String[] args) {
        String[] arr = {"A", "B"};
        System.out.println(arr.length);
        System.out.println(arr[0]);
    }
}
```

输出结果：
```java
2
A
```

输出 / 现象解释：
- 数组一旦创建，长度就固定了。
- 通过索引可以直接访问元素，比如 `arr[0]`。

### 2）List 是什么
- `List` 是 Java 集合框架里的一个接口，不是语言内置数组。
- 特点是：长度可以动态变化，常见实现有 `ArrayList`、`LinkedList`。
- `List` 只能存引用类型，基本类型要靠包装类，比如 `Integer`、`Double`。

```java
import java.util.ArrayList;
import java.util.List;

public class Demo {
    public static void main(String[] args) {
        List<String> list = new ArrayList<>();
        list.add("A");
        list.add("B");
        System.out.println(list.size());
        System.out.println(list.get(0));
    }
}
```

输出结果：
```java
2
A
```

输出 / 现象解释：
- `List` 可以动态扩容，不像数组那样长度固定。
- 这也是集合比数组更灵活的地方。

#### 为什么说 Array 中不能直接用泛型
- 因为数组和泛型的底层机制不一样。
- 数组在运行时会记住自己的元素类型，比如 `String[]`、`Number[]` 运行时是可检查的。
- 但泛型大多依赖类型擦除，运行时很多具体类型参数会被擦掉。
- 一个“运行时保留元素类型”的数组，遇上一个“运行时擦掉类型参数”的泛型，放在一起就容易产生类型安全问题。

#### 直接看两个典型错误
### 1）不能直接创建 `T[]`
```java
class Box<T> {
    public void test() {
        // T[] arr = new T[10];  编译报错
    }
}
```

输出结果：
- 编译报错，不能创建类型变量数组。

输出 / 现象解释：
- 因为运行时并不知道 `T` 到底是什么具体类型。
- 既然不知道真实类型，就没法安全地创建对应的数组。

### 2）不能直接创建参数化类型数组
```java
import java.util.List;

public class Demo {
    public static void main(String[] args) {
        // List<String>[] arr = new List<String>[10];  编译报错
    }
}
```

输出结果：
- 编译报错：generic array creation。

输出 / 现象解释：
- `List<String>` 的泛型信息运行时会被擦除。
- 但数组又要求运行时能准确知道元素类型。
- 这两套机制冲突，所以 Java 直接禁止这种创建方式。

#### 如果强行允许，会有什么问题
```java
import java.util.ArrayList;
import java.util.List;

public class Demo {
    public static void main(String[] args) {
        List[] arr = new List[1];
        Object[] objs = arr;
        objs[0] = new ArrayList<Integer>();

        List<String> list = arr[0];
        // String s = list.get(0);  这里后续就可能埋下类型问题
        System.out.println(arr.length);
    }
}
```

输出结果：
```java
1
```

输出 / 现象解释：
- 这个例子是在模拟“数组运行时类型检查”和“泛型擦除”混在一起时的风险。
- 如果把泛型数组完全开放，编译器就更难保证类型安全。
- 所以 Java 选择在语言层面直接限制“泛型数组创建”。

#### 那是不是完全不能和数组一起用
- 也不是。
- 常见做法有两个：
  1. 用 `Object[]` 做底层存储，再在受控位置强转
  2. 用反射创建数组，比如 `Array.newInstance()`
- 但这些都属于“绕开限制的工程写法”，不是说 Java 原生支持直接 `new T[10]`。

#### 一个常见替代方案
```java
import java.lang.reflect.Array;

public class Demo<T> {
    private final T[] arr;

    @SuppressWarnings("unchecked")
    public Demo(Class<T> clazz, int size) {
        arr = (T[]) Array.newInstance(clazz, size);
    }

    public static void main(String[] args) {
        Demo<String> demo = new Demo<>(String.class, 2);
        System.out.println("ok");
    }
}
```

输出结果：
```java
ok
```

输出 / 现象解释：
- 这里不是直接 `new T[2]`。
- 而是借助 `Class<T>` 和反射在运行时拿到真实类型，再创建数组。
- 这也是泛型数组场景下最常见的替代方案。

#### 这题面试到底怎么答
- Java 里不能直接创建泛型数组，比如 `new T[10]` 或 `new List<String>[10]` 都不行。
- 根本原因是数组和泛型的类型机制不同：数组是运行时保留元素类型的，泛型大多依赖类型擦除。
- 如果直接允许泛型数组创建，类型安全会更难保证，所以 Java 在语法层面做了限制。
- 真要实现类似能力，通常会用 `Object[]`、反射 `Array.newInstance()`，或者干脆优先用 `List<T>` 来替代数组。

#### 面试易错点 / 补充点
- “不能用泛型数组”更准确地说，是“不能直接创建泛型数组”。
- `List<String>[] arr;` 这种声明讨论起来比较细，面试里不建议一开始就把自己带进实现细节，先抓住 `new T[]` 和 `new List<String>[]` 不行这个主线。
- 数组是协变的，泛型默认是不变的，这也是两者容易出问题的根源之一。
- 真正做业务开发时，如果你开始纠结泛型数组，很多时候说明更适合用 `List<T>`。

#### 面试回答模板
- Java 中不能直接创建泛型数组，比如 `new T[10]`、`new List<String>[10]` 都会编译报错。
- 原因是数组在运行时会保留元素类型，而泛型大多采用类型擦除，运行时很多具体泛型参数已经不存在了，这两套机制放在一起会带来类型安全问题。
- 所以 Java 选择在语法层面禁止直接创建泛型数组。如果确实需要，一般会通过 `Array.newInstance()` 或 `Object[]` 间接实现，或者直接用 `List<T>` 替代。

#### 面试简答版
- Java 里不能直接创建泛型数组。
- 因为数组是运行时保留类型的，泛型大多是类型擦除，两者机制冲突。
- 常见替代方案是 `Array.newInstance()` 或直接用 `List<T>`。

## 8. 判断 `ArrayList<String>` 与 `ArrayList<Integer>` 是否相等？

#### 先给结论
- 如果你问的是它们的 `Class` 是否相同，答案是相同。
- 如果你问的是两个对象本身 `equals()` 是否相等，那要看里面存的元素是否相同，和泛型参数本身没有直接关系。
- 这题面试通常真正想考的是：`ArrayList<String>` 和 `ArrayList<Integer>` 在运行时拿到的 `Class` 是一样的，这正好说明了类型擦除。

#### 核心解释
- `ArrayList<String>` 和 `ArrayList<Integer>` 在源码层面看起来是两种不同的泛型写法。
- 但编译完成后，具体的泛型类型参数大多会被擦除。
- 所以运行时它们对应的类信息其实都是同一个：`ArrayList.class`。
- 也就是说，泛型参数 `String`、`Integer` 主要作用在编译期，不是运行期去区分两个不同类。

#### 简单代码示例
### 1）判断 `Class` 是否相同
```java
import java.util.ArrayList;

public class Demo {
    public static void main(String[] args) {
        ArrayList<String> a = new ArrayList<>();
        ArrayList<Integer> b = new ArrayList<>();

        Class<?> c1 = a.getClass();
        Class<?> c2 = b.getClass();

        System.out.println(c1 == c2);
    }
}
```

输出结果：
```java
true
```

输出 / 现象解释：
- `a` 看起来是 `ArrayList<String>`。
- `b` 看起来是 `ArrayList<Integer>`。
- 但运行时拿到的 `Class` 都是同一个 `ArrayList.class`。
- 这就是类型擦除最经典的证明之一。

### 2）判断对象 `equals()` 是否相等
```java
import java.util.ArrayList;

public class Demo {
    public static void main(String[] args) {
        ArrayList<String> a = new ArrayList<>();
        ArrayList<Integer> b = new ArrayList<>();

        System.out.println(a.equals(b));
    }
}
```

输出结果：
```java
true
```

输出 / 现象解释：
- 这里两个列表都是空列表。
- `ArrayList.equals()` 比较的是元素序列是否相同。
- 因为两个列表都为空，所以这里结果是 `true`。
- 这和一个是 `String` 泛型、一个是 `Integer` 泛型没有直接关系。

### 3）元素不同则 `equals()` 不相等
```java
import java.util.ArrayList;

public class Demo {
    public static void main(String[] args) {
        ArrayList<String> a = new ArrayList<>();
        ArrayList<Integer> b = new ArrayList<>();

        a.add("1");
        b.add(1);

        System.out.println(a.equals(b));
    }
}
```

输出结果：
```java
false
```

输出 / 现象解释：
- `equals()` 比较的是元素内容。
- `"1"` 是字符串，`1` 是整数，它们不相等。
- 所以列表也不相等。

#### 这题真正想考什么
- 大多数情况下，题目并不是在考你 `ArrayList.equals()` 的源码细节。
- 真正想考的是：为什么 `ArrayList<String>` 和 `ArrayList<Integer>` 在运行时得到的是同一个 `Class`。
- 这个现象背后的原因就是泛型类型擦除。
- 也就是说：泛型参数主要用于编译期检查，运行时很多具体类型参数已经被擦掉了。

#### String 和 Integer 到底“体现”在哪里
- 它们主要体现在编译阶段。
- 比如：
  - `ArrayList<String>` 编译器会限制你只能放 `String`
  - `ArrayList<Integer>` 编译器会限制你只能放 `Integer`
- 取元素时，编译器也会按对应泛型类型帮你做类型检查和必要的转换处理。
- 但到了运行时，JVM 不会因为一个是 `String`，一个是 `Integer`，就创建两种不同的 `ArrayList` 类。

#### 面试易错点 / 补充点
- 这题里的“是否相等”要先分清到底问的是 `Class` 相等，还是对象 `equals()` 相等。
- 如果是 `getClass()` 比较，答案通常是 `true`，因为类型擦除后运行时类相同。
- 如果是 `equals()` 比较，要看元素内容，不是单纯看泛型参数。
- 这题最常见的答法是直接把“运行时类相同”归因到类型擦除，这是对的，但最好补一句“泛型参数仍然在编译期参与类型检查”。

#### 面试回答模板
- `ArrayList<String>` 和 `ArrayList<Integer>` 在运行时通过 `getClass()` 获取到的 `Class` 是相同的，比较结果通常是 `true`。
- 原因是 Java 泛型底层采用类型擦除，具体泛型参数主要在编译期做类型检查，运行时很多类型参数已经被擦除了，所以它们运行时本质上都是同一个 `ArrayList.class`。
- 但如果题目问的是两个列表对象通过 `equals()` 是否相等，那还要看它们内部元素是否相同，因为 `equals()` 比较的是内容，不是泛型声明。

#### 面试简答版
- 比较 `getClass()`，结果是 `true`。
- 因为 Java 泛型有类型擦除，运行时它们都是同一个 `ArrayList.class`。
- 如果比较 `equals()`，那要看列表里的元素内容。
