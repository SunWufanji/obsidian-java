# 对象相等判断

## 1. `==` 和 `equals()` 的区别是什么？

#### 核心区别
- `==` 是运算符，`equals()` 是 `Object` 类中的方法。
- `==` 对于基本类型比较的是值是否相等。
- `==` 对于引用类型比较的是两个引用是否指向同一个对象。
- `equals()` 默认也是比较对象地址，但很多类会重写它，用来比较“内容是否相等”。

#### 基本类型中的 `==`
```java
int a = 10;
int b = 10;
System.out.println(a == b); // true
```

- 这里 `a` 和 `b` 都是基本类型。
- `==` 比较的是数值本身，所以结果为 `true`。

#### 引用类型中的 `==`
```java
String s1 = new String("abc");
String s2 = new String("abc");

System.out.println(s1 == s2); // false
```

- `s1` 和 `s2` 内容一样，但它们是两个不同对象。
- 所以 `==` 比较地址时结果为 `false`。

#### `equals()` 的默认行为
```java
class Person {
    private String name;
}
```

- 如果一个类没有重写 `equals()`，那么它继承的是 `Object` 的实现。
- `Object` 中的 `equals()` 默认比较的也是地址，本质上和 `==` 类似。
- 所以“`equals()` 一定比较内容”这句话是不准确的，要看类有没有重写。

#### `String` 中的 `equals()`
```java
String s1 = new String("abc");
String s2 = new String("abc");

System.out.println(s1 == s2);      // false
System.out.println(s1.equals(s2)); // true
```

- `String` 重写了 `equals()` 方法。
- 它比较的是字符串内容，而不是对象地址。
- 所以虽然 `s1` 和 `s2` 不是同一个对象，但内容相同，`equals()` 返回 `true`。

#### 运行时到底怎么判断
- `==` 由 JVM 直接按运算符规则处理，不走“方法重写”。
- `equals()` 是方法调用，运行时会根据对象的实际类型执行对应类里重写后的实现。
- 所以 `equals()` 能否比较内容，取决于这个类自己怎么定义“相等”。

#### 面试常见误区
- `==` 不一定都比较地址，基本类型时比较的是值。
- `equals()` 不一定都比较内容，如果没重写，默认比较的还是地址。
- 比较字符串内容时，应该优先用 `equals()`，不要用 `==`。
- 如果可能出现 `null`，建议写成 `"abc".equals(str)`，避免空指针异常。

#### 延伸补充：重写 `equals()` 时为什么通常要同时重写 `hashCode()`
- 因为 Java 规定：如果两个对象通过 `equals()` 判断相等，那么它们的 `hashCode()` 也必须相等。
- 否则在 `HashMap`、`HashSet` 这类基于哈希的集合中会出现逻辑错误。
- 所以面试里如果问到 `equals()`，顺手提一句 `hashCode()`，通常会加分。

#### 面试回答模板
- `==` 和 `equals()` 的区别主要在于比较规则不同。
- `==` 是运算符，基本类型比较的是值，引用类型比较的是是否指向同一个对象。
- `equals()` 是方法，`Object` 默认实现比较的也是地址，但很多类比如 `String` 重写了它，用来比较内容是否相等。
- 所以判断对象内容是否相等，通常看这个类有没有重写 `equals()`。

#### 面试简答版
- `==`：基本类型比值，引用类型比地址。
- `equals()`：默认比地址，重写后通常比内容。
- 例如 `String` 比较内容要用 `equals()`，不能只看 `==`。

## 2. 介绍下 `hashCode()`？

#### `hashCode()` 是什么
- `hashCode()` 是 `Object` 类中的一个方法，用来返回对象的哈希值，返回类型是 `int`。
- 它主要服务于哈希类集合，比如 `HashMap`、`HashSet`、`Hashtable`。
- 你可以把它理解成：对象进入哈希容器时用于快速定位位置的一个“索引线索”。

#### 它的核心作用
- 如果集合里每个对象都只靠 `equals()` 挨个比较，效率会很低。
- 所以 Java 会先用 `hashCode()` 计算一个哈希值，快速决定对象大概落在哪个桶（bucket）里。
- 只有哈希值相同，或者落入同一桶时，才会进一步调用 `equals()` 做精确比较。
- 所以 `hashCode()` 负责“快速筛选”，`equals()` 负责“最终确认”。

#### 在集合里是怎么运行的
- 以 `HashSet` 为例，插入对象时先调用 `hashCode()`。
- 根据哈希值计算出对象应该存放的位置。
- 如果这个位置没有元素，就直接存进去。
- 如果这个位置已经有元素，再调用 `equals()` 比较是否逻辑相等。
- 如果 `equals()` 返回 `true`，说明是重复元素，不会重复存入；如果返回 `false`，说明只是哈希冲突，还会继续共存。

#### `hashCode()` 和 `equals()` 的关系
- 如果两个对象通过 `equals()` 判断为 `true`，那么它们的 `hashCode()` 必须相同。
- 如果两个对象的 `hashCode()` 相同，它们的 `equals()` 不一定为 `true`，这叫哈希冲突。
- 所以这两句一定要记住：
  - `equals()` 相等 -> `hashCode()` 必须相等
  - `hashCode()` 相等 -> `equals()` 不一定相等

#### 为什么重写 `equals()` 时通常也要重写 `hashCode()`
- 因为如果你只重写了 `equals()`，但没重写 `hashCode()`，逻辑上相等的两个对象可能会得到不同的哈希值。
- 这样它们放进 `HashSet` 时可能被分到不同位置，导致去重失败。
- 或者作为 `HashMap` 的 key 时，存得进去却取不出来。
- 所以在实际开发里，`equals()` 和 `hashCode()` 通常要成对重写。

#### 一个典型例子
```java
class Person {
    private String name;

    public Person(String name) {
        this.name = name;
    }

    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;
        if (obj == null || getClass() != obj.getClass()) return false;
        Person person = (Person) obj;
        return java.util.Objects.equals(name, person.name);
    }

    @Override
    public int hashCode() {
        return java.util.Objects.hash(name);
    }
}
```

#### 这个例子想说明什么
- 这里认为 `name` 相同的两个 `Person` 是“逻辑上相等”的。
- 那么 `equals()` 比较 `name`，`hashCode()` 也必须基于 `name` 计算。
- 只有这样，两者规则才一致。
- 这样把 `Person` 放进 `HashSet`、作为 `HashMap` 的 key 时，结果才正确。

#### 面试常见误区
- `hashCode()` 不是对象内存地址。
- `hashCode()` 不是唯一值，不同对象可能产生相同的哈希值。
- `hashCode()` 相同，不代表两个对象一定相等。
- 判断对象逻辑是否相等，最终还得看 `equals()`。

#### 一句话理解
- `hashCode()` 用来提高哈希容器查找和去重的效率，`equals()` 用来判断对象逻辑上是否真正相等。

#### 面试回答模板
- `hashCode()` 是 `Object` 类的方法，用来返回对象的哈希值，主要用于 `HashMap`、`HashSet` 等哈希容器中。
- 哈希容器通常先根据 `hashCode()` 定位对象的大概存储位置，再通过 `equals()` 判断对象是否真正相等。
- 如果两个对象 `equals()` 相等，那么它们的 `hashCode()` 必须相等。
- 所以在重写 `equals()` 时，通常也必须重写 `hashCode()`，否则可能导致集合中的查找、去重出现问题。

#### 面试简答版
- `hashCode()` 返回对象的哈希值，主要用于哈希集合提高效率。
- `equals()` 相等的两个对象，`hashCode()` 必须相等。
- `hashCode()` 相等的两个对象，`equals()` 不一定相等。
- 所以重写 `equals()` 时一般也要一起重写 `hashCode()`。

## 3. 为什么要有 `hashCode()`？

#### 先说结论
- 因为只靠 `equals()` 判断对象是否相等，效率太低。
- `hashCode()` 的存在，是为了让哈希类集合先快速缩小比较范围，再决定是否需要调用 `equals()`。
- 所以它本质上是一个“提高查找、去重、定位效率”的机制。

#### 如果没有 `hashCode()` 会怎样
- 假设 `HashSet` 里已经放了几万条数据。
- 现在新来一个对象，如果没有 `hashCode()`，那就只能拿这个对象和集合里已有对象一个个做 `equals()` 比较。
- 这样查找和去重的成本就会很高，效率很差。
- 也就是说，如果没有 `hashCode()`，哈希集合就很难体现“快”的优势。

#### 有了 `hashCode()` 之后怎么运行
- 先调用对象的 `hashCode()`，得到一个整数。
- 再根据这个整数计算对象应该进入哪个桶（bucket）。
- 如果这个桶里没有元素，直接放入或直接判断不存在。
- 如果这个桶里已经有元素，再对桶里的少量对象调用 `equals()` 精确比较。
- 所以 `hashCode()` 的作用不是替代 `equals()`，而是帮 `equals()` 减少工作量。

#### 你可以把它想象成什么
- `hashCode()` 像“先分组”。
- `equals()` 像“组内再逐个确认”。
- 没有 `hashCode()`，就相当于所有对象都挤在一个大组里，只能全表扫描。
- 有了 `hashCode()`，对象先被分散到不同组里，比较范围就小很多。

#### 为什么说它是为哈希容器服务的
- 像 `HashMap`、`HashSet` 这类集合的核心优势就是查找快。
- 它们之所以快，前提就是先靠 `hashCode()` 快速定位，再靠 `equals()` 最终确认。
- 所以 `hashCode()` 是哈希结构成立的重要基础之一。

#### 为什么不能只用 `hashCode()`
- 因为不同对象可能算出同一个哈希值，这叫哈希冲突。
- 所以 `hashCode()` 只能粗略判断“可能在这儿”，不能直接说明“这两个对象一定相等”。
- 因此最终还需要 `equals()` 来做准确判断。

#### 最本质的一句话
- `hashCode()` 是用空间分桶和整数映射的方式，换取更高的查找和去重效率。

#### 面试回答模板
- `hashCode()` 的存在主要是为了提高哈希集合的操作效率。
- 如果没有 `hashCode()`，集合在查找、去重时就只能依赖 `equals()` 逐个比较，效率会很低。
- 有了 `hashCode()` 后，集合可以先根据哈希值定位对象所在桶，再通过 `equals()` 做精确判断。
- 所以 `hashCode()` 负责快速定位，`equals()` 负责最终确认，两者配合使用。

#### 面试简答版
- 因为只靠 `equals()` 逐个比较太慢。
- `hashCode()` 可以让哈希集合先快速定位，再小范围调用 `equals()`。
- 所以它存在的核心目的就是提高查找和去重效率。

## 4. `hashCode()`、`equals()` 两种方法是什么关系？

#### 核心关系
- `equals()` 用来判断两个对象在逻辑上是否相等。
- `hashCode()` 用来为对象生成哈希值，主要服务于哈希类集合。
- 两者的关系是：哈希容器通常先看 `hashCode()`，再看 `equals()`。
- 所以它们不是互相替代关系，而是协作关系。

#### Java 规范中最重要的约束
- 如果两个对象通过 `equals()` 判断为 `true`，那么它们的 `hashCode()` 必须相同。
- 如果两个对象的 `hashCode()` 相同，它们的 `equals()` 不一定为 `true`。

#### 为什么会这样
- 因为哈希值只是为了快速定位，不是为了精确判断对象是否相等。
- 不同对象完全可能被分配到同一个桶里，这就是哈希冲突。
- 所以 `hashCode()` 相同只能说明“可能相等”或“可能落在同一位置”，不能直接说明“逻辑上一定相等”。
- 真正判断逻辑相等，还得靠 `equals()`。

#### 在 `HashMap` / `HashSet` 里是怎么配合工作的
- 先调用 `hashCode()`，确定对象大概存放在哪个桶里。
- 如果桶为空，直接存入。
- 如果桶中已有元素，再调用 `equals()` 判断是否是同一个逻辑对象。
- 所以：
  - `hashCode()` 决定查找范围
  - `equals()` 决定最终是否相等

#### 为什么重写一个通常就要重写另一个
- 如果你重写了 `equals()`，把“相等”的规则改了，但 `hashCode()` 还是老规则，就会出问题。
- 比如两个对象 `equals()` 明明相等，但 `hashCode()` 不同，它们在哈希集合里可能进不同桶。
- 这样 `HashSet` 去重会失败，`HashMap` 查找也可能异常。
- 所以在实际开发里，只要重写 `equals()`，一般就必须一起重写 `hashCode()`。

#### 一个非常常见的结论
- `equals()` 是“逻辑相等”的标准。
- `hashCode()` 是“哈希定位”的依据。
- 两者规则必须保持一致，但职责并不相同。

#### 面试常见误区
- 误区一：`hashCode()` 相同就说明对象相等。错，可能只是哈希冲突。
- 误区二：只重写 `equals()` 就够了。错，在哈希集合场景下一般不够。
- 误区三：`hashCode()` 决定对象内容是否相等。错，它只参与定位和分桶。

#### 面试回答模板
- `equals()` 和 `hashCode()` 是两个相互配合的方法。
- `equals()` 用来判断两个对象逻辑上是否相等，`hashCode()` 用来生成哈希值，方便哈希集合快速定位对象。
- Java 规定：如果两个对象 `equals()` 相等，那么它们的 `hashCode()` 必须相等；反过来 `hashCode()` 相等，`equals()` 不一定相等。
- 因此在重写 `equals()` 时，通常也必须重写 `hashCode()`，否则会导致 `HashMap`、`HashSet` 等集合出现去重和查找问题。

#### 面试简答版
- `equals()` 判断逻辑是否相等，`hashCode()` 用于哈希定位。
- `equals()` 相等的对象，`hashCode()` 必须相等。
- `hashCode()` 相等的对象，`equals()` 不一定相等。
- 所以重写 `equals()` 时一般也要一起重写 `hashCode()`。

## 5. 为什么重写 `equals()` 方法必须重写 `hashCode()` 方法？

#### 先给结论
- 因为 `equals()` 和 `hashCode()` 在哈希类集合中是配套工作的。
- 如果只重写 `equals()`，不重写 `hashCode()`，逻辑上相等的对象可能会得到不同的哈希值。
- 这样就会破坏 `HashMap`、`HashSet` 等集合的查找和去重规则。

#### 根本原因
- Java 规定：如果两个对象通过 `equals()` 判断相等，那么它们的 `hashCode()` 也必须相等。
- 这是一个必须遵守的约定，不然哈希容器就无法正常工作。
- 因为哈希容器不是先用 `equals()`，而是先用 `hashCode()` 定位桶，再用 `equals()` 最终比较。

#### 如果不重写会发生什么
- 假设你重写了 `equals()`，规定两个 `Person` 只要 `name` 相同就算相等。
- 但如果没有重写 `hashCode()`，那它们仍然可能继承 `Object` 默认实现，得到不同的哈希值。
- 这样这两个逻辑上相等的对象就可能进入不同桶。
- 结果就是：哈希集合根本不会把它们放到一起比较，逻辑就错了。

#### 一个典型错误场景
```java
class Person {
    private String name;

    public Person(String name) {
        this.name = name;
    }

    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;
        if (obj == null || getClass() != obj.getClass()) return false;
        Person person = (Person) obj;
        return java.util.Objects.equals(name, person.name);
    }
}
```

```java
Set<Person> set = new HashSet<>();
set.add(new Person("Tom"));
set.add(new Person("Tom"));

System.out.println(set.size());
```

- 从业务逻辑看，这两个 `Person("Tom")` 应该算重复元素。
- 但如果没有重写 `hashCode()`，它们可能拿到不同哈希值，进不同桶。
- 结果 `HashSet` 可能把它们都存进去，去重失败。

#### 正确做法
- 只要 `equals()` 的判断依据变了，`hashCode()` 的计算依据也要跟着变。
- 也就是说，`equals()` 比较哪些字段，`hashCode()` 一般也要基于这些字段计算。

```java
class Person {
    private String name;

    public Person(String name) {
        this.name = name;
    }

    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;
        if (obj == null || getClass() != obj.getClass()) return false;
        Person person = (Person) obj;
        return java.util.Objects.equals(name, person.name);
    }

    @Override
    public int hashCode() {
        return java.util.Objects.hash(name);
    }
}
```

#### 为什么这样就对了
- 因为两个 `name` 相同的对象，`equals()` 返回 `true`。
- 同时它们的 `hashCode()` 也会相同。
- 这样它们先进入同一个桶，再通过 `equals()` 判断重复，哈希集合逻辑才一致。

#### 从职责上怎么理解
- `hashCode()` 决定对象先去哪一桶。
- `equals()` 决定桶里的两个对象是不是同一个逻辑对象。
- 如果 `equals()` 说“它俩一样”，而 `hashCode()` 却把它们分到不同桶，就前后矛盾了。

#### 面试常见误区
- 不是说语法层面“必须”一起重写，不重写代码也能编译。
- 这里的“必须”是指：为了保证逻辑正确，在涉及哈希容器时必须这样做。
- 如果你的类永远不会作为 `HashMap` key 或放进 `HashSet`，短期看问题不明显，但这依然违背约定，设计上不规范。

#### 面试回答模板
- 因为 `equals()` 和 `hashCode()` 在哈希集合中是配合使用的。
- 哈希集合先根据 `hashCode()` 定位桶，再通过 `equals()` 判断对象是否真正相等。
- Java 规定：如果两个对象 `equals()` 相等，那么它们的 `hashCode()` 也必须相等。
- 所以如果只重写 `equals()` 不重写 `hashCode()`，就可能导致 `HashSet` 去重失败、`HashMap` 查找异常，因此通常要一起重写。

#### 面试简答版
- 因为 `equals()` 相等的对象，`hashCode()` 必须相等。
- 哈希集合先看 `hashCode()`，再看 `equals()`。
- 如果只重写 `equals()` 不重写 `hashCode()`，会导致哈希集合的去重和查找出现问题。
