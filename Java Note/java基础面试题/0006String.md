# 字符串

## 1. `String`、`StringBuffer`、`StringBuilder` 的区别是什么？

#### 核心区别
- `String` 是不可变字符串，内容一旦创建就不能修改。
- `StringBuffer` 是可变字符串，方法大多带同步控制，线程安全。`synchronized` `/ˈsɪŋkrənaɪzd/`
- `StringBuilder` 也是可变字符串，但不保证同步，不是线程安全的，单线程下通常更快。

#### 1）可变性不同
- `String` 不可变，任何“修改”操作本质上都会生成新字符串对象。
- `StringBuffer` 和 `StringBuilder` 可变，`append()`、`insert()`、`delete()` 等操作通常直接在原对象上改。

```java
String s = "a";
s += "b"; // 实际上生成了新对象

StringBuilder sb = new StringBuilder("a");
sb.append("b"); // 直接在原对象上追加
```

#### 2）线程安全不同
- `String` 因为不可变，所以天然适合共享，线程安全。
- `StringBuffer` 是线程安全的，因为很多方法做了同步控制。
- `StringBuilder` 不保证同步，多个线程同时修改同一个实例时不安全。

#### 3）性能不同
- 频繁拼接字符串时，`String` 性能通常最差，因为会创建很多中间对象。
- `StringBuffer` 因为有同步开销，通常比 `StringBuilder` 慢。
- `StringBuilder` 在单线程字符串拼接场景下通常性能最好。

#### 4）使用场景不同
- 少量字符串操作、常量字符串、配置项、返回值等：优先用 `String`。
- 单线程下频繁拼接字符串：优先用 `StringBuilder`。
- 多线程下多个线程共享并修改同一个字符串缓冲区：才考虑 `StringBuffer`。

#### 运行方式怎么理解
- `String`：每次修改本质上都是创建新对象，所以旧对象不会变。
- `StringBuilder`：内部维护可扩容的字符序列，追加时通常直接改当前对象内容。
- `StringBuffer`：和 `StringBuilder` 类似，但关键方法多了一层同步，保证并发安全。

#### 为什么说 `String` 拼接慢
- 因为 `String` 不可变，像 `s = s + "x"` 这种循环拼接时，每次都可能生成新对象。
- 对象多了，就会增加内存分配和垃圾回收压力。
- 所以在循环或高频拼接中，一般不建议直接用 `String` 累加。

#### `+` 拼接是不是一定慢
- 不一定。
- 如果是编译期常量拼接，比如 `"a" + "b"`，编译器通常会直接优化成 `"ab"`。
- 如果是运行期频繁拼接，底层通常会借助 `StringBuilder` 或 JDK 的字符串拼接机制来完成。
- 但从编码习惯上说，循环里高频拼接仍然推荐显式使用 `StringBuilder`。

#### 面试常见误区
- `String` 线程安全，不是因为它“加了锁”，而是因为它不可变。
- `StringBuffer` 线程安全，不代表它适合所有场景；单线程拼接通常优先用 `StringBuilder`。
- `StringBuilder` 不是“绝对更好”，它只是更适合单线程高频拼接。

#### 一句话总结
- `String`：不可变，安全，适合普通场景。
- `StringBuffer`：可变，线程安全，性能相对较低。
- `StringBuilder`：可变，非线程安全，单线程拼接性能最好。

#### 面试回答模板
- `String`、`StringBuffer`、`StringBuilder` 的主要区别在于可变性、线程安全和性能。
- `String` 是不可变对象，修改时会生成新对象；`StringBuffer` 和 `StringBuilder` 都是可变字符序列。
- `StringBuffer` 的方法大多是同步的，线程安全；`StringBuilder` 不保证同步，单线程下通常性能更好。
- 所以一般普通字符串使用 `String`，单线程频繁拼接用 `StringBuilder`，多线程共享修改时才考虑 `StringBuffer`。

#### 面试简答版
- `String` 不可变，`StringBuffer` 和 `StringBuilder` 可变。
- `StringBuffer` 线程安全，`StringBuilder` 非线程安全。
- 单线程频繁拼接优先 `StringBuilder`，多线程共享修改才考虑 `StringBuffer`。

## 2. `String` 为什么要设计成不可变的？

#### 先给结论
- `String` 设计成不可变，主要是为了安全性、性能和可共享性。
- 这不是单纯的语法设计，而是和字符串常量池、哈希缓存、类加载、安全校验、多线程共享等场景紧密相关。

#### 原因一：便于实现字符串常量池
- Java 中字符串字面量使用非常频繁，比如 `"abc"`、`"hello"`。
- 为了节省内存，JVM 会把相同内容的字符串放到字符串常量池中复用。
- 如果 `String` 是可变的，那么多个引用共同指向池中的同一个字符串时，只要有一个地方改了内容，其他地方也会跟着变。
- 这样常量池就失去了“安全复用”的意义。

```java
String a = "hello";
String b = "hello";
```

- 这里 `a` 和 `b` 可能直接指向常量池中的同一个字符串对象。
- 正因为字符串不可变，JVM 才敢放心复用这块内存。

#### 原因二：线程安全，适合共享
- `String` 不可变，所以多个线程可以同时读取同一个字符串对象，而不用担心某个线程把内容改掉。
- 也就是说，`String` 的线程安全不是靠加锁实现的，而是靠“状态不能变”实现的。
- 所以它非常适合作为共享对象、配置对象、常量对象使用。

#### 原因三：安全性更高
- 很多敏感场景都会用到字符串，比如：
  - 数据库连接地址
  - 文件路径
  - URL
  - 类名
  - 网络请求参数
  - 反射参数
- 如果 `String` 可变，那么这些内容在使用过程中就可能被偷偷篡改，带来严重安全问题。
- 不可变设计可以保证：字符串对象一旦创建，它表示的含义就不会变。

#### 原因四：适合作为哈希表的 key
- `String` 经常被拿来做 `HashMap` 的 key。
- 如果字符串内容可以变，那么它的 `hashCode()` 也可能变化。
- 这样对象放进哈希表后，桶位置可能就不对了，后续查找会出问题。
- 正因为 `String` 不可变，所以它的哈希值稳定，适合作为 key。

#### 原因五：可以缓存 `hashCode()`，提升性能
- `String` 的哈希值常常会被频繁使用。
- 因为 `String` 内容不可变，所以它的 `hashCode()` 一旦算出来，就可以缓存起来重复使用。
- 以后再次调用 `hashCode()` 时，很多时候不需要重新计算。
- 这也是 `String` 在哈希集合中表现较好的原因之一。

#### 从实现上怎么保证不可变
- `String` 类本身是 `final` 的，不能被继承后破坏不可变性。
- 内部用于存储内容的字段不会对外暴露。
- 没有提供修改字符内容的公开方法。
- 所有看起来像“修改”的操作，比如 `replace()`、`substring()`、`concat()`，本质上都是返回新字符串。

#### 一个最容易理解的例子
```java
String s1 = "abc";
String s2 = s1.replace("a", "x");

System.out.println(s1); // abc
System.out.println(s2); // xbc
```

- `s1` 并没有被改掉。
- `replace()` 返回的是一个新字符串 `s2`。
- 这就是不可变对象最典型的表现。

#### 面试常见误区
- `String` 不可变，不代表引用不能变。
- 比如 `String s = "a"; s = "b";`，变的是 `s` 这个引用的指向，不是原来那个 `"a"` 对象内容被改了。
- 另外，`final String` 只是让引用不能再指向别的对象，不是 `String` 不可变的根本原因；`String` 自身的类设计才是关键。

#### 一句话总结
- `String` 不可变，是为了让它能够安全复用、线程共享、稳定作为哈希 key，并提升整体性能与安全性。

#### 面试回答模板
- `String` 之所以设计成不可变，主要是出于安全性、性能和共享性的考虑。
- 不可变使得字符串常量池可以安全复用同一对象，也让 `String` 天然线程安全。
- 同时因为内容稳定，`String` 可以安全地作为 `HashMap` 的 key，并缓存 `hashCode()` 提高性能。
- 此外，很多敏感参数都使用字符串表示，不可变也能避免内容被篡改带来的安全问题。

#### 面试简答版
- `String` 不可变主要有四个原因：便于常量池复用、天然线程安全、更安全、适合做哈希 key。
- 因为内容不会变，所以可以共享、可以缓存哈希值，也不容易在使用过程中被篡改。

## 3. 字符型常量和字符串常量的区别？

#### 核心区别
- 字符型常量是 `char` 类型，用单引号表示，例如 `'a'`。
- 字符串常量是 `String` 类型，用双引号表示，例如 `"a"`。
- `char` 属于基本数据类型，表示单个字符；`String` 属于引用类型，表示一串字符。

#### 1）表示形式不同
- 字符型常量使用单引号：`'A'`
- 字符串常量使用双引号：`"A"`

```java
char c = 'A';
String s = "A";

System.out.println(c);
System.out.println(s);
```

输出结果：
```java
A
A
```

#### 2）含义不同
- `char` 表示一个单独字符，本质上存的是一个 Unicode 字符。
- `String` 表示由多个字符组成的字符串对象。

#### 3）长度不同
- `char` 只能表示一个字符，不能表示多个字符。
- `String` 可以表示 0 个、1 个或多个字符。
- 例如：
  - `'a'` 合法
  - `"a"` 合法
  - `"abc"` 合法
  - `'ab'` 非法

#### 4）数据类型不同
- `char` 是基本类型。
- `String` 是引用类型，本质是一个对象。

#### 5）运算方式不同
- `char` 可以参与算术运算，运算时通常会提升为 `int`。
- `String` 不能做普通算术运算，但可以参与字符串拼接。

```java
System.out.println('A' + 1);
System.out.println("A" + 1);
```

输出结果：
```java
66
A1
```

解释：
- `'A' + 1` 中，`'A'` 会按字符编码参与计算，所以结果是 `66`。
- `"A" + 1` 是字符串拼接，所以结果是 `"A1"`。

#### 6）默认值不同
- 成员变量中的 `char` 默认值是 `'\u0000'`。
- 成员变量中的 `String` 默认值是 `null`。

```java
class Demo {
    char c;
    String s;

    public static void main(String[] args) {
        Demo d = new Demo();
        System.out.println((int) d.c);
        System.out.println(d.s);
    }
}
```

输出结果：
```java
0
null
```

解释：
- `char` 的默认值是空字符，打印成整数时是 `0`。
- `String` 的默认值是 `null`。

#### 7）内存和语义不同
- `char` 本身就是一个值。
- `String` 变量里保存的是字符串对象的引用。
- 所以从语义上说，`char` 更像“一个字符值”，`String` 更像“一个字符串对象”。

#### 面试易错点
- `'a'` 和 `"a"` 不是同一种类型。
- `char` 不是字符串，即使它看起来只有一个字符。
- `String` 可以是空串 `""`，但这不等于 `null`。
- `'A' + 1` 是数值运算，`"A" + 1` 是字符串拼接。

#### 面试回答模板
- 字符型常量和字符串常量主要区别在于类型、表示形式和用途不同。
- 字符型常量是 `char` 类型，使用单引号，表示单个字符，属于基本数据类型。
- 字符串常量是 `String` 类型，使用双引号，表示一串字符，属于引用类型。
- 另外，`char` 可以参与算术运算，而 `String` 主要参与字符串拼接。

#### 面试简答版
- `'a'` 是字符常量，类型是 `char`；`"a"` 是字符串常量，类型是 `String`。
- `char` 表示单个字符，`String` 表示一串字符。
- `char` 是基本类型，`String` 是引用类型。

## 4. 什么是字符串常量池？

#### 先给结论
- 面试里说的“字符串常量池”，通常指的是字符串驻留池，也就是 HotSpot 里的 `StringTable`。
- 它的作用是复用内容相同的字符串，避免重复创建对象，节省内存并提升比较效率。
- 简单理解：相同内容的字符串字面量，JVM 会尽量只保留一份并共享使用。

#### 为什么需要字符串常量池
- Java 程序里字符串字面量非常多，比如 `"java"`、`"hello"`、`"abc"`。
- 如果每次出现一个相同字面量都重新建对象，会造成大量重复对象，浪费内存。
- 所以 JVM 维护了一个共享区域，用来保存已经“驻留（intern）”的字符串。
- 后面再遇到相同内容时，就优先复用已有对象。

#### 最经典的例子
```java
String a = "hello";
String b = "hello";

System.out.println(a == b);
```

输出结果：
```java
true
```

解释：
- `"hello"` 是字符串字面量。
- 它会进入字符串常量池。
- `a` 和 `b` 都直接指向池中的同一个字符串对象，所以 `==` 为 `true`。

#### 和 `new String()` 有什么区别
```java
String a = "hello";
String b = new String("hello");

System.out.println(a == b);
System.out.println(a.equals(b));
```

输出结果：
```java
false
true
```

解释：
- `a` 指向常量池中的 `"hello"`。
- `new String("hello")` 会额外在堆上再创建一个新对象，所以 `b` 指向的是堆中对象。
- 两者内容相同，所以 `equals()` 为 `true`；但不是同一个对象，所以 `==` 为 `false`。

#### `intern()` 是做什么的
- `intern()` 的作用是：返回当前字符串在字符串常量池中的规范引用。
- 如果池里已经有内容相同的字符串，就返回池里的那个引用。
- 如果池里没有，就把当前字符串加入池中，并返回池中的引用。

```java
String a = new String("abc");
String b = a.intern();
String c = "abc";

System.out.println(a == c);
System.out.println(b == c);
```

输出结果：
```java
false
true
```

解释：
- `a` 是堆中通过 `new` 创建的新对象。
- `b = a.intern()` 返回的是池中的 `"abc"`。
- `c` 也是池中的 `"abc"`。
- 所以 `b == c` 为 `true`，而 `a == c` 为 `false`。

#### 编译期拼接和运行期拼接的区别
```java
String a = "ab";
String b = "a" + "b";
String x = "a";
String c = x + "b";

System.out.println(a == b);
System.out.println(a == c);
System.out.println(a == c.intern());
```

输出结果：
```java
true
false
true
```

解释：
- `"a" + "b"` 是编译期常量拼接，编译器会直接优化成 `"ab"`，所以 `b` 指向常量池。
- `x + "b"` 是运行期拼接，会生成新的字符串对象，所以 `c` 默认不和池中 `"ab"` 同一个引用。
- 调用 `c.intern()` 后，就能拿到池中的 `"ab"` 引用。

#### 你截图里说的几个“常量池”到底怎么区分
很多资料会同时提到 3 个概念：

- `class` 文件常量池
- 运行时常量池
- 字符串常量池

它们不是同一个东西，容易混。

#### 1）`class` 文件常量池
- 这是 `.class` 文件里自带的一张常量表。
- 它在编译后就存在，属于类文件结构的一部分。
- 里面放的是字面量和符号引用，比如类名、方法名、字段名、字符串字面量等信息。
- 它更偏“编译产物”。

#### 2）运行时常量池
- 类被 JVM 加载后，`class` 文件常量池中的内容会被加载到内存中，形成运行时常量池。
- 它是 JVM 运行时的数据结构，每个类/类型都有一个。
- 它服务于类解析、符号引用解析、方法调用等运行时行为。
- 它更偏“类加载后的运行时表示”。

#### 3）字符串常量池
- 这是专门用于保存驻留字符串的共享池。
- 不是每个类一个，而是整个 JVM 里共享的一份逻辑字符串池。
- 它服务于字符串复用和 `intern()` 机制。
- 面试里问“字符串常量池”，通常就是在问这个。

#### 三者一句话区分
- `class` 文件常量池：在 `.class` 文件里。
- 运行时常量池：类加载后进入 JVM 运行时。
- 字符串常量池：专门保存驻留字符串，供全局复用。

#### JDK 版本上的常见说法怎么记
- 老资料常说 JDK 6 时字符串池在永久代。
- 从 JDK 7 开始，HotSpot 把字符串池调整到了堆上。
- JDK 8 移除了永久代，换成了元空间。
- 面试时如果不确定细节，最稳妥的说法是：现代 Java 中字符串常量池逻辑上是全局共享的，HotSpot 实现里字符串池位于堆上。

#### 为什么 `String` 不可变和常量池有关
- 正因为 `String` 不可变，JVM 才能放心让多个引用共享同一个池中对象。
- 如果字符串可变，一个地方改内容，其他地方都会受影响，那常量池就失去意义了。
- 所以“字符串常量池”和“String 不可变”是强关联知识点。

#### 面试常见误区
- “字符串常量池”和“运行时常量池”不是同一个概念。
- `new String("abc")` 不等于只创建一个对象，实际会涉及池中对象和堆中对象的关系。
- `==` 比较的是引用，不能直接拿它判断字符串内容是否相等。
- 面试里说字符串常量池，通常默认指驻留字符串池，不是泛指所有常量池。

#### 一句话总结
- 字符串常量池就是 JVM 为了复用相同字符串而维护的一块共享机制，相同字面量通常只保留一份，通过 `intern()` 也可以手动拿到池中的规范引用。

#### 面试回答模板
- 字符串常量池本质上是 JVM 为了复用字符串对象而设计的一种共享机制，面试里通常指字符串驻留池，也就是 `StringTable`。
- 字符串字面量和常量表达式的结果通常会进入池中，后续再出现相同内容时可以直接复用，节省内存。
- `new String("abc")` 会在堆中创建新对象，而 `intern()` 可以返回池中的规范引用。
- 需要注意，字符串常量池、`class` 文件常量池和运行时常量池不是同一个概念。

#### 面试简答版
- 字符串常量池是 JVM 用来复用相同字符串的一种共享机制。
- 相同的字符串字面量通常只会在池里保留一份。
- `new String()` 会创建堆对象，`intern()` 可以返回池中的引用。
- 它和 `class` 文件常量池、运行时常量池不是一回事。

## 5. `intern()` 是什么意思？

#### 核心含义
- `intern()` 是 `String` 的一个方法，用来返回当前字符串在字符串常量池中的规范引用。
- 如果常量池中已经有内容相同的字符串，就直接返回池中的那个引用。
- 如果没有，就把该字符串加入池中，再返回池中的引用。
- 你可以把它理解成：把当前字符串“对齐”到字符串常量池里那一份标准对象上。

#### 最经典的例子
```java
String a = new String("abc");
String b = a.intern();
String c = "abc";

System.out.println(a == c);
System.out.println(b == c);
```

输出结果：
```java
false
true
```

解释：
- `a` 是通过 `new` 创建的堆对象。
- `b = a.intern()` 后，拿到的是常量池中 `"abc"` 的引用。
- `c` 作为字面量，通常直接指向池中的 `"abc"`。
- 所以 `b == c` 为 `true`，而 `a == c` 为 `false`。

#### 它和 `new String()` 的关系
- `new String("abc")` 创建的是堆中的新对象。
- `intern()` 不是把当前变量“改成池对象”，而是返回池中的那份引用。
- 所以很多时候：

```java
String s = new String("abc");
System.out.println(s == s.intern());
```

输出结果通常是：
```java
false
```

解释：
- `s` 是堆对象。
- `s.intern()` 返回的是池对象。
- 这两个引用通常不是同一个。

#### 编译期和运行期拼接时的表现
```java
String a = "ab";
String b = "a" + "b";
String x = "a";
String c = x + "b";

System.out.println(a == b);
System.out.println(a == c);
System.out.println(a == c.intern());
```

输出结果：
```java
true
false
true
```

解释：
- `"a" + "b"` 是编译期常量拼接，编译器会直接优化成 `"ab"`，所以 `b` 指向池中对象。
- `x + "b"` 是运行期拼接，`c` 通常是新生成的堆对象。
- `c.intern()` 返回池中的 `"ab"`，所以和 `a` 引用相同。

#### 面试易错点
- `intern()` 返回的是池中的引用，不一定返回当前对象本身。
- `intern()` 不等于“复制一份字符串”，它更像“去池里拿统一引用”。
- 面试里看到 `==` 时，要先分清谁在堆里，谁在池里。

#### 面试回答模板
- `intern()` 是 `String` 的一个方法，用来返回当前字符串在字符串常量池中的规范引用。
- 如果池里已经有相同内容的字符串，就直接返回池中的引用；如果没有，就把它放进池里再返回。
- 它的主要作用是让相同内容的字符串尽量复用，减少重复对象。

#### 面试简答版
- `intern()` 的作用是返回字符串常量池中的引用。
- 池里有就直接返回，没有就放进去再返回。
- 它的本质是让相同内容的字符串共享池中的同一份对象。

## 6. `intern()` 在 JDK 6 / 7 / 8 里有哪些常见面试说法？

#### 先说不变的部分
- 不管 JDK 6、7 还是 8，`intern()` 的对外语义基本不变：返回字符串常量池中的规范引用。
- 真正常被问到的差别，主要在于 HotSpot 的底层实现和内存位置变化。

#### JDK 6 的常见说法
- JDK 6 中，字符串常量池常被认为位于永久代（PermGen）。
- 对于通过 `new` 创建的堆中字符串，调用 `intern()` 时，池里通常维护的是另一份池中引用。
- 所以在很多 JDK 6 语境下，堆对象和池对象通常不是同一个引用。

```java
String s = new String("abc");
System.out.println(s == s.intern());
```

常见理解结果：
```java
false
```

#### JDK 7 的常见说法
- JDK 7 开始，HotSpot 把字符串池调整到了堆上。
- 如果池里原本没有该字符串，那么调用 `intern()` 时，HotSpot 常会直接把当前堆对象的引用记录到池中。
- 所以在某些场景下，会出现 `s == s.intern()` 为 `true`。

```java
String s = new StringBuilder("he").append("llo").toString();
System.out.println(s == s.intern());
```

常见结果：
```java
true
```

解释：
- 如果池里之前还没有 `"hello"`，那么 `intern()` 可能直接返回当前这个堆对象对应的池引用。

#### JDK 8 的常见说法
- JDK 8 移除了永久代，改成元空间（Metaspace）。
- 但字符串池这部分实现思路基本延续 JDK 7。
- 所以关于 `intern()` 的常见理解，JDK 8 和 JDK 7 大体一致。

#### 为什么有时又会是 `false`
```java
String s = new StringBuilder("ja").append("va").toString();
System.out.println(s == s.intern());
```

常见结果：
```java
false
```

解释：
- 因为 `"java"` 往往在 JVM 启动早期就已经存在于池中。
- 这时 `intern()` 返回的是池里早就存在的 `"java"`，而不是你刚拼出来的堆对象 `s`。
- 所以这个结果是否为 `true`，关键要看池里之前是否已经有这个字符串。

#### 最稳的版本对比记法
- JDK 6：池在永久代，常见理解是“堆对象和池对象通常分开”。
- JDK 7：池移到堆上，某些情况下 `intern()` 会直接复用当前堆对象。
- JDK 8：移除永久代，但 `intern()` 的常见行为整体延续 JDK 7。

#### 面试回答模板
- `intern()` 的语义在各版本中基本一致，都是返回字符串常量池中的规范引用。
- 版本差异主要在于底层实现：JDK 6 中字符串池常被认为在永久代，JDK 7 开始迁移到堆中，JDK 8 则延续 JDK 7 的思路。
- 因此在 JDK 7/8 中，如果池里原来没有该字符串，某些场景下会出现 `s == s.intern()` 为 `true`；而 JDK 6 中常见理解通常不是这样。

#### 面试简答版
- `intern()` 的语义没变，变的是底层实现。
- JDK 6 常说字符串池在永久代；JDK 7 开始字符串池移到堆上；JDK 8 延续 JDK 7。
- 所以 `s == s.intern()` 在 JDK 7/8 某些场景下可能为 `true`，但不是绝对的。

## 7. `String str = "aaa"` 与 `String str = new String("aaa")` 一样吗？`new String("aaa")` 创建了几个字符串对象？

#### 先给结论
- 不一样。
- `String str = "aaa"` 通常直接让 `str` 指向字符串常量池中的 `"aaa"`。
- `String str = new String("aaa")` 一定会在堆上新建一个 `String` 对象。
- 至于一共创建几个对象，要看字符串常量池里之前有没有 `"aaa"`。

#### 第一问：两种写法一样吗
- 从内容上看，它们的字符串内容一样，`equals()` 一般为 `true`。
- 从引用上看，它们通常不一样，`==` 一般为 `false`。
- 因为前者通常指向池中对象，后者指向堆中新对象。

```java
String a = "aaa";
String b = new String("aaa");

System.out.println(a == b);
System.out.println(a.equals(b));
```

输出结果：
```java
false
true
```

解释：
- `a` 通常指向池中的 `"aaa"`。
- `b` 指向 `new` 出来的堆对象。
- 所以引用不同，但内容相同。

#### 第二问：`new String("aaa")` 到底创建几个对象

#### 常见面试回答
- 常见回答是：1 个或 2 个。
- 如果字符串常量池中原本没有 `"aaa"`，那么通常会有：
  - 1 个池中对象 `"aaa"`
  - 1 个堆中对象 `new String("aaa")`
- 这时通常可理解为创建了 2 个对象。

- 如果字符串常量池中已经有 `"aaa"`，那么 `new String("aaa")` 只需要再创建 1 个堆对象。
- 这时通常可理解为创建了 1 个对象。

#### 为什么会这样
- 字符串字面量 `"aaa"` 会和常量池机制发生关系。
- `new String("aaa")` 里的 `"aaa"` 这个字面量本身，可能早就已经在池里了。
- `new` 的职责是再额外创建一个堆对象，所以堆对象这一个是一定有的。
- 争议点通常只在池中的 `"aaa"` 是否需要这次顺带新建。

#### 最稳的说法
- 面试时最稳妥的回答是：
  - `new String("aaa")` 至少会创建 1 个堆对象；
  - 如果常量池里原本没有 `"aaa"`，那么还会额外涉及 1 个池中对象；
  - 所以总数通常说成 1 个或 2 个，取决于池中是否已存在该字面量。

#### 对比例子 1：池中已存在时
```java
String x = "aaa";
String y = new String("aaa");

System.out.println(x == y);
System.out.println(x == y.intern());
```

输出结果：
```java
false
true
```

解释：
- `x` 先让 `"aaa"` 进入或使用池中对象。
- `y` 再 `new` 时，只会新建堆对象。
- `y.intern()` 返回池中的 `"aaa"`，所以和 `x` 相同。

#### 对比例子 2：只看 `new String("aaa")`
```java
String y = new String("aaa");
System.out.println(y == "aaa");
System.out.println(y.equals("aaa"));
```

输出结果：
```java
false
true
```

解释：
- `"aaa"` 代表池中对象。
- `y` 是堆对象。
- 所以引用不同，内容相同。

#### 面试易错点
- 不要机械地背“永远创建 2 个对象”。
- 更准确的说法是：至少 1 个，最多通常按面试语境回答为 2 个，取决于池中是否已存在字面量。
- `String str = "aaa"` 不会每次都新建对象，它优先走字符串常量池复用。

#### 面试回答模板
- `String str = "aaa"` 和 `String str = new String("aaa")` 不一样。
- 前者通常直接指向字符串常量池中的对象，后者一定会在堆上再创建一个新对象，所以两者 `==` 一般为 `false`，但 `equals()` 通常为 `true`。
- 至于 `new String("aaa")` 创建几个对象，最稳妥的说法是：至少创建 1 个堆对象；如果常量池里原本没有 `"aaa"`，还会额外涉及 1 个池中对象，所以常见面试答案是 1 个或 2 个。

#### 面试简答版
- 不一样：一个通常指向池中对象，一个是堆中新对象。
- `==` 一般为 `false`，`equals()` 一般为 `true`。
- `new String("aaa")` 至少创建 1 个堆对象；如果池中原本没有 `"aaa"`，通常还会涉及 1 个池中对象，所以常见答案是 1 个或 2 个。

## 8. `String` 是最基本的数据类型吗？

#### 结论
- 不是。
- Java 中最基本的数据类型只有 8 种：
  - `byte`
  - `short`
  - `int`
  - `long`
  - `float`
  - `double`
  - `char`
  - `boolean`
- `String` 不属于基本类型，而是引用类型。

#### 为什么 `String` 不是基本类型
- 因为 `String` 本质上是一个类，定义在 `java.lang.String` 中。
- 既然是类，就说明它是对象类型，不是 primitive type（基本类型）。
- 它可以调用方法，比如 `length()`、`substring()`、`equals()`，这也是它是对象的直接体现。

```java
String s = "abc";
System.out.println(s.length());
```

输出结果：
```java
3
```

解释：
- `s` 能调用成员方法，说明它是对象，不是基本类型。

#### 为什么很多人会误以为 `String` 很像基本类型
- 因为 `String` 用起来特别像基本类型，写法很简洁：

```java
String s = "hello";
```

- 不像普通对象那样总是显式写 `new`。
- 另外，`String` 在 Java 中使用频率极高，又有常量池优化，所以初学者容易误以为它是特殊的基本类型。
- 但本质上它仍然是一个引用类型。

#### 面试补充
- `String` 虽然不是基本类型，但它确实是 Java 中一个非常特殊、非常核心的引用类型。
- 比如：
  - 支持字面量写法
  - 有字符串常量池
  - 是不可变对象
  - 常被当作 key 使用

#### 面试回答模板
- `String` 不是 Java 的基本数据类型。
- Java 的基本数据类型只有 8 种，而 `String` 本质上是 `java.lang.String` 类的对象，属于引用类型。
- 只是因为它支持字面量写法并且使用非常频繁，所以看起来比较像基本类型。

#### 面试简答版
- `String` 不是基本类型，是引用类型。
- Java 的基本类型只有 8 种，`String` 本质上是一个类。

## 9. `String` 有哪些特性？

#### 特性一：不可变
- `String` 一旦创建，内容就不能修改。
- 所有看起来像修改字符串的方法，本质上都是返回一个新字符串。

```java
String s1 = "abc";
String s2 = s1.replace("a", "x");

System.out.println(s1);
System.out.println(s2);
```

输出结果：
```java
abc
xbc
```

解释：
- `s1` 没变。
- `replace()` 返回了新对象 `s2`。

#### 特性二：支持字符串常量池复用
- 相同内容的字符串字面量通常只会在常量池中保留一份。
- 后面再出现相同字面量时，会优先复用池中的对象。

```java
String a = "hello";
String b = "hello";

System.out.println(a == b);
```

输出结果：
```java
true
```

解释：
- `a` 和 `b` 通常都指向常量池中的同一个 `"hello"` 对象。

#### 特性三：`String` 类是 `final`
- `String` 被 `final` 修饰，不能被继承。
- 这样可以防止子类破坏字符串的不可变性和安全性。

```java
public final class String
```

含义：
- 不能写 `class MyString extends String`

#### 特性四：适合做哈希 key
- 因为 `String` 不可变，所以内容稳定，`hashCode()` 也稳定。
- 这使它非常适合作为 `HashMap` 的 key。
- 同时 `String` 的哈希值还可以被缓存，进一步提高效率。

#### 特性五：线程安全
- `String` 的线程安全不是靠加锁，而是靠不可变。
- 多个线程共享同一个字符串对象时，不会出现内容被改掉的问题。

#### 面试补充
- `String` 之所以在 Java 中这么特殊，就是因为它同时具备：
  - 不可变
  - 常量池复用
  - `final`
  - 哈希友好
  - 天然适合共享

#### 面试回答模板
- `String` 的主要特性有：不可变、支持常量池复用、类被 `final` 修饰、适合做哈希 key、线程安全。
- 其中最核心的特性是不可变，因为很多其他特性都是建立在不可变基础上的，比如常量池、安全性和哈希稳定性。

#### 面试简答版
- `String` 的核心特性是不可变。
- 它还支持常量池复用、类是 `final` 的、适合作为哈希 key，并且天然线程安全。

## 10. 在使用 `HashMap` 的时候，用 `String` 做 key 有什么好处？

#### 先给结论
- `String` 非常适合做 `HashMap` 的 key。
- 因为它不可变、`hashCode()` 稳定、哈希值可缓存，而且字符串在业务中又非常常见。

#### 好处一：不可变，key 稳定
- `HashMap` 的 key 一旦放入集合，就不应该再变化。
- 如果 key 内容变了，`hashCode()` 也可能跟着变，桶位置就乱了，后续查找可能失败。
- `String` 内容不可变，所以放入 `HashMap` 后它的逻辑值不会变化。

#### 好处二：`hashCode()` 稳定且可缓存
- `String` 的哈希值只和内容有关。
- 因为内容不会变，所以它的 `hashCode()` 算出来后可以缓存起来。
- 再次使用时不需要重复计算，查找效率更高。

#### 好处三：`equals()` 语义清晰
- `String` 已经重写了 `equals()`，按内容比较。
- 所以两个内容相同的字符串会被认为是同一个逻辑 key。
- 这和实际业务语义非常契合，比如用户名、订单号、路径、编码等。

#### 好处四：字符串在业务场景中最常见
- 很多系统中的 key 天然就是字符串：
  - 用户名
  - id 编码
  - URL
  - 路径
  - 配置项
  - token
- 所以 `String` 既常见，又刚好满足哈希 key 的理想特征。

#### 一个简单例子
```java
import java.util.HashMap;
import java.util.Map;

public class Demo {
    public static void main(String[] args) {
        Map<String, Integer> map = new HashMap<>();
        map.put("Tom", 100);

        System.out.println(map.get("Tom"));
    }
}
```

输出结果：
```java
100
```

解释：
- `"Tom"` 作为 key，内容稳定，哈希规则清晰，查找结果可靠。

#### 如果 key 可变会有什么风险
- 假设你自定义一个可变对象作为 key，放进 `HashMap` 后又改了对象中的关键字段。
- 那么它的 `hashCode()` 和 `equals()` 规则可能都会变化。
- 这时你再去 `get()`，很可能找不到原来的值。
- 所以 `String` 这种不可变 key 就显得非常合适。

#### 面试回答模板
- `String` 适合做 `HashMap` 的 key，主要因为它不可变。
- 不可变意味着字符串内容稳定，`hashCode()` 稳定，不会在放入 `HashMap` 后因为内容变化而导致查找失败。
- 同时 `String` 已经正确重写了 `equals()` 和 `hashCode()`，并且哈希值可以缓存，所以效率和可靠性都很好。

#### 面试简答版
- `String` 做 `HashMap` key 的最大好处是不可变。
- 因为它内容稳定、`hashCode()` 稳定、可缓存，而且 `equals()` 语义清晰，所以非常适合作为 key。
