# 反射

## 1. 什么是反射？

#### 先给结论
- 反射就是：程序在运行时动态获取类的信息，并且动态操作类或对象的机制。
- 通过反射，程序可以在运行时拿到一个类的：
  - 类名
  - 构造器
  - 成员变量
  - 方法
  - 注解
- 还可以进一步：
  - 创建对象
  - 调用方法
  - 访问或修改属性

#### 通俗理解
- 正常写代码时，我们通常在编译期就知道要操作哪个类、调用哪个方法。
- 反射则是：编译时不完全确定，运行时再决定要操作谁。
- 所以反射最大的特点就是“动态”。

#### 为什么叫“反射”
- 因为程序平时是“你写类，然后代码去用类”。
- 反射相当于程序运行时反过来“观察类自己”，把类的内部结构再取出来分析和操作。
- 所以叫反射。

#### 一个最简单的例子
```java
Class<?> clazz = Class.forName("java.lang.String");
System.out.println(clazz.getName());
```

输出结果：
```java
java.lang.String
```

解释：
- 这里程序在运行时拿到了 `String` 类对应的 `Class` 对象。
- 然后再通过它获取类名。
- 这就是最基础的反射操作。

#### 反射通常能做什么

### 1）获取类信息
- 获取类名
- 获取父类
- 获取接口
- 获取构造器
- 获取字段
- 获取方法

### 2）动态创建对象
```java
Class<?> clazz = String.class;
Object obj = clazz.getDeclaredConstructor().newInstance();
System.out.println(obj);
```

输出结果：
```java

```

解释：
- 这里动态创建了一个空字符串对象。
- `println` 打印空字符串，所以输出看起来是空白。

### 3）动态调用方法
```java
import java.lang.reflect.Method;

public class Demo {
    public static void main(String[] args) throws Exception {
        String str = "abc";
        Method method = String.class.getMethod("toUpperCase");
        Object result = method.invoke(str);
        System.out.println(result);
    }
}
```

输出结果：
```java
ABC
```

解释：
- 这里通过反射拿到 `toUpperCase()` 方法
- 再在运行时动态调用它

#### 反射的核心入口是什么
- Java 反射最核心的入口是 `Class` 类。
- 先拿到一个类对应的 `Class` 对象，后面才能继续获取构造器、字段、方法等信息。

#### 常见获取 `Class` 对象的 3 种方式
```java
Class<?> c1 = String.class;
Class<?> c2 = "abc".getClass();
Class<?> c3 = Class.forName("java.lang.String");

System.out.println(c1 == c2);
System.out.println(c2 == c3);
```

输出结果：
```java
true
true
```

解释：
- 同一个类在 JVM 里通常只对应一个 `Class` 对象。
- 所以这 3 种方式拿到的通常是同一个对象。

#### 反射有什么优点
- 灵活，扩展性强
- 适合做通用框架
- 编译时不依赖具体实现类
- 可以实现配置驱动、插件化、动态加载

#### 反射有什么缺点
- 性能通常不如直接调用
- 代码可读性差
- 绕过封装时有一定安全风险
- 出问题时不如普通代码直观

#### 反射常见应用场景
- Spring / Spring Boot 的 IOC 和依赖注入
- MyBatis 的对象封装
- ORM 框架
- 注解解析
- 动态代理
- JDBC 驱动加载

#### 面试常见误区
- 反射不是“只能获取信息”，它还可以动态创建对象、调用方法、改字段。
- 反射不是“完全不能用”，虽然有性能损耗，但在框架层非常常见。
- 反射的核心不是背 API，而是理解“运行时动态获取并操作类信息”。

#### 面试回答模板
- 反射是 Java 在运行时动态获取类信息并动态操作对象的一种机制。
- 通过反射，可以在运行时获取类的构造器、成员变量、方法等信息，还可以动态创建对象、调用方法和访问属性。
- 反射的核心入口是 `Class` 对象，它广泛应用于 Spring、MyBatis、动态代理、注解解析等框架技术中。
- 它的优点是灵活、扩展性强，缺点是性能相对较低、代码可读性较差。

#### 面试简答版
- 反射就是程序在运行时动态获取类信息并操作类或对象的机制。
- 通过反射可以拿到类的构造器、字段、方法，也可以动态创建对象和调用方法。
- 它常用于 Spring、MyBatis、动态代理等框架中。

## 2. 反射机制的优缺点有哪些？

#### 先给结论
- 反射最大的优点是“灵活”和“动态”，最大的缺点是“性能、可读性和安全性”。
- 它非常适合框架开发，但不适合在普通业务代码里到处滥用。

#### 反射的优点

### 1）灵活性强，支持运行时动态操作
- 普通代码通常在编译期就确定要操作哪个类、哪个方法。
- 反射可以在运行时再决定“加载哪个类”“调用哪个方法”。
- 这让程序具备更强的动态性和扩展性。

```java
public class Demo {
    public static void main(String[] args) throws Exception {
        String className = "java.lang.StringBuilder";
        Class<?> clazz = Class.forName(className);
        Object obj = clazz.getDeclaredConstructor().newInstance();
        System.out.println(obj.getClass().getSimpleName());
    }
}
```

输出结果：
```java
StringBuilder
```

解释：
- 这里代码在编译时没有写死 `new StringBuilder()`。
- 而是在运行时根据类名字符串动态创建对象。
- 这正是反射“动态”的体现。

### 2）扩展性强，适合做通用框架
- 框架开发往往不能把业务类写死。
- 通过反射，框架可以不依赖具体实现类，而是在运行时装配对象、调用方法。
- 这也是 Spring、MyBatis、JDBC 驱动加载等大量使用反射的原因。

### 3）可以突破编译期限制
- 有些信息在编译期不确定，但运行时能拿到。
- 反射允许程序在运行时检查类结构、访问方法、处理注解。
- 所以它特别适合做 IOC、AOP、ORM、注解解析、动态代理等底层能力。

### 4）有利于解耦
- 调用方不需要在编译期直接依赖具体类。
- 只要运行时能拿到类名、方法名、注解信息，就能继续工作。
- 这有助于实现配置驱动、插件化、模块化设计。

#### 反射的缺点

### 1）性能通常比直接调用差
- 普通方法调用在编译期和运行期都更直接。
- 反射需要额外做方法查找、权限检查、参数封装等操作。
- 所以在高频热点路径中，反射通常不如直接调用高效。

### 2）可读性和可维护性较差
- 普通代码一眼能看出调用了哪个方法、操作了哪个类。
- 反射代码常常是字符串、`Class`、`Method`、`Field` 混在一起。
- 出问题时排查难度更高，也更不直观。

### 3）可能破坏封装性
- 通过反射可以访问私有成员。
- 比如 `setAccessible(true)` 可以绕过一部分访问控制。
- 这虽然增强了灵活性，但也意味着更高的安全风险和更差的封装性。

### 4）缺少编译期安全检查
- 普通代码里方法名写错，编译期往往就会报错。
- 反射很多错误要到运行时才暴露，比如：
  - 类名写错
  - 方法名写错
  - 参数不匹配
- 所以反射更容易出现运行时异常。

#### 一个能体现“运行时才报错”的例子
```java
import java.lang.reflect.Method;

public class Demo {
    public static void main(String[] args) throws Exception {
        Method method = String.class.getMethod("toUppercase");
        System.out.println(method);
    }
}
```

运行结果：
```java
Exception in thread "main" java.lang.NoSuchMethodException
```

解释：
- 这里方法名写错了，应该是 `toUpperCase`。
- 但因为是反射，错误不会在编译期暴露，而是在运行时抛异常。

#### 应该怎么理解“反射适合框架，不适合滥用”
- 框架层追求通用性、动态性，反射很有价值。
- 业务层更多追求清晰、稳定、易维护，直接调用通常更合适。
- 所以反射并不是“不能用”，而是“要用在对的地方”。

#### 面试常见误区
- 误区一：反射只有缺点。错，框架体系几乎离不开它。
- 误区二：反射可以随便替代普通调用。错，普通业务代码里滥用反射通常得不偿失。
- 误区三：反射慢到不能用。错，它有性能成本，但框架层依然大量使用。

#### 面试回答模板
- 反射的优点主要是灵活、动态、扩展性强，能够在运行时获取类信息并动态创建对象、调用方法，因此特别适合做通用框架。
- 它常用于 Spring、MyBatis、动态代理、注解解析等场景，有助于实现解耦和配置驱动。
- 反射的缺点主要是性能不如直接调用、代码可读性较差、可能破坏封装，并且很多错误只能在运行时发现。
- 所以反射非常适合框架层，但不适合在普通业务代码中滥用。

#### 面试简答版
- 反射的优点：灵活、动态、扩展性强，适合框架开发。
- 反射的缺点：性能较低、可读性差、可能破坏封装、错误多在运行时暴露。
- 一句话：框架层常用，业务层慎用。

## 3. 如何获取反射中的 `Class` 对象？

#### 先给结论
- 获取 `Class` 对象最常见有 3 种方式：
  1. `类名.class`
  2. `对象.getClass()`
  3. `Class.forName("全类名")`
- 这 3 种方式拿到的，通常都是同一个类在 JVM 中对应的同一个 `Class` 对象。

#### 为什么 `Class` 对象这么重要
- 反射的核心入口就是 `Class`。
- 只有先拿到 `Class` 对象，后面才能继续获取：
  - 构造器
  - 字段
  - 方法
  - 注解
- 所以“怎么拿到 `Class`”是反射第一步。

### 1）`类名.class`
- 这种方式最直接。
- 适合在编译期就已经知道具体类的时候使用。

```java
Class<?> c1 = String.class;
System.out.println(c1.getName());
```

输出结果：
```java
java.lang.String
```

解释：
- 这里直接通过类名拿到了 `String` 对应的 `Class` 对象。
- 不需要先创建对象。

### 2）`对象.getClass()`
- 这种方式适合你手里已经有对象实例的时候。
- 通过对象本身，反过来拿到它所属类的 `Class` 对象。

```java
String str = "hello";
Class<?> c2 = str.getClass();
System.out.println(c2.getName());
```

输出结果：
```java
java.lang.String
```

解释：
- `str` 是一个 `String` 对象。
- 所以 `str.getClass()` 拿到的是 `String` 的 `Class` 对象。

### 3）`Class.forName("全类名")`
- 这种方式最能体现反射的“动态性”。
- 适合在运行时通过字符串类名去加载类。

```java
Class<?> c3 = Class.forName("java.lang.String");
System.out.println(c3.getName());
```

输出结果：
```java
java.lang.String
```

解释：
- 这里类名是字符串形式。
- 程序在运行时根据这个字符串去找到对应类。
- 这是框架中非常常见的方式。

#### 这 3 种方式得到的是不是同一个对象
```java
Class<?> c1 = String.class;
Class<?> c2 = "abc".getClass();
Class<?> c3 = Class.forName("java.lang.String");

System.out.println(c1 == c2);
System.out.println(c2 == c3);
```

输出结果：
```java
true
true
```

解释：
- 同一个类在 JVM 中通常只会对应一个 `Class` 对象。
- 所以这 3 种方式拿到的通常是同一个对象引用。

#### 三种方式怎么选

### 用 `类名.class`
- 编译期就知道具体类
- 代码最简洁
- 最常见

### 用 `对象.getClass()`
- 已经有对象实例
- 想知道这个对象实际是什么类

### 用 `Class.forName()`
- 运行时才知道类名
- 类名可能来自配置文件、数据库、注解、外部输入
- 框架开发中特别常见

#### 补充：基本类型也能拿到 `Class` 对象
```java
Class<?> c1 = int.class;
Class<?> c2 = Integer.class;

System.out.println(c1.getName());
System.out.println(c2.getName());
```

输出结果：
```java
int
java.lang.Integer
```

解释：
- 基本类型和引用类型也都有各自对应的 `Class` 对象。

#### 面试常见误区
- `Class.forName()` 需要写全类名，不是简单类名。
- `对象.getClass()` 必须先有对象。
- `类名.class` 最直接，但前提是编译期已经知道这个类。
- 这 3 种方式通常拿到的是同一个类对应的同一个 `Class` 对象。

#### 面试回答模板
- 获取反射中的 `Class` 对象常见有 3 种方式：`类名.class`、`对象.getClass()`、`Class.forName("全类名")`。
- `类名.class` 适合编译期已知类型的场景；`对象.getClass()` 适合手里已经有对象实例；`Class.forName()` 适合运行时根据类名字符串动态加载类。
- 这 3 种方式拿到的通常都是同一个类在 JVM 中对应的唯一 `Class` 对象。

#### 面试简答版
- 获取 `Class` 对象常用 3 种方式：`类名.class`、`对象.getClass()`、`Class.forName("全类名")`。
- 前两种偏静态，`Class.forName()` 更能体现反射的动态加载能力。

## 4. Java 反射 API 有几类？

#### 先给结论
- Java 反射 API 可以理解为围绕“类、构造器、方法、字段”这几个核心对象展开。
- 面试里最常见、最核心的几类是：
  - `Class`
  - `Constructor`
  - `Method`
  - `Field`
- 如果再扩展一点，还常会提到：
  - `Modifier`
  - `Array`
  - `Annotation`
  - `Proxy`

#### 1）`Class`：反射的入口
- `Class` 是反射的核心入口。
- 先拿到 `Class` 对象，后面才能继续获取构造器、方法、字段等信息。
- 它负责描述“这个类本身”。

```java
Class<?> clazz = String.class;
System.out.println(clazz.getName());
```

输出结果：
```java
java.lang.String
```

#### 2）`Constructor`：表示构造器
- `Constructor` 用来表示类的构造方法。
- 它常用于通过反射动态创建对象。

```java
import java.lang.reflect.Constructor;

public class Demo {
    public static void main(String[] args) throws Exception {
        Constructor<String> constructor = String.class.getConstructor(String.class);
        String str = constructor.newInstance("abc");
        System.out.println(str);
    }
}
```

输出结果：
```java
abc
```

解释：
- 这里通过反射拿到 `String(String)` 构造器
- 再动态创建字符串对象

#### 3）`Method`：表示方法
- `Method` 用来表示成员方法。
- 它常用于运行时动态调用方法。

```java
import java.lang.reflect.Method;

public class Demo {
    public static void main(String[] args) throws Exception {
        Method method = String.class.getMethod("toUpperCase");
        Object result = method.invoke("abc");
        System.out.println(result);
    }
}
```

输出结果：
```java
ABC
```

#### 4）`Field`：表示成员变量
- `Field` 用来表示类中的属性。
- 它可以在运行时获取或修改对象字段值。

```java
import java.lang.reflect.Field;

class Person {
    public String name = "Tom";
}

public class Demo {
    public static void main(String[] args) throws Exception {
        Person p = new Person();
        Field field = Person.class.getField("name");
        System.out.println(field.get(p));
    }
}
```

输出结果：
```java
Tom
```

#### 5）`Modifier`：表示修饰符
- `Modifier` 用来解析类、方法、字段的修饰符。
- 比如判断一个方法是不是 `public`、一个类是不是 `final`。

```java
import java.lang.reflect.Method;
import java.lang.reflect.Modifier;

public class Demo {
    public static void main(String[] args) throws Exception {
        Method method = String.class.getMethod("length");
        System.out.println(Modifier.isPublic(method.getModifiers()));
    }
}
```

输出结果：
```java
true
```

#### 6）`Array`：反射方式操作数组
- `Array` 提供了通过反射动态创建和访问数组的能力。
- 它在框架和底层工具类里偶尔会用到。

```java
import java.lang.reflect.Array;

public class Demo {
    public static void main(String[] args) {
        Object arr = Array.newInstance(int.class, 3);
        Array.set(arr, 0, 10);
        System.out.println(Array.get(arr, 0));
    }
}
```

输出结果：
```java
10
```

#### 7）`Annotation`：反射读取注解
- 反射还可以配合注解一起工作。
- 框架经常通过反射读取类、方法、字段上的注解。

```java
Deprecated annotation = String.class.getAnnotation(Deprecated.class);
System.out.println(annotation);
```

输出结果通常是：
```java
null
```

解释：
- `String` 类本身没有 `@Deprecated`
- 所以拿到的是 `null`
- 这个例子主要是说明“反射也能读取注解”

#### 8）`Proxy`：动态代理
- `Proxy` 严格说不完全只是“基础反射类”，但常和反射一起考。
- 它用于在运行时动态生成代理对象。
- Spring AOP、JDK 动态代理等都和它相关。

#### 面试里最核心要记哪几个
- 如果面试官问“Java 反射 API 有几类”，最稳的回答是：
  - `Class`
  - `Constructor`
  - `Method`
  - `Field`
- 这四个是核心中的核心。
- 其他像 `Modifier`、`Array`、`Annotation`、`Proxy`，属于扩展知识点。

#### 一句话串起来
- `Class` 管类本身
- `Constructor` 管构造器
- `Method` 管方法
- `Field` 管属性

#### 面试回答模板
- Java 反射 API 主要围绕类的元信息展开，最核心的几类是 `Class`、`Constructor`、`Method` 和 `Field`。
- 其中 `Class` 是反射入口，`Constructor` 用于构造对象，`Method` 用于调用方法，`Field` 用于访问成员变量。
- 此外还常见 `Modifier`、`Array`、`Annotation`、`Proxy` 等扩展类，用于修饰符解析、数组操作、注解解析和动态代理。

#### 面试简答版
- Java 反射 API 最核心的 4 类是：`Class`、`Constructor`、`Method`、`Field`。
- 它们分别对应：类、构造器、方法、属性。
- 其他常见扩展还有 `Modifier`、`Array`、`Annotation`、`Proxy`。

## 5. 反射使用的步骤？

#### 先给结论
- 反射最常见的使用流程就是 4 步：获取 `Class` -> 获取成员 -> 创建对象或准备目标对象 -> 执行反射操作。
- 面试里真正要答出来的，不只是“背步骤”，而是每一步分别在干什么、最后靠哪些 API 落地。

#### 核心解释
- 第一步先拿到类的 `Class` 对象，因为它是反射的总入口。
- 第二步从 `Class` 里找到你要操作的成员，比如构造器、方法、字段。
- 第三步如果你要调实例方法、读实例字段，一般还得先准备目标对象。
- 第四步才是真正执行，比如用 `newInstance()` 创建对象、用 `invoke()` 调方法、用 `get()/set()` 操作字段。

### 第一步：获取 `Class` 对象
```java
Class<?> clazz = Class.forName("java.lang.StringBuilder");
System.out.println(clazz.getName());
```

输出结果：
```java
java.lang.StringBuilder
```

输出 / 现象解释：
- `Class` 是反射入口。
- 先拿到类对应的 `Class` 对象，后面才能继续拿构造器、方法、字段。

### 第二步：获取要操作的成员
```java
Constructor<String> constructor = String.class.getConstructor(String.class);
Method method = String.class.getMethod("toUpperCase");
Field field = Person.class.getField("name");
```

输出结果：
- 这一步通常没有直接输出，它的作用是“先把要操作的成员找出来”。

输出 / 现象解释：
- `Constructor` 表示构造器。
- `Method` 表示方法。
- `Field` 表示字段。
- 这一阶段只是“查找”，还没有真正执行。

### 第三步：创建对象或准备目标对象
- 如果后面要调用实例方法、访问实例字段，通常要先有一个对象。
- 这个对象可以普通 `new` 出来，也可以反射创建。

```java
Object obj = StringBuilder.class.getDeclaredConstructor().newInstance();
System.out.println(obj.getClass().getSimpleName());
```

输出结果：
```java
StringBuilder
```

输出 / 现象解释：
- 这里通过无参构造器动态创建了一个 `StringBuilder` 对象。
- 本质上还是在调构造器，只不过是反射方式调用。

### 第四步：执行反射操作
#### 调用方法
```java
import java.lang.reflect.Method;

public class Demo {
    public static void main(String[] args) throws Exception {
        Method method = String.class.getMethod("toUpperCase");
        Object result = method.invoke("abc");
        System.out.println(result);
    }
}
```

输出结果：
```java
ABC
```

#### 访问字段
```java
import java.lang.reflect.Field;

class Person {
    public String name = "Tom";
}

public class Demo {
    public static void main(String[] args) throws Exception {
        Person p = new Person();
        Field field = Person.class.getField("name");
        System.out.println(field.get(p));
    }
}
```

输出结果：
```java
Tom
```

输出 / 现象解释：
- `invoke()` 用来执行方法。
- `get()` / `set()` 用来读取或修改字段值。
- 这一步才是把前面查到的成员真正“跑起来”。

### 一个完整的小例子
```java
import java.lang.reflect.Constructor;
import java.lang.reflect.Method;

class Apple {
    private int price;

    public Apple() {
    }

    public int getPrice() {
        return price;
    }

    public void setPrice(int price) {
        this.price = price;
    }
}

public class Demo {
    public static void main(String[] args) throws Exception {
        Class<?> clazz = Class.forName("Apple");
        Constructor<?> constructor = clazz.getConstructor();
        Object apple = constructor.newInstance();

        Method setPrice = clazz.getMethod("setPrice", int.class);
        Method getPrice = clazz.getMethod("getPrice");

        setPrice.invoke(apple, 14);
        System.out.println(getPrice.invoke(apple));
    }
}
```

输出结果：
```java
14
```

输出 / 现象解释：
- 先拿 `Class`。
- 再拿构造器和方法。
- 再创建对象。
- 最后通过反射给对象赋值并取值。
- 这就是一个完整的“查找 -> 创建 -> 执行”的流程。

### 一句话记忆步骤
- 先拿类，再拿成员，再准备对象，最后执行操作。

#### 面试易错点 / 补充点
- 不是一上来就 `invoke()`，而是必须先拿到 `Method` 对象。
- 不是所有反射操作都必须先创建对象；如果是静态成员，可以不依赖实例。
- 获取到私有成员后，很多时候还不能直接用，通常要配合 `setAccessible(true)`。
- `getMethod()`、`getField()` 这些 API 只是“查找”，真正执行是后续步骤。

#### 面试回答模板
- 反射的使用步骤一般可以概括成 4 步：先获取类的 `Class` 对象，再获取要操作的构造器、方法或字段，然后创建对象或准备目标实例，最后通过反射 API 执行操作。
- 如果是创建对象，常见写法是 `getDeclaredConstructor().newInstance()`；如果是调用方法，常见写法是 `Method.invoke()`；如果是操作字段，则会用到 `Field.get()` 和 `Field.set()`。
- 所以反射的本质流程其实就是：先定位成员，再在运行时动态执行。

#### 面试简答版
- 反射步骤就是：获取 `Class` -> 获取成员 -> 准备对象 -> 执行操作。
- 调方法常用 `invoke()`，创建对象常用 `newInstance()`，操作字段常用 `get()/set()`。

## 6. `invoke()`、`newInstance()`、`getMethod()` 和 `getDeclaredMethod()` 怎么理解？

#### 先给结论
- `invoke()`：执行 `Method` 对象代表的方法。
- `newInstance()`：执行构造器，动态创建对象。
- `getMethod()`：查找 `public` 方法，包含继承来的 `public` 方法。
- `getDeclaredMethod()`：查找当前类自己声明的方法，权限不限，但不包含继承方法。
- 你可以把它们分成两类：`getMethod()` / `getDeclaredMethod()` 负责“找”，`invoke()` / `newInstance()` 负责“执行”。

#### 核心解释
- 反射不是一上来就能执行，通常要先查到成员，再执行成员。
- `getMethod()` 和 `getDeclaredMethod()` 是先把方法“找出来”。
- 找到 `Method` 之后，再通过 `invoke()` 真正调用。
- 找到 `Constructor` 之后，再通过 `newInstance()` 真正创建对象。
- 所以这几个 API 放在一起理解，最容易记住它们的分工。

### 1）`invoke()` 是什么意思
- `invoke` 直译就是“调用、执行”。
- 在反射里，它表示：通过 `Method` 对象，在运行时动态调用一个方法。
- 常见写法是：`method.invoke(目标对象, 参数1, 参数2...)`。
- 如果调的是实例方法，第一个参数就是目标对象；如果调的是静态方法，第一个参数可以传 `null`。

```java
import java.lang.reflect.Method;

public class Demo {
    public static void main(String[] args) throws Exception {
        Method method = String.class.getMethod("toUpperCase");
        Object result = method.invoke("abc");
        System.out.println(result);
    }
}
```

输出结果：
```java
ABC
```

输出 / 现象解释：
- `getMethod("toUpperCase")` 先拿到方法的“说明书”。
- `invoke("abc")` 才是真正让这个方法在字符串对象上执行。
- 返回值会被统一当成 `Object` 返回，所以这里用 `Object result` 接收。

```java
import java.lang.reflect.Method;

public class Demo {
    public static void main(String[] args) throws Exception {
        Method method = Math.class.getMethod("max", int.class, int.class);
        Object result = method.invoke(null, 3, 5);
        System.out.println(result);
    }
}
```

输出结果：
```java
5
```

输出 / 现象解释：
- `Math.max()` 是静态方法，不依赖某个具体对象。
- 所以 `invoke()` 的第一个参数传 `null` 也可以。

### 2）`newInstance()` 是什么意思
- `newInstance()` 的核心作用就是：通过反射调用构造器来创建对象。
- 它本质上相当于把普通代码里的 `new Xxx(...)` 放到运行时再决定。
- 现在更推荐的写法不是 `Class.newInstance()`，而是 `构造器对象.newInstance(...)`。

```java
Class<?> clazz = StringBuilder.class;
Object obj = clazz.getDeclaredConstructor().newInstance();
System.out.println(obj.getClass().getSimpleName());
```

输出结果：
```java
StringBuilder
```

输出 / 现象解释：
- `getDeclaredConstructor()` 先拿到无参构造器。
- `newInstance()` 再真正执行这个构造器。
- 这一句本质上就相当于 `new StringBuilder()`。

```java
import java.lang.reflect.Constructor;

public class Demo {
    public static void main(String[] args) throws Exception {
        Constructor<String> constructor = String.class.getConstructor(String.class);
        String str = constructor.newInstance("abc");
        System.out.println(str);
    }
}
```

输出结果：
```java
abc
```

输出 / 现象解释：
- 这里调的是有参构造器 `String(String)`。
- 所以这段代码本质上相当于 `new String("abc")`。

### 3）`getMethod()` 和 `getDeclaredMethod()` 的区别
- `getMethod()`：只找 `public` 方法，而且会向父类方向继续找。
- `getDeclaredMethod()`：只找“当前类自己声明的方法”，但权限不限，`private` 也能拿到。
- 所以它们的核心区别就是：查找范围不同。

```java
class Parent {
    public void show() {
    }
}

class Child extends Parent {
    private void test() {
    }
}

public class Demo {
    public static void main(String[] args) throws Exception {
        System.out.println(Child.class.getMethod("show").getName());
        System.out.println(Child.class.getDeclaredMethod("test").getName());
    }
}
```

输出结果：
```java
show
test
```

输出 / 现象解释：
- `show()` 是从父类继承来的 `public` 方法，所以 `getMethod()` 能拿到。
- `test()` 是 `Child` 自己声明的 `private` 方法，所以要用 `getDeclaredMethod()`。
- 反过来记也很重要：`getMethod("test")` 会失败，因为它不是 `public`；`getDeclaredMethod("show")` 也会失败，因为它不是 `Child` 自己声明的。

#### 面试易错点 / 补充点
- `getDeclaredMethod()` 能拿到 `private` 方法，不等于你就能直接调，很多时候还要 `setAccessible(true)`。
- `Class.newInstance()` 现在不推荐了，更建议用 `getDeclaredConstructor().newInstance()`。
- `invoke()` 的返回值类型统一是 `Object`；如果原方法返回 `void`，那拿到的结果就是 `null`。
- 如果被反射调用的方法内部自己抛异常，外层常常会看到 `InvocationTargetException`，这是面试里常见追问点。

#### 面试回答模板
- 这几个反射 API 可以分成“查找”和“执行”两类。`getMethod()` 和 `getDeclaredMethod()` 负责查找方法，其中前者查 `public` 且包含继承方法，后者查当前类自己声明的方法且不受访问修饰符限制。
- 查到 `Method` 之后，可以通过 `invoke()` 在运行时动态执行这个方法；查到构造器之后，可以通过 `newInstance()` 动态创建对象。
- 所以它们连起来的完整流程其实就是：先找到要操作的成员，再在运行时把它执行起来。

#### 面试简答版
- `invoke()`：调用方法。
- `newInstance()`：创建对象。
- `getMethod()`：找 `public` 方法，包含继承。
- `getDeclaredMethod()`：找本类声明的方法，包含 `private`，不包含继承。


## 7. 为什么引入反射概念？反射机制的应用有哪些？

#### 先给结论
- 引入反射，本质上是为了解决“编译期不知道具体类，但运行期又要获取类信息、创建对象、调用方法”的问题。
- 它让程序具备了运行时动态装配能力，所以特别适合框架、容器、插件化、注解解析这类场景。
- 反射不是为了替代普通 `new` 和普通方法调用，而是为了补上“运行时动态处理”的能力。

#### 核心解释
- 普通 Java 代码更偏“静态”：写代码时通常就已经知道要用哪个类、调哪个方法。
- 但很多场景下，类名、方法名、注解信息、配置内容，只有运行时才能确定。
- 这时候如果没有反射，框架就很难做到通用化、可扩展、可插拔。
- 所以反射的核心价值可以概括成一句话：让程序在运行时认识类、操作类、使用类。

#### 为什么需要反射
### 1）为了支持运行时动态加载和动态创建对象
- 有些类不是在代码里写死的，而是来自配置文件、注解、数据库、外部输入。
- 这时候程序只能在运行时根据类名去加载类、创建对象。

```java
public class Demo {
    public static void main(String[] args) throws Exception {
        String className = "java.lang.StringBuilder";
        Class<?> clazz = Class.forName(className);
        Object obj = clazz.getDeclaredConstructor().newInstance();
        System.out.println(obj.getClass().getSimpleName());
    }
}
```

输出结果：
```java
StringBuilder
```

输出 / 现象解释：
- 编译时并没有直接写死 `new StringBuilder()`。
- 而是在运行时根据字符串类名动态加载并创建对象。
- 这就是反射存在的直接意义。

### 2）为了让框架和业务代码解耦
- 框架不能把每个业务类都提前写死。
- 它只能在运行时扫描类、读取注解、创建对象、注入依赖。
- 所以像 Spring、MyBatis 这类框架，本质上都大量依赖反射。

### 3）为了在运行时获取类的完整结构信息
- 反射不仅能创建对象，还能枚举类的构造器、字段、方法、注解。
- 这让程序具备了“自描述”和“自检查”的能力。
- 比如框架可以根据字段名自动封装对象，根据注解决定执行逻辑。

#### 反射机制的应用有哪些
### 1）框架中的 IOC / 依赖注入
- Spring 在启动时会扫描类、读取注解、创建 Bean、完成注入。
- 这个过程中就会用反射创建对象、调用构造器、给字段赋值。

### 2）ORM / MyBatis 对象封装
- 查询数据库后，框架要把结果集封装成 Java 对象。
- 它通常会通过反射找到字段或 setter 方法，再把值放进去。

### 3）注解解析
- 很多框架会在运行时读取类、方法、字段上的注解。
- 然后根据注解决定注册 Bean、处理路由、做权限校验、生成代理逻辑。

### 4）动态代理和 AOP
- JDK 动态代理底层就和反射密切相关。
- 比如 Spring AOP 会在方法调用前后织入增强逻辑，本质上离不开运行时动态处理。

### 5）插件化、配置驱动、通用工具开发
- 如果系统支持“用户传一个类名，系统就自动加载对应实现类”，背后通常就是反射。
- 这类场景常见于插件系统、规则引擎、可扩展组件设计。

### 6）JDBC 驱动加载
- 这是面试里很经典的例子。
- 早期 JDBC 经常通过 `Class.forName()` 显式加载数据库驱动类，然后再建立连接。
- 这说明程序在编译时并不依赖某个具体驱动实现，而是在运行时决定加载哪种驱动。
- 不过要注意：从 JDBC 4 开始，很多驱动已经支持自动加载，所以现在不一定非要手写 `Class.forName()`。

```java
Class.forName("com.mysql.cj.jdbc.Driver");
System.out.println("driver loaded");
```

输出结果：
```java
driver loaded
```

输出 / 现象解释：
- 这段代码的重点不是打印字符串，而是“按类名把驱动类加载进来”。
- 在老项目或面试题里，这经常被当成反射应用的经典例子。

#### 怎么理解“为什么要引入反射”
- 因为 Java 想给程序一种“运行时再决定”的能力。
- 普通写法擅长处理确定的对象，反射擅长处理不确定、可扩展、可配置的对象。
- 所以反射不是给普通 CRUD 提速的，而是给框架、底层能力、通用组件增加灵活性的。

#### 面试易错点 / 补充点
- 反射不是专门为了“访问 private 成员”才引入的，这只是它的一个能力，不是核心目的。
- 反射的核心价值是动态性、扩展性、解耦，而不是炫技。
- 不是所有地方都应该用反射；业务代码里滥用反射，会让代码更难读、更难维护、性能也更差。
- 说 JDBC 时最好补一句：老写法常见 `Class.forName()`，但现代 JDBC 驱动很多支持自动加载。

#### 面试回答模板
- 引入反射，主要是为了解决编译期不能确定具体类型，但运行期又需要动态获取类信息、创建对象、调用方法的问题。
- 它让程序具备了运行时动态加载、动态装配和动态调用的能力，因此特别适合框架开发。
- 反射的典型应用包括 Spring 的 IOC 和依赖注入、MyBatis 的对象封装、注解解析、动态代理，以及早期 JDBC 中通过 `Class.forName()` 加载驱动。
- 所以一句话总结，反射是 Java 为了增强动态性和扩展性提供的一种底层机制。

#### 面试简答版
- 引入反射，是为了让程序在运行时动态获取类信息、创建对象、调用方法。
- 它主要用于框架开发和通用能力实现，比如 IOC、ORM、注解解析、动态代理、JDBC 驱动加载。
- 本质关键词就是：动态、解耦、扩展。

## 8. 反射机制的原理是什么？

#### 先给结论
- 反射的原理可以概括成一句话：JVM 先把类加载出来，Java 再通过 `Class` 对象拿到类的元信息，并在运行时动态操作构造器、方法、字段。
- 所以反射不是“凭空分析源码”，而是基于 JVM 里已经加载好的类元数据工作。
- 从执行链路看，大致就是：`Class.forName()` 加载类 -> 获取 `Class` 对象 -> 从 `Class` 中查方法、字段、构造器 -> 通过 `invoke()` / `newInstance()` 真正执行。

#### 核心解释
- 每个被 JVM 加载的类，运行时都会有一个对应的 `Class` 对象。
- 这个 `Class` 对象可以理解成“这个类在 JVM 里的运行时说明书”。
- 反射拿到的 `Method`、`Field`、`Constructor`，本质上都是基于这个 `Class` 对象继续查到的成员描述信息。
- 所以反射机制真正的根，不在 `Method.invoke()`，而在“类被加载后，JVM 把类信息交给 `Class` 对象体系对外暴露”。 

#### 可以把整个原理拆成 4 步理解
### 1）先通过类加载，拿到 `Class` 对象
- 比如 `Class.forName("com.demo.User")`，并不是 Java 自己去解析源码。
- 它本质上是让类加载器去定位、加载、链接这个类，成功后返回这个类对应的 `Class` 对象。
- 这也是为什么 `Class` 是整个反射体系的入口。

```java
Class<?> clazz = Class.forName("java.lang.String");
System.out.println(clazz.getName());
```

输出结果：
```java
java.lang.String
```

输出 / 现象解释：
- 这一步说明：反射的第一步不是调方法，而是先拿到运行时类对象 `Class`。
- 没有 `Class`，后面的方法、字段、构造器都无从谈起。

### 2）`Class` 对象里会管理这类的反射元信息
- 你截图里的图二想表达的核心是对的：`Class` 对象内部会关联方法、字段、构造器等反射数据。
- 当你调用 `getMethod()`、`getDeclaredMethod()`、`getConstructor()` 这类 API 时，不是每次都重新从零开始找。
- 旧版 JDK 源码里可以看到，`Class` 内部有 `ReflectionData`，里面会缓存 `Method[]`、`Field[]`、`Constructor[]` 之类的信息。
- 所以反射并不是“每次都完全裸查 JVM”，而是“先看缓存，没有再向 JVM 要，再缓存起来”。

### 3）查到成员后，返回的是反射对象，不是直接执行结果
- 比如 `getMethod("sayHello", String.class)` 返回的是一个 `Method` 对象。
- 这个对象只是“方法描述信息 + 调用入口”，还没真正执行方法。
- 同理，`getDeclaredConstructor()` 返回的是 `Constructor` 对象，也还没真正创建实例。

```java
import java.lang.reflect.Method;

public class Demo {
    public static void main(String[] args) throws Exception {
        Method method = String.class.getMethod("toUpperCase");
        System.out.println(method.getName());
    }
}
```

输出结果：
```java
toUpperCase
```

输出 / 现象解释：
- 这里只是“找到”了方法，还没有执行。
- 所以你要把“找成员”和“执行成员”分开理解。

### 4）真正执行时，再通过 `invoke()` 或 `newInstance()` 进入底层调用
- `Method.invoke(obj, args...)` 表示在运行时调用方法。
- `Constructor.newInstance(args...)` 表示在运行时调用构造器创建对象。
- 老文章里常见 `Class.newInstance()`，但这个 API 现在已经不推荐，更建议用 `getDeclaredConstructor().newInstance()`。

```java
import java.lang.reflect.Method;

public class Demo {
    public static void main(String[] args) throws Exception {
        Method method = String.class.getMethod("toUpperCase");
        Object result = method.invoke("abc");
        System.out.println(result);
    }
}
```

输出结果：
```java
ABC
```

输出 / 现象解释：
- `getMethod()` 是先找到方法。
- `invoke()` 才是让这个方法在目标对象上真正执行。
- 所以反射的本质流程是：先拿类信息，再拿成员信息，最后动态执行。

#### 补充：为什么文章里说 `invoke()` 超过 15 次会变快
- 这个“超过 15”说的不是方法业务逻辑执行了 15 行，也不是参数长度超过 15。
- 它说的是：同一个反射调用入口，在旧版 JDK 实现里累计调用次数超过 15 次之后，会切换实现方式。
- 旧版反射有一个叫 `inflation` 的机制：前几次先走 native 反射调用，启动快；调用多了之后，再生成一个字节码版的 `MethodAccessor`，后续调用走这个生成类，通常更快。
- 所以“15”本质上是一个阈值，意思是“调用次数超过这个阈值，就从初始方案切到优化后的方案”。

```java
Method method = MyClass.class.getMethod("myMethod");

for (int i = 1; i <= 20; i++) {
    method.invoke(new MyClass());
}
```

输出结果：
- 假设 `myMethod()` 有打印，那会打印 20 次。
- 但底层实现上，旧版 JDK 里前 15 次和后面几次，可能已经不是完全同一种调用路径了。

输出 / 现象解释：
- 前几次：通常先走 native 方式，优点是启动简单，省去立刻生成字节码的成本。
- 超过阈值后：JDK 会生成一个专门的 accessor 类，后续反射调用改走这个 accessor。
- 所以文章里说“超过 15”实际上是在解释旧版 JDK 的一个性能优化策略。

#### 这个 15 是固定的吗
- 在老实现里，默认阈值通常就是 15，所以很多文章都会直接写“超过 15 次”。
- 但你面试最好别把它背成绝对铁律，因为这属于 JDK 内部实现细节，不是 Java 语言规范。
- 面试更稳的说法是：旧版 JDK 反射调用有 inflation 优化机制，前期走 native，达到一定调用次数后会切到生成字节码的 accessor 实现。

#### 要注意版本差异
- 你这篇文章讲的思路，主要对应旧版 JDK 的反射实现。
- OpenJDK 的旧源码里，`NativeMethodAccessorImpl` 确实有 `numInvocations > ReflectionFactory.inflationThreshold()` 这样的判断。
- 但从 JDK 18 开始，官方通过 JEP 416 重写了核心反射实现，底层改为基于 method handle，不再靠这套“15 次后 inflation”的老机制来讲主线。
- 所以如果面试官问原理，你可以先答旧实现；如果追问版本差异，再补一句：新版本底层实现已经调整了。

#### 你截图里那段代码，原理上怎么走
```java
Class<?> actionClass = Class.forName("MyClass");
Object action = actionClass.getDeclaredConstructor().newInstance();
Method method = actionClass.getMethod("myMethod");
method.invoke(action);
```

输出结果：
- 如果 `myMethod()` 里有打印，就执行打印；如果没有，就没有直接输出。

输出 / 现象解释：
- 第 1 行：让类加载器把 `MyClass` 加载进 JVM，并返回它的 `Class` 对象。
- 第 2 行：通过构造器反射创建 `MyClass` 实例。
- 第 3 行：从 `Class` 对象中查找名为 `myMethod` 的方法。
- 第 4 行：通过 `Method.invoke()` 真正执行该方法。
- 这 4 行代码刚好就是反射原理最简化的一条链路。

#### 为什么图里说会 copy 一份 `Method` / `Constructor`
- 这个点你可以简单记成：`Class` 内部缓存的是底层反射对象，对外返回时通常不会直接把内部缓存原对象暴露出去，而是返回一个副本或包装后的对象。
- 这样做的目的，核心是隔离内部缓存和外部使用状态，避免外部直接污染内部共享数据。
- 面试里你不用把这个细节背得太死，知道“有缓存、有查找、有复制返回”就够了。

#### 这题面试到底该怎么答
- 反射机制的原理，本质上是基于 JVM 类加载后的运行时元数据。
- 类被加载后，JVM 会为它创建对应的 `Class` 对象；`Class` 对象再维护该类的方法、字段、构造器等反射信息。
- 当我们调用 `getMethod()`、`getField()`、`getConstructor()` 时，本质上是在 `Class` 对象及其缓存的元数据里查找成员；找到后，再通过 `invoke()` 或 `newInstance()` 做动态调用。
- 所以反射的底层逻辑可以概括为：类加载 + 元数据封装 + 成员查找 + 动态执行。

#### 面试易错点 / 补充点
- 反射不是直接读取 `.java` 源码，而是操作 JVM 已加载类的运行时信息。
- `Class.forName()` 的重点是先得到 `Class` 对象，不是直接“new 对象”。
- `getMethod()` / `getDeclaredMethod()` 只是查找，不是执行；执行要靠 `invoke()`。
- 老代码里的 `Class.newInstance()` 现在不推荐，面试最好说现代写法：`getDeclaredConstructor().newInstance()`。
- 文章里的很多源码分析更贴近 JDK 8 实现；面试回答时抓主线即可，不要陷进过细的 JDK 私有实现。

#### 面试回答模板
- 反射机制的原理，是 JVM 在类加载后会为每个类创建一个对应的 `Class` 对象，这个对象里可以拿到类的构造器、方法、字段等运行时元信息。
- 当程序调用 `Class.forName()`、`getMethod()`、`getField()` 这些 API 时，本质上是在基于 `Class` 对象查找类的元数据；查找到后，再通过 `invoke()` 或 `newInstance()` 完成动态调用和实例化。
- 同时，`Class` 内部通常还会对这些反射数据做缓存，所以反射并不是每次都从零开始扫描。
- 一句话总结，反射的底层逻辑就是：类加载后暴露运行时元数据，再基于这些元数据完成动态操作。

#### 面试简答版
- 反射原理就是：类先被 JVM 加载，生成 `Class` 对象；再通过 `Class` 获取方法、字段、构造器等信息；最后通过 `invoke()`、`newInstance()` 动态执行。
- 关键字就 4 个：类加载、`Class` 对象、元数据、动态调用。

## 9. 反射和封装矛盾吗？

#### 先给结论
- 不矛盾，但它们之间确实存在张力。
- 封装的目标是“隐藏内部实现、控制外部访问”。
- 反射的目标是“在运行时获取和操作类的信息”。
- 所以更准确的说法是：反射并不天然否定封装，但它在某些场景下可以绕过封装边界，尤其是配合 `setAccessible(true)` 时。

#### 核心解释
- 封装强调的是：对象应该通过公开的接口暴露能力，内部细节尽量隐藏。
- 正常开发时，我们通常通过 `public` 方法访问对象，而不是直接碰 `private` 字段或 `private` 方法。
- 反射本身先做的是“观察和定位类信息”，这一步不一定破坏封装。
- 真正容易和封装产生冲突的是：通过反射直接访问私有成员，甚至绕过访问控制。

#### 可以怎么理解两者的关系
### 1）反射本身不是错
- 反射是 Java 提供的一种底层能力。
- 框架、容器、注解解析、依赖注入、ORM 映射，这些都大量依赖反射。
- 所以不能简单说“反射就是破坏封装”，这太绝对。

### 2）绕过访问控制时，会削弱封装
- 如果我们通过反射直接拿 `private` 字段或 `private` 方法，再强行访问，这就等于跳过了类本来设计好的访问边界。
- 这时候封装确实被削弱了。
- 所以很多书或面试题会说：反射可以“破坏封装性”。

### 3）工程上要区分“能力存在”和“是否滥用”
- Java 允许你这样做，不代表业务代码里应该到处这样做。
- 在框架层，反射是合理工具。
- 在普通业务代码里，滥用反射访问私有成员，通常会让代码更脆弱、更难维护，也更违背封装思想。

#### 简单代码示例
### 1）正常通过公开方法访问，不破坏封装
```java
class User {
    private String name = "Tom";

    public String getName() {
        return name;
    }
}

public class Demo {
    public static void main(String[] args) {
        User user = new User();
        System.out.println(user.getName());
    }
}
```

输出结果：
```java
Tom
```

输出 / 现象解释：
- 这里是通过 `public getName()` 访问对象状态。
- 这完全符合封装思想。

### 2）通过反射访问私有字段，会绕过封装
```java
import java.lang.reflect.Field;

class User {
    private String name = "Tom";
}

public class Demo {
    public static void main(String[] args) throws Exception {
        User user = new User();

        Field field = User.class.getDeclaredField("name");
        field.setAccessible(true);

        System.out.println(field.get(user));
    }
}
```

输出结果：
```java
Tom
```

输出 / 现象解释：
- `name` 本来是 `private`，按正常设计不该被外部直接访问。
- 但通过反射配合 `setAccessible(true)`，还是拿到了它。
- 这说明反射确实可以绕过封装边界。

#### 这题面试到底怎么答
- 如果问“反射和封装矛盾吗”，最稳的回答是：两者不完全矛盾，但反射在某些场景下可以突破封装限制。
- 封装强调隐藏内部实现、通过公开接口访问对象；反射则提供了在运行时获取并操作类信息的能力。
- 在正常框架场景里，反射是一种必要的底层技术；但如果通过反射强行访问私有成员，就会削弱封装性。
- 所以更准确的表达不是“反射否定封装”，而是“反射提供了绕过封装的能力，是否使用得当取决于场景”。

#### 补充：**反射使用的最小化原则**
- 所谓“最小化原则”，就是能不用反射就不用，必须用时也尽量把范围、次数和影响控制到最小。
- 先走正常调用、接口抽象、公开 API；只有这些方式解决不了时，再考虑反射。
- 如果必须用反射，优先访问 `public` 成员，尽量不要动 `private` 成员，更不要随意 `setAccessible(true)`。
- 反射最好集中在框架层、工具层，不要散落到普通业务代码里到处都是。
- 反射拿到的 `Method`、`Field`、`Constructor` 能缓存就缓存，避免重复查找带来额外开销。
- 一句话记：少用、慎用、集中用、能收口就收口。

#### 面试易错点 / 补充点
- 不要直接回答“矛盾”或者“完全不矛盾”，这两个都太绝对。
- 更稳的说法是：反射本身是能力，中性；但它确实可能绕过封装。
- `getDeclaredField()`、`getDeclaredMethod()` 只是拿到成员信息，真正突破访问限制通常还要配合 `setAccessible(true)`。
- 框架层用反射是合理的，业务层滥用反射访问私有成员通常不推荐。

#### 面试回答模板
- 反射和封装并不是完全矛盾的。封装强调隐藏实现细节、通过公开接口访问对象，而反射提供的是一种运行时获取和操作类信息的能力。
- 在正常情况下，反射可以作为框架底层能力使用，并不一定破坏封装；但如果通过反射配合 `setAccessible(true)` 去访问私有字段或私有方法，就相当于绕过了访问控制，这会削弱封装性。
- 所以更准确的说法是：反射并不否定封装，但它确实提供了突破封装边界的能力，关键看是否在合适场景下使用。

#### 面试简答版
- 不完全矛盾。
- 封装强调隐藏内部细节，反射强调运行时动态操作类信息。
- 反射本身是中性的，但配合 `setAccessible(true)` 可以绕过封装。
- 所以它不是天然反封装，而是“有能力突破封装边界”。

## 10. 反射修改对象，会影响原对象吗？

#### 先给结论
- 会不会影响，关键看反射操作的是不是“同一个对象实例”。
- 如果反射拿到的是原对象本身，并且你通过反射修改了它的字段或调用了会改状态的方法，那么原对象一定会变。
- 反射本身不会自动拷贝对象，它只是换了一种访问对象的方式。
- 只有在你通过反射重新 `newInstance()` 创建了一个新对象时，修改它才不会影响原对象。

#### 核心解释
- 反射不是“复制一份对象再操作”，而是“拿着对象引用去动态访问它”。
- 所以它和普通代码本质上一样：如果你操作的是同一个对象实例，那状态就会被改掉。
- 你可以把反射理解成“绕开编译期静态写法，用运行时方式去调字段、方法、构造器”。
- 访问方式变了，但对象是不是同一个，并没有变。

#### 最关键的一句话
- 同一个对象：改了就会影响原对象。
- 新创建的对象：改了不会影响原对象。

#### 简单代码示例
### 1）通过反射改同一个对象的字段，原对象会变
```java
import java.lang.reflect.Field;

class User {
    private String name = "Tom";

    @Override
    public String toString() {
        return "User{name='" + name + "'}";
    }
}

public class Demo {
    public static void main(String[] args) throws Exception {
        User user = new User();

        Field field = User.class.getDeclaredField("name");
        field.setAccessible(true);
        field.set(user, "Jerry");

        System.out.println(user);
    }
}
```

输出结果：
```java
User{name='Jerry'}
```

输出 / 现象解释：
- `field.set(user, "Jerry")` 操作的就是 `user` 这个对象本身。
- 所以改完之后，原来的 `user` 状态也变了。
- 这和普通代码 `user.name = ...` 在本质上是同一类事情，只是这里是反射方式访问。

### 2）通过反射调用会修改状态的方法，原对象也会变
```java
import java.lang.reflect.Method;

class User {
    private String name = "Tom";

    public void setName(String name) {
        this.name = name;
    }

    @Override
    public String toString() {
        return "User{name='" + name + "'}";
    }
}

public class Demo {
    public static void main(String[] args) throws Exception {
        User user = new User();

        Method method = User.class.getMethod("setName", String.class);
        method.invoke(user, "Jerry");

        System.out.println(user);
    }
}
```

输出结果：
```java
User{name='Jerry'}
```

输出 / 现象解释：
- `invoke(user, "Jerry")` 的目标对象就是 `user`。
- 方法内部改的是这个对象的状态，所以原对象自然也变了。

### 3）如果反射新建的是另一个对象，就不会影响原对象
```java
class User {
    private String name = "Tom";

    public void setName(String name) {
        this.name = name;
    }

    @Override
    public String toString() {
        return "User{name='" + name + "'}";
    }
}

public class Demo {
    public static void main(String[] args) throws Exception {
        User user1 = new User();
        User user2 = User.class.getDeclaredConstructor().newInstance();

        user2.setName("Jerry");

        System.out.println(user1);
        System.out.println(user2);
        System.out.println(user1 == user2);
    }
}
```

输出结果：
```java
User{name='Tom'}
User{name='Jerry'}
false
```

输出 / 现象解释：
- `user2` 是通过反射新创建的对象，不是 `user1`。
- 所以改 `user2` 不会影响 `user1`。
- 这里关键不是“有没有用反射”，而是“是不是同一个对象”。

#### 如果你问的是“方法参数会不会一起变”
- 如果参数本身是对象引用，并且你通过反射修改了这个对象内部状态，那么外面看到的对象状态也会变。
- 但如果只是想通过反射“把方法里的参数变量重新指向另一个对象”，外面的引用不会跟着变，因为 Java 传参本质上还是值传递。

#### 面试易错点 / 补充点
- 不要把“反射”理解成“拷贝对象后再操作”，这是错的。
- 反射操作字段、调用方法时，本质上还是在操作目标对象实例。
- 真正决定原对象会不会变的，不是“有没有反射”，而是“操作的是不是同一个实例”。
- 如果方法内部修改的是对象状态，会生效；如果只是修改参数引用指向，外部不会跟着变。

#### 面试回答模板
- 反射修改对象后，原对象会不会变化，关键看反射操作的是不是同一个对象实例。
- 如果通过反射修改的是原对象本身的字段，或者调用的是原对象上会改变状态的方法，那么原对象一定会变，因为反射不会自动复制对象，它只是换了一种访问方式。
- 只有在通过反射重新创建了一个新对象时，修改这个新对象才不会影响原对象。
- 所以这题的本质不是“反射会不会改原对象”，而是“你操作的是不是同一个实例”。

#### 面试简答版
- 看是不是同一个对象。
- 同一个对象：反射改了，原对象就会变。
- 新对象：改了不会影响原对象。
- 反射本身不会自动拷贝对象。

## 11. Spring 框架底层用反射了吗？

#### 先给结论
- 用了，而且用得很多。
- 但更准确的说法是：Spring 底层不只是反射，还会结合动态代理、字节码增强、类路径扫描、注解解析等机制一起工作。
- 面试里最稳的回答是：Spring 的 IOC、依赖注入、Bean 创建、注解处理等环节大量依赖反射。
- 工厂模式 创建对象

#### 核心解释
- Spring 之所以能做到“你只写类和注解，框架帮你创建对象、注入依赖、调用方法”，核心原因之一就是它能在运行时动态拿到类的信息。
- 而这个“运行时拿类信息、构造对象、设字段、调方法”的能力，正是反射提供的。
- 所以如果问 Spring 底层有没有用反射，答案是肯定的。

#### Spring 里哪些地方会用到反射
### 1）创建 Bean
- Spring 在 IOC 容器启动时，需要根据类信息创建对象。
- 这个过程常常会通过构造器反射实例化对象。

### 2）依赖注入
- 比如 `@Autowired` 注入字段、setter、构造器参数，本质上都要先定位成员，再把值塞进去。
- 这里就会用到反射访问字段、方法、构造器。

### 3）读取注解
- Spring 大量使用注解，比如 `@Component`、`@Autowired`、`@Transactional`、`@Bean`。
- 框架运行时要先拿到类、方法、字段上的注解信息，这个过程也依赖反射。

### 4）调用生命周期方法或业务方法
- 容器有时需要动态调用某些方法，比如初始化方法、销毁方法，或者某些框架扩展点方法。
- 这些运行时调用同样可能通过反射完成。

#### 但 Spring 不只是反射
- 很多同学会把 Spring 底层简单概括成“全靠反射”，这个说法不够完整。
- 更准确地说：
  - IOC / 依赖注入 / 注解解析：大量用反射
  - AOP：除了反射，还会结合 JDK 动态代理、CGLIB 代理
  - 某些场景还会用 ASM、字节码增强等手段
- 所以反射是重要基础能力，但不是 Spring 唯一的底层机制。

#### 一个很短的理解方式
- 没有反射，Spring 很难在运行时：
  - 找到类
  - 创建对象
  - 读取注解
  - 注入依赖
  - 动态调用方法
- 所以可以说：反射是 Spring IOC 和很多基础能力的重要支撑。

#### 面试易错点 / 补充点
- 不要回答成“Spring 只有反射”，这太绝对。
- 更稳的说法是：Spring 大量依赖反射，但同时也结合代理和字节码技术。
- 如果面试官追问“哪块最典型”，优先答：Bean 实例化、依赖注入、注解解析。

#### 面试回答模板
- Spring 底层大量使用了反射，尤其是在 IOC、依赖注入、Bean 创建和注解解析这些环节。
- 比如容器在运行时需要根据类信息创建 Bean、给字段或构造器注入依赖、读取类和方法上的注解，这些能力都离不开反射。
- 不过 Spring 底层不只有反射，还会结合动态代理、CGLIB、字节码增强等机制一起工作。
- 所以更准确的说法是：反射是 Spring 底层的重要基础能力之一，但不是唯一手段。
