# 集合注意事项

> 参考文章：[Java集合使用注意事项总结 - JavaGuide](https://javaguide.cn/java/collection/java-collection-precautions-for-use.html)
>
> 这一篇不要当成“零散补充”去背，它本质上是在讲：集合在真实开发里最容易踩的坑有哪些，为什么会踩，面试时应该怎么解释。

## 1. 集合判空为什么优先用 isEmpty()？

### 先给结论

判断集合是否为空，优先使用 `isEmpty()`，不要习惯性写成 `size() == 0`。

### 核心理解

- `isEmpty()` 语义更直接，一眼就能看出是在判空。
- 很多常见集合里，`isEmpty()` 和 `size()` 的时间复杂度都差不多。
- 但不是所有集合都这样，尤其是某些并发集合，`size()` 可能不是 `O(1)`。
- 也就是说，这题面试不能只答“都能用”，还要补一句“`isEmpty()` 可读性更好，而且有些集合性能更稳妥”。

### 典型例子

像 `ConcurrentLinkedQueue` 这种并发队列：

- `isEmpty()` 的判断方式比较轻量
- `size()` 往往需要遍历统计元素个数

所以：

- 判断空不空，用 `isEmpty()`
- 真要拿元素个数，再用 `size()`

### 简单代码示例

```java
Queue<Integer> queue = new ConcurrentLinkedQueue<>();

System.out.println(queue.isEmpty());
System.out.println(queue.size() == 0);
```

### 输出结果

```java
true
true
```

### 输出 / 现象解释

虽然结果一样，但底层代价可能不一样。面试官想听的重点不是“都能判空”，而是“有些集合 `size()` 可能更重”。

### 面试易错点 / 补充点

- 不要机械地认为所有集合的 `size()` 都是 `O(1)`。
- 并发集合里，这个区别更值得注意。
- 可读性上，`isEmpty()` 也明显更贴近业务语义。

### 面试回答模板

集合判空我一般优先用 `isEmpty()`。一方面可读性更好，另一方面有些集合的 `size()` 不一定是常数时间，比如某些并发集合可能需要遍历统计，所以 `isEmpty()` 更稳妥。

### 面试简答版

集合判空优先 `isEmpty()`，因为语义更清晰，而且部分集合的 `size()` 代价可能更高。

## 2. 使用 Stream 的 toMap() 时要注意什么？

### 先给结论

使用 `Collectors.toMap()` 时有两个高频坑：

- `value` 不能为 `null`，否则可能抛 `NullPointerException`
- `key` 重复时，如果不写合并函数，可能直接报错

### 核心理解

`Collectors.toMap()` 底层会调用 `Map.merge()`。

而 `merge()` 内部会对传入的 `value` 做非空校验，所以如果 `value` 是 `null`，就会直接抛空指针异常。

这题面试要注意一点：

- 不是说 `HashMap` 不能存 `null`
- 而是 `Collectors.toMap()` 这套收集逻辑，在内部就把 `null value` 拦住了

### 简单代码示例

```java
List<Person> list = Arrays.asList(
        new Person("jack", "18163138123"),
        new Person("martin", null)
);

Map<String, String> map = list.stream()
        .collect(Collectors.toMap(Person::getName, Person::getPhone));
```

### 输出结果

```java
Exception in thread "main" java.lang.NullPointerException
```

### 输出 / 现象解释

报错点不在 `HashMap.put()`，而是在 `Collectors.toMap()` 调用 `merge()` 的过程中。

### 更稳妥的写法

```java
Map<String, String> map = list.stream()
        .collect(Collectors.toMap(
                Person::getName,
                p -> p.getPhone() == null ? "" : p.getPhone(),
                (a, b) -> a
        ));
```

### 面试易错点 / 补充点

- `toMap()` 不写合并函数时，遇到重复 key 也会报错。
- 所以实际开发里，通常要同时考虑：
  - `value` 是否可能为 `null`
  - `key` 是否可能重复
- 如果数据来源不稳定，建议显式写 `mergeFunction`。

### 面试回答模板

`Collectors.toMap()` 我会重点注意两个问题：第一，value 不能是 `null`，因为底层走的是 `Map.merge()`；第二，key 冲突时如果不写合并函数，也会抛异常。所以真实开发里我一般会先处理 `null`，并显式写好合并策略。

### 面试简答版

`toMap()` 的两个常见坑：`value` 为 `null` 会 NPE，`key` 重复不写合并函数会报错。

## 3. 为什么不要在 foreach 里直接 remove/add？

### 先给结论

不要在 `foreach` 循环里直接调用集合自己的 `remove()` 或 `add()`，否则很容易触发 `ConcurrentModificationException`。

### 核心理解

- `foreach` 底层本质上还是迭代器 `Iterator`
- 但你在循环体里如果直接调的是集合本身的 `remove/add`
- 这就相当于“迭代器还在遍历，集合结构却被外部偷偷改了”
- 于是触发 fail-fast 机制

这题的关键不是背异常名，而是理解：

`foreach` 不是不能删，而是“不能用错误的方式删”。

### 错误代码示例

```java
List<Integer> list = new ArrayList<>(Arrays.asList(1, 2, 3, 4));

for (Integer num : list) {
    if (num % 2 == 0) {
        list.remove(num);
    }
}
```

### 输出结果

```java
Exception in thread "main" java.util.ConcurrentModificationException
```

### 正确写法 1：使用 Iterator.remove()

```java
List<Integer> list = new ArrayList<>(Arrays.asList(1, 2, 3, 4));
Iterator<Integer> iterator = list.iterator();

while (iterator.hasNext()) {
    Integer num = iterator.next();
    if (num % 2 == 0) {
        iterator.remove();
    }
}

System.out.println(list);
```

### 输出结果

```java
[1, 3]
```

### 正确写法 2：使用 removeIf()

```java
List<Integer> list = new ArrayList<>(Arrays.asList(1, 2, 3, 4));
list.removeIf(num -> num % 2 == 0);
System.out.println(list);
```

### 输出结果

```java
[1, 3]
```

### 输出 / 现象解释

`Iterator.remove()` 是和迭代器配套的删除方式，删除动作是“被迭代器感知到的”，所以不会破坏遍历状态。

### 面试易错点 / 补充点

- 这不只是多线程问题，单线程一样会触发。
- `ConcurrentModificationException` 体现的是 fail-fast 机制。
- `java.util` 下面的大多数常见集合都是 fail-fast 风格。
- 并发场景下，通常要么用并发集合，要么自己做好同步控制。

### 面试回答模板

foreach 底层其实还是迭代器。如果在遍历过程中直接调用集合自己的 `remove/add`，就会导致迭代器感知到结构被外部修改，从而触发 fail-fast，抛出 `ConcurrentModificationException`。正确做法是用 `Iterator.remove()`，或者在合适场景下用 `removeIf()`。

### 面试简答版

foreach 里不能直接删元素，否则容易触发 fail-fast。正确做法是 `Iterator.remove()` 或 `removeIf()`。

## 4. 集合去重为什么优先考虑 Set？

### 先给结论

如果目的是去重，优先考虑 `Set`，不要习惯性用 `List.contains()` 一边遍历一边判重。

### 核心理解

- `HashSet` 底层依赖 `HashMap`
- `HashSet.contains()` 本质上走的是 `HashMap.containsKey()`
- 平均情况下查询效率接近 `O(1)`
- 而 `ArrayList.contains()` 需要线性遍历，时间复杂度是 `O(n)`

所以当数据量大时：

- `HashSet` 去重更自然
- `List.contains()` 去重会明显更慢

### 简单代码示例

```java
List<Integer> list = Arrays.asList(1, 2, 2, 3, 3, 3);
Set<Integer> set = new HashSet<>(list);
System.out.println(set);
```

### 输出结果

```java
[1, 2, 3]
```

### 输出 / 现象解释

`Set` 的语义本来就是“元素唯一”，所以拿它做去重比在 `List` 上反复 `contains()` 更符合数据结构设计。

### 面试易错点 / 补充点

- 如果还想保留插入顺序，用 `LinkedHashSet`。
- 如果还想排序去重，用 `TreeSet`。
- 如果对象去重不生效，通常要检查 `hashCode()` 和 `equals()` 是否正确重写。

### 面试回答模板

集合去重我一般优先用 `Set`。像 `HashSet` 底层基于 `HashMap`，判重效率高；如果用 `List.contains()` 做去重，每次都要遍历查找，数据量一大性能差距会很明显。如果还要保序，我会考虑 `LinkedHashSet`。

### 面试简答版

去重优先用 `Set`，因为 `HashSet` 判重效率更高，`List.contains()` 需要遍历。

## 5. 集合转数组为什么推荐 toArray(new T[0])？

### 先给结论

集合转数组时，优先使用 `toArray(new T[0])`，不要只用无参 `toArray()`。

### 核心理解

- 无参 `toArray()` 返回的是 `Object[]`
- 如果业务上你需要的是 `String[]`、`Integer[]` 这种具体类型数组
- 就必须传入一个类型一致的数组模板

`new T[0]` 的关键点不是“长度是 0”，而是：

- 用它告诉 JVM 返回值应该是什么类型
- 0 长度只是一个模板写法，不是最终结果长度

### 简单代码示例

```java
List<String> list = Arrays.asList("A", "B", "C");
String[] arr = list.toArray(new String[0]);
System.out.println(Arrays.toString(arr));
```

### 输出结果

```java
[A, B, C]
```

### 输出 / 现象解释

`new String[0]` 只是为了告诉方法：“我要的返回类型是 `String[]`。”真正返回的数组长度会根据集合元素个数决定。

### 面试易错点 / 补充点

- `list.toArray()` 返回的是 `Object[]`，不能想当然当成 `String[]` 用。
- 这道题很多人只会背结论，但面试更想听到“类型模板”的理解。

### 面试回答模板

集合转数组我会用 `toArray(new T[0])`。因为无参 `toArray()` 返回的是 `Object[]`，类型不够准确；传一个空的目标类型数组，本质上是给返回值提供类型模板，这样拿到的就是目标类型数组。

### 面试简答版

集合转数组推荐 `toArray(new T[0])`，因为它能保证返回目标类型数组，而不是 `Object[]`。

## 6. Arrays.asList() 有哪些高频坑？

### 先给结论

`Arrays.asList()` 最常见的两个坑：

- 基本类型数组传进去，结果和预期不一样
- 返回的 List 不能随意 `add/remove/clear`

### 核心理解

`Arrays.asList()` 返回的不是 `java.util.ArrayList`，而是 `java.util.Arrays` 里的内部类 `Arrays$ArrayList`。

它更像是：

- 基于原数组包装出来的一个“定长视图”
- 不是一个真正意义上可自由扩容缩容的 `ArrayList`

### 6.1 基本类型数组为什么会踩坑？

### 简单代码示例

```java
int[] arr = {1, 2, 3};
List<int[]> list = Arrays.asList(arr);

System.out.println(list.size());
System.out.println(list.get(0)[0]);
```

### 输出结果

```java
1
1
```

### 输出 / 现象解释

传进去的是一个 `int[]` 对象，不是三个独立的 `Integer` 元素，所以整个数组被当成了 List 里的一个元素。

### 正确写法

```java
Integer[] arr = {1, 2, 3};
List<Integer> list = Arrays.asList(arr);
System.out.println(list);
```

### 输出结果

```java
[1, 2, 3]
```

### 6.2 为什么 add/remove/clear 会报错？

### 简单代码示例

```java
List<String> list = Arrays.asList("A", "B", "C");
list.add("D");
```

### 输出结果

```java
Exception in thread "main" java.lang.UnsupportedOperationException
```

### 输出 / 现象解释

因为这个 List 的长度和底层数组绑定，是固定大小的，所以不支持结构性修改。

### 正确写法

```java
List<String> list = new ArrayList<>(Arrays.asList("A", "B", "C"));
list.add("D");
System.out.println(list);
```

### 输出结果

```java
[A, B, C, D]
```

### 补充：数组转可变集合的常见方式

```java
List<String> list1 = new ArrayList<>(Arrays.asList("a", "b", "c"));

Integer[] arr = {1, 2, 3};
List<Integer> list2 = Arrays.stream(arr).collect(Collectors.toList());

int[] nums = {1, 2, 3};
List<Integer> list3 = Arrays.stream(nums).boxed().collect(Collectors.toList());
```

### 面试易错点 / 补充点

- `Arrays.asList()` 返回的不是我们常说的 `java.util.ArrayList`。
- 它通常支持 `set()`，因为改值不涉及长度变化。
- 真正想得到可变集合，最常见写法是 `new ArrayList<>(Arrays.asList(...))`。
- 基本类型数组如果要转集合，通常要借助包装类型或 `boxed()`。

### 面试回答模板

`Arrays.asList()` 经常有两个坑。第一，基本类型数组传进去时，整个数组会被当成一个元素；第二，它返回的不是 `java.util.ArrayList`，而是一个固定大小的内部类 List，所以不能做 `add/remove/clear`。如果我想要真正可变的集合，一般会再包一层 `new ArrayList<>(Arrays.asList(...))`。

### 面试简答版

`Arrays.asList()` 不是普通 `ArrayList`，基本类型数组会踩坑，而且返回的 List 不能随意增删。

## 7. 这一篇在面试里怎么串起来说？

### 一句话总纲

这篇的核心不是背 API，而是理解集合在真实开发中的几个常见坑：判空、遍历删除、去重、数组转换、`toMap()` 收集，这些题本质上都在考你对集合行为和底层实现的理解。

### 面试简答版

集合使用注意事项里，我会重点记住六类坑：判空优先 `isEmpty()`、`toMap()` 注意 `null` 和重复 key、`foreach` 不能乱删、去重优先 `Set`、集合转数组用 `toArray(new T[0])`、`Arrays.asList()` 不是普通可变集合。
