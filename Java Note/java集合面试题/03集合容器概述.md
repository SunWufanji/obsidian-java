# 集合容器概述

## 1. 什么是集合

### 先给结论
![[Pasted image 20260501175703.png]]
集合是 Java 提供的一种容器，用来存储多个数据。

数组也能存多个数据，但集合比数组更灵活、功能更强。

### 核心解释

- Java 集合主要分两大体系：`Collection` 和 `Map`
- `Collection` 存的是单列数据，下面又分 `List`、`Set`、`Queue`
- `Map` 存的是键值对，也就是 `key-value`

简单来说：

- `Collection`：存一个个单独的元素
- `Map`：存一对对的键值对

### 简单代码示例

```java
// Collection：存单个元素
List<String> list = new ArrayList<>();
list.add("Java");

// Map：存键值对
Map<String, Integer> map = new HashMap<>();
map.put("age", 25);
```

### 面试简答版

- 集合是 Java 提供的容器，用来存储多个数据。
- 主要分 `Collection`（单列）和 `Map`（键值对）两大体系。

---

## 2. 集合的特点

### 先给结论

集合相比数组，主要有以下几个特点：

1. **长度可变**：集合容量不够时可以自动扩容，数组必须提前定好长度
2. **只能存对象**：集合里只能放引用类型，不能放基本类型（但自动装箱后可以放包装类）
3. **功能丰富**：集合提供了排序、查找、去重、队列操作等现成方法
4. **多种类型**：不同场景可以选择不同集合，比如要唯一就用 `Set`，要键值对就用 `Map`

### 核心解释

- 数组长度固定，一旦创建就不能变；集合可以动态扩容。
- 数组可以存基本类型，也可以存对象；集合只能存对象。
- 如果要把基本类型放进集合，Java 会通过自动装箱把它转成对应的包装类。
- 集合提供了很多开箱即用的功能，比如排序、二分查找、线程安全包装等。

### 面试简答版

- 集合长度可变，数组长度固定。
- 集合只能存对象（基本类型需要装箱）。
- 集合功能更丰富，不同场景有不同的实现类可选。

## 3. 集合和数组的区别

| 对比点  | 数组          | 集合               |
| ---- | ----------- | ---------------- |
| 长度   | 固定，创建后不可变   | 可变，可自动扩容         |
| 存储类型 | 基本类型 + 引用类型 | 只能存引用类型          |
| 功能   | 只有基本的存取操作   | 提供排序、查找、去重等丰富API |
| 类型安全 | 编译时就确定类型    | 泛型支持，同样编译期类型安全   |
| 性能   | 更高，直接内存操作   | 有一定封装开销          |
| 适用场景 | 固定长度的简单数据   | 复杂业务、数据量不确定的场景   |

### 一句话总结

> 数组适合固定长度、简单高效的场景；集合适合长度不确定、需要丰富操作的场景。

### 面试简答版

- 数组长度固定，集合长度可变。
- 数组能存基本类型，集合只能存对象。
- 集合功能更丰富，不同场景有不同实现。

## 4. 使用集合框架的好处

### 先给结论

集合框架的核心好处就四个字：**开箱即用**。

你不用自己从零实现数据结构，Java 已经帮你实现好了，而且经过了充分测试和优化。

### 具体的好处

1. **减少开发工作量**：不用自己写链表、树、哈希表，直接用现成的 `ArrayList`、`LinkedList`、`HashMap` 等
2. **提高代码质量**：JDK 集合类经过大量测试和优化，比自己撸的数据结构更稳定
3. **统一操作方式**：所有集合类都实现了统一的接口（`Collection`、`Map` 等），用起来有一致的 API 风格
4. **降低维护成本**：统一的接口意味着换实现类很方便，比如把 `ArrayList` 换成 `LinkedList`，调用代码基本不用改
5. **可扩展性好**：需要特殊功能时，可以基于已有集合类自己扩展

### 面试简答版

- 不用从零实现数据结构，开箱即用。
- 统一的接口，换实现类成本低。
- 经过大量测试，稳定可靠。

## 5. 常用的集合类有哪些？

### 先给结论

按大类来分，用得最多的主要是这几类：

| 大类 | 常用实现类 | 内容可否重复 | 有无索引 | 是否有序 |
|------|-----------|-------------|---------|---------|
| List | `ArrayList`、`LinkedList`、`Vector` | 可重复 | 有索引 | 有序（按插入顺序） |
| Set | `HashSet` | 不可重复 | 无索引 | 无序 |
| Set | `LinkedHashSet` | 不可重复 | 无索引 | 有序（按插入顺序） |
| Set | `TreeSet` | 不可重复 | 无索引 | 排序（按自然顺序/比较器） |
| Map | `HashMap` | key 不可重复 | 无索引 | 无序 |
| Map | `LinkedHashMap` | key 不可重复 | 无索引 | 有序（按插入顺序/访问顺序） |
| Map | `TreeMap` | key 不可重复 | 无索引 | 排序（按自然顺序/比较器） |
| Map | `ConcurrentHashMap` | key 不可重复 | 无索引 | 无序 |
| Queue | `LinkedList`、`ArrayDeque` | 可重复 | 无索引 | 有序（FIFO，按插入顺序） |
| Queue | `PriorityQueue` | 可重复 | 无索引 | 排序（按优先级） |

### 简单记忆

- **要按索引快速访问** → `ArrayList`
- **要经常头尾增删** → `LinkedList`
- **要去重** → `HashSet`
- **要去重还要排序** → `TreeSet`
- **要键值对** → `HashMap`
- **要键值对还要有序** → `LinkedHashMap`
- **要用栈/队列** → `ArrayDeque`
- **要线程安全的 List** → `Vector`
- **要线程安全的 Map** → `ConcurrentHashMap`

### 面试简答版

- `List`：`ArrayList`（数组）、`LinkedList`（链表）、`Vector`（线程安全）
- `Set`：`HashSet`（哈希）、`TreeSet`（红黑树排序）
- `Map`：`HashMap`（哈希）、`TreeMap`（红黑树排序）、`ConcurrentHashMap`（线程安全）
- `Queue`：`ArrayDeque`（数组双端队列）、`PriorityQueue`（优先队列）

## 6. List，Set，Map三者的区别？

### 先给结论

| 对比点       | List                     | Set                        | Map                           |
| --------- | ------------------------ | -------------------------- | ----------------------------- |
| 存储方式      | 按顺序存单个元素                 | 存单个元素，不重复                  | 存键值对（key-value）               |
| 是否有序      | 有序（保持插入顺序）               | 不一定（HashSet 无序，TreeSet 排序） | 不一定（HashMap 无序，TreeMap 排序）    |
| 是否可重复     | 允许重复元素                   | 不允许重复                      | key 不能重复，value 可以重复           |
| 是否可存 null | 可以                       | 可以（HashSet 能存一个 null）      | key 只能一个 null，value 可以多个 null |
| 实现类举例     | `ArrayList`、`LinkedList` | `HashSet`、`TreeSet`        | `HashMap`、`TreeMap`           |
![[Pasted image 20260501181154.png]]
### 核心理解

- **List**：想保留插入顺序、允许重复就用它
- **Set**：想自动去重就用它
- **Map**：想用键快速定位值就用它

这三者不是同一套继承关系：
- `List` 和 `Set` 都继承自 `Collection` 接口
- `Map` 是独立的接口体系，不继承 `Collection`

### 面试简答版

- `List`：有序、可重复，和 `Collection` 同体系。
- `Set`：不可重复，和 `Collection` 同体系。
- `Map`：键值对存储，独立体系，和 `Collection` 无关。

## 7. 集合框架底层数据结构

### 先给结论

| 接口 | 实现类 | 底层数据结构 |
|------|--------|-------------|
| List | `ArrayList` | **动态数组**（Object[]） |
| List | `LinkedList` | **双向链表** |
| List | `Vector` | **动态数组**（Object[]，线程安全） |
| Set | `HashSet` | **基于 HashMap**（底层就是 HashMap） |
| Set | `LinkedHashSet` | **继承 HashSet**，底层是 LinkedHashMap |
| Set | `TreeSet` | **红黑树**（底层是 TreeMap） |
| Map | `HashMap` | **数组 + 链表 + 红黑树**（JDK 8+） |
| Map | `LinkedHashMap` | **HashMap + 双向链表**（保持顺序） |
| Map | `TreeMap` | **红黑树** |
| Map | `Hashtable` | **数组 + 链表**（线程安全，不允许 null） |
| Map | `ConcurrentHashMap` | **数组 + 链表 + 红黑树**（分段锁 → CAS + synchronized） |
| Queue | `ArrayDeque` | **循环数组** |
| Queue | `PriorityQueue` | **二叉堆**（Object[] 数组实现） |

### 核心解释

#### Collection 体系

**1. List**

- **ArrayList**：底层是 `Object[]` 数组（动态数组），初始容量 10，满了扩容 1.5 倍。查询快 O(1)，中间插入/删除慢（需要移位）。
- **Vector**：底层也是 `Object[]` 数组，和 ArrayList 一样，但所有方法加了 `synchronized`，线程安全但性能差。扩容翻倍（ArrayList 1.5 倍）。
- **LinkedList**：底层是**双向链表**（JDK 6 及之前是双向循环链表，JDK 7+ 改为双向链表）。每个节点存前后指针。头尾操作 O(1)，中间操作 O(n)，占内存比 ArrayList 大。

**2. Set**

Set 的底层都是"借" Map 实现的：

- **HashSet（无序，唯一）**：底层基于 `HashMap` 实现，采用 HashMap 来保存元素。把存进来的元素当 Map 的 key，value 全指向一个固定的 `PRESENT` 对象（`private static final Object PRESENT = new Object()`）。利用 HashMap key 不可重复的特性实现去重。
- **LinkedHashSet（有序，唯一）**：继承自 HashSet，内部通过 `LinkedHashMap` 来实现。类似于 LinkedHashMap 在 HashMap 基础上加双向链表一样，LinkedHashSet 也保留了插入顺序。
- **TreeSet（有序，唯一）**：底层是 `TreeMap`（**红黑树**，自平衡的排序二叉树），利用 TreeMap 的 key 排序特性来实现元素的排序和去重。

#### Map 体系

- **HashMap**：
  - **JDK 1.8 之前**：数组 + 链表。数组是 HashMap 的主体，链表是为了解决哈希冲突（**拉链法**）。新节点采用头插法，并发扩容时可能死循环。
  - **JDK 1.8+**：数组 + 链表 + **红黑树**。当链表长度 > 阈值（默认 8）且数组长度 ≥ 64 时，链表转化为红黑树，减少搜索时间；长度 ≤ 6 时红黑树退化为链表。改为尾插法，解决死循环问题。默认负载因子 0.75。

- **LinkedHashMap**：继承自 `HashMap`，底层仍然是**数组 + 链表或红黑树**（拉链式散列结构）。在此基础上，**增加了一条双向链表**，使得结构可以保持键值对的插入顺序。同时通过对链表进行相应操作，也实现了访问顺序相关逻辑（`accessOrder = true` 可用于 LRU 缓存）。

- **Hashtable**：数组 + 链表组成（拉链法），所有方法用 `synchronized` 加锁，线程安全但性能差。key 和 value 都不允许为 null。

- **TreeMap**：**红黑树**（自平衡的排序二叉树），保证 key 按自然顺序或比较器排序。增删查都是 O(log n)。

- **ConcurrentHashMap**：
  - **JDK 7**：**Segment 分段锁**，将数据分为多个 Segment，每个 Segment 独立加锁，并发度等于 Segment 数量。
  - **JDK 8+**：改用 **CAS + synchronized** 锁链表/红黑树头节点，并发性能大幅提升，底层结构和 HashMap 一致（数组 + 链表 + 红黑树）。

### 简单记忆口诀

> ArrayList 是数组，LinkedList 是链表
> HashSet 借 HashMap，TreeSet 靠 TreeMap
> HashMap 数组链表红黑树
> 红黑树是 TreeMap
> ConcurrentHashMap CAS + synchronized

### 面试回答模板

> **Set 体系** — 底层全部是借 Map 实现的。HashSet 底层是 HashMap（元素当 key），TreeSet 底层是 TreeMap（红黑树排序）。

## 8. 哪些集合类是线程安全的？

### 先给结论

按来源分三类，一张表说清：

| 分类          | 类名                               | 线程安全机制                        | 特点                  |
| ----------- | -------------------------------- | ----------------------------- | ------------------- |
| **遗留类**     | `Vector`                         | 方法加 `synchronized`            | JDK 1.0 就有，性能差，基本不用 |
| **遗留类**     | `Hashtable`                      | 方法加 `synchronized`            | JDK 1.0 就有，性能差，基本不用 |
| **包装类**     | `Collections.synchronizedList()` | 包装所有方法加 `synchronized` 块      | 包装任意非线程安全集合         |
| **包装类**     | `Collections.synchronizedSet()`  | 同上                            | 同上                  |
| **包装类**     | `Collections.synchronizedMap()`  | 同上                            | 同上                  |
| **JUC 并发类** | **`ConcurrentHashMap`**          | CAS + synchronized（JDK 8+）    | 高并发首选，读操作无锁         |
| **JUC 并发类** | `CopyOnWriteArrayList`           | 写时复制（ReentrantLock）           | 读多写少场景              |
| **JUC 并发类** | `CopyOnWriteArraySet`            | 写时复制（底层 CopyOnWriteArrayList） | 读多写少的 Set           |
| **JUC 并发类** | `ConcurrentLinkedQueue`          | CAS 无锁队列                      | 高性能并发队列             |
| **JUC 并发类** | `ConcurrentLinkedDeque`          | CAS 无锁双端队列                    | 同上，支持双端操作           |
| **JUC 并发类** | `BlockingQueue` 系                | ReentrantLock + Condition     | 阻塞队列，常用于生产者-消费者     |
| **JUC 并发类** | `ConcurrentSkipListMap`          | CAS + 跳表数据结构                  | 有序并发 Map            |
| **JUC 并发类** | `ConcurrentSkipListSet`          | 底层 ConcurrentSkipListMap      | 有序并发 Set            |

### 核心解释

线程安全的集合类一共有三个来源：

**1. 遗留类（JDK 1.0 时代）**

- **Vector**：和 ArrayList 一样底层是动态数组，但所有方法都加了 `synchronized`。性能差，扩容翻倍。现在基本被 `ArrayList` + `Collections.synchronizedList()` 或 `CopyOnWriteArrayList` 取代。
- **Hashtable**：和 HashMap 一样底层是数组 + 链表，但所有方法加了 `synchronized`。性能差，key 和 value 都不允许 null。现在基本被 `ConcurrentHashMap` 取代。

**2. 同步包装类（JDK 1.2）**

`Collections` 工具类提供了三个静态方法，可以把任意非线程安全的集合包装成线程安全的：

```java
List<String> safeList = Collections.synchronizedList(new ArrayList<>());
Set<String> safeSet = Collections.synchronizedSet(new HashSet<>());
Map<String, Integer> safeMap = Collections.synchronizedMap(new HashMap<>());
```

原理很简单：包装类内部持有一个 `mutex` 对象，所有方法都在这个 mutex 上同步。
注意：**迭代遍历时仍需手动加锁**，包装类不会在迭代器上加锁。

**3. JUC 并发集合（JDK 1.5+，重点）**

### 🔥 ConcurrentHashMap 详解（面试重点）

#### 为什么需要 ConcurrentHashMap？

| 方案 | 问题 |
|------|------|
| `Hashtable` | 所有方法加 `synchronized`，并发时相当于串行，性能极差 |
| `Collections.synchronizedMap()` | 同上，整个 Map 一把锁 |
| `ConcurrentHashMap` | **锁粒度细化**，只锁部分区域，读操作几乎无锁 |

**核心思想：降低锁粒度，尽量让读写互不干扰。**

---

#### JDK 7 实现：Segment 分段锁

```
            ┌──────────────────────────────┐
            │      ConcurrentHashMap        │
            ├──────────┬─────────┬──────────┤
            │ Segment 0 │Segment 1│Segment 2 │  ← 默认 16 个 Segment
            ├─────┬─────┼────┬────┼────┬─────┤
            │Entry[]   │Entry[]   │Entry[]   │  ← 每个 Segment 相当于一个 HashMap
            │ K→V→null │ K→V→K→V │ K→null   │
            └──────────┴──────────┴──────────┘
```

- **结构**：`Segment[]` + `HashEntry[]` + 链表
- **锁粒度**：每个 `Segment` 一把锁，默认 16 个 Segment，理论并发度 16
- **put 流程**：先定位到 Segment，再对 Segment 加锁，然后在 HashEntry 数组里操作
- **get 流程**：不加锁，用 `volatile` 保证可见性
- **size 流程**：先无锁统计 2 次，若结果一致则返回，不一致则锁所有 Segment 再统计
- **问题**：Segment 数量固定，扩容麻烦；头插法仍有死循环风险

```java
// JDK 7 ConcurrentHashMap 核心结构（简化理解）
static final class Segment<K,V> extends ReentrantLock {
    transient volatile HashEntry<K,V>[] table;
}

static final class HashEntry<K,V> {
    final int hash;
    final K key;
    volatile V value;
    volatile HashEntry<K,V> next;  // 链表
}
```

#### JDK 8+ 实现：CAS + synchronized + 数组 + 链表 + 红黑树

```
            ┌──────────────────────────────────────────┐
            │          ConcurrentHashMap                │
            ├────┬────┬────┬────┬────┬────┬────┬───────┤
            │null│Node│null│null│Node│null│null│ TreeNode│  ← Node[] 数组
            ├────┼─┬──┴────┴────┼─┬──┴────┴────┴───┬───┤
            │    │ Node → Node  │ │  TreeNode ←→ TreeNode  │
            │    └──────────────┘ └──────────────────┘
            │      链表（长度≤7）          红黑树（长度≥8）
            └──────────────────────────────────────────┘
```

**核心变化（JDK 7 → JDK 8）：**

| 对比项 | JDK 7 | JDK 8+ |
|--------|-------|--------|
| 数据结构 | Segment + HashEntry + 链表 | Node + 链表 + 红黑树 |
| 锁机制 | 继承 ReentrantLock（分段锁） | **CAS + synchronized**（锁头节点） |
| 锁粒度 | 一个 Segment 一把锁（锁一段区域） | 一个桶一把锁（锁单个链表/红黑树头） |
| 并发度 | 固定 16 | 数组长度，更大 |
| 扩容 | 每个 Segment 独立扩容 | 整个数组一起扩容，多线程协助 |
| 链表插入 | 头插法 | **尾插法**（避免死循环） |
| 红黑树 | 无 | 链表 ≥ 8 且数组 ≥ 64 转红黑树 |

---

#### put() 方法执行流程（JDK 8）

```
1. 计算 key 的 hash，定位桶下标
       ↓
2. 桶为空？ → CAS 无锁插入新 Node（乐观锁）
       ↓ 桶不为空
3. 桶头节点 hash = MOVED（-1）？ → 协助扩容（helpTransfer）
       ↓ 不是扩容
4. synchronized(桶头节点) {     ← 只锁这一个桶
       遍历链表/红黑树：
       - key 已存在 → 覆盖 value
       - key 不存在 → 尾插法插入新 Node
   }
       ↓
5. 链表长度 ≥ 8 且数组长度 ≥ 64？ → 链表转红黑树（treeifyBin）
       ↓
6. 数组元素总数 > 负载因子 × 容量？ → 扩容（transfer）
```

```java
// JDK 8 put 核心逻辑（简化示意）
final V putVal(K key, V value, boolean onlyIfAbsent) {
    // 1. 计算 hash
    int hash = spread(key.hashCode());
    int binCount = 0;
    
    for (Node<K,V>[] tab = table;;) {
        Node<K,V> f; int n, i, fh;
        
        // 2. 数组为空 → 初始化
        if (tab == null || (n = tab.length) == 0)
            tab = initTable();
        
        // 3. 桶为空 → CAS 无锁插入
        else if ((f = tabAt(tab, i = (n - 1) & hash)) == null) {
            if (casTabAt(tab, i, null, new Node<K,V>(hash, key, value, null)))
                break;
        }
        
        // 4. 正在扩容 → 协助迁移
        else if ((fh = f.hash) == MOVED)
            tab = helpTransfer(tab, f);
        
        // 5. 桶不为空 → synchronized 锁头节点
        else {
            synchronized (f) {
                // 遍历链表或红黑树，插入或覆盖
                // ...
            }
        }
    }
    // 6. 判断是否需要扩容
    addCount(1L, binCount);
    return null;
}
```

---

#### get() 方法执行流程（JDK 8）

```
1. 计算 key 的 hash，定位桶下标
       ↓
2. 桶头节点 hash 匹配？ → 直接返回 value
       ↓ 不匹配
3. 头节点 hash < 0？ → 红黑树查找（TreeBin）
       ↓ 不是红黑树
4. 遍历链表查找
       ↓
5. 没找到 → 返回 null
```

**关键**：`get()` **全程无锁**。Node 的 value 和 next 都是 `volatile` 修饰，保证可见性。

```java
// get 方法核心（简化）
public V get(Object key) {
    Node<K,V>[] tab; Node<K,V> e, p; int n, eh; K ek;
    int h = spread(key.hashCode());
    
    if ((tab = table) != null && (n = tab.length) > 0 &&
        (e = tabAt(tab, (n - 1) & h)) != null) {
        
        // 头节点刚好是要找的
        if ((eh = e.hash) == h) {
            if ((ek = e.key) == key || (ek != null && key.equals(ek)))
                return e.val;
        }
        // 红黑树查找
        else if (eh < 0)
            return (p = e.find(h, key)) != null ? p.val : null;
        // 链表遍历查找
        while ((e = e.next) != null) {
            if (e.hash == h &&
                ((ek = e.key) == key || (ek != null && key.equals(ek))))
                return e.val;
        }
    }
    return null;
}
```

---

#### size() / mappingCount() （JDK 8）

JDK 7 的 size() 需要无锁统计 2 次，不一致就锁全表。JDK 8 改进了：

- 内部维护一个 `CounterCell[]` 数组，每个线程 put/remove 时 CAS 更新自己的 CounterCell
- `size()` 时**无锁累加**所有 CounterCell 的值，不需要锁全表
- 并发极高时多个线程各自更新自己的 CounterCell，减少 CAS 竞争

```java
// JDK 8 size() 本质是累加 counterCells
final long sumCount() {
    CounterCell[] cs = counterCells;
    long sum = baseCount;
    if (cs != null) {
        for (CounterCell c : cs)
            if (c != null)
                sum += c.value;
    }
    return sum;
}
```

---

#### 扩容（transfer）：多线程协助扩容

JDK 8 最巧妙的设计之一：

- 扩容时把数组分成多个段（stride），每个线程认领一段去迁移
- 其他线程 put 时如果发现正在扩容（hash = MOVED），**会主动帮忙迁移**（helpTransfer）
- 迁移完一个桶就把它标记为 ForwardingNode，后续线程就知道这个桶已迁完
- **多线程并行迁移**，扩容速度大幅提升

---

#### ConcurrentHashMap 关键面试问题

**Q1：ConcurrentHashMap 怎么保证线程安全？**
> JDK 8 通过 **CAS + synchronized** 保证。put 时桶为空用 CAS 无锁插入，桶不为空用 synchronized 锁链表/红黑树头节点。get 全程无锁，靠 volatile 保证可见性。

**Q2：ConcurrentHashMap 和 Hashtable 的区别？**
> Hashtable 锁整个 Map，所有方法串行；ConcurrentHashMap JDK 8 只锁单个桶头节点，读无锁，并发度是数组长度级别。

**Q3：ConcurrentHashMap 的 size() 怎么实现的？**
> 内部维护 `baseCount` + `CounterCell[]`，多线程各自 CAS 更新自己的计数单元，size() 时直接累加，不需要锁。

**Q4：ConcurrentHashMap 在 JDK 7 和 JDK 8 有什么区别？**
> 数据结构从 Segment + 链表 → Node + 链表 + 红黑树；锁从 ReentrantLock 分段锁 → CAS + synchronized 锁头节点；并发度 16 → 数组长度级别；头插法 → 尾插法；增加了多线程协助扩容。

**Q5：ConcurrentHashMap 的 key 和 value 可以为 null 吗？**
> 都不可以为 null。HashMap 允许是因为它是单线程的，能通过 containsKey 区分。ConcurrentHashMap 中 get 返回 null 无法区分是"key 不存在"还是"value 就是 null"，在多线程环境下做二次检查也不安全。

---

- **CopyOnWriteArrayList**：读操作完全无锁，写操作时复制一份新数组（ReentrantLock 保护），写完替换原数组引用。适合**读多写极少**的场景（如白名单、配置），不适合写频繁的场景（每次写都复制数组，开销大）。

- **CopyOnWriteArraySet**：底层就是 CopyOnWriteArrayList，和 HashSet 借 HashMap 类似。

- **ConcurrentLinkedQueue**：基于 CAS 的无锁队列，性能高，适合高并发下的生产者-消费者。

- **BlockingQueue 系**（`ArrayBlockingQueue`、`LinkedBlockingQueue` 等）：支持阻塞的线程安全队列。`put()` 队列满时阻塞等待，`take()` 队列空时阻塞等待。常用在线程池的任务队列。

- **ConcurrentSkipListMap / ConcurrentSkipListSet**：基于跳表（SkipList）数据结构的有序并发 Map/Set，替代 `TreeMap`/`TreeSet` 的并发版本。

### 哪几个最常用？

```
日常开发中最常用的线程安全集合（按频率排序）：

1. ConcurrentHashMap   — 取代 Hashtable / synchronizedMap
2. CopyOnWriteArrayList — 读多写少场景的 List
3. BlockingQueue 系     — 线程池、生产者-消费者
4. ConcurrentLinkedQueue — 高并发非阻塞队列
5. ConcurrentSkipListMap — 需要有序的并发 Map
```

### 一句话总结

> `Vector` 和 `Hashtable` 是遗留方案，性能差；`synchronizedXxx` 是简单包装，性能一般；**JUC 包下的并发集合**才是真正的现代方案，优先选它们。

### 面试回答模板

> **面试官：哪些集合类是线程安全的？**
>
> 可以分为三类：
>
> 第一类是**遗留类**，Vector 和 Hashtable，JDK 1.0 就有了，所有方法都加 synchronized，性能差，现在基本不用了。
>
> 第二类是 **Collections.synchronizedXxx 包装类**，JDK 1.2 提供，优点是能把任意非线程安全的集合包成线程安全的，但性能依然一般，而且迭代时要自己手动加锁。
>
> 第三类是 **JUC 包下的并发集合**，这才是重点。ConcurrentHashMap 是高频考点，JDK 8 以后用 CAS + synchronized 锁头节点，读操作无锁，性能最好。CopyOnWriteArrayList 适合读多写少场景，写时复制。还有 BlockingQueue、ConcurrentLinkedQueue 等具体场景的并发集合。
>
> 实际开发中，ConcurrentHashMap 用得最多，基本全面替代了 Hashtable。

### 面试简答版

- 遗留类：`Vector`、`Hashtable`（方法级 synchronized，性能差）
- 包装类：`Collections.synchronizedList/set/map`（简单包装，迭代需手动加锁）
- JUC 并发类：`ConcurrentHashMap`、`CopyOnWriteArrayList`、`ConcurrentLinkedQueue`、`BlockingQueue` 系、`ConcurrentSkipListMap` 等（真正的现代方案）

## 9. Java 集合的快速失败机制 "fail-fast"

### 先讲场景

你写了一段代码，遍历一个集合，遍历到一半时顺手删了一个元素：

```java
List<String> list = new ArrayList<>(List.of("A", "B", "C", "D"));

for (String s : list) {       // 增强 for 循环
    if ("B".equals(s)) {
        list.remove(s);        // 遍历时删元素
    }
}
```

跑一下，**报错了**：

```
Exception in thread "main" java.util.ConcurrentModificationException
```

明明代码看起来没什么问题，为什么要报错？

这就是 Java 的 **fail-fast（快速失败）机制**——它发现你在遍历的时候"偷偷"改了集合，直接抛异常拦住你，而不是让程序带着 bug 继续跑下去。

---

### 为什么报错？用个比喻理解

想象你排队做核酸：

> 你是**志愿者**（迭代器），拿着一张名单挨个叫人。
>
> 实际排队的**队伍**就是集合本身。
>
> 志愿者手上有一个**计数器**（`expectedModCount`），记着"队伍人数应该是 4 个"。
>
> 这时有个人从队伍里走了（`list.remove()`），队伍人数变了。但志愿者手上的计数器没更新，还是 4。
>
> 志愿者叫下一个人时，发现**实际人数和计数器对不上**——"不对啊，我记的是 4 个，怎么只有 3 个了？有人偷偷动了队伍！"
>
> 于是志愿者直接罢工：**"ConcurrentModificationException！"**

用代码的角度说就是：

```
集合内部有一个计数器 modCount（记录"结构被改了多少次"）
迭代器创建时，把 modCount 抄一份存下来，叫 expectedModCount

每次 it.next() 时，都会检查：
  modCount == expectedModCount？  → 一致就正常走
  modCount != expectedModCount？  → 说明有人动了集合 → 抛异常
```

所以代码的执行过程是这样的：

```
初始：modCount = 0
创建迭代器：expectedModCount = 0

next() 拿 A → 检查 0 == 0 → ✅ → 打印 A
next() 拿 B → 检查 0 == 0 → ✅ → 打印 B
           → list.remove("B") → modCount 变成 1 ❗
next() 拿 C → 检查 1 == 0 → ❌ → 抛异常！
```

**关键**：`list.remove()` 改了集合的结构（少了一个元素），`modCount` 增加了，但迭代器不知道。下次 `next()` 时一对比发现不一致，直接罢工。

---

### 那怎么正确删元素？

**方式一：用迭代器自己的 remove()**

```java
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    String s = it.next();
    if ("B".equals(s)) {
        it.remove();     // ✅ 迭代器自己的 remove，会同步更新计数器
    }
}
```

迭代器的 `remove()` 会顺手把自己的 `expectedModCount` 也更新一下，这样下次检查时两边就对得上了。

**方式二：用 removeIf()（JDK 8，最推荐）**

```java
list.removeIf("B"::equals);   // 一行搞定
```

---

### 什么时候 modCount 会变？

| 操作 | modCount 变不变 | 举例 |
|------|:---:|------|
| `add()` | ✅ 变 | 加了一个元素 |
| `remove()` | ✅ 变 | 少了一个元素 |
| `clear()` | ✅ 变 | 全清空了 |
| `set()` | ❌ 不变 | 只是替换某个位置的值，结构没变 |

简单记：**结构变了（增/删/清空）就加 1，改值不影响。**

---

### 那多线程遍历呢？

如果你一个线程在遍历，另一个线程同时在改，同样会抛异常：

```java
// 线程 1：遍历
new Thread(() -> {
    for (String s : list) System.out.println(s);
}).start();

// 线程 2：修改
new Thread(() -> {
    list.add("E");  // 并发改 → 大概率抛异常
}).start();
```

**多线程的正确做法**：
- 用 `ConcurrentHashMap`、`CopyOnWriteArrayList` 等线程安全集合
- 或者遍历时手动加锁

---

### fail-fast vs fail-safe

fail-fast 不是唯一的方案。有些集合用的是 **fail-safe**（安全失败）机制：

| | fail-fast | fail-safe |
|---|---|---|
| **什么意思** | 检测到修改就**抛异常**拦住你 | 不抛异常，自己处理 |
| **怎么做到的** | 直接在原集合上遍历 | 在**快照副本**上遍历（拷贝一份出来遍历） |
| **代表** | `ArrayList`、`HashMap`、`HashSet` | `CopyOnWriteArrayList`、`ConcurrentHashMap` |
| **数据一致性** | 强一致（看到的就是最新的） | 弱一致（遍历的是旧快照，可能不是最新数据） |
| **性能** | 一般 | 写的时候要拷贝，开销大 |

```java
// fail-safe 例子：CopyOnWriteArrayList 不会抛异常
List<String> list = new CopyOnWriteArrayList<>(List.of("A", "B", "C"));
for (String s : list) {
    if ("B".equals(s)) {
        list.remove(s);  // ✅ 不报错
        list.add("X");   // ✅ 也不报错
    }
}
System.out.println(list);  // [A, C, X]
// 注意：遍历时看到的仍然是 [A, B, C]，因为遍历的是旧快照
```

### 面试易错点

1. **fail-fast 不是 100% 准确的**：文档说它是 **best-effort**（尽力而为），不保证一定能检测到所有并发修改。不能依赖它来做业务逻辑的正确性判断。

2. **增强 for 循环 = 迭代器**：写成 `for (String s : list)` 底层就是迭代器，同样会触发 fail-fast。它不是"另一种更安全的遍历方式"。

3. **单线程也会触发**：不需要多线程，单线程遍历时直接调 `list.remove()` 一样报错。

4. **Set 和 Map 也有**：不只是 List，`HashMap`、`HashSet`、`TreeMap` 等非线程安全集合都是 fail-fast 的。

---

### 面试回答

> **面试官：说说 fail-fast？**
>
> fail-fast 是 Java 集合的一种保护机制。当你用迭代器遍历集合时，如果集合结构被修改了，迭代器会立即抛 `ConcurrentModificationException`，而不是让程序带着错误继续跑。
>
> 原理是：集合内部有个计数器 `modCount`，迭代器创建时把它记下来作为 `expectedModCount`。每次 `next()` 时对比一下，不一致就说明有人动了集合，直接抛异常。
>
> 遍历时想删元素不能用 `list.remove()`，要用迭代器自己的 `remove()`，或者 JDK 8 的 `removeIf()`。
>
> 还有一个概念叫 fail-safe，像 `CopyOnWriteArrayList` 和 `ConcurrentHashMap`，它们遍历的是快照副本，不会抛异常，但数据可能不是最新的。

### 面试简答版

- fail-fast：遍历时发现集合被改了，立即抛异常，**不是为了修复问题，而是为了早发现问题**
- 原理：`modCount`（集合的计数器）vs `expectedModCount`（迭代器记下来的），不相等就抛
- 正确删元素：用 `it.remove()` 或 `removeIf()`
- 增强 for 循环 = 迭代器，一样会触发
- fail-safe：在快照副本上遍历，不抛异常但数据可能不是最新的


## 10. 怎么确保一个集合不能被修改？

### 先讲场景

假设你写了一个方法，返回一周七天的名字：

```java
public List<String> getWeekDays() {
    return Arrays.asList("周一", "周二", "周三", "周四", "周五", "周六", "周日");
}
```

调用的人拿到的 List 可以随便改——`list.add("周八")`、`list.remove("周一")`，你的数据就被破坏了。

这个问题本质上就是：**你给了别人一个集合，怎么保证他只能看不能改？**

Java 给了几种办法，我挑两个最常用的讲。

---

### 方法一：用 List.of() 直接创建"死数据"（最简单）

JDK 9 开始，Java 提供了 `List.of()`、`Set.of()`、`Map.of()`：

```java
List<String> list = List.of("A", "B", "C");
Set<String> set = Set.of("A", "B", "C");
Map<String, Integer> map = Map.of("A", 1, "B", 2);
```

这种集合就像**水泥封死的箱子**——焊死了，谁也打不开。任何想改的操作都会直接报错：

```java
List<String> list = List.of("A", "B", "C");
list.add("D");    // ❌ 报错：UnsupportedOperationException
list.set(0, "X"); // ❌ 报错
list.remove("A"); // ❌ 报错
```

**适合场景**：你明确知道集合里有哪些元素，而且永远不改。比如一周七天、省份列表、配置项。

---

### 方法二：把已有的集合"锁起来"（Collections.unmodifiableList）

如果你已经从别的地方拿到了一个集合，比如从数据库查出来的 List，不能直接 `List.of()`，那就用这个：

```java
List<String> list = new ArrayList<>();
list.add("A");
list.add("B");

List<String> locked = Collections.unmodifiableList(list);
locked.add("C"); // ❌ 报错：UnsupportedOperationException
```

**⚠️ 但是有个坑，新手容易踩进去：**

```java
List<String> list = new ArrayList<>(List.of("A", "B"));
List<String> locked = Collections.unmodifiableList(list);

list.add("C");  // 原集合还能改！

System.out.println(locked); // [A, B, C] ← 被锁的那个也变了！
```

这怎么理解呢？

> 想象你有个透明玻璃柜子，里面放了一堆东西，柜子门锁上了（这就是 `unmodifiableList`）。别人不能从柜门伸手进去改。
>
> 但是——**如果柜子本身在别人手里**（`list` 这个原集合），他可以打开柜子从后面把东西塞进去！玻璃柜里看到的自然也跟着变了。

**所以正确用法是**：先拷贝一份，再把拷贝锁上：

```java
List<String> list = new ArrayList<>(List.of("A", "B"));
List<String> safe = Collections.unmodifiableList(new ArrayList<>(list));
// 现在把 list 丢掉，只用 safe
```

或者更简单，用 JDK 10+ 的 `List.copyOf()`，一步到位：

```java
List<String> safe = List.copyOf(list);  // 拷贝 + 不可变，一步完成
```

---

### 容易搞混的几个点

**1. `final` 和集合不可变不是一回事**

```java
final List<String> list = new ArrayList<>();
list = new ArrayList<>();  // ❌ 报错：final 不让换对象
list.add("A");             // ✅ 不报错！内容照样改
```

`final` 只是说"这个变量不能指向别的对象"，但对象里面的内容想怎么改都行。相当于贴了个标签说"不许换盒子"，但盒子里装什么你管不着。

**2. `Arrays.asList()` 半锁不锁**

```java
List<String> list = Arrays.asList("A", "B", "C");
list.set(0, "X");  // ✅ 可以改元素
list.add("D");     // ❌ 报错：不能增加/删除
```

它的规矩是：不能改变长度（不能加不能删），但可以替换已有的元素。不是真正的不可变。

**3. 对外暴露集合时要注意**

```java
class Student {
    private List<String> courses;
    
    // ❌ 危险：外部拿着这个 List 可以直接 add/remove，把你内部数据改乱了
    public List<String> getCourses() {
        return courses;
    }
    
    // ✅ 安全：外面只能看，不能改
    public List<String> getCourses() {
        return Collections.unmodifiableList(courses);
        // 或者 return List.copyOf(courses);
    }
}
```

### 面试回答

> **面试官：怎么确保一个集合不能被修改？**
>
> 主要有两种方式：
>
> 一是 `List.of()`、`Set.of()`（JDK 9+），直接创建不可变集合，最简单。
>
> 二是 `Collections.unmodifiableList()`，给已有集合套一层"只读壳子"。但要注意这只是个壳子，原集合改了它也会跟着变，所以最好配合 `List.copyOf()` 先拷贝再锁。
>
> 另外容易搞混的是：`final List` 只是引用不能变，内容照样能改，不等于不可变。

### 一句话记住

> `List.of()` = 水泥封死，谁也动不了
> `Collections.unmodifiableList()` = 玻璃柜子上锁，但柜子本身在别人手里就危险
> `final` = 不让换盒子，但盒子里装什么管不着
