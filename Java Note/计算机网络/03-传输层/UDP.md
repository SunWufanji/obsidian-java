# UDP

## 1. UDP 是什么？

UDP（用户数据报协议）是一种**简单的、无连接的**传输层协议。它只负责把数据"扔"过去，不管对方收没收到、顺序对不对。

打个比方：TCP 像寄挂号信（有回执、保证送达），UDP 像往邻居门口扔报纸——扔了就走了，看没看、丢没丢不关我事。

UDP 的特点：
- **无连接**：不用建立连接，直接发
- **不可靠**：不保证送达，不保证顺序
- **高效**：头部只有 8 字节（TCP 至少 20 字节），开销小
- **面向报文**：一次发一个完整的数据报，不会拆分合并
- **支持一对多**：可以广播、组播

## 2. TCP 和 UDP 的区别？
![[Pasted image 20260522201423.png]]
打个比方：TCP 像打电话——先拨号接通（建立连接），说话有顺序，丢了对方会说"你再说一遍"；UDP 像寄明信片——写完就扔进邮筒，不管对方在不在、收没收到。

| 对比项 | TCP | UDP |
|-------|-----|-----|
| 是否连接 | 面向连接（三次握手） | 无连接 |
| 是否可靠 | 可靠传输，使用流量控制和拥塞控制 | 不可靠传输，不使用流量控制和拥塞控制 |
| 是否有序 | 有序，消息在传输过程中可能会乱序，TCP 会重新排序 | 无序 |
| 传输速度 | 慢 | 快 |
| 连接对象个数 | 只能是一对一通信 | 支持一对一、一对多、多对一和多对多交互通信 |
| 传输方式 | 面向字节流 | 面向报文 |
| 首部开销 | 首部最小 20 字节，最大 60 字节 | 首部开销小，仅 8 字节 |
| 适用场景 | 适用于要求可靠传输的应用，例如文件传输 | 适用于实时应用（IP 电话、视频会议、直播等） |

**什么时候用 TCP？**
- 需要可靠传输：文件下载、网页浏览、邮件发送
- 数据不能出错或丢失：银行转账、订单提交

**什么时候用 UDP？**
- 需要速度：视频通话、在线游戏、直播
- 可以容忍丢包：语音通话偶尔丢一帧无所谓
- 天然支持一对多：DNS 查询、组播

**面试简答版：**
> TCP 用于在传输层有必要实现可靠传输的情况，UDP 用于对高速传输和实时性有较高要求的通信。两者应该根据应用目的按需使用。

## 3. 如何用 UDP 实现可靠传输？

UDP 本身不可靠，但我们可以在应用层自己实现可靠性。常见的方案：

**核心思路**：在 UDP 基础上加上类似 TCP 的机制。

1. **序列号**：给每个数据包编号，解决乱序问题
2. **确认应答（ACK）**：接收方收到后回复 ACK
3. **超时重传**：发送后没收到 ACK 就重传
4. **滑动窗口**：控制发送速率，防止接收方溢出
5. **拥塞控制**：参考 TCP 的慢启动、拥塞避免

**实际案例**：
- **KCP 协议**：基于 UDP 的可靠传输协议，比 TCP 更快（牺牲约 10%-20% 的带宽换取更低的延迟）
- **QUIC 协议**：Google 开发的基于 UDP 的传输协议，HTTP/3 就是基于 QUIC

**为什么不直接用 TCP？** 因为 TCP 的某些机制（如重传等待时间、队头阻塞）在某些场景下太慢了。基于 UDP 自己实现可以更灵活地控制这些行为。

**面试简答版：**
> 在 UDP 上加序列号、ACK、重传、滑动窗口等机制就能实现可靠传输。实际案例有 KCP 协议和 QUIC 协议。核心优势是可以自定义重传策略，比 TCP 更灵活。
