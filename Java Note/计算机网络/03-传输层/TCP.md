# TCP

## 1. TCP 是什么？

TCP（传输控制协议）是一种**可靠的、面向连接的**传输层协议。它保证数据完整、有序地到达对方，不会丢、不会乱。

打个比方：TCP 就像寄挂号信。你寄出去一封信，邮局会给你回执单，告诉你对方收到了。如果信丢了，邮局会重寄一份。信的顺序也是保证好的，先寄先到。

TCP 的核心特点：
- **面向连接**：通信前必须先建立连接（三次握手）
- **可靠传输**：有确认、有重传、有校验
- **面向字节流**：数据像水流一样，没有明确的消息边界
- **全双工**：双方可以同时发送和接收数据
- **流量控制**：发送方不能把接收方"撑死"
- **拥塞控制**：发送方不能把网络"堵死"

---

## 2. TCP 三次握手是什么？为什么是三次？

三次握手就是客户端和服务器互相确认"我们都能收发消息"的过程。

```
客户端                          服务器
  |  ---- SYN (seq=x) ---->   |    第1次：客户端说"我想建立连接"
  |  <--- SYN+ACK (seq=y,ack=x+1) --- |  第2次：服务器说"好的，我也能收到你"
  |  ---- ACK (ack=y+1) ----> |    第3次：客户端说"好的，我也能收到你"
  |        连接建立             |
```

**为什么是三次，不是两次？**

防止**历史重复连接**的初始化。想象一下：
- 客户端先发了一个 SYN，但网络堵了，没到服务器
- 客户端超时重发，这次成功了，通信完毕关闭连接
- 之前那个堵住的 SYN 突然到了服务器
- 服务器以为是新连接，回复 SYN+ACK，建立了连接
- 但客户端根本没想建立连接，这个连接就浪费了

三次握手的第三次 ACK 就是让服务器确认"这个连接是客户端主动想要的"。

**面试简答版：**
> 三次握手：客户端发 SYN，服务器回 SYN+ACK，客户端再发 ACK。三次的目的是防止历史重复连接的初始化，确保双方都能收发。

---

## 3. TCP 四次挥手是什么？为什么是四次？

四次挥手就是双方都要关闭连接。

```
客户端                          服务器
  |  ---- FIN (seq=u) ---->   |    第1次：客户端说"我发完了"
  |  <--- ACK (ack=u+1) ----  |    第2次：服务器说"知道了，但我可能还有数据要发"
  |  ... 服务器继续发数据 ...    |
  |  <--- FIN (seq=w) ----  |    第3次：服务器说"我也发完了"
  |  ---- ACK (ack=w+1) ----> |    第4次：客户端说"好的"
  |     等待2MSL后关闭         |
```

**为什么是四次而不是三次？**

因为 TCP 是全双工的，每个方向需要独立关闭。服务器收到客户端的 FIN 时，可能还有数据没发完，所以不能立即发 FIN，得先发 ACK 确认，等数据发完再发 FIN。

**为什么客户端要等 2MSL？**
- 确保服务器能收到最后的 ACK（如果丢了，服务器会重传 FIN）
- 让本次连接的所有报文都从网络中消失，避免影响新连接

---

## 4. TCP 如何保证可靠传输？

TCP 通过这些机制保证数据不丢、不错、不乱：

1. **序列号和确认应答**：每个字节都有编号，接收方收到后回 ACK 确认
2. **校验和**：检验数据在传输中有没有损坏
3. **超时重传**：发出去的数据如果一段时间没收到 ACK，就重发
4. **连接管理**：三次握手、四次挥手保证连接的可靠性
5. **流量控制**：接收方通过滑动窗口告诉发送方"我还能接收多少"，防止发太快撑爆缓冲区
6. **拥塞控制**：发送方感知网络状况，网络拥塞时降低发送速率

## 5. TCP 重传机制是怎样的？

TCP 发出数据后会启动一个定时器，如果超时没收到 ACK，就重传。

两种触发重传的方式：

**超时重传**：
- 发送数据后启动定时器
- 超时还没收到 ACK → 重传
- 定时器的 timeout 时间怎么算？RTT（往返时间）的动态计算

**快速重传**：
- 接收方收到乱序的报文时，会重复发送之前那个有序报文的 ACK
- 比如收到 1、3、4（2 丢了），接收方会连续发 3 次 ACK=2
- 发送方连续收到 3 个重复 ACK → 立即重传，不用等超时

**面试简答版：**
> TCP 有两种重传方式：超时重传（定时器到期没收到 ACK 就重传）和快速重传（收到 3 个重复 ACK 就立即重传丢失的报文）。

## 6. TCP 滑动窗口是什么？

滑动窗口就是发送方和接收方各自维护一个"窗口"，控制一次能发多少数据。

- **发送窗口**：发送方根据自己和接收方的情况，决定"我现在最多能发多少数据"
- **接收窗口**：接收方告诉发送方"我的缓冲区还能装多少"

发送方发出窗口内的数据后，不需要等 ACK 就可以继续发。收到 ACK 后，窗口向右滑动，可以发新的数据。

如果接收方窗口为 0，发送方就停止发送，定期探测窗口是否恢复。

**面试简答版：**
> 滑动窗口是 TCP 的流量控制机制，接收方通过窗口大小告诉发送方能发多少数据，发送方在窗口范围内可以连续发送不需要等待 ACK，提高效率。

## 7. TCP 拥塞控制是怎样的？

拥塞控制是防止网络被数据"堵死"。和流量控制不同，流量控制是怕接收方撑不住，拥塞控制是怕网络撑不住。

四个核心算法：

**慢启动**：
- 刚建立连接时，拥塞窗口从 1 开始，每收到一个 ACK 翻倍（指数增长）
- 增长到慢启动阈值（ssthresh）后进入拥塞避免

**拥塞避免**：
- 窗口增长变慢，每个 RTT 只增加 1（线性增长）
- 发现丢包（网络拥塞）→ 阈值减半 → 窗口回到 1（重新慢启动）

**快速恢复**：
- 收到 3 个重复 ACK（快速重传）→ 阈值减半，窗口减半（不是回到 1）
- 然后进入拥塞避免阶段

**快速重传**：
- 收到 3 个重复 ACK → 立即重传丢失的报文（不等超时）

整体策略：**慢启动、快增长、拥塞时快速降**。

**面试简答版：**
> TCP 拥塞控制包括慢启动（指数增长）、拥塞避免（线性增长）、快速重传（3 个重复 ACK 立即重传）和快速恢复（丢包后窗口减半而非回到 1）。目标是平衡发送速度和网络承载能力。

## 8. 如何优化 TCP 性能？

| 优化方向 | 手段 |
|---------|------|
| 减少握手开销 | TCP Fast Open（TFO）、连接复用 |
| 拥塞控制优化 | BBR 算法替代 CUBIC |
| 减少延迟 | Nagle 算法关闭、Delayed ACK 调整 |
| 带宽利用 | 调整窗口大小、启用窗口缩放 |
| 连接管理 | keep-alive、连接池 |

**BBR vs CUBIC**：
- CUBIC：基于丢包的拥塞控制，丢包才降速，延迟较高
- BBR：基于带宽和延迟的拥塞控制，主动探测最优速率，延迟低、吞吐高

Google 全面使用 BBR 后，全球平均延迟降低了 4%，YouTube 在发展中国家的吞吐量提升了 4%。

**面试简答版：**
> 优化 TCP 性能可以使用 BBR 拥塞控制算法、TCP Fast Open 减少握手次数、调整窗口大小提升吞吐量、使用连接池复用连接等。
