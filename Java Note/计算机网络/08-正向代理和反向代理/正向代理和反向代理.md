# 正向代理和反向代理

## 1. 正向代理是什么？

正向代理就是**客户端知道要访问谁，但通过代理服务器去访问**。代理代替客户端去请求目标服务器。

打个比方：你（客户端）想买国外的商品，但自己没法直接买。你找了一个代购（正向代理），代购帮你去买，商家只知道代购来买了，不知道你的真实身份。

**正向代理的用途**：
- **翻墙**：访问被封锁的网站
- **隐藏客户端身份**：服务器不知道真实请求者是谁
- **缓存加速**：缓存常用资源，加速访问
- **访问控制**：公司内网通过正向代理限制员工访问某些网站

---

## 2. 反向代理是什么？

反向代理就是**客户端不知道真正的服务器是谁，代理代替服务器接收请求**。客户端以为在和代理通信，实际代理把请求转发给了后端服务器。

打个比方：你打客服电话（反向代理），客服帮你转接相关部门。你不需要知道具体是哪个部门处理你的问题，客服帮你搞定。

**反向代理的用途**：
- **负载均衡**：把请求分发到多个后端服务器
- **安全防护**：隐藏后端服务器的真实 IP，防止直接攻击
- **SSL 卸载**：HTTPS 的加密解密由代理处理，后端服务器不需要管
- **缓存加速**：缓存静态资源，减轻后端压力
- **压缩**：响应数据压缩后再返回给客户端

---

## 3. 正向代理 vs 反向代理

| 对比项 | 正向代理 | 反向代理 |
|-------|---------|---------|
| 谁发起 | 客户端知道，主动使用 | 服务器配置，客户端不知道 |
| 代理对象 | 代理客户端 | 代理服务器 |
| 客户端知不知道 | 知道目标是谁，但通过代理访问 | 不知道真正服务器是谁 |
| 用途 | 翻墙、匿名访问、缓存 | 负载均衡、安全防护、SSL 卸载 |
| 常见实现 | Squid、Nginx（proxy_pass） | Nginx、HAProxy、F5 |

**一句话总结**：正向代理帮客户端"找服务器"，反向代理帮服务器"接客户端"。

---

## 4. 什么是负载均衡？

负载均衡就是把用户的请求**均匀分配到多台服务器**上，避免单台服务器压力过大。

没有负载均衡：所有请求都打到一台服务器 → 服务器扛不住 → 宕机
有了负载均衡：请求分散到 10 台服务器 → 每台只处理 1/10 的请求 → 稳定运行

---

## 5. 常见的负载均衡算法

多台服务器以对称的方式组成一个服务器集合，每台服务器都具有等价的地位，能互相分担负载。

**轮询法**：将请求按照顺序轮流的分配到服务器上。大锅饭，不能发挥某些高性能服务器的优势。

**随机法**：随机获取一台，和轮询类似。

**哈希法**：通过 IP 地址哈希化来确定要选择的服务器编号。好处是每次客户端访问的服务器都是同一台，能很好地利用 session 或者 cookie。

**加权轮询**：根据服务器性能不同加权。性能好的服务器权重高，分到更多请求。

| 算法 | 原理 | 适用场景 |
|------|------|---------|
| 轮询 | 按顺序一个一个分配 | 服务器性能差不多 |
| 加权轮询 | 性能强的服务器分到更多请求 | 服务器性能有差异 |
| 随机 | 随机选择一台 | 简单场景 |
| 哈希 | 同一 IP 固定访问同一台服务器 | 需要会话保持 |
| 最小连接数 | 分配给当前连接数最少的服务器 | 长连接场景 |

**Nginx 负载均衡配置示例**：
```nginx
upstream backend {
    server 192.168.1.1 weight=3;  # 权重3
    server 192.168.1.2 weight=1;  # 权重1
    server 192.168.1.3 backup;    # 备用服务器
}
```

**面试简答版：**
> 正向代理帮客户端访问服务器（如翻墙），客户端知道目标；反向代理帮服务器接收请求（如负载均衡），客户端不知道后端是谁。负载均衡常见算法有轮询、加权轮询、最小连接数、一致性哈希。
