# RPC

## 1. RPC 是什么？

RPC（远程过程调用）就是让你调用远程服务器上的函数，就像调用本地函数一样简单。

想象你在公司，想让另一个部门帮你查个数据。正常做法是发邮件、等回复、拿到数据。RPC 就像装了一部直通电话，你喊一声"帮我查一下 xxx"，对方查完直接告诉你结果，你感觉就像在跟隔壁工位的人说话。

**为什么要用 RPC？** 单体应用拆成微服务后，服务之间需要互相调用。HTTP 接口虽然也能做，但 RPC 在性能、序列化、服务治理上更专业。

---

## 2. 一次完整的 RPC 调用流程是怎样的？

```
客户端 → 客户端Stub → 网络传输 → 服务端Stub → 服务端
```

详细步骤：

1. **客户端调用**：像调本地函数一样调远程函数
2. **客户端 Stub（存根）**：把函数名、参数等信息打包（序列化）
3. **网络传输**：通过 TCP/UDP 把数据发给服务端
4. **服务端 Stub**：接收数据，解包（反序列化），得到函数名和参数
5. **服务端执行**：调用对应的本地函数，执行业务逻辑
6. **返回结果**：把结果按原路返回给客户端

关键角色：
- **Stub（存根）**：帮客户端和服务端屏蔽网络细节，让远程调用看起来像本地调用
- **注册中心**：服务端启动时注册自己的地址，客户端通过注册中心找到服务端地址
- **序列化/反序列化**：把数据转换成可传输的格式（如 Protobuf、JSON）

## 3. 有了 HTTP，为什么还需要 RPC？

| 对比项 | HTTP | RPC |
|-------|------|-----|
| 协议 | 公开标准，通用 | 可自定义，更灵活 |
| 性能 | 较低（Header 冗余大） | 较高（可定制紧凑协议） |
| 序列化 | JSON 为主（可读但慢） | Protobuf/Thrift（二进制，快） |
| 服务治理 | 需要额外组件 | 内置负载均衡、熔断等 |
| 适用场景 | 对外开放的 API | 内部服务间调用 |

**核心区别**：
- HTTP 适合对外提供的 API（前后端交互），通用性强
- RPC 适合内部微服务之间调用，追求性能和治理能力

**面试简答版：**
> HTTP 是通用协议，适合对外 API；RPC 是为内部服务间调用设计的，性能更高、序列化更紧凑、内置服务治理。两者不是替代关系，而是适用场景不同。

## 4. 设计一个 RPC 框架需要考虑什么？

如果让你从零设计一个 RPC 框架，需要考虑这些方面：

1. **协议设计**：定义请求和响应的数据格式（Header + Body）
2. **序列化**：选择高效的序列化方案（Protobuf、Hessian 等）
3. **网络通信**：基于 TCP/Netty，处理粘包拆包、连接管理
4. **服务注册与发现**：服务端注册地址，客户端发现服务端（ZooKeeper、Nacos）
5. **负载均衡**：客户端有多个服务端实例时，选择哪个？（轮询、随机、一致性哈希）
6. **容错机制**：服务挂了怎么办？（重试、熔断、降级）
7. **超时控制**：防止调用一直等待
8. **序列化协议**：高效、跨语言

## 5. gRPC 和 RPC 有什么区别？

gRPC 是 Google 开源的一个高性能 RPC 框架，基于 HTTP/2 协议，使用 Protobuf 作为序列化协议。

| 对比项 | 普通 RPC | gRPC |
|-------|---------|------|
| 传输协议 | 可自定义 | HTTP/2 |
| 序列化 | 可选 | Protobuf（强制） |
| 跨语言 | 需要自己适配 | 原生支持多语言 |
| 流式传输 | 不支持 | 支持（双向流） |
| 性能 | 取决于实现 | 很高 |

gRPC 的四种调用模式：
- **一元 RPC**：普通请求-响应（最常用）
- **服务端流 RPC**：客户端发一个请求，服务端返回一个流
- **客户端流 RPC**：客户端发送一个流，服务端返回一个响应
- **双向流 RPC**：双方都用流通信

**面试简答版：**
> gRPC 是 Google 开源的 RPC 框架，基于 HTTP/2 + Protobuf，性能高、跨语言支持好、支持流式传输。和普通 RPC 的区别是传输协议和序列化方案更标准化。

## 6. 常见的序列化协议有哪些？

序列化就是把对象变成字节流以便网络传输。反序列化就是反过来。

| 协议 | 格式 | 优点 | 缺点 |
|------|------|------|------|
| JSON | 文本 | 可读性强，通用 | 体积大，解析慢 |
| Protobuf | 二进制 | 体积小，速度快 | 不可读，需要 .proto 文件 |
| Hessian | 二进制 | 跨语言，性能好 | 不太流行了 |
| Thrift | 二进制 | 跨语言，性能好 | 生态不如 Protobuf |

**Protobuf 为什么快？**
- 二进制格式，体积比 JSON 小 3-10 倍
- 编解码速度快
- 字段用编号而非名称，解析高效

**面试简答版：**
> 常见的序列化协议有 JSON（通用但慢）、Protobuf（快但不可读）、Hessian 和 Thrift。RPC 框架一般用 Protobuf，Web API 用 JSON。
