# DNS

## 1. DNS 是什么？

DNS（域名系统）就是互联网的"电话簿"。你在浏览器输入 `www.baidu.com`，DNS 帮你查到它对应的 IP 地址 `110.242.68.3`，然后浏览器才能跟这个 IP 通信。

没有 DNS，你就得记住每个网站的 IP 地址，就像没有通讯录你得记住所有人的手机号一样。

---

## 2. DNS 解析过程是怎样的？

当你在浏览器输入 `www.example.com` 时，DNS 解析分这几步：

```
浏览器缓存 → 操作系统缓存 → hosts文件 → 本地DNS服务器 → 根DNS → 顶级DNS → 权威DNS
```

详细过程：

1. **浏览器缓存**：先查浏览器有没有缓存这个域名的 IP，有就直接用
2. **操作系统缓存**：浏览器没有，查操作系统的 DNS 缓存
3. **hosts 文件**：操作系统也没有，看 hosts 文件里有没有手动配置
4. **本地 DNS 服务器**：都没有，向本地 DNS 服务器（一般是运营商提供的）发起查询
5. **递归查询开始**：
   - 本地 DNS 先查自己的缓存
   - 没有就问**根 DNS 服务器**："你知道 example.com 的 IP 吗？" → 根 DNS 说："我不知道，但 .com 域名归那个服务器管，你去问它"
   - 本地 DNS 问 **.com 顶级 DNS**："example.com 的 IP 是多少？" → 顶级 DNS 说："example.com 归 example.com 的权威 DNS 管"
   - 本地 DNS 问 **example.com 的权威 DNS**："www.example.com 的 IP 是？" → 权威 DNS 回答："是 93.184.216.34"
6. **返回结果**：本地 DNS 把 IP 返回给操作系统，操作系统返回给浏览器，同时本地 DNS 会缓存这个结果

**面试简答版：**
> 先查浏览器缓存，再查操作系统缓存，然后查 hosts 文件，都没有就向本地 DNS 服务器查询。本地 DNS 会依次问根 DNS、顶级 DNS、权威 DNS，最终拿到 IP 地址并缓存。

---

## 3. 递归查询和迭代查询的区别？

- **递归查询**：你问本地 DNS，本地 DNS 负责帮你一路查到底，最后给你答案。你只问了一次。
- **迭代查询**：本地 DNS 去问根 DNS，根 DNS 说"你去问 xxx"，本地 DNS 再去问 xxx，xxx 又说"你去问 yyy"。本地 DNS 跑了好几趟。

实际中：
- 浏览器 → 本地 DNS：**递归查询**（浏览器只问一次，等结果）
- 本地 DNS → 根/顶级/权威 DNS：**迭代查询**（本地 DNS 自己跑腿）

---

## 4. DNS 用了什么协议？

DNS 主要使用 **UDP** 协议，端口 53。因为查询数据量小，UDP 快速简单。

但以下情况会用 **TCP**：
- 响应数据超过 512 字节（UDP 限制）
- 区域传送（主从 DNS 同步数据时）

---

## 5. DNS 缓存的作用？TTL 是什么？

DNS 缓存就是为了避免每次都从根 DNS 开始查，加快解析速度。

**TTL（Time To Live）**：缓存的生存时间，单位是秒。比如 TTL=300，意思是这个缓存结果 300 秒后就过期，过期后要重新查询。

各级缓存都会遵守 TTL：
- 浏览器缓存一般有独立的过期时间
- 操作系统缓存遵循 TTL
- 本地 DNS 服务器缓存遵循 TTL

**面试简答版：**
> DNS 缓存是为了减少重复查询提高速度。TTL 是缓存过期时间，到期后缓存失效需要重新解析。

---

## 6. 域名解析的记录类型有哪些？

| 记录类型 | 含义 | 示例 |
|---------|------|------|
| A | 域名 → IPv4 地址 | example.com → 93.184.216.34 |
| AAAA | 域名 → IPv6 地址 | example.com → 2606:2800:... |
| CNAME | 域名 → 另一个域名（别名） | www.example.com → example.com |
| MX | 邮件服务器地址 | example.com → mail.example.com |
| NS | 域名服务器记录 | example.com → ns1.example.com |
| TXT | 文本记录（常用于验证） | SPF 记录、域名验证 |
| PTR | IP → 域名（反向解析） | 93.184.216.34 → example.com |

**CNAME 的应用场景**：很多 CDN 服务会让域名配一个 CNAME 指向 CDN 提供的域名，再通过 CDN 的 DNS 解析到最近的节点 IP。

---

## 7. 什么是 DNS 泛解析，它有哪些用途？

DNS 泛解析（通配符 DNS）就是用一个 `*` 匹配域名的所有子域名，统一解析到同一个 IP 地址。

### 怎么配的

在 DNS 服务器添加一条 A 记录：

```
*.example.com  →  A  →  93.184.216.34
```

这样：
- `www.example.com` → 93.184.216.34
- `blog.example.com` → 93.184.216.34
- `anything.example.com` → 93.184.216.34

所有不存在的子域名都会解析到这个 IP，不需要逐个添加。

### 用途

**1. 简化域名管理**

公司有几十上百个子域名，不用每个都单独配 A 记录，一条泛解析全搞定。

**2. 快速部署服务**

新上线一个子域名（如 `api.example.com`），不需要等 DNS 生效，泛解析已经兜底了。

**3. 多租户系统**

SaaS 平台给每个租户分配子域名（`tenant1.app.com`、`tenant2.app.com`），泛解析让所有租户域名都指向同一个应用，应用根据域名区分不同租户。

**4. 防止子域名被劫持**

配了泛解析后，攻击者没法通过注册不存在的子域名来做 DNS 劫持（因为通配符已经兜底了）。

### 注意事项

- 泛解析只能配在**最后一级**，比如 `*.example.com` 可以，但 `*.*.example.com` 不行
- 泛解析对**已经存在的子域名不生效**，单独配置的 A 记录优先级更高
- 泛解析可能被用于钓鱼攻击（攻击者泛解析一个域名指向恶意 IP）

**面试简答版：**
> DNS 泛解析用 `*` 匹配所有子域名，统一解析到同一个 IP。适合多租户系统、快速部署、简化管理。注意泛解析优先级低于已存在的单独 A 记录，也可能被利用做钓鱼攻击。
