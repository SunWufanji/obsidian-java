# HTTP

## 1. HTTP 是什么？

HTTP（超文本传输协议）是一种无状态的应用层协议，用于从服务器传输超文本到本地浏览器。你在浏览器输入网址按回车，浏览器就用 HTTP 协议跟服务器说"我要看这个网页"，服务器把网页内容通过 HTTP 回传给你。

**HTTP 协议的主要特点：**

1. **简单快速**：客户端向服务器请求时，只需传送请求方法和路径。服务器响应快速简单
2. **灵活**：HTTP 允许传输任意类型的数据对象。通过 Content-Type 标头，客户端和服务器可以进行内容协商
3. **无状态**：HTTP 协议自身不对请求和响应之间的通信状态进行保存。但可以通过在请求和响应消息中加入 cookie 来实现状态管理
4. **支持 B/S 及 C/S 模式**：最常用于浏览器/服务器（B/S）模式的通信，也支持客户端/服务器（C/S）模式
5. **基于 TCP**：HTTP/1.1 和 HTTP/2 都基于 TCP 传输，HTTP/3 改用 UDP（QUIC 协议）

## 2. HTTP 报文格式长什么样？

HTTP 报文分为**请求报文**和**响应报文**，结构都差不多，就像写信有信头和信体。

### 请求报文格式

1. **请求行**（请求方法 + URI + 协议版本）
2. **请求头部**
3. **空行**
4. **请求主体**

```
GET /sample.jsp HTTP/1.1                    ← 请求行
Accept: image/gif, image/jpeg               ← 请求头部
Accept-Language: zh-cn
Connection: Keep-Alive
Host: localhost
User-Agent: Mozilla/4.0 (compatible; MSIE 5.01; Window NT 5.0)
Accept-Encoding: gzip, deflate

username=jinqiao&password=1234              ← 请求主体
```

### 响应报文格式

1. **状态行**（协议版本 + 状态码 + 原因短语）
2. **响应首部**
3. **空行**
4. **响应主体**

```
HTTP/1.1 200 OK                             ← 状态行
Server: Apache Tomcat/5.0.12               ← 响应首部
Date: Mon, 6 Oct 2003 13:23:42 GMT
Content-Length: 112

                                            ← 空行
<html><head><title>HTTP响应示例</title></head><body>
    Hello HTTP!
</body></html>                              ← 响应主体
```

**三个核心部分**：起始行（请求行/状态行）、首部字段（Header）、主体（Body）。

## 3. GET 和 POST 的区别是什么？

| 对比项 | GET | POST |
|-------|-----|------|
| 参数位置 | URL 的 query string 里 | 请求体里 |
| 参数长度 | 有限制（浏览器/服务器限制） | 理论上无限制 |
| 幂等性 | 幂等（多次请求结果一样） | 非幂等 |
| 安全性 | 参数暴露在 URL 里 | 参数在 body 里，相对安全 |
| 用途 | 获取数据 | 提交数据 |
| 缓存 | 可以被缓存 | 默认不缓存 |

**注意**：GET 也可以有 body（虽然不常见），POST 的 URL 也可以带参数。它们最核心的区别是**语义不同**——GET 用来获取资源，POST 用来提交数据。

### 使用上的区别

- GET 使用 URL 或 Cookie 传参，而 POST 将数据放在 BODY 中，这个是因为 HTTP 协议用法的约定
- GET 方式提交的数据有长度限制，则 POST 的数据则可以非常大，这个是因为它们使用的操作系统和浏览器设置的不同引起的区别
- POST 比 GET 安全，因为数据在地址栏上不可见，这个说法没毛病，但依然不是 GET 和 POST 本身的区别

### 本质区别

GET 和 POST 最大的区别主要是 **GET 请求是幂等性的，POST 请求不是**。这个是它们本质区别。

幂等性是指一次和多次请求某一个资源应该具有同样的副作用。简单来说意味着对同一 URL 的多个请求应该返回同样的结果。

**面试简答版：**
> GET 参数在 URL 里，用于获取数据，幂等可缓存；POST 参数在请求体里，用于提交数据，非幂等。但本质上都是 TCP 连接，区别主要在语义规范上。

## 4. URL 和 URI 有什么区别？

**URI（统一资源标识符）**：标识一个资源的"身份证号"，范围更广。
**URL（统一资源定位符）**：通过"地址"来定位资源，是 URI 的一种。

打个比方：URI 是"张三"（标识这个人），URL 是"北京市海淀区xx路xx号的张三"（通过地址找到这个人）。

```
https://www.example.com:80/path/page.html?q=search#section1
|____|  |______________| |_| |_____________| |________| |______|
协议       域名         端口     路径           查询参数    锚点
```

- `https://www.example.com` 是 URL，也是 URI
- `urn:isbn:0451450523` 是 URI，但不是 URL（URN 格式，通过名字而非地址标识）

**面试简答版：**
> URI 是更广的概念，标识资源的身份；URL 是 URI 的子集，通过地址定位资源。所有 URL 都是 URI，但不是所有 URI 都是 URL。

## 5. 常见的 HTTP 状态码有哪些？

状态码就是服务器告诉客户端"我处理得怎么样了"。

**2xx 成功**：
- `200 OK`：服务器已成功处理了请求。通常，这表示服务器提供了请求的网页
- `204 No Content`：成功但没有返回内容（比如删除操作）
- `206 Partial Content`：断点续传时用，只返回部分内容

**3xx 重定向**：
- `301 Moved Permanently`（永久移动）：请求的网页已永久移动到新位置。服务器返回此响应（对 GET 或 HEAD 请求的响应）时，会自动将请求者转到新位置。搜索引擎会更新链接
- `302 Found`（临时移动）：服务器目前从不同位置的网页响应请求，但请求者应继续使用原有位置来进行以后的请求
- `304 Not Modified`：资源没变，用缓存就行（配合 If-Modified-Since 使用）

**4xx 客户端错误**：
- `400 Bad Request`：客户端请求有语法错误，不能被服务器所理解
- `401 Unauthorized`：未认证（需要登录）
- `403 Forbidden`：服务器收到请求，但是拒绝提供服务
- `404 Not Found`（未找到）：服务器找不到请求的网页，最经典的
- `405 Method Not Allowed`：请求方法不被允许

**5xx 服务器错误**：
- `500 Internal Server Error`（服务器内部错误）：服务器遇到错误，无法完成请求
- `502 Bad Gateway`：网关/代理收到的响应无效
- `503 Service Unavailable`：服务器过载或维护中
- `504 Gateway Timeout`：网关超时

**状态码开头代表类型**：

| 开头 | 类型 | 含义 |
|------|------|------|
| 1xx | 信息性 | 收到请求，继续处理 |
| 2xx | 成功 | 请求被成功接收并处理 |
| 3xx | 重定向 | 需要进一步操作才能完成请求 |
| 4xx | 客户端错误 | 请求有语法错误或无法被执行 |
| 5xx | 服务器错误 | 服务器在处理请求时出错 |

**面试简答版：**
> 2xx 表示成功，3xx 表示重定向，4xx 表示客户端错误，5xx 表示服务器错误。常见的是 200 成功、301/302 重定向、404 不存在、500 服务器内部错误。

## 6. HTTP 内容协商是什么？

内容协商就是客户端和服务器"商量"返回什么格式的内容。比如浏览器支持中文和英文，它会告诉服务器"我两个都能看，你随便给一个"。

协商方式有三种：
- **服务端驱动**（最常用）：通过请求头里的 `Accept`、`Accept-Language`、`Accept-Encoding` 等字段告诉服务器偏好
- **客户端驱动**：服务器返回多种格式，客户端自己选（不常用）
- **透明协商**：中间代理帮着协商（也不常用）

示例请求头：
```
Accept: text/html, application/json   ← 我能看 HTML 和 JSON
Accept-Language: zh-CN, en;q=0.9      ← 优先中文，其次英文
Accept-Encoding: gzip, br             ← 支持 gzip 和 br 压缩
```

服务器根据这些字段返回最匹配的资源，响应头里会用 `Content-Type`、`Content-Language` 等告诉客户端最终选择了什么格式。

## 7. HTTP 分块传输编码是什么？

分块传输（Chunked Transfer Encoding）就是服务器不知道内容总长度时，把响应数据切成一块一块地发。

正常情况：服务器先算好"这个网页一共 5000 字节"，然后一次性发给你。但有时内容是动态生成的（比如边查数据库边输出），算不出总长度，这时候就用分块传输：

```
HTTP/1.1 200 OK
Transfer-Encoding: chunked

7\r\n
Hello, \r\n
6\r\n
World!\r\n
0\r\n
\r\n
```

每一块前面有个十六进制的数字表示这一块的长度，最后用 `0` 表示结束。

**面试简答版：**
> 分块传输是当服务器不知道响应总长度时，把数据切成小块逐个发送，每块标注长度，最后用 0 表示结束。HTTP/1.1 引入，配合流水线使用。

## 8. HTTP 重定向是什么？

重定向就是服务器告诉浏览器："你要找的页面搬新家了，去这个新地址吧"。

**共同点**：301 和 302 状态码都表示重定向，浏览器在拿到服务器返回的这个状态码后会自动跳转到一个新的 URL 地址，这个地址可以从响应的 Location 首部中获取（用户看到的效果就是他输入的地址 A 瞬间变成了另一个地址 B）。

**不同点**：
- **301 永久重定向**：旧地址 A 的资源已经被永久地移除了（这个资源不可访问了），搜索引擎在抓取新内容的同时也将旧的网址交换为重定向之后的网址。比如网站从 HTTP 升级到 HTTPS
- **302 临时重定向**：旧地址 A 的资源还在（仍然可以访问），这个重定向只是临时地从旧地址 A 跳转到地址 B，搜索引擎会抓取新的内容而保存旧的网址。比如用户没登录时跳转到登录页

SEO 中 302 好于 301（因为旧网址还保留着）。

**常见的重定向原因**：
1. 网站调整（如改变网页目录结构）
2. 网页被移到一个新地址
3. 网页扩展名改变（如应用需要把 .php 改成 .html 或 .shtml）

**重定向 vs 转发**：
- **重定向**：浏览器地址栏会变，是两次请求（浏览器自动去请求新地址）
- **转发**：服务器内部跳转，地址栏不变，只有一次请求

**面试简答版：**
> 301 和 302 都是重定向，浏览器自动跳到新地址。301 是永久的，搜索引擎更新索引；302 是临时的，旧地址还保留。SEO 中 302 好于 301。

## 9. HTTP 长连接和短连接？

在 HTTP/1.0 中，默认使用的是**短连接**。也就是说，浏览器和服务器每进行一次 HTTP 操作，就建立一次连接，但任务结束就中断连接。如果客户端浏览器访问的某个 HTML 或其他类型的 Web 页中包含有其他的 Web 资源，如 JavaScript 文件、图像文件、CSS 文件等；当浏览器每遇到这样一个 Web 资源，就会建立一个 HTTP 会话。

但从 HTTP/1.1 起，默认使用**长连接**，用以保持连接特性。使用长连接的 HTTP 协议，会在响应头有加入这行代码：
```
Connection:keep-alive
```

在使用长连接的情况下，当一个网页打开完成后，客户端和服务器之间用于传输 HTTP 数据的 TCP 连接不会关闭，如果客户端再次访问这个服务器上的网页，会继续使用这一条已经建立的连接。Keep-Alive 不会永久保持连接，它有一个保持时间，可以在不同的服务器软件（如 Apache）中设定这个时间。实现长连接要客户端和服务端都支持长连接。

**HTTP 协议的长连接和短连接，实质上是 TCP 协议的长连接和短连接。**

**但有个队头阻塞问题**：虽然连接复用了，但请求必须一个一个排队。浏览器只能同时开 6-8 个 TCP 连接来解决。

**HTTP/2**：在同一个 TCP 连接里引入了**多路复用**，多个请求可以并行传输，彻底解决了应用层的队头阻塞。就像开了多个窗口同时处理不同人的业务。

**面试简答版：**
> HTTP/1.0 每次请求新建连接，HTTP/1.1 默认 keep-alive 复用连接但有队头阻塞，HTTP/2 实现了多路复用，多个请求可在同一连接并行传输。

## 10. Cookie、Session 和 Token 的区别？

因为 HTTP 是无状态的，服务器不记得你是谁。所以需要一些机制来"认人"。

**Cookie**：浏览器本地存储的小数据。登录成功后服务器发一个 Cookie 给浏览器，之后每次请求浏览器自动带上，服务器就认出你了。
- 优点：简单，浏览器自动管理
- 缺点：大小限制（4KB），CSRF 攻击风险

**Session**：服务器端存储的会话信息。服务器给每个用户创建一个 Session，用 Session ID 标识，这个 ID 通过 Cookie 传给浏览器。
- 优点：数据存在服务器端，安全
- 缺点：服务器压力大，分布式环境下要共享 Session

**Token**：服务器生成的一串加密字符串，包含用户信息，不需要服务器存储。最常用的是 JWT（JSON Web Token）。
- 优点：无状态，天然支持分布式，跨域友好
- 缺点：无法主动失效（需要配合黑名单机制）

| 对比项 | Cookie | Session | Token |
|-------|--------|---------|-------|
| 存储位置 | 浏览器 | 服务器 | 客户端 |
| 安全性 | 较低 | 较高 | 较高 |
| 服务器压力 | 低 | 高（要存储） | 低（无状态） |
| 分布式支持 | 不友好 | 需要共享 | 天然支持 |

**面试简答版：**
> Cookie 存在浏览器，Session 存在服务器通过 Cookie 传 ID，Token 是无状态的加密字符串存在客户端。分布式系统推荐用 Token（JWT）。

## 11. HTTP 常用的请求方式有哪些？

| 方法 | 作用 |
|------|------|
| GET | 获取资源 |
| POST | 传输实体主体 |
| PUT | 上传文件 |
| DELETE | 删除文件 |
| HEAD | 和 GET 方法类似，但只返回报文首部，不返回报文实体主体部分 |
| PATCH | 对资源进行部分修改 |
| OPTIONS | 查询指定的 URL 支持的方法 |
| CONNECT | 要求用隧道协议连接代理 |
| TRACE | 服务器会将通信路径返回给客户端 |

为了方便记忆，可以将 PUT、DELETE、POST、GET 理解为客户端对服务端的**增删改查**：
- **PUT**：上传文件，向服务器添加数据，可以看作**增**
- **DELETE**：删除文件，可以看作**删**
- **POST**：传输数据，向服务器提交数据，对服务器数据进行更新，可以看作**改**
- **GET**：获取资源，查询服务器资源，可以看作**查**

**面试简答版：**
> 常用方法有 GET（查）、POST（改）、PUT（增）、DELETE（删）、HEAD（只拿头部）、PATCH（部分修改）。GET/POST 最常用，RESTful 风格会用到全部。

## 12. HTTP 1.0 和 HTTP 1.1 的区别？

| 对比项 | HTTP 1.0 | HTTP 1.1 |
|-------|----------|----------|
| 长连接 | 默认短连接，每次请求新建 TCP 连接 | 默认长连接（`Connection: keep-alive`），一个 TCP 连接传多个请求 |
| 缓存处理 | 使用 `If-Modified-Since`、`Expires` 做缓存判断 | 引入更多缓存控制策略，可供选择的缓存头来控制缓存策略 |
| 带宽优化 | 不支持断点续传，客户端需要整个对象 | 引入 `range` 头域，支持断点续传，返回 206（Partial Content） |
| 错误通知 | 错误码较少 | 新增 24 个错误状态码，如 409（Conflict）、410（Gone） |
| Host 头处理 | 每台服务器绑定唯一 IP，不需要 Host 头 | 支持虚拟主机，请求头必须带 Host 字段，没有会报 400 Bad Request |

**面试简答版：**
> HTTP 1.1 相比 1.0：默认开启长连接、支持断点续传（range 头）、新增错误码、必须带 Host 头支持虚拟主机。

## 13. HTTP 1.1 和 HTTP 2.0 的区别？

HTTP 2.0 相比 HTTP 1.1 支持的特性：

**新的二进制格式**：HTTP 1.1 的解析是基于文本。基于文本协议的格式解析存在天然缺陷，文本的表现形式有多样性，要做到健壮性考虑的场景必然很多。二进制则不同，只认 0 和 1 的组合。基于这种考虑 HTTP 2.0 的协议解析决定采用二进制格式，实现方便且健壮。

**多路复用**：即连接共享，每一个 request 都是用作连接共享机制的。一个 request 对应一个 id，这样一个连接上可以有多个 request，每个连接的 request 可以随机的混杂在一起，接收方可以根据 request 的 id 将 request 再归属到各自不同的服务端请求里面。

**头部压缩**：HTTP 1.1 的头部（header）带有大量信息，而且每次都要重复发送；HTTP 2.0 使用 encoder 来减少需要传输的 header 大小，通讯双方各自 cache 一份 header fields 表，既避免了重复 header 的传输，又减小了需要传输的大小。

**服务端推送**：服务器除了对最初请求的响应外，服务器还可以额外的向客户端推送资源，而无需客户端明确的请求。

| 对比项 | HTTP 1.1 | HTTP 2.0 |
|-------|----------|----------|
| 协议格式 | 文本协议 | 二进制格式 |
| 连接方式 | 一个连接传多个请求（但排队） | 多路复用（多个请求并行） |
| 头部 | 每次重复发送，浪费带宽 | 头部压缩，双方缓存 header 表 |
| 推送 | 客户端请求才响应 | 服务端可以主动推送资源 |

**面试简答版：**
> HTTP 2.0 四大改进：二进制格式更健壮、多路复用解决队头阻塞、头部压缩减少传输量、服务端推送主动发资源。

## 14. HTTP 与 HTTPS 的区别？

| 对比项 | HTTP | HTTPS |
|-------|------|-------|
| 端口 | 80 | 443 |
| 安全性 | 无加密，安全性较差 | 有加密机制，安全性较高 |
| 资源消耗 | 较少 | 由于加密处理，资源消耗更多 |
| 是否需要证书 | 不需要 | 需要 |
| 协议 | 运行在 TCP 协议之上 | 运行在 SSL 协议之上，SSL 运行在 TCP 协议之上 |

**面试简答版：**
> HTTP 明文传输端口 80，HTTPS 加密传输端口 443。HTTPS = HTTP + SSL/TLS，需要证书，安全性高但资源消耗更大。

## 15. HTTPS 的优缺点？

**优点：**

**安全性**：
- 使用 HTTPS 协议可认证用户和服务器，确保数据发送到正确的客户机和服务器
- HTTPS 协议是由 SSL+HTTP 协议构建的可进行加密传输、身份认证的网络协议，要比 HTTP 协议安全，可防止数据在传输过程中不被窃取、改变，确保数据的完整性
- HTTPS 是现行架构下最安全的解决方案，虽然不是绝对安全，但它大幅增加了中间人攻击的成本

**SEO 方面**：谷歌曾在 2014 年 8 月份调整搜索引擎算法，并称"比起同等 HTTP 网站，采用 HTTPS 加密的网站在搜索结果中的排名将会更高"。

**缺点：**
- 在相同网络环境中，HTTPS 相比 HTTP 无论是响应时间还是耗电量都有大幅度上升
- HTTPS 的安全是有范围的，在黑客攻击、服务器劫持等情况下几乎起不到作用
- 在现有的证书机制下，中间人攻击依然有可能发生
- HTTPS 需要更多的服务器资源，也会导致成本的升高

**面试简答版：**
> HTTPS 优点：加密传输防窃取、身份认证防伪造、SEO 排名更高。缺点：响应慢耗电多、服务器成本高、不是绝对安全。

## 16. 讲一讲 HTTPS 的原理？

加密流程按图中的序号分为：

1. 客户端请求 HTTPS 网址，然后连接到 server 的 443 端口（HTTPS 默认端口，类似于 HTTP 的 80 端口）

2. 采用 HTTPS 协议的服务器必须要有一套数字 **CA（Certification Authority）证书**。颁发证书的同时会产生一个私钥和公钥。私钥由服务端自己保存，不可泄漏。公钥则是附带在证书的信息中，可以公开的。证书本身也附带一个证书电子签名，这个签名用来验证证书的完整性和真实性，可以防止证书被篡改

3. 服务器响应客户端请求，将证书传递给客户端，证书包含公钥和大量其他信息，比如证书颁发机构信息、公司信息和证书有效期等

4. 客户端解析证书并对其进行验证。如果证书不是可信机构颁布，或者证书中的域名与实际域名不一致，或者证书已经过期，就会向访问者显示一个警告，由其选择是否还要继续通信

5. 如果证书没有问题，客户端就会从服务器证书中取出服务器的公钥 A。然后客户端还会生成一个**随机码 KEY**，并使用公钥 A 将其加密

6. 客户端把加密后的随机码 KEY 发送给服务器，作为后面**对称加密的密钥**

7. 服务器在收到随机码 KEY 之后会使用私钥 B 将其解密。经过以上这些步骤，客户端和服务器终于建立了安全连接，完美解决了对称加密的密钥泄漏问题，接下来就可以用对称加密愉快地进行通信了

8. 服务器使用密钥（随机码 KEY）对数据进行对称加密并发送给客户端，客户端使用相同的密钥（随机码 KEY）解密数据

9. 双方使用对称加密愉快地传输所有数据

### 为什么用两种加密？

- **非对称加密**（公钥加密、私钥解密）：解决密钥传递问题，但速度慢
- **对称加密**（同一个密钥加密解密）：速度快，但密钥怎么安全地传给对方是个问题
- **HTTPS 的做法**：用非对称加密安全地把对称密钥传过去，之后通信用对称加密，兼顾安全和速度

**面试简答版：**
> HTTPS 原理：客户端连接服务器 443 端口 → 服务器返回 CA 证书（含公钥） → 客户端验证证书 → 客户端生成随机密钥用公钥加密传给服务器 → 服务器私钥解密得到密钥 → 双方用该密钥做对称加密通信。核心是用非对称加密安全交换对称密钥，再用对称加密通信。

## 17. 什么是 Cookie 和 Session？

**什么是 Cookie**

HTTP Cookie（也叫 Web Cookie 或浏览器 Cookie）是服务器发送到用户浏览器并保存在本地的一小块数据，它会在浏览器下次向同一服务器再发起请求时被携带并发送到服务器上。通常，它用于告知服务端两个请求是否来自同一浏览器，如保持用户的登录状态。Cookie 使基于无状态的 HTTP 协议记录稳定的状态信息成为了可能。

Cookie 主要用于以下三个方面：
- **会话状态管理**（如用户登录状态、购物车、游戏分数或其它需要记录的信息）
- **个性化设置**（如用户自定义设置、主题等）
- **浏览器行为跟踪**（如跟踪分析用户行为等）

**什么是 Session**

Session 代表着服务器和客户端一次会话的过程。Session 对象存储特定用户会话所需的属性及配置信息。这样，当用户在应用程序的 Web 页之间跳转时，存储在 Session 对象中的变量将不会丢失，而是在整个用户会话中一直存在下去。当客户端关闭会话，或者 Session 超时失效时会话结束。

---

## 18. Cookie 和 Session 是如何配合的呢？

用户第一次请求服务器的时候，服务器根据用户提交的相关信息，创建对应的 Session，请求返回时将此 Session 的唯一标识信息 **SessionID** 返回给浏览器，浏览器接收到服务器返回的 SessionID 信息后，会将此信息存入到 Cookie 中，同时 Cookie 记录此 SessionID 属于哪个域名。

当用户第二次访问服务器的时候，请求会自动判断此域名下是否存在 Cookie 信息，如果存在自动将 Cookie 信息也发送给服务端，服务端会从 Cookie 中获取 SessionID，再根据 SessionID 查找对应的 Session 信息，如果没有找到说明用户没有登录或者登录失效，如果找到 Session 证明用户已经登录可执行后面操作。

根据以上流程可知，**SessionID 是连接 Cookie 和 Session 的一道桥梁**，大部分系统也是根据此原理来验证用户登录状态。

```
第一次请求：
  浏览器 → 服务器：用户名+密码
  服务器 → 浏览器：SessionID（存在 Cookie 中）

第二次请求：
  浏览器 → 服务器：Cookie 中携带 SessionID
  服务器：查找 Session → 找到则已登录，没找到则未登录
```

## 19. Cookie 和 Session 的区别？

| 对比项 | Cookie | Session |
|-------|--------|---------|
| 作用范围 | 保存在客户端（浏览器） | 保存在服务器端 |
| 存取方式 | 只能保存 ASCII | 可以存任意数据类型，比如 UserId 等 |
| 有效期 | 可设置为长时间保持（如"记住登录"） | 一般失效时间较短，客户端关闭或超时都会失效 |
| 隐私策略 | 存储在客户端，比较容易遭到不法获取 | 存储在服务端，安全性相对好一些 |
| 存储大小 | 单个 Cookie 保存的数据不能超过 4K | Session 可存储数据远高于 Cookie |

**面试简答版：**
> Cookie 存在浏览器，只能存 ASCII，大小 4K 限制，可设长期有效但不安全。Session 存在服务器，能存任意数据，大小无限制，但会超时失效。SessionID 是连接两者的桥梁。

## 20. 如何考虑分布式 Session 问题？

在互联网公司为了可以支撑更大的流量，后端往往需要多台服务器共同来支撑前端用户请求，那如果用户在 A 服务器登录了，第二次请求跑到服务 B 就会出现登录失效问题。

分布式 Session 一般会有以下几种解决方案：

**1. 客户端存储**：直接将信息存储在 cookie 中，cookie 是存储在客户端上的一小段数据，客户端通过 http 协议和服务器进行 cookie 交互，通常用来存储一些不敏感信息

**2. Nginx ip_hash 策略**：服务端使用 Nginx 代理，每个请求按访问 IP 的 hash 分配，这样来自同一 IP 固定访问一个后台服务器，避免了在服务器 A 创建 Session，第二次分发到服务器 B 的现象

**3. Session 复制**：任何一个服务器上的 Session 发生改变（增删改），该节点会把这个 Session 的所有内容序列化，然后广播给所有其它节点

**4. 共享 Session**：服务端无状态话，将用户的 Session 等信息使用缓存中间件（如 Redis）来统一管理，保障分发到每一个服务器的响应结果都一致

**建议采用共享 Session 的方案。**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 客户端存储 | 实现简单 | 不安全，只能存不敏感信息 |
| Nginx ip_hash | 同一 IP 固定一台服务器 | 负载不均衡，IP 变了会失效 |
| Session 复制 | 任何节点都有完整 Session | 广播消耗大，节点多了扛不住 |
| **共享 Session（Redis）** | 集中管理，一致性强，性能好 | 需要额外维护 Redis 集群 |

**面试简答版：**
> 多台服务器时 Session 不共享。解决方案：ip_hash 固定路由、Session 复制广播、或用 Redis 集中存储。推荐 Redis 共享 Session，一致性强性能好。

## 21. 如何理解 HTTP 协议的无状态性？

HTTP 的无状态性是指：**服务器不会记住之前发生过什么**。每次请求对服务器来说都是全新的，它不知道你之前来过，也不关心你接下来要干什么。

### 具体表现

- **请求独立**：每次 HTTP 请求都是独立的，服务器处理完就忘了。你第一次访问和第二次访问，对服务器来说没有任何关联
- **不保存上下文**：服务器不会主动保存"这个用户是谁"、"之前看了什么页面"等信息

### 无状态的好处

- **简化服务器设计**：服务器不需要为每个客户端维护会话状态，处理逻辑更简单
- **易于扩展**：请求可以发到任意一台服务器，不需要"会话粘连"，天然适合分布式和负载均衡
- **缓存友好**：相同的请求可以被缓存直接响应，不需要每次都去后端查

### 无状态的"不便"

HTTP 无状态虽然简化了协议设计，但实际应用中经常需要记住用户状态（比如登录状态、购物车）。解决办法就是前面提到的 Cookie、Session、Token 等机制——在无状态的 HTTP 之上人为地加上"状态"。

打个比方：HTTP 就像一个只有 7 秒记忆的客服。你每次打电话都得重新自我介绍一遍。但如果你有会员卡（Cookie/Token），客服刷一下就能认出你了。

### 无状态 vs 有状态

| 对比项 | 无状态（HTTP） | 有状态（如 TCP） |
|-------|--------------|----------------|
| 服务器是否记住连接 | 不记住 | 记住 |
| 每次请求是否独立 | 独立 | 关联 |
| 扩展性 | 好（任意分发） | 差（需要会话粘连） |
| 适用场景 | Web 服务、API | 传输层通信 |

**面试简答版：**
> HTTP 无状态就是服务器不记住客户端的任何信息，每次请求都是独立的。好处是简化服务器设计、易于扩展和缓存；坏处是需要额外机制（Cookie/Session/Token）来实现状态管理。本质上是"协议层无状态，应用层自己管状态"。

## 22. HTTP Keep-Alive 机制的作用是什么？

HTTP Keep-Alive（也叫持久连接）就是**复用一条 TCP 连接发送多个 HTTP 请求/响应**，而不是每次都新建连接。

### 没有 Keep-Alive（HTTP/1.0 默认）

```
请求1 → 新建TCP连接 → 三次握手 → 发请求 → 收响应 → 四次挥手关闭
请求2 → 新建TCP连接 → 三次握手 → 发请求 → 收响应 → 四次挥手关闭
请求3 → 新建TCP连接 → 三次握手 → 发请求 → 收响应 → 四次挥手关闭
```

每个请求都要经历完整的握手和挥手，开销很大。

### 有 Keep-Alive（HTTP/1.1 默认）

```
请求1 → 新建TCP连接 → 三次握手 → 发请求 → 收响应
请求2 → 复用同一条连接 → 发请求 → 收响应
请求3 → 复用同一条连接 → 发请求 → 收响应
                                    → 四次挥手关闭
```

只需要一次握手，多个请求复用同一条连接。

### 核心优势

**1. 减少延迟**

每次新建 TCP 连接都需要三次握手（至少 1 个 RTT），Keep-Alive 省掉了重复握手的时间。一个网页加载可能有几十个请求，省下来的时间很可观。

**2. 节省资源**

复用连接减少了服务器和客户端需要维护的 TCP 连接数，节省内存和文件描述符。

**3. 减轻服务器压力**

服务器不需要为每个请求都处理连接建立/销毁的开销，可以把更多资源用在处理业务逻辑上。

### 配置方式

服务器通过响应头告诉客户端连接保持多久：

```
Keep-Alive: timeout=5, max=100
```

- `timeout=5`：空闲 5 秒后关闭连接
- `max=100`：最多处理 100 个请求后关闭

### Keep-Alive 的局限

虽然连接复用了，但 HTTP/1.1 的 Keep-Alive 存在**队头阻塞**问题：同一个 TCP 连接上的请求必须排队处理，一个请求慢了，后面的都得等。浏览器的解决方案是同时开 6-8 个 TCP 连接并行请求。

HTTP/2 的多路复用才是彻底解决队头阻塞的方案——多个请求可以在同一个连接上**并行**传输，不需要排队。

**面试简答版：**
> Keep-Alive 让多个 HTTP 请求复用同一条 TCP 连接，省掉重复的三次握手开销，减少延迟、节省资源。HTTP/1.1 默认开启，但有队头阻塞问题；HTTP/2 的多路复用彻底解决了这个问题。

## 23. HTTP/1.1 中的管道化（Pipelining）是什么意思？

管道化就是**客户端在收到前一个响应之前，可以连续发送多个请求**，不用一个一个等响应。

### 没有管道化（默认）

```
客户端 → 请求1 → 服务器
客户端 ← 响应1 ← 服务器
客户端 → 请求2 → 服务器
客户端 ← 响应2 ← 服务器
客户端 → 请求3 → 服务器
客户端 ← 响应3 ← 服务器
```

一来一回，串行处理，每个请求都要等前一个响应回来才能发下一个。

### 有管道化

```
客户端 → 请求1, 请求2, 请求3 → 服务器（一口气发出去）
客户端 ← 响应1, 响应2, 响应3 ← 服务器（按顺序返回）
```

请求不用等响应，连续发出去，减少了等待时间。

### 关键规则

1. **请求必须按序发送**：客户端按 1、2、3 的顺序发
2. **响应必须按序返回**：服务器必须按请求的顺序返回响应 1、2、3，不能乱序
3. **客户端必须记住请求顺序**：因为响应是按序返回的，客户端需要知道第几个响应对应哪个请求

### 为什么不流行？

管道化虽然在规范里定义了，但实际几乎没人用，原因：

**队头阻塞问题没解决**

管道化只是让请求不排队了，但响应还是要排队。如果第一个请求特别慢（比如查数据库），后面所有响应都被堵住，客户端拿不到。

```
请求1: 查数据库（耗时5秒）
请求2: 读缓存（10ms）

即使请求2的响应早就准备好了，也得等请求1的响应发完才能发
```

**实现复杂**

客户端要处理乱序响应的映射（虽然规范要求按序，但实现中可能出错），服务器要确保按序响应，中间代理也可能搞乱顺序。兼容性问题多。

**实际替代方案**

- 浏览器用**多连接并行**（同时开 6-8 个 TCP 连接）绕过这个问题
- HTTP/2 用**多路复用**彻底解决——多个请求可以在同一个连接上真正并行，响应可以乱序返回，每个响应带 stream ID 标识属于哪个请求

### 管道化 vs 多路复用

| 对比项 | 管道化（HTTP/1.1） | 多路复用（HTTP/2） |
|-------|-----------------|-----------------|
| 请求发送 | 可以连续发多个 | 可以连续发多个 |
| 响应顺序 | 必须按请求顺序返回 | 可以乱序返回（带 stream ID） |
| 队头阻塞 | 仍然存在（响应排队） | 彻底解决 |
| 实际使用 | 几乎没人用 | 广泛使用 |

**面试简答版：**
> 管道化允许客户端连续发多个请求不用等响应，但响应必须按请求顺序返回，队头阻塞没解决，实现复杂，实际几乎没人用。HTTP/2 的多路复用才是真正解决队头阻塞的方案。

## 24. HTTP 消息头中的 Content-Type 和 Accept 有什么作用？

这两个头部字段都和**数据格式（MIME 类型）**有关，但用在不同方向。

### Content-Type

告诉对方"**我发给你的是什么格式的数据**"。

- **在请求中**：告诉服务器客户端发的是什么格式
  ```
  Content-Type: application/json    ← 我发的是 JSON
  Content-Type: application/x-www-form-urlencoded  ← 我发的是表单数据
  Content-Type: multipart/form-data  ← 我发的是文件上传
  ```

- **在响应中**：告诉客户端服务器返回的是什么格式
  ```
  Content-Type: text/html; charset=utf-8  ← 返回的是 HTML 页面
  Content-Type: application/json          ← 返回的是 JSON 数据
  Content-Type: image/png                 ← 返回的是 PNG 图片
  ```

### Accept

告诉服务器"**我能接受什么格式的响应**"。只在请求中使用。

```
Accept: text/html, application/json   ← 我能看 HTML 和 JSON
Accept-Language: zh-CN, en;q=0.9      ← 优先中文，其次英文
Accept-Encoding: gzip, br             ← 支持 gzip 和 br 压缩
```

服务器根据 Accept 字段做**内容协商**，返回最匹配的格式。比如客户端说 `Accept: application/json`，服务器就返回 JSON 而不是 HTML。

### 对比

| 对比项 | Content-Type | Accept |
|-------|-------------|--------|
| 作用 | 描述"我发的数据是什么格式" | 描述"我能接受什么格式" |
| 出现在 | 请求和响应都有 | 只在请求中 |
| 方向 | 告诉对方数据格式 | 告诉服务器偏好 |

### 常见 MIME 类型

| MIME 类型 | 说明 |
|----------|------|
| text/html | HTML 页面 |
| text/plain | 纯文本 |
| application/json | JSON 数据 |
| application/xml | XML 数据 |
| application/x-www-form-urlencoded | URL 编码表单 |
| multipart/form-data | 文件上传 |
| image/png | PNG 图片 |
| image/jpeg | JPEG 图片 |

**面试简答版：**
> Content-Type 描述"我发的数据是什么格式"（请求和响应都有），Accept 描述"我能接受什么格式"（只在请求中）。服务器根据 Accept 做内容协商返回对应格式。常见格式有 JSON、HTML、表单、图片等。

## 25. 什么是 WebSocket？它和 HTTP 有什么区别？

WebSocket 是一种在**单个 TCP 连接上实现全双工通信**的协议。和 HTTP 的请求-响应模式不同，WebSocket 连接建立后，客户端和服务器可以**随时互相发数据**，不需要对方先请求。

### 和 HTTP 的核心区别

| 对比项 | HTTP | WebSocket |
|-------|------|-----------|
| 通信模式 | 请求-响应（客户端问，服务器答） | 全双工（双方随时互发） |
| 连接方式 | 短连接或长连接（Keep-Alive） | 持久连接（一直不断） |
| 实时性 | 低（客户端要不断轮询） | 高（服务器可以主动推送） |
| 头部开销 | 每次请求都带完整头部（几百字节） | 数据帧头部很小（2-14 字节） |
| 协议标识 | `http://` / `https://` | `ws://` / `wss://` |
| 服务端能否主动发 | 不能（只能客户端先发） | 可以主动推送给客户端 |

### WebSocket 的工作流程

```
1. 客户端发 HTTP 请求（带 Upgrade 头）：
   GET /chat HTTP/1.1
   Upgrade: websocket
   Connection: Upgrade

2. 服务器返回 101 Switching Protocols：
   HTTP/1.1 101 Switching Protocols
   Upgrade: websocket

3. 协议升级完成，TCP 连接从 HTTP 切换到 WebSocket
   此后双方可以随时互发数据帧

4. 一方关闭连接 → 发 Close 帧 → 连接断开
```

关键点：WebSocket 的连接**始于 HTTP**，通过 `Upgrade` 头部将协议从 HTTP 升级到 WebSocket。升级完成后就不再是 HTTP 了。

### 为什么需要 WebSocket？

HTTP 是"客户端不问，服务器不说"。但有些场景需要服务器**主动推数据**给客户端：

- **实时聊天**：别人发消息时服务器要立刻推给你，不能等你轮询
- **在线游戏**：玩家操作需要实时同步给其他人
- **股票行情**：价格变动要立刻推送给客户端
- **协同编辑**：多个人同时编辑一个文档，变化要实时同步

用 HTTP 实现这些就得不断轮询（每隔几秒问服务器"有新数据吗？"），浪费资源还延迟高。WebSocket 直接让服务器有新数据就推过来。

### 数据帧格式

WebSocket 的数据帧头部只有 2-14 字节，比 HTTP 头部（通常几百字节）小得多：

```
HTTP 请求：200+ 字节头部 + 数据
WebSocket：2 字节头部 + 数据（小消息开销极低）
```

**面试简答版：**
> WebSocket 是全双工协议，连接建立后双方可以随时互发数据，服务器能主动推送。HTTP 是请求-响应模式，客户端不问服务器就不说。WebSocket 通过 HTTP 的 Upgrade 头升级协议，适合聊天、游戏、股票行情等实时场景，数据帧头部也比 HTTP 小得多。

## 26. 如何在 TCP/IP 模型中实现端到端的加密通信？

在 TCP/IP 模型中，端到端加密主要靠 **TLS/SSL 协议**实现，它工作在传输层和应用层之间。

### TLS/SSL 是什么？

TLS（Transport Layer Security）是 SSL 的升级版，提供**加密、认证、完整性**三大安全能力。HTTPS 就是 HTTP + TLS。

```
应用层：HTTP / FTP / SMTP ...
  ↕
TLS/SSL 层（在这里加密解密）
  ↕
传输层：TCP
  ↕
网络层：IP
```

### TLS 握手过程

1. **客户端 Hello**：客户端发送支持的 TLS 版本、加密算法列表、一个随机数
2. **服务器 Hello**：服务器选择加密算法，发回证书（含公钥）、另一个随机数
3. **客户端验证证书**：检查证书是否可信（CA 签名、域名、有效期）
4. **密钥交换**：客户端用公钥加密一个"预主密钥"发给服务器，服务器用私钥解密
5. **生成会话密钥**：双方用两个随机数 + 预主密钥生成相同的对称密钥
6. **加密通信**：之后所有数据都用这个对称密钥加密传输

```
客户端                              服务器
  |  ---- ClientHello ---->         |  支持的加密算法、随机数1
  |  <--- ServerHello + 证书 ----   |  选定算法、公钥、随机数2
  |  ---- 预主密钥（公钥加密）--->   |  用公钥加密传过去
  |       双方用 3 个随机数生成对称密钥 |
  |  <==== 加密数据传输 ====>        |  用对称密钥通信
```

### 为什么用混合加密？

- **非对称加密**（RSA 等）：解决密钥传递问题——公钥加密的东西只有私钥能解，安全地把密钥传过去。但速度慢
- **对称加密**（AES 等）：速度快，适合大量数据加密。但密钥怎么安全传给对方是个问题
- **TLS 的做法**：用非对称加密安全地交换对称密钥，之后通信用对称加密。兼顾安全和速度

### 加密在哪一层？

| 层次 | 加密协议 | 保护范围 |
|------|---------|---------|
| 应用层 | 应用自身加密 | 只保护应用数据 |
| 传输层与应用层之间 | TLS/SSL | 保护应用层所有数据（最常用） |
| 网络层 | IPsec | 保护所有 IP 数据（包括 TCP/UDP 头部） |

实际中最常用的是 TLS（在传输层之上、应用层之下），HTTPS 就是典型应用。

**面试简答版：**
> 端到端加密主要靠 TLS 协议，工作在传输层和应用层之间。握手过程用非对称加密安全交换对称密钥，之后通信用对称加密。HTTPS = HTTP + TLS。TLS 提供加密（防窃听）、认证（证书验证身份）、完整性（防篡改）三大安全能力。

## 27. 描述 WebSocket 如何实现全双工通信？

WebSocket 通过一次 HTTP 升级握手建立持久连接，之后双方可以在**同一个 TCP 连接上同时收发数据**，实现全双工。

### 建立过程（HTTP 升级握手）

```
1. 客户端发 HTTP 请求（带 Upgrade 头）：
   GET /chat HTTP/1.1
   Upgrade: websocket
   Connection: Upgrade
   Sec-WebSocket-Key: dGhlIHNhbXBsZQ==

2. 服务器返回 101 状态码：
   HTTP/1.1 101 Switching Protocols
   Upgrade: websocket
   Connection: Upgrade
   Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

3. 协议切换完成，TCP 连接从 HTTP 变成 WebSocket
```

升级完成后，连接就不再是 HTTP 了，双方可以随时发 WebSocket 数据帧。

### 全双工通信

```
客户端 ←→ 服务器（同一个 TCP 连接）

客户端随时可以发数据给服务器
服务器随时可以发数据给客户端
不需要对方先请求

例：聊天应用
客户端：发送消息 "你好"  ←→ 服务器：推送新消息 "在的"
同时进行，互不阻塞
```

### WebSocket 数据帧

WebSocket 的数据帧头部很小（2-14 字节），支持发送文本和二进制数据：

```
帧格式（简化）：
[操作码][是否分片][数据长度][数据]
  1字节   1字节    可变     载荷

操作码：
  0x1 = 文本帧
  0x2 = 二进制帧
  0x8 = 关闭帧
  0x9 = Ping（心跳检测）
  0xA = Pong（心跳回复）
```

### 为什么 HTTP 做不到全双工？

HTTP 是请求-响应模式：客户端不发请求，服务器就不能发数据。即使有 Server Push（HTTP/2），也是服务器"预测"客户端需要什么资源，主动推过去——但本质上还是客户端先发起请求触发的。

WebSocket 不同，连接建立后是**真正的双向通道**，任何一方都可以主动发数据，不需要对方先请求。

**面试简答版：**
> WebSocket 通过 HTTP 的 Upgrade 头升级协议，建立持久连接后实现全双工——双方可以在同一个 TCP 连接上随时互发数据，不需要对方先请求。数据帧头部只有 2-14 字节，比 HTTP 轻量得多。

## 28. HTTP/2 的服务器推送功能如何工作？

HTTP/2 的服务器推送（Server Push）允许服务器**主动把客户端还没请求的资源发过去**，减少等待时间。

### 工作原理

当客户端请求 `index.html` 时，服务器知道这个页面还需要 `style.css` 和 `app.js`，于是在返回 HTML 的同时，主动把 CSS 和 JS 也推过去。

```
客户端：请求 index.html
服务器：返回 index.html + PUSH_PROMISE 帧（告知将推送 style.css 和 app.js）
服务器：主动推送 style.css
服务器：主动推送 app.js

客户端不用再单独请求 CSS 和 JS，省了 2 次往返
```

### PUSH_PROMISE 帧

服务器通过 **PUSH_PROMISE 帧**告诉客户端"我要推什么资源"。客户端收到后：
- 如果已经缓存了这个资源 → 发送 **RST_STREAM** 帧拒绝推送
- 没缓存 → 接受推送，等待数据到来

### 推送 vs 不推送

```
没有 Server Push：
  客户端 → 请求 index.html → 服务器返回
  客户端解析 HTML，发现需要 style.css
  客户端 → 请求 style.css → 服务器返回
  客户端解析 HTML，发现需要 app.js
  客户端 → 请求 app.js → 服务器返回
  共 3 次请求-响应

有 Server Push：
  客户端 → 请求 index.html → 服务器返回 HTML + 推送 CSS + 推送 JS
  共 1 次请求-响应
```

### 为什么实际用得不多？

1. **容易推错**：服务器猜不准客户端有没有缓存，推了不需要的资源浪费带宽
2. **HTTP/3 的替代**：QUIC 协议有更好的方案（103 Early Hints）
3. **浏览器逐步弃用**：Chrome 等浏览器已经开始移除 Server Push 支持

### 实际替代方案：103 Early Hints

服务器先返回 `103 Early Hints` 状态码，告诉客户端"我建议你先加载这些资源"，客户端自己决定要不要请求。比 Server Push 更灵活，不会推错。

```
客户端 → 请求 index.html
服务器 → 103 Early Hints: Link: <style.css>; rel=preload; as=style
  （客户端开始预加载 CSS，但还没收到 HTML）
服务器 → 200 OK: index.html
  （同时 CSS 已经在加载了）
```

**面试简答版：**
> HTTP/2 Server Push 通过 PUSH_PROMISE 帧让服务器主动推送客户端还没请求的资源（如请求 HTML 时同时推 CSS/JS），减少等待。但容易推错浪费带宽，浏览器已逐步弃用，103 Early Hints 是更好的替代方案。

## 29. HTTP 协议中的幂等性是什么意思？

幂等性就是**一个操作执行一次和执行多次，效果是一样的**。不管调多少次，结果都不会变。

### 哪些方法是幂等的？

| 方法 | 幂等？ | 说明 |
|------|--------|------|
| GET | 幂等 | 获取资源，请求多少次结果都一样 |
| PUT | 幂等 | 更新资源，多次更新同一个值，结果相同 |
| DELETE | 幂等 | 删除资源，删一次和删多次效果一样（资源都没了） |
| HEAD | 幂等 | 和 GET 一样但不返回 body |
| OPTIONS | 幂等 | 查询支持的方法，不会改变状态 |
| POST | **不幂等** | 创建资源，每调一次就多创建一个 |
| PATCH | **不一定** | 如果是设置某个值就幂等，如果是 increment 就不幂等 |

### 为什么幂等性重要？

在网络不可靠的环境下，请求可能超时、重试。如果操作是幂等的，重试多少次都没问题；如果不幂等，重试就会产生副作用。

```
场景：用户点了"支付"按钮
  客户端 → 服务器：POST /pay 100元
  服务器处理成功，但响应超时了
  客户端不知道成功没，重试...

  如果 POST 不幂等：用户被扣了两次钱！
  如果用了幂等性设计（如带唯一订单号）：服务器识别出重复请求，只扣一次
```

### 如何保证非幂等操作的幂等性？

**1. 唯一请求 ID**

每次请求带一个唯一 ID（如 UUID），服务器先查这个 ID 处理过没有，处理过就直接返回之前的结果。

```
客户端：POST /pay {orderId: "abc123", amount: 100}
服务器：查 orderId → 没处理过 → 执行支付 → 返回成功
客户端：重试 POST /pay {orderId: "abc123", amount: 100}
服务器：查 orderId → 已处理过 → 直接返回之前的结果（不重复扣款）
```

**2. 数据库唯一约束**

用数据库的唯一索引防止重复插入。比如订单号加唯一索引，重复插入会报错。

**3. 状态机**

资源有明确的状态流转（待支付 → 已支付 → 已完成），重复操作不会改变已流转的状态。

**面试简答版：**
> 幂等性是操作执行多次效果和一次一样。GET/PUT/DELETE 是幂等的，POST 不幂等。网络重试时幂等性很重要，防止重复操作。保证幂等的方法：唯一请求 ID、数据库唯一约束、状态机。
