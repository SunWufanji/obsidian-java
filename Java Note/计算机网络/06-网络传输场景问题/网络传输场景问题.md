# 网络传输场景问题

## 1. 输入一个 URL 到页面显示，发生了什么？

这是最经典的网络面试题，覆盖了几乎所有网络层的知识点。

```
1. URL 解析 → 2. DNS 解析 → 3. TCP 三次握手 → 4. 发送 HTTP 请求 → 
5. 服务器处理并返回响应 → 6. 浏览器渲染页面 → 7. TCP 四次挥手
```

**详细流程**：

**1. 域名解析（域名变为 IP 地址）**
- 浏览器搜索自己的 DNS 缓存（维护一张域名与 IP 的对应表）
- 若没有，则搜索操作系统的 DNS 缓存（维护一张域名与 IP 的对应表）
- 若没有，则搜索操作系统的 hosts 文件（维护一张域名与 IP 的对应表）

**2. 若都没有，则找 DNS 服务器**
- 找 tcp/ip 参数中设置的首选 dns 服务器，即本地 dns 服务器（递归查询）
- 本地域名服务器查询自己的 dns 缓存，如果没有，则进行迭代查询
- 将本地 dns 服务器将 IP 返回给操作系统，同时缓存 IP

**3. TCP 三次握手，建立 TCP 连接**
- 浏览器会以一个随机端口（1024-65535）向服务端的 web 程序 80 端口发起 TCP 的连接

**4. 建立 TCP 连接后发起 HTTP 请求**

**5. 服务器响应 HTTP 请求**
- 服务器 web 应用程序收到 http 请求后，就开始处理请求
- 处理之后就返回给浏览器 html 文件

**6. 浏览器解析 HTML 代码，并请求 HTML 中的资源**

**7. 浏览器对页面进行渲染，并呈现给用户**

**面试简答版：**
> 先解析 URL，然后 DNS 解析拿到 IP，TCP 三次握手建立连接，发送 HTTP 请求，服务器返回响应，浏览器解析渲染页面，最后四次挥手断开连接。

## 2. 两台主机之间的通信过程是怎样的？

假设主机 A（192.168.1.100）要给主机 B（10.0.0.200）发数据，跨网段通信：

**同网段通信**（都在 192.168.1.x）：
1. A 先用自己的子网掩码判断 B 是否在同一网段
2. 如果在同一网段，ARP 广播找 B 的 MAC 地址
3. 拿到 MAC 地址后，封装成以太网帧直接发给 B

**跨网段通信**：
1. A 判断 B 不在同一网段
2. A 把数据包发给**默认网关**（路由器）
3. 路由器查看路由表，找到下一跳的路由器
4. 数据包经过多个路由器转发，最终到达 B 所在的网络
5. 最后一跳的路由器 ARP 找到 B 的 MAC 地址，把帧发给 B

**每一层封装的变化**：
- 应用层数据
- + TCP 头（传输层）
- + IP 头（网络层，源 IP 和目标 IP 始终不变）
- + 以太网帧头（数据链路层，每经过一跳 MAC 地址会变）

**面试简答版：**
> 先判断是否同网段，同网段 ARP 找 MAC 直接发；跨网段先发给网关，经过多个路由器转发，每跳 MAC 地址变化但 IP 不变，最终到达目标网络。

## 3. 连接一个不存在的 IP 地址，会发生什么？

比如你 ping 一个根本不存在的 IP `192.168.1.999`（无效地址）或者一个没人用的 IP `10.0.0.100`：

**情况一：IP 地址格式无效**（如 192.168.1.999）
- 操作系统直接拒绝，报"参数错误"

**情况二：IP 地址有效但不存在**（如 10.0.0.100）
- 本地 ARP 请求广播：找不到目标主机
- 如果数据包发到网关，网关也转发不出去
- 最终超时，返回"请求超时"或"目标主机不可达"
- 不会建立 TCP 连接（因为 SYN 包根本到不了）

**面试简答版：**
> 如果 IP 格式无效操作系统直接拒绝；IP 有效但不存在则 ARP 找不到目标，数据包经过路由器转发最终超时，返回"请求超时"或"目标主机不可达"。

## 4. 网页线上问题如何排查？

线上网页出问题时，按这个思路排查：

**1. 网络层排查**
- `ping` 目标服务器：能不能通？
- `traceroute`：在哪一跳断了？
- DNS 解析是否正常：`nslookup` / `dig`

**2. 传输层排查**
- TCP 连接是否建立：`telnet` / `nc`
- 端口是否开放：`netstat` / `ss`

**3. 应用层排查**
- HTTP 状态码：200/404/500/502/503
- 查看服务器日志（Nginx、应用日志）
- 检查负载均衡配置

**4. 常见场景**
- **偶发超时**：检查网络丢包率、服务器负载
- **部分用户无法访问**：检查 DNS 解析、CDN 节点、地域限制
- **502 Bad Gateway**：后端服务挂了或连接超时
- **503 Service Unavailable**：服务过载或维护中

## 5. 偶发性网络问题如何排查？

偶发性问题最难排查，因为"出问题的时候你不在"。

**思路**：
1. **看监控和日志**：有没有规律？（时间段、特定接口、特定用户群）
2. **抓包分析**：用 Wireshark/tcpdump 抓取异常时段的网络包
3. **检查超时设置**：连接超时、读超时、重试机制是否合理
4. **排查中间链路**：CDN、负载均衡、代理、网关都可能出问题
5. **压力测试**：高并发下是否出现资源竞争、连接池耗尽
